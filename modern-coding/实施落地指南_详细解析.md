# 实施落地指南 详细解析

> **核心结论**：在现有项目中引入 SDD + Harness Engineering 不需要一步到位。最高 ROI 的起点是 Phase 4（Develop）的 Harness 约束——先建反馈循环，再逐步向上游扩展规范定义。6 个阶段可在 1-6 个月内完成，最快 1 天可落地第一个 Pre-commit Hook。

## 概述

本文档是 Modern Coding 方法论的**操作手册**，面向想在实际项目中落地 SDD 和 Harness Engineering 的开发团队。每个步骤都附有可直接复制的配置文件、代码示例和检查清单。

与理论文档的关系：

## 一、从哪里开始：反直觉的实施顺序

### 1.1 不要从 Phase 1 (Define) 开始

大多数团队的直觉是从头开始：先写需求文档，再做架构设计。**这是错误的。**

原因：
- 没有 Harness 反馈，你无法验证 Spec 的质量
- 团队缺乏 AI 协作经验，无法写出有效的规范
- 前期投入大、收益慢，容易放弃

### 1.2 推荐实施顺序

```
最高 ROI 起点
    ↓
Phase 4 (Develop): 建立 Harness 反馈循环
    ↓
Phase 5 (Deliver): 自动化验证管道
    ↓
Phase 1 (Define): 引入需求 ID 和追溯
    ↓
Phase 2 (Design): ADR 和接口规范
    ↓
Phase 3 (Decompose): 任务编排和依赖图
```

### 1.3 为什么从 Develop 阶段开始

| 优势 | 说明 |
|------|------|
| **即时可见效果** | 第一个 Pre-commit Hook 上线后，当天就能看到错误拦截 |
| **零文档负担** | 不需要先写 Spec 就能开始约束 Agent |
| **经验积累** | 通过 Agent 犯的错误，反向识别需要哪些 Spec |
| **团队认同** | 小胜利建立信心，为后续流程改造铺路 |

## 二、6 阶段实施路线图

### 阶段 1：建立反馈循环（第 1 天 - 第 2 周）

**目标**：让 Agent 的每次代码提交都能在 5 秒内获得结构化反馈。

#### Step 1: 创建 Pre-commit Hook

```bash
#!/bin/bash
# .git/hooks/pre-commit
# 或使用 husky/pre-commit 框架管理

set -e

echo "=== Running Pre-commit Checks ==="

# 1. 编译检查（最基础的反馈）
echo "[1/4] Compile check..."
npm run build 2>&1 || {
    echo ""
    echo "❌ [BUILD-001] 编译失败"
    echo ""
    echo "建议修复:"
    echo "  1. 检查上方的编译错误信息"
    echo "  2. 确保所有导入路径正确"
    echo "  3. 运行 'npm run build' 查看完整输出"
    echo ""
    exit 1
}

# 2. 类型检查
echo "[2/4] Type check..."
npx tsc --noEmit 2>&1 || {
    echo ""
    echo "❌ [TYPE-001] 类型检查失败"
    echo ""
    echo "建议修复:"
    echo "  1. 检查上方的类型错误"
    echo "  2. 确保类型定义文件是最新的"
    echo ""
    exit 1
}

# 3. Lint 检查
echo "[3/4] Lint check..."
npx eslint --max-warnings 0 src/ 2>&1 || {
    echo ""
    echo "❌ [LINT-001] Lint 检查失败"
    echo ""
    echo "自动修复: npx eslint --fix src/"
    echo ""
    exit 1
}

# 4. 格式检查
echo "[4/4] Format check..."
npx prettier --check src/ 2>&1 || {
    echo ""
    echo "❌ [FMT-001] 代码格式不一致"
    echo ""
    echo "自动修复: npx prettier --write src/"
    echo ""
    exit 1
}

echo "✅ All pre-commit checks passed"
```

**错误消息设计原则**（来自 OpenAI 实践）：

每条错误消息必须包含三要素：
1. **错误代码**：`[BUILD-001]`、`[LINT-001]` — 方便追溯和统计
2. **问题描述**：具体说明哪里出了问题
3. **修复建议**：可直接执行的命令或明确的操作步骤

#### Step 2: 创建 AGENTS.md

这是 Agent 的"入职文档"，保持精简（100 行以内）。

```markdown
# AGENTS.md

## 项目概述
{项目名称}：{一句话描述}

## 技术栈
- 语言: TypeScript 5.x
- 运行时: Node.js 20+
- 框架: {框架名称}
- 包管理: pnpm

## 目录结构
```
src/
├── types/       # 类型定义（最底层，无外部依赖）
├── config/      # 配置管理（仅依赖 types）
├── services/    # 业务逻辑（依赖 types + config）
├── api/         # API 路由（依赖 services）
└── utils/       # 工具函数（仅依赖 types）
```

## 依赖规则
- 只能向上依赖：types → config → services → api
- 严禁跨层依赖：api 不能直接导入 types
- utils 只能被其他模块导入，不能导入非 types 模块

## 编码规范
- 所有公开函数必须有 JSDoc 注释
- 错误处理使用自定义 Error 类（见 src/types/errors.ts）
- 异步操作统一使用 async/await
- 禁止使用 any 类型

## 测试规范
- 单元测试位于 `__tests__/` 目录
- 测试文件命名: `{module}.test.ts`
- 最低覆盖率: 80%

## 提交规范
- 格式: `{type}({scope}): {description}`
- type: feat | fix | refactor | test | docs | chore
- 每个提交必须关联需求 ID（如有）: `Refs: FR-XXX-001`

## 详细文档
- [架构设计](docs/ARCHITECTURE.md)
- [安全规范](docs/SECURITY.md)
- [API 规范](specs/openapi.yaml)
```

**关键要点**：
- 控制在 100 行以内，避免挤占上下文空间
- 详细内容通过链接指向子文档（渐进式披露）
- 包含可执行的规则（依赖方向、命名规范），而非模糊的建议

#### Step 3: 配置基础验证管道

#### Step 2: 编写第一个需求文档

从**最近要开发的功能**开始写，不要回溯已有代码。

```markdown
# FR-AUTH-001: 用户登录

## 需求描述
系统 MUST 支持用户通过邮箱和密码登录。

## GWT（Given-When-Then）

### Scenario 1: 登录成功
```gherkin
Given 用户已注册，邮箱为 "user@example.com"
When 用户提交正确的邮箱和密码
Then 系统返回 200 状态码和 JWT Token
And Token 有效期为 24 小时
```

### Scenario 2: 密码错误
```gherkin
Given 用户已注册
When 用户提交错误的密码
Then 系统返回 401 状态码
And 错误消息为 "邮箱或密码错误"
And 系统记录失败尝试次数
```

### Scenario 3: 频率限制
```gherkin
Given 同一 IP 在 1 小时内已失败 5 次
When 用户再次尝试登录
Then 系统返回 429 状态码
And 提示 "请在 {剩余分钟} 分钟后重试"
```

## 约束
- 密码 MUST 使用 bcrypt 哈希（cost factor >= 12）
- 会话 MUST 在 24 小时后过期
- 同一 IP 每小时最多 5 次失败尝试
- 登录事件 MUST 写入审计日志

## 验收标准
- [ ] POST /auth/login API 端点
- [ ] JWT Token 生成和验证
- [ ] 频率限制中间件
- [ ] 审计日志记录
- [ ] 单元测试覆盖率 >= 90%
- [ ] GWT 场景全部通过

## 相关文档
- ADR-001: 认证策略选型
```

#### Step 3: 在代码中添加追溯标签

// @adr ADR-001
export async function login(
  email: string,
  password: string
): Promise<AuthResult> {
  // 验证频率限制 (@require FR-AUTH-001, Scenario 3)
  await rateLimiter.check(getClientIP());

  const user = await userRepo.findByEmail(email);
  if (!user || !await bcrypt.compare(password, user.passwordHash)) {
    // @require FR-AUTH-001, Scenario 2
    await auditLog.record('login_failed', { email });
    throw new AuthError('邮箱或密码错误');
  }

  // @require FR-AUTH-001, Scenario 1
  const token = jwt.sign({ userId: user.id }, SECRET, { expiresIn: '24h' });
  await auditLog.record('login_success', { userId: user.id });
  return { token, expiresIn: 86400 };
}

#### Step 4: 自动化追溯检查脚本

```python
#!/usr/bin/env python3
"""check_traceability.py - 检查代码与需求的追溯关系"""

import os
import re
import sys
import json
from pathlib import Path

def scan_specs(specs_dir: str) -> set:
    """扫描所有需求 ID"""
    ids = set()
    pattern = re.compile(r'^#\s+(FR|NFR|BR)-[\w]+-\d{3}', re.MULTILINE)
    for md_file in Path(specs_dir).rglob('*.md'):
        content = md_file.read_text(encoding='utf-8')
        for match in pattern.finditer(content):
            ids.add(match.group(1) if ':' not in match.group() 
                    else match.group().split(':')[0].strip('# '))
    # 简化：直接提取 ID 模式
    id_pattern = re.compile(r'(FR|NFR|BR)-[\w]+-\d{3}')
    for md_file in Path(specs_dir).rglob('*.md'):
        content = md_file.read_text(encoding='utf-8')
        for match in id_pattern.finditer(content):
            ids.add(match.group())
    return ids

def scan_code(src_dir: str) -> dict:
    """扫描代码中的 @require 标签"""
    refs = {}
    pattern = re.compile(r'@require\s+((?:FR|NFR|BR)-[\w]+-\d{3})')
    for code_file in Path(src_dir).rglob('*'):
        if code_file.suffix in ('.ts', '.js', '.tsx', '.jsx', '.py', '.cpp', '.h'):
            content = code_file.read_text(encoding='utf-8')
            for match in pattern.finditer(content):
                req_id = match.group(1)
                if req_id not in refs:
                    refs[req_id] = []
                refs[req_id].append(str(code_file))
    return refs

def main():
    specs_dir = sys.argv[1] if len(sys.argv) > 1 else 'specs'
    src_dir = sys.argv[2] if len(sys.argv) > 2 else 'src'
    
    spec_ids = scan_specs(specs_dir)
    code_refs = scan_code(src_dir)
    
    errors = []
    
    # 检查未实现的需求
    for spec_id in sorted(spec_ids):
        if spec_id not in code_refs:
            errors.append(f"[TRACE-001] {spec_id}: 需求已定义但无代码实现")
    
    # 检查无需求的实现
    for ref_id in sorted(code_refs.keys()):
        if ref_id not in spec_ids:
            files = ', '.join(code_refs[ref_id])
            errors.append(f"[TRACE-002] {ref_id}: 代码引用了不存在的需求 ({files})")
    
    # 输出结果
    if errors:
        print("❌ 追溯性检查失败:\n")
        for err in errors:
            print(f"  {err}")
        print(f"\n建议修复:")
        print(f"  - TRACE-001: 为需求创建实现代码，或标记为"未计划"")
        print(f"  - TRACE-002: 在 specs/ 中补充需求文档，或修正 @require 标签")
        sys.exit(1)
    else:
        print(f"✅ 追溯性检查通过: {len(spec_ids)} 个需求, {len(code_refs)} 个实现")

if __name__ == '__main__':
    main()
```

#### Step 1: 编写 ADR（架构决策记录）

```markdown
# ADR-001: 认证策略选型

## Status
Accepted

## Context
系统需要用户认证机制。候选方案：
1. Session-based (服务端存储)
2. JWT (无状态 Token)
3. OAuth 2.0 (委托认证)

## Decision
采用 JWT 方案。

## Rationale
- 无状态：适合水平扩展，不需要共享 Session 存储
- 标准化：生态成熟，库支持完善
- 自包含：Token 携带用户信息，减少数据库查询

## Consequences
### Positive
- 服务端无状态，易于扩展
- 跨域友好，支持微服务架构

### Negative
- Token 无法主动撤销（需额外的黑名单机制）
- Token 大小较 Session ID 大

### Mitigations
- 实现 Token 黑名单（Redis，TTL = Token 过期时间）
- Token 过期时间设为 24 小时，减小泄露风险

## References
- FR-AUTH-001: 用户登录需求
```

#### Step 2: 实现架构依赖检查

以下是一个通用的 TypeScript/JavaScript 分层架构检查器：

```javascript
#!/usr/bin/env node
/**
 * check-architecture.js
 * 检查代码是否遵守分层依赖规则
 * 
 * 用法: node check-architecture.js src/
 */

const fs = require('fs');
const path = require('path');

// 定义分层架构（从底层到上层）
const LAYERS = ['types', 'config', 'utils', 'services', 'api'];

// 构建允许的依赖关系
const ALLOWED_DEPS = {};
LAYERS.forEach((layer, index) => {
  // 每一层只能依赖它下面的层
  ALLOWED_DEPS[layer] = LAYERS.slice(0, index);
});
// utils 特殊处理：只能依赖 types
ALLOWED_DEPS['utils'] = ['types'];

function getLayer(filePath) {
  const parts = filePath.split(path.sep);
  for (const layer of LAYERS) {
    if (parts.includes(layer)) return layer;
  }
  return null;
}

function extractImports(content) {
  const imports = [];
  // ES6 imports
  const es6Pattern = /import\s+.*?from\s+['"](.+?)['"]/g;
  // require
  const reqPattern = /require\s*\(\s*['"](.+?)['"]\s*\)/g;

  let match;
  while ((match = es6Pattern.exec(content)) !== null) imports.push(match[1]);
  while ((match = reqPattern.exec(content)) !== null) imports.push(match[1]);
  return imports;
}

function checkFile(filePath, srcDir) {
  const content = fs.readFileSync(filePath, 'utf-8');
  const currentLayer = getLayer(path.relative(srcDir, filePath));
  if (!currentLayer) return [];

  const imports = extractImports(content);
  const violations = [];

  for (const imp of imports) {
    if (!imp.startsWith('.') && !imp.startsWith('/')) continue; // 跳过外部包
    
    const resolved = path.resolve(path.dirname(filePath), imp);
    const targetLayer = getLayer(path.relative(srcDir, resolved));
    if (!targetLayer || targetLayer === currentLayer) continue;

    if (!ALLOWED_DEPS[currentLayer].includes(targetLayer)) {
      violations.push({
        file: path.relative(srcDir, filePath),
        from: currentLayer,
        to: targetLayer,
        import: imp,
      });
    }
  }
  return violations;
}

// 主流程
const srcDir = process.argv[2] || 'src';
const allViolations = [];

function walkDir(dir) {
  for (const entry of fs.readdirSync(dir, { withFileTypes: true })) {
    const full = path.join(dir, entry.name);
    if (entry.isDirectory() && !entry.name.startsWith('.')) {
      walkDir(full);
    } else if (/\.(ts|js|tsx|jsx)$/.test(entry.name) && !entry.name.includes('.test.')) {
      allViolations.push(...checkFile(full, srcDir));
    }
  }
}

walkDir(srcDir);

if (allViolations.length > 0) {
  console.log(`❌ [ARCH-001] 发现 ${allViolations.length} 个分层依赖违规:\n`);
  for (const v of allViolations) {
    console.log(`  ${v.file}`);
    console.log(`    ${v.from} → ${v.to} (❌ 禁止)`);
    console.log(`    import: ${v.import}`);
    console.log(`    建议: 通过 ${ALLOWED_DEPS[v.from].join(' 或 ')} 层间接访问`);
    console.log();
  }
  console.log(`文档: docs/ARCHITECTURE.md#dependency-rules`);
  process.exit(1);
} else {
  console.log(`✅ 架构依赖检查通过 (${LAYERS.length} 层, 0 违规)`);
}
```

  review:
    checklist:
      - "所有新函数都有 @require 标签"
      - "编译无警告"
      - "测试全部通过"
      - "覆盖率 >= 80%"
      - "无架构违规"

# metrics.yaml — 实施效果度量指标

metrics:
  # === 效率指标 ===

### 阶段 6：垃圾回收与长期维护（第 5 月+）

**目标**：防止 Harness 自身腐化。

#### 关键实践

1. **文档漂移扫描**：每周运行脚本检查文档与代码一致性
2. **规则审计**：每月审查 Linter 规则，删除不再需要的、补充新发现的
3. **指标趋势监控**：如果 Agent 完成率下降，说明 Harness 可能需要更新

## 三、常见陷阱及应对

### 陷阱 1: AGENTS.md 膨胀

**症状**：AGENTS.md 超过 200 行，Agent 性能下降

**原因**：所有规则都塞进一个文件

**解决方案**：
```
AGENTS.md (100 行以内，仅包含导航)
├── docs/ARCHITECTURE.md  (架构规则细节)
├── docs/SECURITY.md      (安全规范细节)
├── docs/CODING-STYLE.md  (编码风格细节)
└── docs/API-DESIGN.md    (API 设计规范)
```

### 陷阱 2: Spec 过度设计

**症状**：一个 Spec 文档写了 500 行，指定了每个函数的实现细节

**原因**：混淆了"做什么"和"怎么做"

**解决方案**：
- Spec 只描述**行为**（Given-When-Then），不描述实现
- 如果发现自己在写伪代码，停下来——那应该是代码，不是 Spec
- 控制粒度：一个 FR 文档不超过 100 行

### 陷阱 3: 忽视反馈速度

**症状**：CI 检查需要 10 分钟才完成，Agent 不断提交"试试看"的代码

**原因**：验证循环太慢，Agent 缺乏即时反馈

**解决方案**：
```
分层验证策略:
├── Pre-commit (< 5 秒): 编译 + 类型检查 + 格式
├── Pre-push   (< 30 秒): Lint + 架构检查 + 追溯检查
└── CI/CD      (< 5 分钟): 完整测试 + 覆盖率 + 集成测试
```

### 陷阱 4: 让 Agent 自己审查自己

**症状**：Agent 说"代码看起来没问题"，但实际有逻辑错误

**原因**：LLM 无法可靠地评估自己的输出（来源：Anthropic 研究）

**解决方案**：
- 生成和评估职责分离
- 用确定性工具（编译器、测试、Linter）做主要验证
- AI 评审仅作为补充，不作为唯一验证手段

### 陷阱 5: 一步到位

**症状**：团队花了一个月写 Spec、配 Harness，但还没跑过一次 Agent

**原因**：瀑布思维——试图在开始前做完所有准备

**解决方案**：
- 第 1 天就运行 Agent，观察它犯什么错
- 根据错误模式逐步建立约束
- Mitchell Hashimoto："Every Agent error = Harness improvement opportunity"

## 五、不同项目类型的适配策略

### 全新项目（Green Field）

从 Phase 1 开始完全可行：
1. 项目创建时就建立 `specs/` 目录
2. 每个功能先写 FR 文档
3. 从第一个 commit 开始要求追溯标签

### 遗留项目（Brown Field）

采用"围栏策略"：
1. 对新代码强制要求 Spec + 追溯
2. 旧代码不回溯（除非重构时顺带添加）
3. 用 Linter 的 `--fix-dry-run` 模式先统计违规数量，不强制

```json
// .eslintrc.json — 遗留代码豁免配置
{
  "overrides": [
    {
      "files": ["src/legacy/**/*.ts"],
      "rules": {
        "require-traceability": "warn"  // 旧代码仅警告
      }
    },
    {
      "files": ["src/new/**/*.ts"],
      "rules": {
        "require-traceability": "error"  // 新代码强制
      }
    }
  ]
}
```

### 多语言项目

统一 Harness 层，各语言适配：

```yaml
# harness.yaml — 多语言配置
harness:
  verification:
    steps:
      # 通用检查
      - name: traceability
        command: "python scripts/check_traceability.py specs/ ."
        must_pass: true

      # TypeScript 特定
      - name: ts_typecheck
        command: "npx tsc --noEmit"
        must_pass: true
        applies_to: "**/*.ts"

      # Python 特定
      - name: py_typecheck
        command: "mypy src/"
        must_pass: true
        applies_to: "**/*.py"

      # C++ 特定
      - name: cpp_compile
        command: "cmake --build build"
        must_pass: true
        applies_to: "**/*.cpp"
```
