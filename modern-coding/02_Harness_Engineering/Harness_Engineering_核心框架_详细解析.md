# Harness Engineering 核心框架 详细解析

> **核心结论**：Harness Engineering 是 AI 工程时代的范式转变——它不优化模型本身，而是系统性地设计和优化 AI Agent 运行的工程环境，通过约束机制、反馈回路和持续改进循环，将 AI 编码能力提升 10 倍。

## 概述

Harness Engineering（驾驭工程）是一套围绕 AI Agent 设计和构建的**约束机制、反馈回路、工作流控制和持续改进循环**的系统工程实践。

"Harness"一词的多层含义：
- **工程含义**：在任何工程学科中，harness 都是连接、保护和编排组件的那一层
- **比喻含义**：像马具之于骏马，harness 让 AI 这匹"烈马"既能释放全部力量，又不会偏离轨道
- **系统含义**：包含环境、工具、规则和反馈机制的整个工作环境

## 一、核心哲学

### 1.1 Mitchell Hashimoto 的定义

> "Every time you discover an agent has made a mistake, you take the time to engineer a solution so that it can never make that mistake again."
> 
> —— Mitchell Hashimoto，HashiCorp 创始人，Terraform 作者

### 1.2 OpenAI 的核心洞察

> "Agents aren't hard; the Harness is hard."
> 
> —— Ryan Lopopolo，OpenAI Codex 团队

### 1.3 生产力的悖论

**约束创造自由**：
- 当 GPT-5 或 Claude 可以生成任何东西时，它会浪费大量 token 探索死胡同
- 一个设计良好的 Harness 划定了一条狭窄、明确的成功路径
- 通过提供清晰的边界和有限的工具集，Harness 强制 Agent 更快收敛到正确答案

## 二、三维框架详解

### 维度一：上下文工程（Context Engineering）

**目标**：让 AI 系统能够理解结构和上下文

#### 渐进式文档披露

核心思想：模仿人类工程师的学习路径，从小而稳定的入口开始

```
AGENTS.md（目录，约100行）
├── ARCHITECTURE.md
├── DESIGN.md  
├── docs/
│   ├── design-docs/
│   ├── exec-plans/
│   ├── product-specs/
│   ├── references/
│   ├── FRONTEND.md
│   ├── QUALITY_SCORE.md
│   └── SECURITY.md
```

#### 反面教训

巨大的单一 AGENTS.md 文件会导致：
- 挤占任务和代码的上下文空间
- Agent 局部模式匹配，而非有意识导航
- 文档快速腐烂（无人维护的大文件）
- 难以机械化验证

#### 多通道可观测性接入

Agent 需要运行时直接访问的三大通道：

| 通道 | 工具 | 用途 |
|------|------|------|
| 浏览器驱动 | Chrome DevTools Protocol | 截屏、DOM 快照、UI 操作、bug 复现、修复验证 |
| 日志/指标 | LogQL / PromQL | 运行时状态监控 |
| 代码追溯 | Git 历史、PR 讨论 | 决策上下文理解 |

#### 仓库即知识图谱

从 Agent 视角：**任何无法在运行时访问的东西都不存在**

隐含在 Slack 讨论中的决策对 Agent 不可见，需要将系统结构、架构原则显性化为机器可读形式。

---

### 维度二：架构约束（Architectural Constraints）

**目标**：将工程品味机械化，限制 Agent 的行为范围

#### 分层架构规则

OpenAI 采用严格的依赖流向：

```
Types → Config → Repo → Service → Runtime → UI
        只能向上依赖，严禁跨层
```

#### 自定义 Linter + 结构化测试

关键实践：
- 不依赖 LLM 来监控，用确定性工具
- Lint 错误消息包含自动修复建议
- 违规代码拦截率达 **100%**
- Pre-push hook 在 5 秒内解决常见问题

#### Linter 错误消息设计

```
错误: [ARCH-001] 违反分层依赖规则

位置: src/runtime/Service.ts:45

问题: Runtime 层直接导入了 Types 层的模块
      runtime → types (❌ 禁止)

建议修复:
  1. 通过 Config 层间接引用
  2. 或将共享类型上移到更底层

自动修复命令: npm run fix:arch-001
```

#### 跨切面关注点的显式管理

认证、连接器、遥测、特性开关通过**唯一的 Providers 接口**进入系统，其他一切被禁止。

---

### 维度三：垃圾回收与熵管理（Garbage Collection）

**目标**：对抗系统衰退，保持 Harness 本身的可读性

#### 概念类比

| 传统 GC | Harness GC |
|---------|-----------|
| 清理不再使用的内存 | 清理不再使用的文档、过时规则 |
| 防止内存泄漏 | 防止文档腐化 |
| 自动化运行 | Agent 驱动的自动化重构 |

#### 关键实践

1. **文档漂移扫描**：定期运行专用 Agent，检测文档不一致，发现架构违规
2. **质量评分追踪**：保留可验证的评估指标，预防 Harness 自身腐化
3. **自动化重构 PR**：Agent 定期提交改进 PR，清理技术债

#### 数据洞察

- 89% 的团队已部署可观测性
- 仅 52% 建立评估体系
- **结论**：评估体系是 Harness 成熟的关键分水岭

## 三、反馈回路设计

### 3.1 分层反馈体系

```
Agent 写代码
    ↓
┌─────────────────────┐
│ 本地 Pre-commit      │ ← 快速反馈（5秒）
├─────────────────────┤
│ Pre-push Hook        │ ← 结构化错误消息
├─────────────────────┤
│ CI/CD 管道           │ ← 完整验证
├─────────────────────┤
│ 自动化测试 Agent     │ ← 多 Agent 评审
├─────────────────────┤
│ 可观测性回传         │ ← 运行时反馈
└─────────────────────┘
    ↓
修复迭代 → 最终合并
```

### 3.2 错误消息三要素

每条错误消息必须包含：
1. **问题定位**：文件、行号、具体违规
2. **修复建议**：可执行的自动修复命令
3. **规则引用**：链接到相关规范文档

### 3.3 迭代次数限制

**Stripe 二次重试规则**：
- CI 失败后，Agent 仅允许一次自动修复尝试
- 第二次失败，任务立即人工介入
- 防止 Agent 无限循环浪费资源

## 四、安全设计

Agent 执行环境的安全需要五层防御：

| 层级 | 措施 | 目的 |
|------|------|------|
| 网络隔离 | 仅允许授权端点连接 | 防止访问内网资源 |
| 文件系统 | 只读代码库，写权限仅限工作目录 | 防止意外修改 |
| 工具权限 | 工具白名单 + 参数白名单 + 速率限制 | 防止滥用 |
| 凭证管理 | Vault 集成，Agent 不直接访问敏感凭证 | 防止泄露 |
| 审计追踪 | 记录所有 Agent 操作，完整变更链 | 支持取证和回滚 |

## 五、案例与数据

### OpenAI Codex 体系

最成熟的实践案例，工程师角色根本转变：

**传统模式**：工程师写代码，Agent 辅助
**Agent-First 模式**：Agent 写代码，工程师设计环境

新角色的核心职责：
1. **拆解目标**：大目标 → 小的可执行构建块
2. **描述意图**：通过结构化文档和 prompt 启动 Agent
3. **构建反馈循环**：Agent 卡住时追问"缺什么能力"
4. **验证结果**：把人类品味反馈回系统，改进 Harness

### Stripe Minions 体系

规模最大的实践——周均合并 **1,300 个 AI 生成 PR**：

```
开发者在 Slack/CLI 发任务
    ↓
Agent 生成实现代码 + 运行测试
    ↓
CI 失败? → 自动进入修复循环（最多1次）
    ↓
代码质量符合门禁 → 自动生成 PR
    ↓
人类工程师做最终评审
```

关键要素：
- Devbox 隔离环境 10 秒启动
- 工具访问权限与人类完全一致
- 完整的 CI/CD 流水线自动化

## 六、最佳实践与注意事项

### 推荐做法

1. **文档优先**：从 100 行 AGENTS.md 开始，随错误积累逐步完善
2. **约束硬化**：能写进 Linter 的规则，绝不依赖 Prompt
3. **渐进授权**：分阶段授予 Agent 更大的自主权
4. **失败即改进**：每次 Agent 失败，都是改进 Harness 的机会

### 常见陷阱

1. **过度文档**：巨大的 AGENTS.md 挤占上下文空间
   - 解决：渐进式披露，保持入口精简

2. **依赖 AI 自审**：模型无法可靠评估自己的工作
   - 解决：分离 Generator 和 Evaluator Agent

3. **忽视熵增**：Harness 自身腐化
   - 解决：部署 GC Agent，定期扫描和清理
