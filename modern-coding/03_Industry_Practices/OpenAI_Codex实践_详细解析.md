# OpenAI Codex Harness Engineering 实践 详细解析

> **核心结论**：OpenAI 通过 5 个月的极端实验（零人工代码）证明了 Harness Engineering 的威力——3-7 人团队完成了传统需要 30+ 人的工作量，人均日 PR 数达 3.5 个，效率提升约 10 倍。

## 公司概况

| 维度 | 信息 |
|------|------|
| 公司 | OpenAI |
| 系统/产品 | Codex（内部 AI 编码系统） |
| 团队规模 | 3 人起步，后扩至 7 人 |
| 实验周期 | 5 个月（2025年8月-2026年1月） |
| 代码产出 | ~100 万行 |
| 合并 PR | ~1,500 个 |

## 一、实验设计

### 1.1 极端约束

**规则**：完全禁止人工编写源码

目的：倒逼团队构建能让 Agent 可靠工作的基础设施

### 1.2 核心洞察

> "Agents aren't hard; the Harness is hard."

## 二、Harness 设计

### 2.1 三层体系架构

```
┌─────────────────────────────────────────────────────────┐
│                    反馈循环层                            │
│   Pre-commit → Pre-push → CI/CD → 自动化评审 → 监控     │
├─────────────────────────────────────────────────────────┤
│                    代码质量层                            │
│   自定义 Linter → 静态分析 → 结构化测试 → 架构验证       │
├─────────────────────────────────────────────────────────┤
│                    可观测性栈                            │
│   日志(Loki) → 指标(Prometheus) → 追踪 → UI 截图        │
├─────────────────────────────────────────────────────────┤
│                    基础设施层                            │
│   Git Worktree → Devbox → 工具白名单 → 资源限制         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 上下文策略

#### 渐进式文档披露

```
AGENTS.md（入口，~100行）
├── 项目目标简述
├── 核心架构原则
├── 指向详细文档的链接树
└── Agent 工作流快速参考

docs/
├── ARCHITECTURE.md    # 分层模型、依赖规则
├── DESIGN.md          # 设计决策和模式
├── FRONTEND.md        # 前端规范
├── SECURITY.md        # 安全约束
└── QUALITY_SCORE.md   # 质量评分标准
```

#### 多通道可观测性

| 通道 | 工具 | Agent 能力 |
|------|------|-----------|
| 浏览器驱动 | Chrome DevTools Protocol | 截屏、DOM 快照、UI 操作 |
| 日志查询 | LogQL | 运行时日志分析 |
| 指标查询 | PromQL | 性能指标监控 |
| 代码追溯 | Git blame、PR 讨论 | 理解决策背景 |

### 2.3 约束机制

#### 分层架构规则

```
Types → Config → Repo → Service → Runtime → UI

规则：
- 只能向上依赖
- 严禁跨层依赖
- 跨切面关注点通过 Providers 接口统一管理
```

#### 自定义 Linter

关键特性：
- **100% 违规拦截率**：任何架构违规都被自动捕获
- **5 秒反馈**：Pre-push hook 快速失败
- **自动修复建议**：错误消息包含可执行的修复命令

#### Linter 错误消息示例

```
[ARCH-001] 分层依赖违规

文件: src/runtime/Service.ts:45
问题: Runtime 层直接导入了 Types 层
      runtime → types (❌ 禁止)

建议修复:
  1. 通过 Config 层间接引用
  2. 或将共享类型上移到更底层

自动修复: npm run fix:arch-001
文档: docs/ARCHITECTURE.md#dependency-rules
```

### 2.4 反馈回路

```
Agent 写代码
    ↓
Pre-commit (5秒)
    ↓ 失败 → 结构化错误消息
Pre-push (完整 Lint)
    ↓ 失败 → 自动修复建议
CI/CD (完整测试)
    ↓ 失败 → Agent 自动修复循环
自动化评审 Agent
    ↓ 问题 → 反馈回 Harness 改进
人类最终审核
    ↓
合并
```

## 四、工程师角色转变

### 4.1 角色对比

| 维度 | 传统工程师 | Agent-First 工程师 |
|------|-----------|-------------------|
| 核心工作 | 写代码 | 设计 Harness |
| 技能重点 | 编码能力 | 系统设计能力 |
| 质量保障 | 代码审查 | 规则编码化 |
| 问题处理 | 修复 bug | 改进 Harness |
| 产出度量 | 代码行数 | Agent 成功率 |

### 4.2 新角色核心职责

1. **拆解目标**
   - 将大目标分解为小的可执行构建块
   - 确保 Agent 能理解任务边界

2. **描述意图**
   - 通过结构化文档启动 Agent
   - 编写清晰的 Prompt 和约束

3. **构建反馈循环**
   - Agent 卡住时追问"缺什么能力"
   - 持续优化 Harness 约束

4. **验证结果**
   - 把人类品味反馈回系统
   - 决策是否需要人工介入

## 五、关键经验教训

### 成功因素

1. **极端约束倒逼创新**：零人工代码规则强制团队构建完善的 Harness
2. **文档即代码**：所有约束都有机器可读的表达形式
3. **快速反馈循环**：5 秒内获得 Lint 结果，防止 Agent 浪费时间
4. **渐进式授权**：分阶段扩大 Agent 自主权，确保可控

### 踩过的坑

1. **早期文档过大**
   - 问题：单一 AGENTS.md 超过 1000 行
   - 后果：挤占上下文空间，Agent 效率下降
   - 解决：拆分为渐进式披露结构

2. **依赖 AI 自审**
   - 问题：让 Agent 自己检查自己的代码
   - 后果：大量低级错误漏检
   - 解决：分离 Generator 和 Evaluator

3. **忽视熵增**
   - 问题：文档和规则随时间腐化
   - 后果：Harness 效率逐渐下降
   - 解决：部署专用 GC Agent 定期清理

## 六、可借鉴的实践

### 中小团队适用

1. **从 Pre-commit Hook 开始**：5 秒反馈是最低成本高收益的实践
2. **需求 ID 系统**：确保每个代码变更可追溯
3. **结构化错误消息**：包含修复建议和文档链接

### 大型企业适用

1. **分层架构 Linter**：将架构决策编码为可执行规则
2. **多 Agent 评审**：分离生成和评估职责
3. **完整可观测性栈**：每个 worktree 独立实例

## 五、开放挑战

1. **学徒缺口**：初级工程师如何获得手动开发直觉
2. **大规模老项目迁移**：无 Harness 的遗留系统如何逐步引入
3. **多语言环境**：跨语言的统一 Harness 框架仍在探索
4. **Agent 间协作**：多 Agent 并发修改的合并策略
5. **成本量化**：构建完整 Harness 需要 7 人 × 5 个月
