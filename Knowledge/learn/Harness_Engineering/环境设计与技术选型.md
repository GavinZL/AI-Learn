# Harness Engineering 环境设计与技术选型

> **核心结论**：一套完整的 Harness Engineering 环境需要覆盖基础设施层、可观测性栈、代码质量层和反馈循环四个核心层次，通过技术选型和架构设计构建 Agent 可靠工作的系统基础。

## 一、整体架构概览

Harness Engineering 的环境设计围绕四个核心层展开：

```
┌─────────────────────────────────────────┐
│            反馈循环层                      │
│  Pre-commit → Pre-push → CI/CD → 监控     │
├─────────────────────────────────────────┤
│            代码质量层                      │
│  自定义Linter → 静态分析 → 结构化测试      │
├─────────────────────────────────────────┤
│            可观测性栈                      │
│  日志(Loki) → 指标(Prometheus) → 追踪     │
├─────────────────────────────────────────┤
│            基础设施层                      │
│  沙盒隔离 → 工具约束 → 资源限制 → Git      │
└─────────────────────────────────────────┘
```

## 二、基础设施层

| 组件 | 技术选择 | 目的 |
|------|----------|------|
| 代码隔离 | Docker + 虚拟机 | 沙盒执行，防止 Agent 恶意代码 |
| 工具约束 | 受控的 CLI/API 工具集 | 防止 Agent 随意调用系统命令 |
| 资源限制 | 配额管理 | 防止无限循环和资源耗尽 |
| 版本控制 | Git + 结构化 Hooks | 强制执行架构规则 |

## 三、可观测性栈

```
应用日志层          指标层              追踪层
   ↓                 ↓                   ↓
Victoria Logs  →  Victoria Metrics  →  Jaeger
   ↓                 ↓                   ↓
LogQL查询       PromQL查询        链路追踪
   ↓                 ↓                   ↓
Agent可读      Agent可读        Agent可读
```

**重点**：每个工作树（git worktree）独立实例化，Agent 有完全隔离的观测视图。

## 四、代码质量层

```
Source Code
    ↓
┌─────────────────────┐
│  自定义 Linter       │ ← 架构规则编码化
├─────────────────────┤
│  静态分析工具        │ ← 安全/规范检查
├─────────────────────┤
│  结构化测试          │ ← 契约验证
├─────────────────────┤
│  集成测试            │ ← 端到端验证
└─────────────────────┘
    ↓
必须全部通过 → PR才能合并
```

## 五、反馈循环实现

```
Agent写代码
    ↓
┌─────────────────────┐
│ 本地 Pre-commit      │ ← 快速反馈（5秒）
├─────────────────────┤
│ Pre-push Hook        │ ← 结构化错误消息
├─────────────────────┤
│ CI/CD 管道           │ ← 完整验证
├─────────────────────┤
│ 自动化测试 Agent     │ ← 多 Agent 评审
├─────────────────────┤
│ 可观测性回传         │ ← 运行时反馈
└─────────────────────┘
    ↓
修复迭代 → 最终合并
```

## 七、工具栈建议

| 功能层 | 开源方案 | 商业方案 | 评价 |
|--------|----------|----------|------|
| 版本控制 | Git | GitHub/GitLab | Git 必须，集成平台可选 |
| CI/CD | Jenkins/GitLab CI | GitHub Actions | GitOps 优先 |
| 可观测性 | Loki/Prometheus | DataDog/Splunk | 开源组合足够，规模化考虑商业 |
| 代码分析 | SonarQube | Snyk | SonarQube 可覆盖大部分需求 |
| Linter | ESLint/Pylint | Veracode | 自定义规则是关键，开源工具够用 |
| 容器化 | Docker | - | Docker 必须 |
| 工作流编排 | Airflow | - | 轻量级用 Shell 脚本，复杂用 Airflow |
| API网关 | Kong/Tyk | - | 控制 Agent 权限的关键 |
| MCP协议 | Anthropic MCP | Harness MCP | 标准化 Agent 接入的未来方向 |

## 八、安全设计

Agent 执行环境的安全需要五层防御：

1. **网络隔离**：仅允许到已授权端点的连接，禁止对内网资源的直接访问
2. **文件系统权限**：只读访问代码库，写权限仅限于工作目录
3. **工具权限管理**：工具白名单 + 参数白名单 + 速率限制
4. **凭证管理**：Vault/Sealed Secrets 集成，Agent 无法直接访问敏感凭证
5. **审计追踪**：记录所有 Agent 操作，完整的变更链，支持取证和回滚
