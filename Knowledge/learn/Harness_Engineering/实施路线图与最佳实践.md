# Harness Engineering 实施路线图与最佳实践

> **核心结论**：Harness Engineering 的实施遵循"从简单到复杂、从局部到全局"的渐进路径，通过六个阶段逐步构建完整的 Agent 工作环境。

## 实施六阶段路线图

基于 Mitchell Hashimoto 提出的方法论，结合 OpenAI 和 Stripe 的实践经验。

### 第一阶段：双轨执行（第0-2周）

**目标**：建立对 Agent 能力和局限的认知

- 同一任务由人工和 Agent 分别执行
- 比较结果，发现分歧点
- 记录 Agent 的常见错误模式

**关键产出**：问题模式库

### 第二阶段：习惯养成（第2-4周）

**目标**：逐步建立对 Agent 能力的信心

- 每日下班前 30 分钟启动 Agent 处理深度调研/并行探索任务
- 识别 Agent 最擅长的任务类型
- 建立团队对 AI 辅助开发的工作节奏

**关键产出**：Agent 最擅长的任务清单

### 第三阶段：关键跃迁（第4-8周）

**目标**：建立项目级的 Agent 工作规范

- 创建项目 AGENTS.md 文档（从约100行开始）
- 随错误积累逐步完善规则
- 改造 Linter 输出为 Agent 可读格式
- 配置基础 Git hooks

**关键产出**：可执行的工程约束

### 第四阶段：反馈回路完善（第2-3个月）

**目标**：形成自愈循环

- 在项目中集成可观测性接入（日志/指标/UI 驱动）
- 让 Agent 能直接访问运行时信息
- 建立自动化测试框架

**关键产出**：自愈循环的形成

### 第五阶段：架构防御强化（第3-5个月）

**目标**：机械化的品味编码

- 部署自定义 Linter 和结构化测试
- 实现 100% 的违规拦截率
- 建立分层架构验证体系

**关键产出**：机械化的品味编码

### 第六阶段：垃圾回收机制（第5-6个月+）

**目标**：长期可维护的系统

- 部署文档漂移扫描 Agent
- 建立质量评分体系
- 自动化重构和技术债清理

**关键产出**：长期可维护的系统

## 四大核心原则

### 原则1：从微观管理到宏观设计

错误做法：逐行审查 Agent 代码
正确做法：设计好架构、规则、边界，让 Agent 在框架内自由发挥

### 原则2：从一次性指令到持续闭环

错误做法：prompt 一次 → 期望完美结果
正确做法：建立反馈循环 → 通过自动化测试和监控持续校准

### 原则3：从依赖模型到依赖系统

错误做法：期待单一 AI 模型完美解决
正确做法：用沙盒、Linter、CI/CD、监控构成的系统约束 Agent

### 原则4：用代码和测试来沟通

错误做法：依赖自然语言 prompt
正确做法：代码、架构规则、自动化测试——最精确、最无歧义

## 代码评审的演变

在 Harness Engineering 体系中，代码评审经历了根本性变化：

1. **Agent 自审**：自动检查——Linter、结构化测试、架构验证
2. **多 Agent 评审**：不同 Agent 互相审查代码质量
3. **人类高层评审**：仅在高层架构决策时介入，而非逐行审查

这种分层评审机制使得评审自动化率可达 95% 以上。
