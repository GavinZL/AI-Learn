# 封装与解封装原理

> 从第一性原理理解视频容器格式：多流同步、随机访问与元数据承载

---

## 核心结论（TL;DR）

**封装（Muxing）的本质：将编码后的音频流、视频流、字幕流等多路数据按照特定规则"打包"进一个文件，同时附加时间同步信息和元数据。解封装（Demuxing）则是其逆过程。**

关键要点：
1. **封装格式 ≠ 编码格式**：MP4 是容器，H.264 是编码，两者是"箱子"与"货物"的关系
2. **核心价值四件套**：多流合并、时间同步、随机访问（Seek）、元数据承载
3. **转封装 ≠ 转码**：改变容器不需要重新编码，秒级完成且无质量损失
4. **时间戳是灵魂**：PTS/DTS 机制确保音视频同步，是播放器正确渲染的基础

---

## 一、第一性原理：为什么需要封装

让我们从最基础的问题出发，理解封装存在的必然性。

### 1.1 问题一：多路数据如何合并？

想象你是一个快递分拣员，面前有三条传送带：
- 传送带 A：不断输出视频编码数据（H.264 NAL 单元流）
- 传送带 B：不断输出音频编码数据（AAC 帧流）
- 传送带 C：不断输出字幕数据（文本或图形）

现在你需要把它们装进**同一个包裹**发给用户。问题来了：

```
视频流: [NAL1][NAL2][NAL3][NAL4]...
音频流: [AAC1][AAC2][AAC3][AAC4]...
字幕流: [SUB1][SUB2][SUB3]...
              ↓
        如何合并成一个文件？
              ↓
        [???????????????]
```

**直接拼接行不行？**

```
[NAL1][NAL2]...[NALn][AAC1][AAC2]...[AACn][SUB1]...
```

不行！因为：
- 播放器不知道视频从哪里结束、音频从哪里开始
- 无法边播视频边播音频（必须等视频全部读完）
- 无法跳转到中间位置播放

**正确的做法：交错存储 + 目录索引**

```
[Header/目录]
[Video Pkt 1][Audio Pkt 1][Video Pkt 2][Audio Pkt 2]...
[Index/索引]
```

这就像把多本书装进一个箱子：
- 箱子盖上贴一张清单（Header），说明里面有哪些书
- 每本书用书签标记页码（时间戳）
- 箱子底部放一本目录（Index），记录每本书在箱子里的位置

### 1.2 问题二：音视频如何同步？

这是一个精度问题。看这组数据：

| 流类型 | 帧率/采样率 | 每帧时长 |
|--------|-------------|----------|
| 视频 | 30 fps | 33.33 ms |
| 视频 | 24 fps | 41.67 ms |
| 视频 | 60 fps | 16.67 ms |
| 音频 | 44100 Hz, 1024 samples/frame | 23.22 ms |
| 音频 | 48000 Hz, 1024 samples/frame | 21.33 ms |

视频和音频的"心跳"节奏完全不同！

**没有同步机制会怎样？**

假设视频 30fps，音频每帧 23.22ms：
- 第 100 帧视频应该在 3333ms 显示
- 此时音频已经播放了 3333/23.22 ≈ 143.5 帧

如果简单地"视频播一帧、音频播一帧"交替进行，几秒钟后就会出现严重的口型不同步（说话人嘴巴动了，声音还没出来）。

**解决方案：统一的时间戳系统**

封装格式引入了 **PTS（Presentation Time Stamp，显示时间戳）** 机制：
- 每个视频帧标记：这一帧应该在第 X 毫秒显示
- 每个音频帧标记：这一帧应该在第 Y 毫秒播放
- 播放器根据时间戳协调两路数据的渲染时机

```
Video Frame 1: PTS = 0ms      → 显示
Video Frame 2: PTS = 33ms     → 显示
Audio Frame 1: PTS = 0ms      → 播放
Audio Frame 2: PTS = 23ms     → 播放
Video Frame 3: PTS = 67ms     → 显示
Audio Frame 3: PTS = 46ms     → 播放
...
```

### 1.3 问题三：如何实现随机访问（Seek）？

用户拖动进度条到第 5 分钟，播放器需要：

1. **快速定位**：找到第 5 分钟附近的数据在文件的什么位置
2. **找到关键帧**：视频必须从关键帧（I 帧）开始解码，否则画面花屏
3. **同步音频**：找到对应时间点的音频帧

**没有索引会怎样？**

如果文件没有索引结构，播放器只能：
```
从文件开头 → 逐帧解析 → 读取时间戳 → 
判断是否到达目标时间 → 没到则继续 → ...
```

一个 2 小时的 4K 电影，文件大小可能 50GB，这样遍历一遍可能需要几分钟！

**索引结构的作用**

```
索引表（Index Table）:
┌─────────────┬───────────────┬──────────────┐
│ 时间戳(PTS) │ 文件偏移(Byte)│ 是否关键帧   │
├─────────────┼───────────────┼──────────────┤
│ 0           │ 1024          │ Yes (IDR)    │
│ 33ms        │ 52480         │ No           │
│ 67ms        │ 53200         │ No           │
│ ...         │ ...           │ ...          │
│ 300000ms    │ 1073741824    │ Yes (IDR)    │ ← 第5分钟
│ ...         │ ...           │ ...          │
└─────────────┴───────────────┴──────────────┘
```

有了索引，Seek 操作变成：
```
1. 二分查找索引表 → O(log n) 复杂度
2. 找到目标时间最近的关键帧
3. 直接 fseek() 到对应文件偏移
4. 开始解码播放
```

### 1.4 问题四：如何描述内容属性？

一个视频文件需要携带大量描述信息：

| 类别 | 信息示例 |
|------|----------|
| 技术参数 | 分辨率 1920x1080、帧率 24fps、色彩空间 BT.709 |
| 编码参数 | H.264 Profile High、Level 4.1、参考帧数 4 |
| 音频参数 | AAC-LC、48kHz、立体声、比特率 128kbps |
| 内容信息 | 标题、作者、创建时间、版权信息 |
| 多语言 | 音轨 1: 英语、音轨 2: 中文、字幕: 中/英/日 |
| 章节 | 第一章 00:00:00、第二章 00:15:30... |

这些信息必须有**标准化的存储格式**，否则每个播放器都需要猜测文件结构。

## 二、封装格式的核心架构模型

所有封装格式，无论 MP4、MKV 还是 TS，都遵循类似的分层架构：

```
┌─────────────────────────────────────────────────┐
│              Container（容器层）                 │
│  ┌─────────────────────────────────────────┐   │
│  │           Header / Metadata              │   │
│  │  - 文件格式标识                           │   │
│  │  - 轨道数量和类型                         │   │
│  │  - 时长、创建时间等                       │   │
│  └─────────────────────────────────────────┘   │
│                                                 │
│  ┌──────────────┐    ┌──────────────┐         │
│  │ Video Track  │    │ Audio Track  │   ...   │
│  │ ┌──────────┐ │    │ ┌──────────┐ │         │
│  │ │ Codec    │ │    │ │ Codec    │ │         │
│  │ │ Config   │ │    │ │ Config   │ │         │
│  │ └──────────┘ │    │ └──────────┘ │         │
│  │ ┌──────────┐ │    │ ┌──────────┐ │         │
│  │ │ Packet 1 │ │    │ │ Packet 1 │ │         │
│  │ │ PTS, DTS │ │    │ │ PTS      │ │         │
│  │ │ Data     │ │    │ │ Data     │ │         │
│  │ ├──────────┤ │    │ ├──────────┤ │         │
│  │ │ Packet 2 │ │    │ │ Packet 2 │ │         │
│  │ │ ...      │ │    │ │ ...      │ │         │
│  │ └──────────┘ │    │ └──────────┘ │         │
│  └──────────────┘    └──────────────┘         │
│                                                 │
│  ┌─────────────────────────────────────────┐   │
│  │              Index（索引层）              │   │
│  │  - 关键帧位置表                          │   │
│  │  - 时间戳 → 文件偏移映射                 │   │
│  └─────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### 2.1 容器层（Container）

**定义**：一个文件 = 一个容器。容器是封装格式的最外层边界。

**职责**：
- 文件格式标识（Magic Number）
- 包含一个或多个轨道
- 存储全局元数据

**类比**：容器就像一个快递箱，外面贴着收件人信息（元数据），里面装着多个独立包装的商品（轨道）。

### 2.2 轨道层（Track / Stream）

**定义**：每个轨道对应一路编码数据。

**常见轨道类型**：
- 视频轨道（Video Track）
- 音频轨道（Audio Track）：可以有多条，如不同语言
- 字幕轨道（Subtitle Track）：可以是文本或图形字幕
- 数据轨道（Data Track）：章节信息、时间码等

**轨道包含的信息**：
```
Track {
    track_id: 1,              // 轨道编号
    track_type: "video",      // 轨道类型
    codec: "H.264",           // 编码格式
    codec_config: {...},      // 编码器配置（SPS/PPS 等）
    duration: 7200000,        // 时长（单位取决于 timescale）
    timescale: 90000,         // 时间刻度
    packets: [...]            // 数据包序列
}
```

### 2.3 数据包层（Packet / Sample / Frame）

**定义**：轨道内的基本数据单元，通常对应编码器输出的一帧或一组帧。

**数据包结构**：
```
Packet {
    pts: 180000,              // 显示时间戳
    dts: 180000,              // 解码时间戳
    duration: 3000,           // 持续时间
    size: 45678,              // 数据大小（字节）
    flags: {
        keyframe: true,       // 是否关键帧
        disposable: false,    // 是否可丢弃
    },
    data: [0x00, 0x00, ...]   // 实际编码数据
}
```

### 2.4 编码数据层（Codec Data）

**定义**：数据包内部的实际编码二进制数据。

**重要特性**：
- 封装格式**不关心**编码数据的内部结构
- 封装只负责"搬运"，不负责"理解"
- 就像快递公司不管箱子里装的是书还是衣服

**示例**：
- H.264 的 NAL 单元（NAL Unit）
- AAC 的原始帧（Raw Frame）或 ADTS 帧
- HEVC 的 NAL 单元

## 三、时间模型深入解析

时间模型是封装格式的**灵魂**，理解它才能理解音视频同步的本质。

### 3.1 时间基（Timebase / Timescale）

**定义**：时间戳的精度单位，表示"1 秒被分成多少份"。

**常见 Timescale 值**：

| 格式/场景 | Timescale | 含义 |
|-----------|-----------|------|
| MP4 视频 | 90000 | 1 秒 = 90000 个 tick |
| MP4 音频 | 48000 | 通常等于采样率 |
| MPEG-TS | 90000 | 标准值 |
| MKV | 1000000 | 1 秒 = 1000000 纳秒级精度 |

**计算示例**：

假设 Timescale = 90000：
```
第 0 秒的帧:    PTS = 0 × 90000 = 0
第 1 秒的帧:    PTS = 1 × 90000 = 90000
第 2.5 秒的帧:  PTS = 2.5 × 90000 = 225000
```

反向计算：
```
PTS = 180000
实际时间 = 180000 / 90000 = 2.0 秒
```

**为什么选择 90000？**

历史原因：90000 = 300 × 300 = LCM(24000/1001, 25, 30000/1001, ...)

它能被常见帧率整除或接近整除：
- 30 fps: 90000/30 = 3000（整除）
- 25 fps: 90000/25 = 3600（整除）
- 24 fps: 90000/24 = 3750（整除）
- 29.97 fps: 90000 ÷ (30000/1001) ≈ 3003（近似整除）

### 3.2 PTS（Presentation Time Stamp）

**定义**：帧的**显示时间**，告诉播放器"在什么时刻把这一帧呈现给用户"。

**特点**：
- 单调递增（在正常播放时）
- 决定用户看到的画面顺序
- 音视频同步的基准

**计算公式**：
```
PTS = 帧序号 × (Timescale / 帧率)

示例（30fps, Timescale=90000）:
Frame 0: PTS = 0 × 3000 = 0
Frame 1: PTS = 1 × 3000 = 3000
Frame 2: PTS = 2 × 3000 = 6000
```

### 3.3 DTS（Decoding Time Stamp）

**定义**：帧的**解码时间**，告诉解码器"在什么时刻解码这一帧"。

**关键问题：为什么需要 DTS？PTS 不够吗？**

答案：因为 **B 帧**的存在！

### 3.4 PTS/DTS 与 B 帧的关系

**背景知识**：
- I 帧（Intra）：完整编码，不依赖其他帧
- P 帧（Predictive）：前向参考，依赖前面的 I 或 P 帧
- B 帧（Bidirectional）：双向参考，依赖前后的帧

**核心矛盾**：B 帧需要参考**未来**的帧，但解码是**顺序**进行的。

**示例分析**：

假设显示顺序为：I0 - B1 - B2 - P3

```
显示顺序（用户看到的）: I0 → B1 → B2 → P3
                        ↑    ↑    ↑    ↑
                       显示  显示  显示  显示
                       0ms   33ms 67ms 100ms

但是解码顺序必须是:      I0 → P3 → B1 → B2
                        ↑    ↑    ↑    ↑
                       解码  解码  解码  解码

因为 B1、B2 需要参考 P3，所以 P3 必须先解码！
```

**PTS/DTS 对照表**：

| 帧类型 | 显示顺序 | 解码顺序 | PTS (显示时间) | DTS (解码时间) |
|--------|----------|----------|----------------|----------------|
| I0 | 0 | 0 | 0 | 0 |
| B1 | 1 | 2 | 3000 | 6000 |
| B2 | 2 | 3 | 6000 | 9000 |
| P3 | 3 | 1 | 9000 | 3000 |

观察规律：
- I/P 帧：DTS ≤ PTS
- B 帧：DTS > PTS（因为要等参考帧解码完）
- **无 B 帧时：DTS = PTS**

**图解**：
```
时间轴 →

显示顺序:  I0 ─── B1 ─── B2 ─── P3
           ↑      ↑      ↑      ↑
          PTS=0  PTS=1  PTS=2  PTS=3

解码顺序:  I0 ─── P3 ─── B1 ─── B2
           ↑      ↑      ↑      ↑
          DTS=0  DTS=1  DTS=2  DTS=3

解码器视角：
  解码 I0 → 缓存
  解码 P3 → 缓存（因为 B1/B2 需要它）
  解码 B1 → 输出（参考 I0 和 P3）
  解码 B2 → 输出（参考 I0 和 P3）
  输出 P3
```

### 3.5 音视频同步机制

**核心问题**：视频和音频是两个独立的解码管线，如何保证同步？

**三种主时钟策略**：

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| 音频时钟为主 | 音频连续播放，视频帧根据音频时钟调整（跳帧或重复帧） | 最常用，因为人耳对音频卡顿更敏感 |
| 视频时钟为主 | 视频连续播放，音频采样率微调 | 专业视频编辑 |
| 系统时钟为主 | 使用独立的系统时钟，音视频都向它对齐 | 直播场景 |

**同步容差（Lip Sync Tolerance）**：

人类感知实验表明：
- 音频**领先**视频：可接受 -45ms 到 -125ms（因为声音传播比光慢）
- 音频**落后**视频：可接受 +45ms 到 +90ms
- 实际工程中，±40ms 被认为是"无感知同步"

**播放器同步算法（简化版）**：

```python
# 伪代码：音频时钟为主的同步策略
audio_clock = get_audio_playback_time()  # 当前音频播放位置
video_pts = get_next_video_frame_pts()   # 下一帧视频的 PTS

diff = video_pts - audio_clock

if diff > 40ms:
    # 视频太快，等待
    sleep(diff - 40ms)
    display_frame()
elif diff < -40ms:
    # 视频太慢，跳帧
    drop_frame()
else:
    # 在可接受范围内，正常显示
    display_frame()
```

## 四、索引结构与随机访问

### 4.1 索引的必要性

**没有索引的后果**：

想象一本 1000 页的书没有目录：
- 找第 500 页的内容 → 必须从第 1 页翻到第 500 页
- 找某个章节 → 必须逐页扫描标题

对于视频文件：
- Seek 到 1 小时位置 → 必须解析前 1 小时的所有帧头
- 一个 10GB 的文件可能需要读取几百 MB 才能完成 Seek

### 4.2 前置索引 vs 后置索引

**MP4 的 moov 原子**：

MP4 文件的核心索引信息存储在 `moov` 原子（atom/box）中，它可以放在：

| 位置 | 优点 | 缺点 |
|------|------|------|
| 文件尾（默认） | 写入时无需预估大小 | 必须下载完整文件才能播放 |
| 文件头（faststart） | 可边下边播 | 写入时需要预留空间或后处理 |

**文件结构对比**：

```
默认 MP4 结构：          Faststart MP4 结构：
┌────────────────┐      ┌────────────────┐
│     ftyp       │      │     ftyp       │
├────────────────┤      ├────────────────┤
│                │      │     moov       │  ← 索引在前！
│     mdat       │      │   (索引信息)   │
│  (实际数据)    │      ├────────────────┤
│                │      │                │
├────────────────┤      │     mdat       │
│     moov       │      │  (实际数据)    │
│   (索引信息)   │      │                │
└────────────────┘      └────────────────┘
```

**FFmpeg faststart 处理**：

```bash
# 录制时直接生成 faststart（需要估计文件大小）
ffmpeg -i input.mp4 -movflags faststart output.mp4

# 后处理：将 moov 移到文件头
ffmpeg -i input.mp4 -c copy -movflags +faststart output.mp4
```

**原理**：FFmpeg 读取整个文件，提取 moov 原子，重新写入文件，将 moov 放在 mdat 之前。

### 4.3 流式传输的索引策略

**问题**：直播或流媒体场景，文件没有"结尾"，如何建立索引？

**解决方案：分段 + 周期性索引**

**MPEG-TS 的 PAT/PMT**：
```
┌─────────────────────────────────────────────┐
│ [PAT][PMT][Video][Video][Audio][Video]...   │
│        ↑                                     │
│    每隔约 100ms 重复插入                      │
│                                              │
│ ...[PAT][PMT][Video][Audio][Video]...       │
└─────────────────────────────────────────────┘
```

- PAT（Program Association Table）：节目关联表
- PMT（Program Map Table）：节目映射表，描述流的组成

**fMP4（Fragmented MP4）**：
```
┌────────────────────────────────────────────────┐
│ [moov] [moof+mdat] [moof+mdat] [moof+mdat]...  │
│   ↑        ↑           ↑           ↑          │
│ 初始化  片段1索引+数据 片段2索引+数据 ...       │
└────────────────────────────────────────────────┘
```

- `moov`：只包含初始化信息（编码参数等），不包含样本索引
- `moof`：Movie Fragment，每个片段的索引
- `mdat`：该片段的实际数据

### 4.4 关键帧索引

**为什么 Seek 必须定位到关键帧？**

视频编码的帧依赖关系：
```
I ← P ← P ← P ← I ← P ← P ← P ← I
         ↑
     用户想跳转到这里
         ↓
     但必须从前面的 I 帧开始解码
```

**关键帧索引表（Sync Sample Table）**：

MP4 使用 `stss`（Sync Sample Table）原子存储关键帧索引：
```
stss = [1, 30, 60, 90, 120, ...]
        ↑   ↑   ↑
      第1帧是关键帧
          第30帧是关键帧
              第60帧是关键帧
```

**Seek 算法**：
```python
def seek_to(target_time):
    # 1. 将时间转换为帧号
    target_frame = int(target_time * fps)
    
    # 2. 在关键帧索引中找到 <= target_frame 的最大值
    keyframe = binary_search_floor(stss, target_frame)
    
    # 3. 从关键帧开始解码
    decode_from(keyframe)
    
    # 4. 跳过不需要显示的帧，直到 target_frame
    while current_frame < target_frame:
        decode_and_discard()
```

### 4.5 索引精度与文件大小的权衡

**更精细的索引 = 更快的 Seek，但文件更大**

| 索引粒度 | 优点 | 缺点 |
|----------|------|------|
| 每帧都索引 | Seek 最精确 | 索引表可能占文件 1-5% |
| 只索引关键帧 | 索引表小 | Seek 后需要解码多帧 |
| 每 N 秒索引 | 折中方案 | 通用性好 |

**实际数据**：

一个 2 小时 30fps 视频 = 216,000 帧
- 每帧索引 8 字节 → 索引表 ≈ 1.7MB
- 只索引关键帧（每 2 秒一个） → 索引表 ≈ 28KB

## 五、封装 vs 编码：彻底理清概念

这是音视频领域**最常见的误解**，必须彻底搞清楚。

### 5.1 类比理解

```
封装格式 = 箱子（Container）
编码格式 = 里面的货物（Content）

MP4 文件 = 一个 MP4 箱子，里面装着：
           - H.264 编码的视频
           - AAC 编码的音频
           
MKV 文件 = 一个 MKV 箱子，里面可以装着：
           - H.265 编码的视频
           - FLAC 无损编码的音频
           - SRT 格式的字幕
```

### 5.2 常见误解辨析

| 误解 | 实际情况 | 正确理解 |
|------|----------|----------|
| "MP4 是一种视频格式" | MP4 是封装格式 | MP4 是容器，里面可以装 H.264/H.265/AV1 等多种编码 |
| "转码 = 改封装" | 两者完全不同 | 转封装（Remux）不涉及编解码，转码（Transcode）需要重新编解码 |
| "MKV 质量比 MP4 好" | 封装不影响画质 | 画质由编码参数决定，与封装格式无关 |
| "AVI 很老，质量差" | AVI 只是旧封装 | 可以在 AVI 里装最新的编码（虽然不推荐） |
| "WebM 是谷歌的编码" | WebM 是封装格式 | WebM 是 MKV 的子集，通常装 VP9/AV1 + Opus |

### 5.3 转封装（Remux）vs 转码（Transcode）

**转封装（Remux）**：
- 只改变容器，不动编码数据
- **秒级完成**，因为只是复制数据
- **无质量损失**，因为没有重新编码
- CPU 占用极低

```bash
# 将 MKV 转封装为 MP4（秒级完成）
ffmpeg -i input.mkv -c copy output.mp4
#                    ↑
#              -c copy = 不重新编码，直接复制
```

**转码（Transcode）**：
- 解码原视频 → 处理 → 重新编码
- **耗时很长**，取决于视频时长和编码参数
- **有质量损失**（如果目标码率低于原码率）
- CPU/GPU 占用高

```bash
# 将 H.264 转码为 H.265（可能需要几十分钟）
ffmpeg -i input.mp4 -c:v libx265 -crf 23 output.mp4
#                    ↑
#              指定新的编码器
```

### 5.4 封装格式的兼容性矩阵

不是所有编码都能装进所有容器：

| 封装格式 | 常见视频编码 | 常见音频编码 | 字幕支持 |
|----------|--------------|--------------|----------|
| MP4 | H.264, H.265, AV1 | AAC, MP3, AC3 | 有限 |
| MKV | 几乎所有 | 几乎所有 | 优秀 |
| WebM | VP8, VP9, AV1 | Vorbis, Opus | WebVTT |
| MOV | H.264, H.265, ProRes | AAC, PCM | 有限 |
| TS | H.264, H.265 | AAC, AC3, MP2 | DVB字幕 |
| FLV | H.264 | AAC, MP3 | 无 |

**MKV 被称为"万能容器"**：因为它支持几乎所有编码格式，没有兼容性限制。

## 六、封装格式的分类维度

为后续详解文档铺垫，这里按 MECE（Mutually Exclusive, Collectively Exhaustive）原则分类。

### 6.1 按数据组织方式

**基于原子/Box 的（ISO BMFF 家族）**：
- 代表：MP4、MOV、3GP、fMP4
- 特点：树状层级结构，每个节点是一个 Box
- 结构清晰，易于解析

**基于 EBML 的**：
- 代表：MKV、WebM
- 特点：类似 XML 的二进制格式，可扩展性强
- EBML = Extensible Binary Meta Language

**基于包/流的**：
- 代表：MPEG-TS
- 特点：固定大小的包（188 字节），适合流式传输
- 没有全局索引，依赖周期性的 PAT/PMT

**基于标签的**：
- 代表：FLV
- 特点：简单的 Tag 结构，Header + Tag 序列
- 适合 RTMP 推流

### 6.2 按应用场景

| 场景 | 推荐格式 | 原因 |
|------|----------|------|
| 本地文件存储 | MP4, MKV | 索引完整，支持 Seek |
| 网页播放 | MP4 (H.264), WebM | 浏览器原生支持 |
| 流媒体点播 | fMP4 (DASH), TS (HLS) | 支持分段、自适应码率 |
| 直播推流 | FLV (RTMP), TS (HLS) | 低延迟，协议支持 |
| 专业剪辑 | MOV, MXF | 支持 ProRes/DNxHD 等中间编码 |
| 蓝光光盘 | M2TS | BDMV 规范要求 |

### 6.3 按标准来源

| 来源 | 格式 | 标准文档 |
|------|------|----------|
| ISO/IEC | MP4 | ISO 14496-12/14 (ISOBMFF) |
| ISO/IEC | MPEG-TS | ISO 13818-1 |
| Apple | MOV | QuickTime File Format |
| 开源社区 | MKV | Matroska Specification |
| 开源社区 | WebM | 基于 MKV 的子集 |
| Adobe | FLV | Flash Video Specification |
| SMPTE | MXF | SMPTE ST 377 |

## 七、实际计算示例

### 7.1 时间戳计算

**场景**：一个 30fps 的视频，Timescale = 90000

```
帧号    实际时间      PTS 计算              PTS 值
0       0.000s       0 × 90000/30         0
1       0.033s       1 × 90000/30         3000
2       0.067s       2 × 90000/30         6000
...
100     3.333s       100 × 90000/30       300000
```

**反向计算**：
```
PTS = 450000
实际时间 = 450000 / 90000 = 5.0 秒
帧号 = 5.0 × 30 = 150
```

### 7.2 文件大小估算

**场景**：一个 1080p 30fps H.264 视频，码率 8Mbps，时长 2 小时

```
视频数据量 = 8 Mbps × 7200 秒 = 57,600 Mb = 7,200 MB = 7.2 GB

音频数据量（AAC 128kbps）= 0.128 Mbps × 7200 秒 = 921.6 Mb ≈ 115 MB

封装开销 ≈ 0.1% ~ 1%（索引、元数据等）

总大小 ≈ 7.2 GB + 115 MB + ~70 MB = ~7.4 GB
```

### 7.3 索引表大小估算

**场景**：2 小时 30fps 视频的索引表

```
总帧数 = 30 × 60 × 60 × 2 = 216,000 帧

如果每帧索引 8 字节（文件偏移 + 大小）：
索引表大小 = 216,000 × 8 = 1,728,000 字节 ≈ 1.7 MB

如果只索引关键帧（每 2 秒一个 = 每 60 帧）：
关键帧数 = 216,000 / 60 = 3,600 个
索引表大小 = 3,600 × 8 = 28,800 字节 ≈ 28 KB
```

## 九、常见封装格式速览

| 格式 | 全称 | 特点 | 典型用途 |
|------|------|------|----------|
| MP4 | MPEG-4 Part 14 | 最通用，浏览器原生支持 | 网络视频、本地存储 |
| MKV | Matroska | 功能最强，支持几乎所有编码 | 高清收藏、多音轨/字幕 |
| MOV | QuickTime | Apple 生态，支持 ProRes | 视频编辑 |
| TS | MPEG Transport Stream | 固定包大小，流式传输 | 广播电视、HLS |
| FLV | Flash Video | 结构简单，低延迟 | RTMP 直播 |
| WebM | Web Media | 开源，浏览器支持 | 网页视频 |
| AVI | Audio Video Interleave | 古老但兼容性好 | 遗留系统 |
| MXF | Material Exchange Format | 专业级，元数据丰富 | 广播、后期制作 |

## 十、总结

### 核心概念回顾

1. **封装的本质**：多流打包 + 时间同步 + 随机访问 + 元数据
2. **时间模型**：Timebase 定义精度，PTS 控制显示，DTS 控制解码
3. **索引结构**：前置索引支持流式播放，关键帧索引支持快速 Seek
4. **封装 ≠ 编码**：容器与内容的关系，转封装与转码的区别

### 选择封装格式的决策树

```
需要什么功能？
│
├─ 网页播放 → MP4 (H.264) 或 WebM
│
├─ 本地存储，多音轨/字幕 → MKV
│
├─ 流媒体点播
│   ├─ DASH → fMP4
│   └─ HLS → TS 或 fMP4
│
├─ 直播推流
│   ├─ RTMP → FLV
│   └─ HLS/低延迟 → TS
│
├─ 专业剪辑 → MOV (ProRes) 或 MXF
│
└─ 最大兼容性 → MP4
```
