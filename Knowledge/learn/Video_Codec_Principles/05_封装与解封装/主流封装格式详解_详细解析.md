# 主流封装格式详解

> MP4、MKV、TS、FLV、MOV 五大封装格式的结构剖析与场景选型

---

## 核心结论（TL;DR）

**没有"最好"的封装格式，只有最适合场景的格式。** 封装格式的选择取决于你的具体需求：存储还是流媒体、兼容性要求、功能需求以及延迟要求。

关键要点：
1. **MP4** 是通用性最强的格式，适合点播和存储，但流媒体场景需要 faststart 处理
2. **MKV** 功能最丰富（多音轨、多字幕、章节），适合本地收藏和复杂媒体
3. **TS** 为广播设计，天然适合流媒体，但开销较大
4. **FLV** 结构简单、低延迟，是 RTMP 直播的标准格式
5. **MOV** 是 MP4 的"超集"，专业影视制作的首选

---

## 一、封装格式 MECE 分类总览

### 1.1 按数据组织方式分类

封装格式可以按照其数据组织方式进行互斥且完整（MECE）的分类：

```
封装格式数据组织方式
├── 树状层级结构（Box/Atom/EBML）
│   ├── ISO BMFF 系列：MP4、MOV、3GP、HEIF
│   └── EBML 系列：MKV、WebM
├── 固定长度包结构
│   └── MPEG-TS（188字节包）
└── 标签序列结构
    └── FLV（Tag 序列）
```

### 1.2 五大格式对比总表

| 格式 | 数据组织 | 基于标准 | 流式友好 | 编辑友好 | 编解码器支持 | 典型场景 |
|------|---------|---------|---------|---------|------------|---------|
| **MP4** | Box/Atom 树 | ISO BMFF | 中（需faststart） | 中 | H.264/H.265/AAC/AV1 | 点播、存储 |
| **MOV** | Box/Atom 树 | QuickTime | 中 | 高 | 同MP4+ProRes | 专业剪辑 |
| **MKV** | EBML 树 | Matroska | 低 | 中 | 几乎所有 | 存储、字幕 |
| **TS** | 固定包 | MPEG-2 | 高 | 低 | H.264/H.265/AAC/AC3 | 广播、HLS |
| **FLV** | 标签序列 | Adobe | 高 | 低 | H.264/H.265/AAC | RTMP直播 |

### 1.3 历史演进时间线

```
1991  QuickTime MOV 发布（Apple）
      │
1995  MPEG-TS 标准化（ISO 13818-1，广播电视）
      │
2001  ISO BMFF 标准化（ISO 14496-12）
      │
2002  MKV 1.0 发布（Matroska 项目）
      │
2003  FLV 随 Flash Player 6 发布（Macromedia）
      │
2003  MP4 标准化（ISO 14496-14）
      │
2010  WebM 发布（Google，基于 MKV）
      │
2016  fMP4 in HLS（Apple，iOS 10+）
      │
2018  CMAF 标准化（统一 DASH 和 HLS）
```

## 二、MP4 格式深度解析

### 2.1 历史与标准

MP4 是当今最广泛使用的视频封装格式，其标准体系如下：

| 标准 | 名称 | 描述 |
|------|------|------|
| **ISO 14496-12** | ISO Base Media File Format (ISOBMFF) | 定义了 Box 结构的基础框架 |
| **ISO 14496-14** | MP4 File Format | 基于 ISOBMFF 的 MP4 具体规范 |
| **ISO 14496-15** | Carriage of NAL unit structured video | 定义 AVC/HEVC 在 MP4 中的封装 |

**与 QuickTime MOV 的关系**：
- MOV 是 Apple 在 1991 年开发的 QuickTime 容器格式
- MP4 是 MOV 的标准化版本，可以视为 MOV 的"子集"
- 两者共享 Box/Atom 结构，但 MP4 剔除了 Apple 私有扩展

### 2.2 Box（Atom）基础结构

MP4 文件由嵌套的 Box（也称 Atom）组成。每个 Box 的基本结构：

```
┌─────────────────────────────────────────────────────────────┐
│                        Box 结构                              │
├─────────────┬─────────────┬─────────────────────────────────┤
│  Size (4B)  │  Type (4B)  │         Data (变长)              │
│  Box总大小   │  Box类型     │  Box内容（可包含子Box）          │
└─────────────┴─────────────┴─────────────────────────────────┘

特殊情况：
- Size = 1：使用 Extended Size（紧随 Type 后的 8 字节表示大小）
- Size = 0：Box 延伸到文件末尾
```

**Box 类型命名规则**：
- 4 个 ASCII 字符
- 全小写：标准 Box（如 `ftyp`, `moov`, `mdat`）
- 首字母大写：用户自定义扩展

### 2.3 MP4 核心 Box 层级结构

```
MP4 文件结构
│
├── ftyp              → 文件类型标识（必须在最前面）
│   ├── major_brand    → 主品牌（isom, mp41, mp42, dash...）
│   ├── minor_version  → 次版本
│   └── compatible_brands → 兼容品牌列表
│
├── moov              → 影片元数据容器（Movie Box）
│   │
│   ├── mvhd          → 影片头（Movie Header）
│   │   ├── creation_time   → 创建时间（1904年起的秒数）
│   │   ├── modification_time → 修改时间
│   │   ├── timescale       → 时间刻度（每秒多少单位）
│   │   ├── duration        → 时长（timescale 单位）
│   │   └── rate/volume     → 播放速率/音量
│   │
│   ├── trak          → 轨道容器（每个轨道一个）
│   │   │
│   │   ├── tkhd      → 轨道头（Track Header）
│   │   │   ├── track_id    → 轨道ID（从1开始）
│   │   │   ├── duration    → 轨道时长
│   │   │   ├── width/height → 视频宽高（视频轨道）
│   │   │   └── volume      → 音量（音频轨道）
│   │   │
│   │   ├── edts      → 编辑列表（可选）
│   │   │   └── elst  → 编辑列表条目
│   │   │
│   │   └── mdia      → 媒体容器
│   │       │
│   │       ├── mdhd  → 媒体头（Media Header）
│   │       │   ├── timescale → 媒体时间刻度（视频常为帧率的倍数）
│   │       │   ├── duration  → 媒体时长
│   │       │   └── language  → 语言代码
│   │       │
│   │       ├── hdlr  → 处理器引用（Handler Reference）
│   │       │   └── handler_type → vide/soun/subt/meta
│   │       │
│   │       └── minf  → 媒体信息（Media Information）
│   │           │
│   │           ├── vmhd/smhd/hmhd → 视频/音频/提示媒体头
│   │           │
│   │           ├── dinf  → 数据信息
│   │           │   └── dref → 数据引用（通常指向自身文件）
│   │           │
│   │           └── stbl  → 样本表（Sample Table，核心索引区）
│   │               │
│   │               ├── stsd → 样本描述（编码参数）
│   │               ├── stts → 解码时间-样本映射
│   │               ├── ctts → 合成时间偏移（B帧时需要）
│   │               ├── stss → 同步样本表（关键帧索引）
│   │               ├── stsc → 样本-Chunk映射
│   │               ├── stsz/stz2 → 样本大小表
│   │               └── stco/co64 → Chunk偏移表
│   │
│   └── udta          → 用户数据（可选，元数据）
│
├── mdat              → 媒体数据容器（实际音视频数据）
│
└── free/skip         → 填充 Box（可选，用于对齐或保留空间）
```

### 2.4 样本表（stbl）详解——MP4 索引的核心

样本表是 MP4 格式中最关键的部分，它建立了时间到文件位置的完整映射。

#### 2.4.1 stsd（Sample Description）

存储编解码器参数，是播放器解码的依据：

```
stsd 结构
├── entry_count: 描述条目数量（通常为1）
└── SampleEntry[]: 样本描述数组
    │
    ├── 视频：avc1/hvc1/av01...
    │   ├── width, height    → 分辨率
    │   ├── frame_count      → 每个样本的帧数（通常为1）
    │   └── avcC/hvcC/av1C   → 编码器配置
    │       ├── SPS (Sequence Parameter Set)
    │       ├── PPS (Picture Parameter Set)
    │       └── 其他编码参数
    │
    └── 音频：mp4a/ac-3/ec-3...
        ├── channelcount     → 声道数
        ├── samplesize       → 采样位深
        ├── samplerate       → 采样率
        └── esds/dac3/dec3   → 编码器特定配置
```

#### 2.4.2 stts（Decoding Time to Sample）

建立帧序号与解码时间的映射：

```
stts 结构（采用游程编码压缩）
├── entry_count: 条目数
└── entries[]:
    ├── sample_count: 连续相同duration的样本数
    └── sample_delta: 每个样本的duration（timescale单位）

示例（25fps视频，timescale=25000）：
┌───────────────┬───────────────┐
│ sample_count  │ sample_delta  │
├───────────────┼───────────────┤
│     1000      │     1000      │  ← 1000帧，每帧40ms
└───────────────┴───────────────┘

帧号 → DTS 计算：
帧0: DTS = 0
帧1: DTS = 1000
帧2: DTS = 2000
...
```

#### 2.4.3 ctts（Composition Time to Sample）

处理 B 帧时，PTS（显示时间）与 DTS（解码时间）不同：

```
ctts 结构
├── entry_count: 条目数
└── entries[]:
    ├── sample_count: 样本数
    └── sample_offset: PTS = DTS + offset

示例（含B帧的GOP: I B B P）：
┌───────────────┬───────────────┐
│ sample_count  │ sample_offset │
├───────────────┼───────────────┤
│       1       │     2000      │  ← I帧：PTS = DTS + 2000
│       2       │     0         │  ← B帧：PTS = DTS
│       1       │     2000      │  ← P帧：PTS = DTS + 2000
└───────────────┴───────────────┘
```

#### 2.4.4 stss（Sync Sample Table）

记录关键帧（I帧）的位置，Seek 操作的关键：

```
stss 结构
├── entry_count: 关键帧数量
└── sample_number[]: 关键帧的帧号（从1开始）

示例（每60帧一个关键帧）：
sample_number = [1, 61, 121, 181, ...]

注意：如果没有 stss，表示所有帧都是关键帧（如纯I帧视频）
```

#### 2.4.5 stsc（Sample to Chunk）

样本按 Chunk 分组存储，stsc 描述这种分组关系：

```
stsc 结构
├── entry_count: 条目数
└── entries[]:
    ├── first_chunk: 起始 chunk 号（从1开始）
    ├── samples_per_chunk: 每个 chunk 的样本数
    └── sample_description_index: 使用的 stsd 条目索引

示例：
┌─────────────┬───────────────────┬─────────────────────────┐
│ first_chunk │ samples_per_chunk │ sample_description_idx  │
├─────────────┼───────────────────┼─────────────────────────┤
│      1      │        10         │           1             │
│     101     │         5         │           1             │
└─────────────┴───────────────────┴─────────────────────────┘

解释：
- Chunk 1-100：每个 chunk 包含 10 个样本
- Chunk 101+：每个 chunk 包含 5 个样本
```

#### 2.4.6 stsz（Sample Size）

记录每个样本的大小（字节数）：

```
stsz 结构
├── sample_size: 固定大小（如果所有样本大小相同）
├── sample_count: 样本总数
└── entry_size[]: 每个样本的大小（sample_size=0时使用）

示例（变长样本）：
sample_size = 0
sample_count = 5
entry_size = [45320, 8921, 7845, 9012, 48723]
             ↑ I帧    ↑ B帧  ↑ B帧  ↑ B帧  ↑ P帧
```

#### 2.4.7 stco/co64（Chunk Offset）

记录每个 Chunk 在文件中的字节偏移：

```
stco 结构（32位偏移，最大支持4GB文件）
├── entry_count: chunk 数量
└── chunk_offset[]: 每个 chunk 的文件偏移

co64 结构（64位偏移，大文件）
├── entry_count: chunk 数量
└── chunk_offset[]: 每个 chunk 的文件偏移（64位）

示例：
chunk_offset = [48, 1048624, 2097200, ...]
               ↑ 第1个chunk在文件的第48字节位置
```

### 2.5 Seek 流程详解

当用户拖动进度条到目标时间点时，播放器执行以下步骤：

```
Seek 到目标时间 T 的流程：

1. 时间 → 帧号（使用 stts）
   ┌─────────────────────────────────────┐
   │ 目标时间 T = 10.5s                   │
   │ timescale = 1000                    │
   │ 目标 DTS = 10500                    │
   │                                     │
   │ 遍历 stts，累加 sample_delta        │
   │ 找到 DTS >= 10500 的第一个样本      │
   │ → 样本号 N = 263                    │
   └─────────────────────────────────────┘
              ↓
2. 找最近的关键帧（使用 stss）
   ┌─────────────────────────────────────┐
   │ 在 stss 中查找 <= N 的最大关键帧号   │
   │ stss = [1, 61, 121, 181, 241, ...]  │
   │ → 最近关键帧 K = 241                 │
   └─────────────────────────────────────┘
              ↓
3. 定位 Chunk（使用 stsc）
   ┌─────────────────────────────────────┐
   │ 根据样本号 K=241 和 stsc 规则       │
   │ 计算该样本所在的 chunk 号           │
   │ → Chunk C = 25                      │
   └─────────────────────────────────────┘
              ↓
4. 获取 Chunk 偏移（使用 stco）
   ┌─────────────────────────────────────┐
   │ 从 stco 获取 chunk 25 的文件偏移    │
   │ → offset = 12345678                 │
   └─────────────────────────────────────┘
              ↓
5. 计算样本在 Chunk 内的偏移（使用 stsz）
   ┌─────────────────────────────────────┐
   │ 累加该 chunk 内在目标样本之前的      │
   │ 所有样本大小                        │
   │ → 最终文件偏移 = 12345678 + 内部偏移 │
   └─────────────────────────────────────┘
              ↓
6. 定位并解码
   ┌─────────────────────────────────────┐
   │ fseek() 到计算出的文件位置          │
   │ 从关键帧 K=241 开始解码             │
   │ 解码到目标帧 N=263 后开始显示       │
   └─────────────────────────────────────┘
```

### 2.6 Fragmented MP4（fMP4）

#### 2.6.1 传统 MP4 的问题

```
传统 MP4 结构：
┌─────────────────────────────────────────┐
│ ftyp │ moov（完整索引）│ mdat（所有数据）│
└─────────────────────────────────────────┘

问题：
1. moov 必须包含完整索引 → 必须等录制结束才能写入
2. 录制中断 → moov 丢失 → 文件无法播放
3. 流媒体必须先下载 moov → 需要 faststart 处理
```

#### 2.6.2 fMP4 结构

```
fMP4 结构：
┌─────────────────────────────────────────────────────────────────┐
│ ftyp │ moov（初始化段）│ moof+mdat │ moof+mdat │ moof+mdat │...│
│      │   ↓            │  ↓         ↓           ↓              │
│      │ 仅编码参数     │ 片段1     片段2       片段3            │
│      │ 无样本索引     │ 每段独立  每段独立    每段独立          │
└─────────────────────────────────────────────────────────────────┘

moof（Movie Fragment）结构：
├── mfhd  → 片段序号
└── traf  → 轨道片段
    ├── tfhd → 轨道片段头（默认样本参数）
    ├── tfdt → 轨道片段解码时间（基础DTS）
    └── trun → 轨道运行（样本数组：大小、duration、偏移）
```

#### 2.6.3 fMP4 的应用场景

| 场景 | 说明 |
|------|------|
| **DASH** | Dynamic Adaptive Streaming over HTTP，原生使用 fMP4 |
| **HLS** | 自 iOS 10 起支持 fMP4（之前仅支持 TS） |
| **CMAF** | Common Media Application Format，统一 DASH 和 HLS 的 fMP4 规范 |
| **低延迟直播** | LL-HLS、LL-DASH 使用极小的 fMP4 片段（200-500ms） |

### 2.7 MP4 工具命令

```bash
# 查看 Box 结构
ffprobe -v error -show_format -show_streams input.mp4

# 详细查看 Box 层级
ffprobe -v error -show_entries format input.mp4 -of json
mp4dump input.mp4  # 需要安装 Bento4

# 查看特定 Box 内容
mp4dump --verbosity 3 input.mp4 | head -100

# 将 moov 移到文件头部（faststart）
ffmpeg -i input.mp4 -c copy -movflags +faststart output.mp4

# 生成 fMP4
ffmpeg -i input.mp4 -c copy -movflags +frag_keyframe+empty_moov output.mp4

# 查看关键帧位置
ffprobe -v error -select_streams v -show_entries frame=pict_type,pts_time \
  -of csv input.mp4 | grep "I"
```

### 2.8 MP4 的限制

| 限制 | 说明 |
|------|------|
| 编解码器支持 | 不支持 VP8/VP9（需要 WebM） |
| 录制安全性 | moov 在结尾时，录制中断导致文件损坏 |
| 编辑灵活性 | 修改需要重写索引，成本高 |
| 字幕支持 | 仅支持 tx3g 等有限格式 |
| 多字幕切换 | 支持但不如 MKV 方便 |

## 三、MOV 格式解析

### 3.1 与 MP4 的关系

```
关系图：

QuickTime MOV (1991, Apple)
       │
       ├─────────────────────┐
       │                     │
       ↓                     ↓
ISO BMFF (2001)        Apple 私有扩展
       │               (ProRes, Timecode, etc.)
       ↓
    MP4 (2003)

结论：MOV 是 MP4 的"超集"，MP4 是 MOV 的标准化子集
```

### 3.2 MOV 独有特性

#### 3.2.1 ProRes 编解码器

```
ProRes 系列（Apple 专业编解码器）：

┌────────────────┬────────────┬─────────────────┐
│ 编解码器        │ 比特率     │ 用途            │
├────────────────┼────────────┼─────────────────┤
│ ProRes 422 Proxy│ ~45 Mbps  │ 离线编辑        │
│ ProRes 422 LT   │ ~102 Mbps │ 轻量级制作      │
│ ProRes 422      │ ~147 Mbps │ 标准广播级      │
│ ProRes 422 HQ   │ ~220 Mbps │ 高质量广播      │
│ ProRes 4444     │ ~330 Mbps │ 带Alpha通道     │
│ ProRes 4444 XQ  │ ~500 Mbps │ 最高质量        │
│ ProRes RAW      │ 可变      │ RAW数据封装     │
└────────────────┴────────────┴─────────────────┘

特点：
- 帧内编码（All-I），每帧独立，编辑友好
- 支持 10-bit/12-bit 色深
- 仅 Apple 设备/软件支持编码
```

#### 3.2.2 时间码轨道（Timecode Track）

```
MOV 时间码结构：

trak (handler_type = 'tmcd')
└── mdia
    └── minf
        └── stbl
            └── stsd
                └── tmcd
                    ├── frame_duration
                    ├── number_of_frames
                    └── flags (drop_frame, etc.)

时间码格式：HH:MM:SS:FF（时:分:秒:帧）
- Drop Frame (DF): 29.97fps 视频使用，跳过特定帧号以同步实际时间
- Non-Drop Frame (NDF): 24fps, 25fps 使用
```

#### 3.2.3 Edit List（非破坏性编辑）

```
edts/elst 结构：

edts
└── elst
    └── entries[]
        ├── segment_duration: 在影片时间线上的持续时间
        ├── media_time: 媒体数据的起始时间（-1表示空白）
        └── media_rate: 播放速率（1.0正常，2.0两倍速）

使用场景：
┌────────────────────────────────────────────────────────────┐
│ 原始媒体：  |======A======|======B======|======C======|    │
│                                                            │
│ Edit List 定义播放顺序：                                    │
│   [B段, 正常速度] → [A段前半, 2倍速] → [1秒空白] → [C段]    │
│                                                            │
│ 实际播放：  |===B===|=A=|空白|======C======|               │
│                                                            │
│ 优点：不修改原始媒体数据，仅修改元数据                       │
└────────────────────────────────────────────────────────────┘
```

#### 3.2.4 丰富的元数据

```
MOV 支持的元数据类型（udta/meta Box）：

┌──────────────────┬─────────────────────────────────┐
│ 元数据类型        │ 描述                            │
├──────────────────┼─────────────────────────────────┤
│ 相机信息          │ 制造商、型号、镜头信息          │
│ GPS 位置          │ 拍摄地点经纬度                  │
│ 面部检测区域      │ 画面中人脸的位置和大小          │
│ 创作者信息        │ 导演、制片人、版权              │
│ 场景/镜头信息     │ 场号、镜次、拍摄日期            │
│ 色彩空间信息      │ 色域、传输特性、矩阵系数        │
│ HDR 元数据        │ 最大/平均亮度、色彩容量         │
└──────────────────┴─────────────────────────────────┘
```

### 3.3 MOV vs MP4 对比

| 特性 | MOV | MP4 |
|------|-----|-----|
| 标准组织 | Apple 私有 | ISO 标准 |
| ProRes 支持 | 原生支持 | 理论可行但罕见 |
| 时间码轨道 | 完整支持 | 有限支持 |
| Edit List | 完整支持 | 基本支持 |
| 元数据丰富度 | 非常丰富 | 标准化但有限 |
| 兼容性 | Apple 生态 | 通用 |
| 典型用途 | 专业制作 | 分发和存储 |

## 四、MKV（Matroska）格式解析

### 4.1 EBML 基础

MKV 使用 EBML（Extensible Binary Meta Language），一种类似 XML 的二进制格式：

```
EBML 元素结构：

┌──────────────────────────────────────────────────────────┐
│   Element ID   │   Data Size   │        Data            │
│   (变长1-4B)   │   (变长1-8B)   │      (变长)            │
└──────────────────────────────────────────────────────────┘

ID 和 Size 使用 VINT（Variable Integer）编码：
- 首字节的前导1位数决定字节数
- 1xxxxxxx = 1字节，最大值 127
- 01xxxxxx xxxxxxxx = 2字节，最大值 16383
- 以此类推...

示例：
ID = 0x1A45DFA3 (EBML Header)
  ├── 首字节 0x1A = 00011010
  ├── 前导0个数 = 3 → 4字节ID
  └── 完整ID = 0x1A45DFA3

优点：
- 自描述性强
- 节省空间（常用ID用短编码）
- 可扩展性好
```

### 4.2 MKV 核心结构

```
MKV 文件结构：

EBML Header (ID: 0x1A45DFA3)
├── EBMLVersion: 1
├── EBMLReadVersion: 1
├── DocType: "matroska" 或 "webm"
└── DocTypeVersion: 4

Segment (ID: 0x18538067)           → 主容器
│
├── SeekHead (ID: 0x114D9B74)      → 元素位置索引
│   └── Seek[]
│       ├── SeekID: 目标元素ID
│       └── SeekPosition: 相对Segment的偏移
│
├── Info (ID: 0x1549A966)          → 段信息
│   ├── TimestampScale: 纳秒/刻度（默认1000000=1ms）
│   ├── Duration: 时长（TimestampScale单位）
│   ├── MuxingApp: 封装程序
│   └── WritingApp: 写入程序
│
├── Tracks (ID: 0x1654AE6B)        → 轨道定义
│   └── TrackEntry[]
│       ├── TrackNumber: 轨道号
│       ├── TrackUID: 唯一ID
│       ├── TrackType: 1=视频, 2=音频, 17=字幕
│       ├── CodecID: "V_MPEG4/ISO/AVC", "A_AAC", etc.
│       ├── CodecPrivate: 编解码器配置数据
│       ├── Video (视频轨道专用)
│       │   ├── PixelWidth/PixelHeight
│       │   ├── DisplayWidth/DisplayHeight
│       │   └── Colour (色彩信息)
│       └── Audio (音频轨道专用)
│           ├── SamplingFrequency
│           ├── Channels
│           └── BitDepth
│
├── Chapters (ID: 0x1043A770)      → 章节信息
│   └── EditionEntry[]
│       └── ChapterAtom[]
│           ├── ChapterUID
│           ├── ChapterTimeStart/End
│           └── ChapterDisplay (标题、语言)
│
├── Tags (ID: 0x1254C367)          → 元数据标签
│   └── Tag[]
│       ├── Targets (作用范围)
│       └── SimpleTag[]
│           ├── TagName
│           └── TagString
│
├── Cues (ID: 0x1C53BB6B)          → Seek索引
│   └── CuePoint[]
│       ├── CueTime: 时间戳
│       └── CueTrackPositions[]
│           ├── CueTrack: 轨道号
│           └── CueClusterPosition: Cluster偏移
│
├── Attachments (ID: 0x1941A469)   → 附件
│   └── AttachedFile[]
│       ├── FileName
│       ├── FileMimeType
│       └── FileData
│
└── Cluster[] (ID: 0x1F43B675)     → 数据簇（实际媒体数据）
    ├── Timestamp: 簇基准时间
    └── SimpleBlock[] / BlockGroup[]
        ├── TrackNumber
        ├── Timestamp offset (相对簇基准)
        ├── Flags (keyframe, invisible, etc.)
        └── Frame data
```

#### 4.3.4 有序章节（Ordered Chapters）

### 4.5 MKV 的优缺点

## 五、MPEG-TS（Transport Stream）格式解析

### 5.1 设计哲学

MPEG-TS 为**不可靠传输信道**设计，核心特点：

```
设计目标：
┌─────────────────────────────────────────────────────────────┐
│ 1. 容错性：丢包后能快速恢复                                  │
│ 2. 随时调入：任意位置开始接收即可播放                        │
│ 3. 多节目复用：一个流承载多个节目（电视频道）                │
│ 4. 固定速率：适合广播传输                                   │
└─────────────────────────────────────────────────────────────┘

应用场景：
- 数字电视广播（DVB-T/S/C）
- ATSC（美国数字电视标准）
- HLS（HTTP Live Streaming）
- 蓝光光盘（M2TS）
```

### 5.2 固定 188 字节包结构

```
TS 包结构（188 字节）：

┌─────────────────────────────────────────────────────────────┐
│ Byte 0  │ Bytes 1-2        │ Byte 3     │ Bytes 4-187      │
├─────────┼──────────────────┼────────────┼──────────────────┤
│ Sync    │ PID + Flags      │ Continuity │ Payload          │
│ 0x47    │ 13-bit PID等     │ + Adapt    │ (最多184字节)    │
└─────────┴──────────────────┴────────────┴──────────────────┘

详细字段：

Byte 0: sync_byte = 0x47（同步字节，包起始标识）

Bytes 1-2 (16 bits):
├── transport_error_indicator (1 bit): 传输错误标志
├── payload_unit_start_indicator (1 bit): 负载单元起始
├── transport_priority (1 bit): 传输优先级
└── PID (13 bits): 包标识符（0x0000-0x1FFF）

Byte 3 (8 bits):
├── transport_scrambling_control (2 bits): 加扰控制
├── adaptation_field_control (2 bits): 适配域控制
│   ├── 00: 保留
│   ├── 01: 仅负载
│   ├── 10: 仅适配域
│   └── 11: 适配域 + 负载
└── continuity_counter (4 bits): 连续计数器（0-15循环）

Bytes 4+:
├── [Adaptation Field]（可选）
│   ├── length (1 byte)
│   ├── flags (1 byte): 包含PCR标志等
│   ├── PCR (6 bytes)（可选）
│   └── 填充字节
└── Payload（PES数据或PSI表）
```

### 5.3 为什么是 188 字节？

```
历史原因（ATM 适配）：

ATM Cell = 53 bytes (5 header + 48 payload)

188 = 4 × 47 = 4 × (48 - 1)

4个TS包正好映射到8个ATM信元：
┌───────────────────────────────────────────────────┐
│ TS包1(188B) │ TS包2(188B) │ TS包3(188B) │ TS包4(188B) │
│                    752 bytes                       │
│ = 16 × 47 = 16 × ATM payload(with 1-byte header)  │
│ ≈ 8 × ATM cells(with null padding)                │
└───────────────────────────────────────────────────┘

其他好处：
- 小包便于错误恢复（丢失一个包损失小）
- 固定大小简化硬件处理
- 便于计算比特率
```

### 5.4 PID（Packet Identifier）分配

```
PID 分配规则：

┌────────────┬─────────────────────────────────────┐
│ PID        │ 用途                                │
├────────────┼─────────────────────────────────────┤
│ 0x0000     │ PAT (Program Association Table)    │
│ 0x0001     │ CAT (Conditional Access Table)     │
│ 0x0002     │ TSDT (Transport Stream Description)│
│ 0x0010     │ NIT/ST (Network Information Table) │
│ 0x0011     │ SDT/BAT/ST (Service Description)   │
│ 0x0012     │ EIT/ST (Event Information Table)   │
│ 0x0013     │ RST/ST (Running Status Table)      │
│ 0x0014     │ TDT/TOT/ST (Time & Date Table)     │
│ 0x0020-1FFE│ 用户定义（节目PMT、音视频流）       │
│ 0x1FFF     │ Null packet（填充包）               │
└────────────┴─────────────────────────────────────┘
```

### 5.5 PSI（节目特定信息）

#### 5.5.1 PAT（Program Association Table）

```
PAT 结构（PID = 0x0000）：

┌─────────────────────────────────────────────────────────────┐
│ table_id: 0x00                                              │
│ section_length: 表长度                                      │
│ transport_stream_id: TS流标识                               │
│ version_number: 版本号（变化时+1）                          │
│ current_next_indicator: 当前/下一个                         │
│                                                             │
│ 节目列表：                                                  │
│ ┌───────────────────┬───────────────────┐                  │
│ │ program_number    │ PID               │                  │
│ ├───────────────────┼───────────────────┤                  │
│ │ 0                 │ 0x0010 (NIT)      │                  │
│ │ 1                 │ 0x0100 (PMT for program 1) │         │
│ │ 2                 │ 0x0200 (PMT for program 2) │         │
│ └───────────────────┴───────────────────┘                  │
│                                                             │
│ CRC_32: 校验                                                │
└─────────────────────────────────────────────────────────────┘
```

#### 5.5.2 PMT（Program Map Table）

```
PMT 结构（PID 由 PAT 指定）：

┌─────────────────────────────────────────────────────────────┐
│ table_id: 0x02                                              │
│ section_length: 表长度                                      │
│ program_number: 节目号                                      │
│ PCR_PID: 携带PCR的包的PID                                   │
│                                                             │
│ 流列表：                                                    │
│ ┌───────────────┬─────────┬─────────────────────┐          │
│ │ stream_type   │ PID     │ 描述                │          │
│ ├───────────────┼─────────┼─────────────────────┤          │
│ │ 0x1B          │ 0x0101  │ H.264 视频          │          │
│ │ 0x24          │ 0x0101  │ H.265 视频          │          │
│ │ 0x0F          │ 0x0102  │ AAC 音频            │          │
│ │ 0x81          │ 0x0103  │ AC-3 音频           │          │
│ │ 0x06          │ 0x0104  │ 字幕                │          │
│ └───────────────┴─────────┴─────────────────────┘          │
│                                                             │
│ CRC_32: 校验                                                │
└─────────────────────────────────────────────────────────────┘

stream_type 常见值：
┌────────────┬─────────────────────────────────────┐
│ stream_type│ 含义                                │
├────────────┼─────────────────────────────────────┤
│ 0x01       │ MPEG-1 Video                        │
│ 0x02       │ MPEG-2 Video                        │
│ 0x03       │ MPEG-1 Audio                        │
│ 0x04       │ MPEG-2 Audio                        │
│ 0x0F       │ AAC Audio                           │
│ 0x1B       │ H.264/AVC Video                     │
│ 0x24       │ H.265/HEVC Video                    │
│ 0x81       │ AC-3 Audio                          │
│ 0x87       │ E-AC-3 Audio                        │
└────────────┴─────────────────────────────────────┘
```

### 5.6 PES（Packetized Elementary Stream）

```
PES 包结构：

┌─────────────────────────────────────────────────────────────┐
│ packet_start_code_prefix: 0x000001 (3 bytes)               │
│ stream_id: 流类型标识 (1 byte)                             │
│   - 0xE0-0xEF: 视频流                                      │
│   - 0xC0-0xDF: 音频流                                      │
│ PES_packet_length: 包长度 (2 bytes，0表示未知)             │
│                                                             │
│ [可选字段]                                                  │
│ ├── PTS_DTS_flags: 是否包含PTS/DTS                         │
│ ├── PTS (5 bytes): 显示时间戳 (90kHz)                      │
│ └── DTS (5 bytes): 解码时间戳 (90kHz)                      │
│                                                             │
│ PES_packet_data: 编码帧数据                                │
└─────────────────────────────────────────────────────────────┘

PES 切分为 TS 包：

PES包（可能数KB）
│
├──→ TS包1 (payload_unit_start = 1, 包含PES头)
├──→ TS包2 (payload_unit_start = 0)
├──→ TS包3 (payload_unit_start = 0)
└──→ TS包N (最后一包可能需要填充)
```

### 5.7 PCR（Program Clock Reference）

```
PCR 用途：

发送端                           接收端
┌──────────┐                    ┌──────────┐
│ 27MHz    │  ──TS流──→        │ PLL锁相  │
│ 系统时钟  │                    │ 恢复时钟  │
└──────────┘                    └──────────┘
    ↓                               ↑
  生成PCR ──────────────────────── 读取PCR

PCR 结构（6 字节，48 bits）：
┌──────────────────────────────────────────────────────────┐
│ PCR_base (33 bits)  │ Reserved (6 bits) │ PCR_ext (9 bits)│
├─────────────────────┴───────────────────┴─────────────────┤
│ PCR = PCR_base × 300 + PCR_ext                            │
│ 单位：27MHz 时钟周期                                       │
│ PCR_base 对应 90kHz（与 PTS/DTS 相同）                     │
└──────────────────────────────────────────────────────────┘

PCR 插入频率：
- 标准要求：至少每 100ms 一次
- 实际常见：每 40ms 一次
```

### 5.8 TS 在现代流媒体中的角色

#### 5.8.1 HLS（HTTP Live Streaming）

```
HLS 架构：

媒体服务器
│
├── playlist.m3u8（播放列表）
│   #EXTM3U
│   #EXT-X-VERSION:3
│   #EXT-X-TARGETDURATION:10
│   #EXT-X-MEDIA-SEQUENCE:0
│   #EXTINF:9.009,
│   segment0.ts
│   #EXTINF:9.009,
│   segment1.ts
│   ...
│
├── segment0.ts（TS 切片，约 6-10 秒）
├── segment1.ts
└── segment2.ts

播放流程：
1. 客户端下载 m3u8
2. 解析获取 TS 切片 URL
3. 下载 TS 切片并播放
4. 定期刷新 m3u8 获取新切片
```

#### 5.8.2 TS 正在被 fMP4 替代

```
HLS 格式演进：

iOS 10 之前：仅支持 TS
iOS 10+：支持 fMP4

fMP4 相比 TS 的优势：
┌─────────────────────┬────────────────┬────────────────┐
│ 特性                │ TS             │ fMP4           │
├─────────────────────┼────────────────┼────────────────┤
│ 头部开销            │ ~5%            │ ~1%            │
│ 加密粒度            │ 188字节包      │ 样本级         │
│ DRM 兼容            │ 需要适配       │ 原生支持       │
│ CMAF 兼容           │ 否             │ 是             │
│ 低延迟支持          │ 较难           │ 原生支持       │
└─────────────────────┴────────────────┴────────────────┘

趋势：新项目推荐使用 fMP4
```

### 5.9 TS 优缺点

| 优点 | 缺点 |
|------|------|
| 天然流式友好 | 头部开销大（~3-5%） |
| 容错性强（丢包可恢复） | 索引粗糙（Seek 不精确） |
| 随时可调入（PSI 重复） | 编辑困难 |
| 多节目复用 | 不支持多字幕 |
| 广播设备广泛支持 | 现代场景逐渐被 fMP4 取代 |

### 5.10 TS 工具命令

```bash
# 查看 TS 结构
ffprobe -v error -show_format -show_streams input.ts

# 分析 TS 包
tsanalyze input.ts  # 需要 TSDuck

# 查看 PAT/PMT
ffprobe -v error -show_programs input.ts

# 提取特定 PID
ffmpeg -i input.ts -map 0:v -c copy video.ts

# TS 转 MP4
ffmpeg -i input.ts -c copy output.mp4

# MP4 转 TS（HLS 准备）
ffmpeg -i input.mp4 -c copy -bsf:v h264_mp4toannexb output.ts

# HLS 切片
ffmpeg -i input.mp4 -c copy -f hls -hls_time 10 \
  -hls_list_size 0 playlist.m3u8

# 查看 PCR
ffprobe -v error -select_streams v -show_entries packet=pts_time,dts_time \
  -of csv input.ts | head
```

## 六、FLV（Flash Video）格式解析

### 6.1 历史背景

```
FLV 时间线：

2002  Macromedia 发布 Flash Player 6，引入 FLV
2005  Flash Video 成为网络视频主流（YouTube 采用）
2007  Adobe 发布 RTMP 协议规范
2010  HTML5 <video> 开始挑战 Flash
2017  Adobe 宣布停止 Flash 开发
2020  Flash Player 正式终止

FLV 的遗产：
- RTMP 协议至今仍是直播推流的主流
- FLV 结构简单，低延迟，直播友好
- 国内直播平台广泛使用（扩展支持 H.265）
```

### 6.2 FLV 文件结构

```
FLV 文件结构：

┌─────────────────────────────────────────────────────────────┐
│                    FLV Header (9 bytes)                     │
├─────────────────────────────────────────────────────────────┤
│ Signature (3B)  │ Version (1B) │ Flags (1B) │ DataOffset(4B)│
│ "FLV"           │ 0x01         │ 0x05       │ 0x00000009    │
│                 │              │ bit0=audio │ (Header大小)  │
│                 │              │ bit2=video │               │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                 Previous Tag Size 0 (4 bytes)               │
│                        = 0x00000000                         │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                         Tag 1                               │
├──────────┬────────────────────────────────────┬─────────────┤
│ Tag Type │ Data Size │ Timestamp │ StreamID   │   Tag Data  │
│ (1B)     │ (3B)      │ (3B+1B)   │ (3B)       │   (变长)    │
└──────────┴────────────────────────────────────┴─────────────┘

┌─────────────────────────────────────────────────────────────┐
│                 Previous Tag Size 1 (4 bytes)               │
│                   = Tag1 Header(11B) + Tag1 Data            │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                         Tag 2                               │
└─────────────────────────────────────────────────────────────┘
                            ...
```

### 6.3 Tag 类型详解

```
Tag 类型：

┌───────────┬────────────────────────────────────────────────┐
│ Tag Type  │ 描述                                           │
├───────────┼────────────────────────────────────────────────┤
│ 8         │ Audio Tag                                      │
│ 9         │ Video Tag                                      │
│ 18        │ Script Data Tag (onMetaData)                   │
└───────────┴────────────────────────────────────────────────┘
```

#### 6.3.1 Video Tag 结构

```
Video Tag Data：

┌─────────────────────────────────────────────────────────────┐
│ Byte 0 (Video Tag Header)                                   │
├───────────────────────────────────┬─────────────────────────┤
│ Frame Type (4 bits)               │ CodecID (4 bits)        │
│ 1=keyframe, 2=inter frame         │ 7=AVC(H.264)            │
│ 3=disposable inter, 4=generated   │ 12=HEVC(扩展)           │
│ 5=video info/command              │ 13=AV1(扩展)            │
└───────────────────────────────────┴─────────────────────────┘

AVC (H.264) 特有字段：
┌─────────────────────────────────────────────────────────────┐
│ AVCPacketType (1 byte)                                      │
│ 0=AVC sequence header (SPS/PPS)                             │
│ 1=AVC NALU                                                  │
│ 2=AVC end of sequence                                       │
├─────────────────────────────────────────────────────────────┤
│ CompositionTime (3 bytes, signed)                           │
│ CTS = PTS - DTS (用于B帧)                                   │
├─────────────────────────────────────────────────────────────┤
│ Video Data                                                  │
│ - AVCPacketType=0: AVCDecoderConfigurationRecord            │
│ - AVCPacketType=1: NAL units (length-prefixed)              │
└─────────────────────────────────────────────────────────────┘
```

#### 6.3.2 Audio Tag 结构

```
Audio Tag Data：

┌─────────────────────────────────────────────────────────────┐
│ Byte 0 (Audio Tag Header)                                   │
├─────────────┬────────────┬────────────────┬─────────────────┤
│ SoundFormat │ SoundRate  │ SoundSize      │ SoundType       │
│ (4 bits)    │ (2 bits)   │ (1 bit)        │ (1 bit)         │
│ 10=AAC      │ 0=5.5kHz   │ 0=8-bit        │ 0=Mono          │
│ 2=MP3       │ 1=11kHz    │ 1=16-bit       │ 1=Stereo        │
│             │ 2=22kHz    │                │                 │
│             │ 3=44kHz    │                │                 │
└─────────────┴────────────┴────────────────┴─────────────────┘

AAC 特有字段：
┌─────────────────────────────────────────────────────────────┐
│ AACPacketType (1 byte)                                      │
│ 0=AAC sequence header (AudioSpecificConfig)                 │
│ 1=AAC raw frame                                             │
├─────────────────────────────────────────────────────────────┤
│ Audio Data                                                  │
└─────────────────────────────────────────────────────────────┘
```

#### 6.3.3 Script Tag（onMetaData）

```
Script Tag 结构：

┌─────────────────────────────────────────────────────────────┐
│ AMF (Action Message Format) 编码                            │
├─────────────────────────────────────────────────────────────┤
│ AMF String: "onMetaData"                                    │
├─────────────────────────────────────────────────────────────┤
│ AMF Object/ECMAArray:                                       │
│ {                                                           │
│   "duration": 120.5,        // 时长（秒）                   │
│   "width": 1920,            // 视频宽度                     │
│   "height": 1080,           // 视频高度                     │
│   "videodatarate": 2500,    // 视频码率（kbps）             │
│   "framerate": 30,          // 帧率                         │
│   "videocodecid": 7,        // 视频编解码器（7=AVC）        │
│   "audiodatarate": 128,     // 音频码率（kbps）             │
│   "audiosamplerate": 44100, // 音频采样率                   │
│   "audiosamplesize": 16,    // 音频位深                     │
│   "stereo": true,           // 是否立体声                   │
│   "audiocodecid": 10,       // 音频编解码器（10=AAC）       │
│   "filesize": 38572846,     // 文件大小                     │
│   "keyframes": {            // 关键帧索引（可选，重要）     │
│     "times": [0, 2.0, 4.0, ...],                            │
│     "filepositions": [1234, 56789, 112345, ...]             │
│   }                                                         │
│ }                                                           │
└─────────────────────────────────────────────────────────────┘
```

### 6.4 FLV 的 Keyframe 索引

```
FLV Seek 机制：

有 keyframes 索引：
┌─────────────────────────────────────────────────────────────┐
│ 目标时间 T = 10.5s                                          │
│                                                             │
│ 在 onMetaData.keyframes.times 中二分查找 <= T 的最大值      │
│ → 找到 times[5] = 10.0s                                     │
│                                                             │
│ 对应 filepositions[5] = 12345678                            │
│                                                             │
│ fseek(12345678) 并从该关键帧开始解码                        │
└─────────────────────────────────────────────────────────────┘

无 keyframes 索引：
┌─────────────────────────────────────────────────────────────┐
│ 只能顺序扫描 Tag，效率极低                                  │
│ 或者使用 Previous Tag Size 从文件末尾往前扫描               │
│                                                             │
│ 许多播放器在这种情况下禁用 Seek                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.5 Enhanced FLV

为了支持 H.265 和 AV1，国内厂商推动了 FLV 扩展：

```
Enhanced FLV 扩展（非官方标准）：

视频编解码器扩展：
┌───────────────┬────────────────────────────────┐
│ CodecID       │ 编解码器                        │
├───────────────┼────────────────────────────────┤
│ 7             │ H.264/AVC（原有）              │
│ 12            │ H.265/HEVC（扩展）             │
│ 13            │ AV1（扩展）                    │
│ 14            │ VP9（部分实现）                │
└───────────────┴────────────────────────────────┘

HEVC 封装（类似 AVC）：
┌─────────────────────────────────────────────────────────────┐
│ Frame Type (4 bits) │ CodecID=12 (4 bits)                   │
├─────────────────────────────────────────────────────────────┤
│ HEVCPacketType (1 byte)                                     │
│ 0=HEVC sequence header (HEVCDecoderConfigurationRecord)     │
│ 1=HEVC NALU                                                 │
├─────────────────────────────────────────────────────────────┤
│ CompositionTime (3 bytes)                                   │
├─────────────────────────────────────────────────────────────┤
│ Video Data (NAL units)                                      │
└─────────────────────────────────────────────────────────────┘

支持厂商：
- 金山云
- 腾讯云
- 阿里云
- B站（开源 FLV.js 支持 HEVC）
```

### 6.6 FLV 在直播中的地位

```
RTMP/FLV 直播架构：

推流端                    服务器                    播放端
┌──────────┐            ┌──────────┐            ┌──────────┐
│ 编码器   │   RTMP    │ 直播CDN  │   HTTP    │ 播放器   │
│ (OBS等)  │ ───────→  │          │ ───────→  │          │
│          │   FLV     │          │   FLV     │          │
└──────────┘            └──────────┘            └──────────┘

RTMP 推流：FLV 数据通过 RTMP 协议传输
HTTP-FLV 拉流：FLV 数据通过 HTTP 持续传输

优点：
- 延迟低（1-3秒）
- 协议简单
- 广泛支持

正在被替代：
- WebRTC：更低延迟（<1秒）
- LL-HLS：HTTP 友好，支持 CDN 缓存
- SRT：可靠 UDP 传输
```

### 6.7 FLV 的局限

| 局限 | 说明 |
|------|------|
| 不支持多音轨 | 只能有一个音频流 |
| 字幕支持有限 | 依赖 onTextData 事件，兼容性差 |
| H.265 需要扩展 | 非标准扩展，兼容性问题 |
| 不支持 4GB+ 文件 | Data Size 字段仅 3 字节 |
| Seek 依赖 onMetaData | 缺失时无法精确 Seek |

### 6.8 FLV 工具命令

```bash
# 查看 FLV 结构
ffprobe -v error -show_format -show_streams input.flv

# 查看 onMetaData
ffprobe -v error -show_entries format_tags input.flv

# 注入 keyframes 索引
flvmeta -U input.flv
yamdi -i input.flv -o output.flv

# FLV 转 MP4
ffmpeg -i input.flv -c copy output.mp4

# MP4 转 FLV
ffmpeg -i input.mp4 -c copy -f flv output.flv

# RTMP 推流
ffmpeg -re -i input.mp4 -c copy -f flv rtmp://server/live/stream

# HTTP-FLV 拉流
ffplay http://server/live/stream.flv
```

## 九、常见问题与最佳实践

### 9.1 MP4 录制中断后如何恢复？

```
问题：传统 MP4 的 moov 在文件末尾，录制中断导致文件无法播放

解决方案：

1. 使用 fMP4 格式录制
   ffmpeg -i input -c copy -movflags +frag_keyframe+empty_moov output.mp4

2. 定期刷新 moov（ffmpeg 的 -movflags faststart）
   需要定时重新封装

3. 使用 untrunc 等工具修复
   untrunc good.mp4 broken.mp4

4. 使用更安全的格式（MKV/TS）进行录制
```

### 9.2 如何让 MP4 支持流式播放？

```
问题：moov 在末尾时，必须下载完整文件才能播放

解决方案：

1. faststart 处理（将 moov 移到 ftyp 后面）
   ffmpeg -i input.mp4 -c copy -movflags +faststart output.mp4

2. 使用 fMP4 格式
   适用于流媒体场景

3. 服务器端支持 Range 请求
   允许客户端先请求文件末尾的 moov
```

### 9.3 MKV 为什么在某些设备上不支持？

```
原因：
- MKV 是社区开发的格式，不是国际标准
- 硬件解码器通常不直接支持 MKV 容器
- 需要软件解封装后再硬件解码

解决方案：
- 使用软件播放器（VLC、PotPlayer）
- 转封装为 MP4（不重新编码）
  ffmpeg -i input.mkv -c copy output.mp4
- 仅存储时使用 MKV，分发时转为 MP4
```

### 9.4 直播选择 FLV 还是 HLS？

```
对比：

┌─────────────┬─────────────────┬─────────────────┐
│ 指标        │ FLV (HTTP-FLV)  │ HLS (TS/fMP4)   │
├─────────────┼─────────────────┼─────────────────┤
│ 延迟        │ 1-3秒           │ 6-30秒          │
│ CDN缓存     │ 不友好          │ 友好            │
│ 兼容性      │ 需flv.js        │ 原生支持iOS     │
│ 自适应码率  │ 困难            │ 原生支持        │
│ DRM         │ 不支持          │ 支持            │
└─────────────┴─────────────────┴─────────────────┘

建议：
- 追求低延迟：FLV + WebSocket/HTTP长连接
- 追求兼容性：HLS
- 平衡方案：LL-HLS (fMP4, 延迟2-4秒)
```
