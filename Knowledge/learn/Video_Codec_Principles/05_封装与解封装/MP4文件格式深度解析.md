# MP4文件格式深度解析

> 从ISO BMFF标准到底层Box结构，全面剖析MP4封装格式的内部机制

---

## 核心结论（TL;DR）

**MP4（MPEG-4 Part 14）是基于ISO Base Media File Format（ISOBMFF）的通用封装格式，采用树状Box结构组织数据，通过样本表（stbl）建立时间到文件位置的完整映射，支持多流同步、随机访问和流式播放。**

关键要点：
1. **Box/Atom是MP4的基本单元**：每个Box由Size（4字节）、Type（4字节）和Data（变长）组成，形成树状层级结构
2. **moov是核心元数据容器**：包含轨道定义、编码参数和样本索引，决定文件的可播放性
3. **stbl样本表是Seek的关键**：stts/stss/stsc/stsz/stco协同工作，实现时间到字节偏移的精确映射
4. **fMP4解决流式痛点**：将传统MP4的集中式索引拆分为片段式结构，支持边录边播和自适应流媒体
5. **moov位置决定流式能力**：前置moov支持faststart，后置moov需完整下载才能播放

## 一、MP4的历史背景与标准体系

### 1.1 历史演进脉络

```
1991  QuickTime MOV 发布（Apple）
       │
       ├── 引入Box/Atom结构
       ├── 支持多轨道媒体
       └── 时间戳同步机制
       │
2001  ISO BMFF 标准化（ISO 14496-12）
       │
       ├── 基于MOV的通用化改造
       ├── 去除Apple私有扩展
       └── 定义基础Box框架
       │
2003  MP4 标准化（ISO 14496-14）
       │
       ├── 基于ISOBMFF的MP4具体规范
       ├── 定义音视频轨道封装
       └── 成为网络视频主流格式
       │
2016  fMP4 in HLS（Apple iOS 10+）
       │
       ├── 支持Fragmented MP4
       └── 统一DASH和HLS的容器格式
       │
2018  CMAF 标准化
       └── Common Media Application Format
```

### 1.2 标准文档体系

| 标准编号 | 全称 | 核心内容 |
|----------|------|----------|
| **ISO 14496-12** | ISO Base Media File Format (ISOBMFF) | Box结构基础框架、文件组织原则、时间模型 |
| **ISO 14496-14** | MP4 File Format | 基于ISOBMFF的MP4具体实现、音视频轨道定义 |
| **ISO 14496-15** | Carriage of NAL unit structured video | AVC/HEVC在MP4中的封装规范、NAL单元映射 |

### 1.3 MP4与QuickTime MOV的关系

```
┌─────────────────────────────────────────────────────────────┐
│                    QuickTime MOV (1991)                     │
│                    Apple 专有格式                            │
│                                                             │
│  ┌─────────────────────┐    ┌─────────────────────────────┐ │
│  │   ISO BMFF 标准     │    │   Apple 私有扩展             │ │
│  │   (2001, 标准化)    │    │   (ProRes, Timecode等)       │ │
│  │                     │    │                             │ │
│  │  • 核心Box结构       │    │  • 专业编解码器支持          │ │
│  │  • 基础轨道模型      │    │  • 时间码轨道               │ │
│  │  • 标准元数据        │    │  • 编辑列表(非破坏性)        │ │
│  │  • 通用兼容性        │    │  • 丰富元数据               │ │
│  └──────────┬──────────┘    └─────────────────────────────┘ │
│             │                                               │
│             ↓                                               │
│      MP4 (2003)                                             │
│      通用封装格式                                            │
└─────────────────────────────────────────────────────────────┘
```

**关键区别**：
- MOV是Apple专有格式，功能更丰富但兼容性受限
- MP4是国际标准，去除私有扩展，通用性最强
- 两者共享相同的Box/Atom基础结构，工具通常可互操作

## 二、Box/Atom的基本结构与层级关系

### 2.1 Box的基本结构

MP4文件由**Box**（也称Atom）组成，每个Box遵循统一格式：

```
┌─────────────────────────────────────────────────────────────┐
│                      Box 基本结构                            │
├─────────────┬─────────────┬─────────────────────────────────┤
│  Size (4B)  │  Type (4B)  │         Data (变长)              │
│  Box总大小   │  Box类型     │  Box内容（可包含子Box）          │
├─────────────┴─────────────┴─────────────────────────────────┤
│  32位无符号整数 │ 4字符ASCII │  数据内容或嵌套Box              │
│  包含自身大小   │ 如"moov"  │  长度 = Size - 8               │
└─────────────────────────────────────────────────────────────┘

特殊情况处理：
┌─────────────────────────────────────────────────────────────┐
│ Size = 0: Box延伸到文件末尾（仅用于mdat等大数据Box）          │
│ Size = 1: 使用Extended Size（紧随Type后的8字节表示大小）      │
│           用于超过4GB的Box                                    │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Box类型命名规则

| 命名模式 | 示例 | 含义 |
|----------|------|------|
| 全小写 | `ftyp`, `moov`, `mdat` | 标准Box，ISO定义 |
| 首字母大写 | `Uuid`, `Meta` | 用户自定义扩展 |
| 特定前缀 | `avc1`, `mp4a` | 编解码器特定Box |

### 2.3 MP4文件的Box层级结构

```
MP4文件结构（树状层级）
│
├── ftyp (File Type Box)                    【文件类型标识】
│   ├── major_brand: "isom", "mp41", "mp42", "dash"
│   ├── minor_version: 版本号
│   └── compatible_brands[]: 兼容品牌列表
│
├── moov (Movie Box)                        【影片元数据容器 - 核心】
│   │
│   ├── mvhd (Movie Header Box)             【影片头】
│   │   ├── creation_time: 创建时间（1904年起的秒数）
│   │   ├── modification_time: 修改时间
│   │   ├── timescale: 影片时间刻度（每秒单位数）
│   │   ├── duration: 影片时长（timescale单位）
│   │   ├── rate: 播放速率（默认0x00010000 = 1.0）
│   │   ├── volume: 音量（默认0x0100 = 1.0）
│   │   └── next_track_id: 下一个轨道ID
│   │
│   ├── trak (Track Box) × N                【轨道容器】
│   │   │
│   │   ├── tkhd (Track Header Box)         【轨道头】
│   │   │   ├── track_id: 轨道唯一ID（从1开始）
│   │   │   ├── duration: 轨道时长
│   │   │   ├── width/height: 视频宽高（16.16定点数）
│   │   │   ├── volume: 音频音量
│   │   │   └── flags: 轨道标志（启用、在电影中、预览）
│   │   │
│   │   ├── edts (Edit Box) [可选]          【编辑列表】
│   │   │   └── elst (Edit List Box)        【编辑条目】
│   │   │       ├── segment_duration: 段持续时间
│   │   │       ├── media_time: 媒体起始时间
│   │   │       └── media_rate: 播放速率
│   │   │
│   │   └── mdia (Media Box)                【媒体容器】
│   │       │
│   │       ├── mdhd (Media Header Box)     【媒体头】
│   │       │   ├── timescale: 媒体时间刻度（视频常为帧率倍数）
│   │       │   ├── duration: 媒体时长
│   │       │   └── language: 语言代码（3字符压缩）
│   │       │
│   │       ├── hdlr (Handler Reference Box)【处理器引用】
│   │       │   └── handler_type: "vide"(视频)/"soun"(音频)/"subt"(字幕)
│   │       │
│   │       └── minf (Media Information Box)【媒体信息】
│   │           │
│   │           ├── vmhd/smhd/hmhd          【视频/音频/提示头】
│   │           │
│   │           ├── dinf (Data Information Box)【数据信息】
│   │           │   └── dref (Data Reference Box)【数据引用】
│   │           │       └── url: 数据位置（通常是self-contained）
│   │           │
│   │           └── stbl (Sample Table Box) 【样本表 - 核心索引区】
│   │               │
│   │               ├── stsd (Sample Description Box) 【样本描述】
│   │               ├── stts (Decoding Time to Sample Box) 【解码时间映射】
│   │               ├── ctts (Composition Time to Sample Box) 【合成时间偏移】
│   │               ├── stss (Sync Sample Box) 【同步样本（关键帧）】
│   │               ├── stsc (Sample to Chunk Box) 【样本到Chunk映射】
│   │               ├── stsz (Sample Size Box) 【样本大小】
│   │               └── stco/co64 (Chunk Offset Box) 【Chunk偏移】
│   │
│   └── udta (User Data Box) [可选]         【用户数据/元数据】
│       └── meta: 标题、作者、版权等信息
│
├── mdat (Media Data Box)                   【媒体数据容器】
│   └── 实际的音视频编码数据（H.264 NAL单元、AAC帧等）
│
└── free/skip (Free Space Box) [可选]       【填充Box】
    └── 用于对齐或预留空间
```

### 2.4 文件布局的两种模式

```
传统MP4布局（moov在末尾）：
┌─────────────────────────────────────────────────────────────┐
│  ftyp  │        mdat (所有媒体数据)        │      moov      │
│  (小)  │        (可能数GB)                 │    (索引)      │
└─────────────────────────────────────────────────────────────┘
问题：必须下载完整文件才能读取moov，无法边下边播

Faststart MP4布局（moov在前）：
┌─────────────────────────────────────────────────────────────┐
│  ftyp  │      moov (索引)      │        mdat (媒体数据)      │
│  (小)  │      (相对小)          │        (可能数GB)           │
└─────────────────────────────────────────────────────────────┘
优势：先下载moov获取索引，可立即开始播放，支持Seek
```

## 三、核心Box详细解析

### 3.1 ftyp（File Type Box）

**作用**：标识文件类型和兼容性，必须位于文件开头。

```
ftyp Box结构：
┌─────────────────────────────────────────────────────────────┐
│ Box Header (8 bytes)                                        │
│ ├── size: 4 bytes (整个ftyp的大小)                          │
│ └── type: "ftyp" (4 bytes)                                  │
├─────────────────────────────────────────────────────────────┤
│ major_brand (4 bytes): 主品牌标识                            │
│   - "isom": ISO Base Media                                  │
│   - "mp41": MP4 version 1                                   │
│   - "mp42": MP4 version 2 (支持B帧)                         │
│   - "dash": DASH流媒体                                       │
│   - "avc1": AVC基础                                          │
│                                                             │
│ minor_version (4 bytes): 次版本号                            │
│                                                             │
│ compatible_brands[] (4 bytes each): 兼容品牌列表             │
│   - 表示播放器应支持哪些品牌特性                             │
│   - 通常包含主品牌和基础品牌如"isom"、"mp41"                │
└─────────────────────────────────────────────────────────────┘

示例：
major_brand = "mp42"
minor_version = 0
compatible_brands = ["isom", "mp42", "avc1"]
含义：这是一个MP4 v2文件，兼容ISOBMFF基础、MP4 v2和AVC视频
```

### 3.2 moov（Movie Box）

**作用**：影片级元数据容器，包含所有轨道的定义和索引信息。

**关键特性**：
- 一个文件只有一个moov
- 包含mvhd（影片头）和多个trak（轨道）
- 大小取决于视频长度和索引粒度（通常占文件0.1%-1%）

### 3.3 mvhd（Movie Header Box）

```
mvhd Box结构（Full Box，含version和flags）：
┌─────────────────────────────────────────────────────────────┐
│ version (1 byte): 通常为0或1                                │
│ flags (3 bytes): 通常为0                                    │
├─────────────────────────────────────────────────────────────┤
│ version 0 字段：                                             │
│ ├── creation_time (4 bytes): 创建时间（秒，从1904-01-01起）  │
│ ├── modification_time (4 bytes): 修改时间                    │
│ ├── timescale (4 bytes): 影片时间刻度（每秒单位数）          │
│ │   常见值：1000（毫秒）、90000（视频常用）、48000（音频）    │
│ ├── duration (4 bytes): 影片时长（timescale单位）            │
│ │   例：timescale=1000, duration=120000 → 120秒              │
│ ├── rate (4 bytes): 播放速率（16.16定点数，0x00010000=1.0）  │
│ ├── volume (2 bytes): 音量（8.8定点数，0x0100=1.0）          │
│ ├── reserved (10 bytes)                                     │
│ ├── matrix (36 bytes): 视频变换矩阵（默认单位矩阵）          │
│ ├── pre_defined (24 bytes)                                  │
│ └── next_track_id (4 bytes): 下一个可用轨道ID                │
├─────────────────────────────────────────────────────────────┤
│ version 1 字段（支持64位时间）：                             │
│ ├── creation_time (8 bytes)                                 │
│ ├── modification_time (8 bytes)                             │
│ ├── timescale (4 bytes)                                     │
│ └── duration (8 bytes)                                      │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 trak（Track Box）

**作用**：定义单个媒体轨道（视频、音频、字幕等）。

**子Box关系**：
```
trak
├── tkhd: 轨道基本属性（ID、时长、宽高/音量）
├── edts [可选]: 非线性编辑信息
└── mdia: 媒体数据描述
    ├── mdhd: 媒体时间刻度、时长、语言
    ├── hdlr: 轨道类型处理器
    └── minf: 媒体信息
        ├── vmhd/smhd: 视频/音频特定头
        ├── dinf: 数据位置引用
        └── stbl: 样本表（核心索引）
```

### 3.5 mdat（Media Data Box）

**作用**：存储实际的编码媒体数据。

```
mdat Box结构：
┌─────────────────────────────────────────────────────────────┐
│ Box Header (8 bytes)                                        │
│ ├── size: 4 bytes (可能为0或1，表示扩展大小)                 │
│ └── type: "mdat" (4 bytes)                                  │
├─────────────────────────────────────────────────────────────┤
│ Data: 实际的音视频编码数据                                   │
│                                                             │
│ 视频数据（H.264/AVC）：                                       │
│ ├── 通常以4字节长度前缀的NAL单元序列                          │
│ ├── NAL单元类型：SPS(7)、PPS(8)、IDR(5)、非IDR(1)           │
│ └── 格式：AVCC（MP4封装格式，区别于Annex B的start code）     │
│                                                             │
│ 音频数据（AAC）：                                             │
│ ├── 原始AAC帧（无ADTS头）                                    │
│ └── 解码参数在stsd的esds中定义                               │
└─────────────────────────────────────────────────────────────┘

注意：mdat中的数据没有内置时间戳，时间信息完全由moov中的stbl索引定义
```

## 四、样本表（stbl）各组件详解

### 4.1 stbl概述

**样本表（Sample Table）是MP4最核心的索引结构**，它建立了从时间到文件位置的完整映射。

```
stbl中的Box协同工作机制：

时间轴 ────────────────────────────────────────────────→

┌─────────────────────────────────────────────────────────────┐
│ stts: 时间 → 样本号                                          │
│   输入：目标时间 T                                           │
│   输出：样本号 N                                             │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ stss: 找关键帧                                               │
│   输入：样本号 N                                             │
│   输出：最近的关键帧号 K（K ≤ N）                            │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ stsc: 样本号 → Chunk号                                       │
│   输入：样本号 K                                             │
│   输出：Chunk号 C                                            │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ stco: Chunk号 → 文件偏移                                     │
│   输入：Chunk号 C                                            │
│   输出：Chunk在文件中的字节偏移 O                            │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ stsz: 计算样本在Chunk内的偏移                                │
│   输入：样本号 K，Chunk起始样本号                            │
│   输出：样本在Chunk内的字节偏移                              │
└─────────────────────────────────────────────────────────────┘

最终：文件偏移 = Chunk偏移 + 样本在Chunk内的偏移
```

### 4.2 stsd（Sample Description Box）

**作用**：定义每个样本的格式和解码参数。

```
stsd结构：
┌─────────────────────────────────────────────────────────────┐
│ Full Box Header                                             │
│ ├── version: 0                                              │
│ └── flags: 0                                                │
├─────────────────────────────────────────────────────────────┤
│ entry_count (4 bytes): 描述条目数量（通常为1）               │
├─────────────────────────────────────────────────────────────┤
│ SampleEntry[]: 样本描述数组                                  │
│                                                             │
│ 视频轨道（avc1/avc3/hvc1/hev1）：                            │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ SampleEntry Header (8 bytes)                            │ │
│ │ ├── reserved: 6 bytes                                   │ │
│ │ └── data_reference_index: 2 bytes                       │ │
│ │                                                         │ │
│ │ VisualSampleEntry Fields:                               │ │
│ │ ├── pre_defined: 2 bytes                                │ │
│ │ ├── reserved: 2 bytes                                   │ │
│ │ ├── pre_defined: 3 × 4 bytes                            │ │
│ │ ├── width: 2 bytes (像素宽度)                           │ │
│ │ ├── height: 2 bytes (像素高度)                          │ │
│ │ ├── horizresolution: 4 bytes (水平分辨率，16.16定点)     │ │
│ │ ├── vertresolution: 4 bytes (垂直分辨率)                 │ │
│ │ ├── reserved: 4 bytes                                   │ │
│ │ ├── frame_count: 2 bytes (每样本帧数，通常为1)           │ │
│ │ ├── compressorname: 32 bytes (编码器名称)                │ │
│ │ ├── depth: 2 bytes (颜色深度，通常为0x0018=24位)         │ │
│ │ └── pre_defined: 2 bytes                                │ │
│ │                                                         │ │
│ │ avcC (AVC Decoder Configuration Record):                │ │
│ │ ├── configurationVersion: 1 byte                        │ │
│ │ ├── AVCProfileIndication: 1 byte (Profile)              │ │
│ │ ├── profile_compatibility: 1 byte                       │ │
│ │ ├── AVCLevelIndication: 1 byte (Level)                  │ │
│ │ ├── lengthSizeMinusOne: 1 byte (NAL长度字段大小-1)      │ │
│ │ ├── numOfSequenceParameterSets: 1 byte                  │ │
│ │ │   └── SPS[]: 序列参数集（包含分辨率、Profile等）       │ │
│ │ ├── numOfPictureParameterSets: 1 byte                   │ │
│ │ │   └── PPS[]: 图像参数集                               │ │
│ │ └── [可选] 其他NAL单元                                   │ │
│ └─────────────────────────────────────────────────────────┘ │
│                                                             │
│ 音频轨道（mp4a）：                                           │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ SampleEntry Header (8 bytes)                            │ │
│ │                                                         │ │
│ │ AudioSampleEntry Fields:                                │ │
│ │ ├── reserved: 4 bytes (2 × 2)                           │ │
│ │ ├── channelcount: 2 bytes (声道数，1=单声道，2=立体声)   │ │
│ │ ├── samplesize: 2 bytes (采样位深，通常为16)            │ │
│ │ ├── pre_defined: 2 bytes                                │ │
│ │ ├── reserved: 2 bytes                                   │ │
│ │ └── samplerate: 4 bytes (采样率，16.16定点)              │ │
│ │                                                         │ │
│ │ esds (Elementary Stream Descriptor):                    │ │
│ │ ├── ES_Descriptor: 流描述符                             │ │
│ │ ├── DecoderConfigDescriptor: 解码器配置                 │ │
│ │ │   ├── objectTypeIndication: 编码类型 (0x40=AAC)       │ │
│ │ │   ├── streamType: 流类型                              │ │
│ │ │   ├── bufferSizeDB: 缓冲区大小                        │ │
│ │ │   ├── maxBitrate: 最大比特率                          │ │
│ │ │   ├── avgBitrate: 平均比特率                          │ │
│ │ │   └── DecoderSpecificInfo: 解码器特定信息             │ │
│ │ │       └── AudioSpecificConfig: AAC配置               │ │
│ │ └── SLConfigDescriptor: 同步层配置                      │ │
│ └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 stts（Decoding Time to Sample Box）

**作用**：建立解码时间（DTS）到样本号的映射。

```
stts结构：
┌─────────────────────────────────────────────────────────────┐
│ Full Box Header                                             │
│ ├── version: 0                                              │
│ └── flags: 0                                                │
├─────────────────────────────────────────────────────────────┤
│ entry_count (4 bytes): 条目数量                             │
├─────────────────────────────────────────────────────────────┤
│ entries[]: 每个条目包含                                       │
│ ├── sample_count (4 bytes): 连续相同duration的样本数         │
│ └── sample_delta (4 bytes): 每个样本的duration               │
│     （以media timescale为单位）                              │
└─────────────────────────────────────────────────────────────┘

示例（30fps视频，timescale=90000）：
┌─────────────────────────────────────────────────────────────┐
│ entry_count = 1                                             │
│                                                             │
│ entry[0]:                                                   │
│ ├── sample_count = 3000  (假设100秒视频)                    │
│ └── sample_delta = 3000  (每帧3000单位 = 33.33ms)           │
│                                                             │
│ 帧号 → DTS计算：                                             │
│ 帧0: DTS = 0                                                │
│ 帧1: DTS = 3000                                             │
│ 帧2: DTS = 6000                                             │
│ ...                                                         │
│ 帧N: DTS = N × 3000                                         │
└─────────────────────────────────────────────────────────────┘

变帧率示例：
┌─────────────────────────────────────────────────────────────┐
│ entry[0]: sample_count=100, sample_delta=3000  (30fps)      │
│ entry[1]: sample_count=50,  sample_delta=1500  (60fps)      │
│ entry[2]: sample_count=100, sample_delta=3000  (30fps)      │
│                                                             │
│ 表示：前100帧30fps，中间50帧60fps，后100帧30fps              │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 ctts（Composition Time to Sample Box）

**作用**：处理B帧时的PTS（显示时间）与DTS（解码时间）差异。

```
ctts结构：
┌─────────────────────────────────────────────────────────────┐
│ Full Box Header                                             │
├─────────────────────────────────────────────────────────────┤
│ entry_count (4 bytes)                                       │
├─────────────────────────────────────────────────────────────┤
│ entries[]:                                                  │
│ ├── sample_count (4 bytes)                                  │
│ └── sample_offset (4 bytes): PTS - DTS 的偏移量              │
│     （有符号整数，version 1支持负数）                        │
└─────────────────────────────────────────────────────────────┘

示例（GOP: I B B P，timescale=3000）：
┌─────────────────────────────────────────────────────────────┐
│ 帧序  解码序  DTS    显示序  PTS    offset = PTS - DTS      │
├─────────────────────────────────────────────────────────────┤
│ I0    0      0      0      0       0                        │
│ B1    2      6000   1      3000    -3000                    │
│ B2    3      9000   2      6000    -3000                    │
│ P3    1      3000   3      9000    6000                     │
├─────────────────────────────────────────────────────────────┤
│ ctts entries:                                               │
│ entry[0]: sample_count=1, sample_offset=0    (I帧)          │
│ entry[1]: sample_count=2, sample_offset=-3000 (B帧)         │
│ entry[2]: sample_count=1, sample_offset=6000  (P帧)         │
└─────────────────────────────────────────────────────────────┘

注意：
• 无B帧的视频通常没有ctts Box
• PTS = DTS + sample_offset
• 对于B帧，sample_offset通常为负（PTS < DTS）
```

### 4.5 stss（Sync Sample Box）

**作用**：记录关键帧（I帧/IDR帧）的位置，是Seek操作的关键。

```
stss结构：
┌─────────────────────────────────────────────────────────────┐
│ Full Box Header                                             │
├─────────────────────────────────────────────────────────────┤
│ entry_count (4 bytes): 关键帧数量                           │
├─────────────────────────────────────────────────────────────┤
│ sample_number[] (4 bytes each): 关键帧的样本号（从1开始）    │
└─────────────────────────────────────────────────────────────┘

示例（每60帧一个关键帧，共180帧）：
┌─────────────────────────────────────────────────────────────┐
│ entry_count = 3                                             │
│                                                             │
│ sample_number[0] = 1    (第1帧，I帧)                        │
│ sample_number[1] = 61   (第61帧，I帧)                       │
│ sample_number[2] = 121  (第121帧，I帧)                      │
│                                                             │
│ 注意：如果没有stss Box，表示所有帧都是关键帧（如All-I编码）  │
└─────────────────────────────────────────────────────────────┘
```

### 4.6 stsc（Sample to Chunk Box）

**作用**：定义样本（Sample）到Chunk的映射关系。

```
stsc结构：
┌─────────────────────────────────────────────────────────────┐
│ Full Box Header                                             │
├─────────────────────────────────────────────────────────────┤
│ entry_count (4 bytes)                                       │
├─────────────────────────────────────────────────────────────┤
│ entries[]:                                                  │
│ ├── first_chunk (4 bytes): 此条目适用的起始Chunk号（从1开始）│
│ ├── samples_per_chunk (4 bytes): 每个Chunk包含的样本数       │
│ └── sample_description_index (4 bytes): 使用的stsd条目索引   │
└─────────────────────────────────────────────────────────────┘

示例：
┌─────────────────────────────────────────────────────────────┐
│ entry_count = 2                                             │
│                                                             │
│ entry[0]:                                                   │
│ ├── first_chunk = 1                                         │
│ ├── samples_per_chunk = 10                                  │
│ └── sample_description_index = 1                            │
│                                                             │
│ entry[1]:                                                   │
│ ├── first_chunk = 101                                       │
│ ├── samples_per_chunk = 5                                   │
│ └── sample_description_index = 1                            │
├─────────────────────────────────────────────────────────────┤
│ 解释：                                                       │
│ • Chunk 1-100: 每个Chunk包含10个样本                         │
│ • Chunk 101+: 每个Chunk包含5个样本                           │
│                                                             │
│ 为什么要分Chunk？                                            │
│ • 减少stco表大小（只需记录Chunk偏移，不用记录每帧偏移）      │
│ • 允许不同Chunk有不同样本数（如场景变化时调整）              │
└─────────────────────────────────────────────────────────────┘

样本号 → Chunk号计算：
┌─────────────────────────────────────────────────────────────┐
│ 给定样本号 N，求所在Chunk：                                  │
│                                                             │
│ 1. 遍历stsc条目，确定N落在哪个范围                           │
│ 2. 计算前面所有Chunk包含的样本总数                           │
│ 3. 确定N所在的Chunk号                                        │
│                                                             │
│ 示例：N=150                                                  │
│ • Chunk 1-100 共包含 100×10 = 1000个样本                     │
│ • 150 > 1000? 否，所以落在前100个Chunk中                     │
│ • Chunk号 = ceil(150 / 10) = 15                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.7 stsz（Sample Size Box）

**作用**：记录每个样本的大小（字节数）。

```
stsz结构：
┌─────────────────────────────────────────────────────────────┐
│ Full Box Header                                             │
├─────────────────────────────────────────────────────────────┤
│ sample_size (4 bytes):                                      │
│   • 如果所有样本大小相同：此字段为固定大小值                 │
│   • 如果样本大小不同：此字段为0                              │
├─────────────────────────────────────────────────────────────┤
│ sample_count (4 bytes): 样本总数                            │
├─────────────────────────────────────────────────────────────┤
│ entry_size[] (4 bytes each) [仅当sample_size=0时存在]       │
│   • 每个样本的字节大小                                       │
└─────────────────────────────────────────────────────────────┘

示例（变长样本，典型情况）：
┌─────────────────────────────────────────────────────────────┐
│ sample_size = 0  (样本大小不同)                             │
│ sample_count = 5                                            │
│                                                             │
│ entry_size[0] = 45320  (I帧，较大)                          │
│ entry_size[1] = 8921   (B帧)                                │
│ entry_size[2] = 7845   (B帧)                                │
│ entry_size[3] = 9012   (P帧)                                │

│ sample_size = 4096  (每个样本4096字节)                      │
│ sample_count = 1000                                         │
│                                                             │
│ entry_size[] 不存在（节省空间）                              │
└─────────────────────────────────────────────────────────────┘
```

**作用**：记录每个Chunk在文件中的字节偏移。

## 五、Seek操作的实现机制

### 5.1 Seek流程详解

当用户拖动进度条到目标时间T时，播放器执行以下步骤：

```
┌─────────────────────────────────────────────────────────────┐
│                    Seek到目标时间T的完整流程                  │
└─────────────────────────────────────────────────────────────┘

步骤1: 时间 → 样本号（使用stts）
┌─────────────────────────────────────────────────────────────┐
│ 输入: 目标时间 T = 10.5秒                                    │
│       timescale = 90000                                      │
│                                                              │
│ 计算: 目标DTS = T × timescale = 10.5 × 90000 = 945000       │
│                                                              │
│ 遍历stts:                                                    │
│   累加 sample_count × sample_delta                           │
│   找到DTS >= 945000的第一个样本                              │
│                                                              │
│ 输出: 目标样本号 N = 315 (假设每帧3000单位)                  │
└─────────────────────────────────────────────────────────────┘
                      │
                      ▼
步骤2: 找最近关键帧（使用stss）
┌─────────────────────────────────────────────────────────────┐
│ 输入: 目标样本号 N = 315                                     │
│       stss = [1, 61, 121, 181, 241, 301, 361, ...]          │
│                                                              │
│ 二分查找: 找到 <= N 的最大关键帧号                            │
│   301 <= 315 < 361                                           │
│                                                              │
│ 输出: 关键帧号 K = 301                                       │
│       （需要从第301帧开始解码，解码15帧后显示第315帧）       │
└─────────────────────────────────────────────────────────────┘
                      │
                      ▼
步骤3: 样本号 → Chunk号（使用stsc）
┌─────────────────────────────────────────────────────────────┐
│ 输入: 关键帧号 K = 301                                       │
│       stsc entries:                                          │
│         [first_chunk=1, samples_per_chunk=10]                │
│         [first_chunk=101, samples_per_chunk=5]               │
│                                                              │
│ 计算:                                                        │
│   Chunk 1-100: 100 × 10 = 1000个样本                         │
│   301 <= 1000，所以落在前100个Chunk中                        │
│   Chunk号 = ceil(301 / 10) = 31                              │
│                                                              │
│ 输出: Chunk C = 31                                           │
└─────────────────────────────────────────────────────────────┘
                      │
                      ▼
步骤4: 获取Chunk偏移（使用stco）
┌─────────────────────────────────────────────────────────────┐
│ 输入: Chunk C = 31                                           │
│                                                              │
│ 查表: stco[30]（数组索引从0开始）                            │
│       stco = [48, 1048624, 2097200, ..., 3145728, ...]       │
│                                                              │
│ 输出: Chunk偏移 O = 3145728（假设值）                        │
└─────────────────────────────────────────────────────────────┘
                      │
                      ▼
步骤5: 计算样本在Chunk内的偏移（使用stsz）
┌─────────────────────────────────────────────────────────────┐
│ 输入: 关键帧号 K = 301                                       │
│       Chunk 31的起始样本号 = 301                             │
│       （假设301正好是一个Chunk的开始）                       │
│                                                              │
│ 如果K不在Chunk开始位置：                                     │
│   需要累加该Chunk内前面所有样本的大小                        │
│   内部偏移 = sum(stsz[Chunk起始..K-1])                       │
│                                                              │
│ 输出: 内部偏移 = 0（K正好是Chunk第一个样本）                 │
└─────────────────────────────────────────────────────────────┘
                      │
                      ▼
步骤6: 定位并解码
┌─────────────────────────────────────────────────────────────┐
│ 最终文件偏移 = Chunk偏移 + 内部偏移                          │
│              = 3145728 + 0                                   │
│              = 3145728                                       │
│                                                              │
│ 操作:                                                        │
│   1. fseek(file, 3145728, SEEK_SET)                          │
│   2. 从第301帧（关键帧）开始读取并解码                       │
│   3. 连续解码到第315帧                                       │
│   4. 从第315帧开始显示                                       │
│                                                              │
│ 注意: 从关键帧到目标帧之间的帧需要解码但不显示               │
│       （这是视频编码的帧依赖关系决定的）                     │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 Seek复杂度分析

| 操作 | 数据结构 | 时间复杂度 | 说明 |
|------|----------|------------|------|
| 时间→样本号 | stts（游程编码） | O(条目数) | 通常条目数很少（<100） |
| 找关键帧 | stss（有序数组） | O(log K) | K为关键帧数，二分查找 |
| 样本→Chunk | stsc | O(条目数) | 通常条目数很少 |
| 获取Chunk偏移 | stco（数组索引） | O(1) | 直接索引 |
| 计算内部偏移 | stsz（累加） | O(每Chunk样本数) | 通常<100 |

**总体Seek时间复杂度：O(log K)**，其中K为关键帧数量，效率非常高。

### 5.3 Seek精度与关键帧间隔

```
关键帧间隔对Seek体验的影响：

场景A: 关键帧间隔 = 2秒 (GOP=60@30fps)
┌─────────────────────────────────────────────────────────────┐
│ 用户Seek到 00:05:23.500                                     │
│ 最近关键帧 = 00:05:22.000                                   │
│ 需要解码帧数 = 1.5秒 × 30fps = 45帧                         │
│ 解码延迟 = 45帧 × 5ms/帧 = 225ms（可接受）                  │
└─────────────────────────────────────────────────────────────┘

场景B: 关键帧间隔 = 10秒 (GOP=300@30fps)
┌─────────────────────────────────────────────────────────────┐
│ 用户Seek到 00:05:23.500                                     │
│ 最近关键帧 = 00:05:20.000                                   │
│ 需要解码帧数 = 3.5秒 × 30fps = 105帧                        │
│ 解码延迟 = 105帧 × 5ms/帧 = 525ms（明显延迟）               │
└─────────────────────────────────────────────────────────────┘

权衡：
• 短GOP（1-2秒）: Seek响应快，但压缩率低，文件大
• 长GOP（4-10秒）: 压缩率高，文件小，但Seek延迟大
• 推荐：点播2-4秒，直播1-2秒
```

## 六、Fragmented MP4（fMP4）结构详解

### 6.1 传统MP4的问题

```
传统MP4结构：
┌─────────────────────────────────────────────────────────────┐
│ ftyp │ moov（完整索引）│ mdat（所有媒体数据）│               │
└─────────────────────────────────────────────────────────────┘
         ↑
         └── 问题1: 必须等录制完成才能写入moov
         └── 问题2: 录制中断 → moov丢失 → 文件损坏
         └── 问题3: 流媒体必须先下载完整moov
```

### 6.2 fMP4结构原理

```
fMP4结构：
┌─────────────────────────────────────────────────────────────────┐
│ ftyp │ moov（初始化）│ moof+mdat │ moof+mdat │ moof+mdat │...   │
│      │   段0        │   片段1   │   片段2   │   片段3   │      │
└─────────────────────────────────────────────────────────────────┘

各组成部分：
┌─────────────────────────────────────────────────────────────┐
│ moov（Movie Box）- 初始化段                                  │
│ ├── 包含mvhd（影片头）                                       │
│ ├── 包含trak（轨道定义）                                     │
│ ├── 包含完整的stsd（样本描述）                               │
│ └── 不包含样本索引（stts/stss/stsc/stsz/stco）              │
│     这些索引移到每个moof中                                   │
├─────────────────────────────────────────────────────────────┤
│ moof（Movie Fragment Box）- 片段头                           │
│ ├── mfhd（Movie Fragment Header）: 片段序号                  │
│ └── traf（Track Fragment Box）: 轨道片段                     │
│     ├── tfhd（Track Fragment Header）: 轨道片段头            │
│     ├── tfdt（Track Fragment Base Media Decode Time）: 基础DTS│
│     └── trun（Track Fragment Run Box）: 样本运行信息         │
│         ├── sample_count: 样本数量                           │
│         ├── data_offset: 数据在mdat中的偏移                  │
│         └── samples[]: 每个样本的信息                        │
│             ├── sample_duration                              │
│             ├── sample_size                                  │
│             ├── sample_flags（是否关键帧等）                 │
│             └── sample_composition_time_offset（PTS偏移）    │
├─────────────────────────────────────────────────────────────┤
│ mdat（Media Data Box）- 片段数据                             │
│ └── 该片段的实际音视频编码数据                                │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 fMP4 vs 传统MP4对比

| 特性 | 传统MP4 | fMP4 |
|------|---------|------|
| **moov位置** | 文件头或尾 | 仅文件头 |
| **moov内容** | 完整索引 | 仅初始化信息 |
| **索引分布** | 集中式 | 分散式（每个片段） |
| **录制安全性** | 低（中断损坏） | 高（每个片段独立） |
| **流式播放** | 需faststart | 原生支持 |
| **自适应码率** | 不支持 | 原生支持（DASH/HLS） |
| **文件开销** | 低（~0.1%） | 略高（~0.5%） |

### 6.4 fMP4的应用场景

```
场景1: DASH（Dynamic Adaptive Streaming over HTTP）
┌─────────────────────────────────────────────────────────────┐
│ 服务器提供：                                                 │
│ ├── manifest.mpd（MPD描述文件）                              │
│ ├── init.mp4（初始化段，包含moov）                          │
│ ├── segment_1.m4s（片段1: moof+mdat）                       │
│ ├── segment_2.m4s（片段2: moof+mdat）                       │
│ └── ...                                                      │
│                                                              │
│ 播放流程：                                                   │
│ 1. 下载init.mp4获取解码参数                                  │
│ 2. 根据带宽选择合适码率的片段                                │
│ 3. 顺序下载segment_x.m4s并播放                               │
│ 4. 网络变化时切换不同码率的片段                              │
└─────────────────────────────────────────────────────────────┘

场景2: HLS with fMP4（iOS 10+）
┌─────────────────────────────────────────────────────────────┐
│ 服务器提供：                                                 │
│ ├── playlist.m3u8                                          │
│ ├── init.mp4                                               │
│ ├── segment0.m4s                                           │
│ ├── segment1.m4s                                           │
│ └── ...                                                      │
│                                                              │
│ m3u8内容：                                                   │
│ #EXTM3U                                                      │
│ #EXT-X-VERSION:6                                             │
│ #EXT-X-TARGETDURATION:6                                      │
│ #EXT-X-MAP:URI="init.mp4"                                   │
│ #EXTINF:6.000,                                               │
│ segment0.m4s                                                 │
│ #EXTINF:6.000,                                               │
│ segment1.m4s                                                 │
└─────────────────────────────────────────────────────────────┘

场景3: 低延迟直播（LL-HLS / LL-DASH）
┌─────────────────────────────────────────────────────────────┐
│ 使用极小的片段（200-500ms）：                                │
│                                                              │
│ 传统HLS延迟: 3-4个片段 × 6秒 = 18-24秒                      │
│ LL-HLS延迟:  3-4个片段 × 0.5秒 = 1.5-2秒                    │
│                                                              │
│ 关键技术：                                                   │
│ • 部分片段预加载（PART-HOLD-BACK）                          │
│ • HTTP/2服务器推送                                           │
│ • 阻塞式播放列表重新加载                                     │
└─────────────────────────────────────────────────────────────┘
```

### 6.5 CMAF（Common Media Application Format）

```
CMAF是ISO/IEC 23000-19标准，统一了DASH和HLS的fMP4格式：

┌─────────────────────────────────────────────────────────────┐
│ 统一前：                                                     │
│ ├── DASH: fMP4 + MPD                                        │
│ └── HLS: TS + m3u8 或 fMP4 + m3u8（Apple特定格式）          │
│                                                              │
│ 统一后（CMAF）：                                             │
│ ├── 统一的fMP4格式（CMAF Fragment）                         │
│ ├── DASH: CMAF + MPD                                        │
│ └── HLS: CMAF + m3u8                                        │
│                                                              │
│ 优势：同一套媒体文件同时服务DASH和HLS客户端                  │
│       减少存储和转码成本                                     │
└─────────────────────────────────────────────────────────────┘
```

## 九、MP4中数据存储的结构

### 9.1 视频数据存储（H.264/AVC）

```
MP4中的H.264数据格式（AVCC）：

Annex B格式（原始NAL流）：
[Start Code: 00 00 00 01] [NAL Header] [NAL Payload]
[Start Code: 00 00 00 01] [NAL Header] [NAL Payload]...

AVCC格式（MP4封装格式）：
[Length: 4 bytes] [NAL Unit]
[Length: 4 bytes] [NAL Unit]...

差异：
┌─────────────────────────────────────────────────────────────┐
│ Annex B使用start code分隔NAL单元                            │
│ AVCC使用4字节长度前缀分隔NAL单元                            │
│                                                             │
│ 转换：                                                       │
│ Annex B → AVCC: 替换start code为长度值                      │
│ AVCC → Annex B: 替换长度值为start code                      │
└─────────────────────────────────────────────────────────────┘

mdat中的实际存储：
┌─────────────────────────────────────────────────────────────┐
│ mdat Box                                                    │
│ ├── [4 bytes: NAL length = 23] [23 bytes: SPS]              │
│ ├── [4 bytes: NAL length = 7]  [7 bytes: PPS]               │
│ ├── [4 bytes: NAL length = 45000] [45000 bytes: IDR帧]      │
│ ├── [4 bytes: NAL length = 8000]  [8000 bytes: P帧]         │
│ └── ...                                                      │
└─────────────────────────────────────────────────────────────┘

SPS/PPS存储位置：
• 在stsd的avcC Box中存储一份（解码器配置）
• 在mdat的每个IDR帧前也可能重复存储（确保解码恢复）
```

### 9.2 音频数据存储（AAC）

```
MP4中的AAC数据格式：

ADTS格式（原始传输流）：
[ADTS Header: 7 bytes] [AAC Raw Data]
[ADTS Header: 7 bytes] [AAC Raw Data]...

MP4格式（无ADTS头）：
[AAC Raw Data]
[AAC Raw Data]...

差异：
┌─────────────────────────────────────────────────────────────┐
│ ADTS每个帧有独立的头部（采样率、声道等）                     │
│ MP4中这些信息集中在stsd的esds Box中                         │
│ MP4中的AAC帧没有ADTS头，是裸帧                              │
└─────────────────────────────────────────────────────────────┘

mdat中的实际存储：
┌─────────────────────────────────────────────────────────────┐
│ mdat Box                                                    │
│ ├── [1024 samples: AAC frame 1]                             │
│ ├── [1024 samples: AAC frame 2]                             │
│ ├── [1024 samples: AAC frame 3]                             │
│ └── ...                                                      │
│                                                             │
│ 注意：每个AAC帧大小由stsz定义，不是固定的                     │
└─────────────────────────────────────────────────────────────┘

esds中的解码配置：
┌─────────────────────────────────────────────────────────────┐
│ objectTypeIndication: 0x40 (MPEG-4 AAC)                     │
│ audioObjectType: 2 (AAC-LC)                                 │
│ samplingFrequencyIndex: 4 (44100Hz)                         │
│ channelConfiguration: 2 (立体声)                            │
└─────────────────────────────────────────────────────────────┘
```

### 9.3 时间戳存储机制

```
时间戳的存储位置：

┌─────────────────────────────────────────────────────────────┐
│ 全局时间刻度（mvhd）                                         │
│ ├── timescale: 影片级时间基准（如1000表示毫秒）             │
│ └── duration: 影片总时长                                     │
├─────────────────────────────────────────────────────────────┤
│ 媒体时间刻度（mdhd）                                         │
│ ├── timescale: 轨道级时间基准                                │
│ │   视频常用: 90000（兼容多种帧率）                          │
│ │   音频常用: 采样率（44100, 48000等）                       │
│ └── duration: 轨道总时长                                     │
├─────────────────────────────────────────────────────────────┤
│ 样本时间戳（stts/ctts）                                      │
│ ├── stts: 每个样本的DTS（解码时间戳）                        │
│ │   计算: DTS[n] = sum(stts[0..n-1].sample_delta)           │
│ │                                                      │
│ └── ctts: 每个样本的PTS偏移（如果有B帧）                     │
│     计算: PTS[n] = DTS[n] + ctts[n].sample_offset           │
├─────────────────────────────────────────────────────────────┤
│ 实际时间计算                                                 │
│ 实际时间(秒) = 时间戳值 / timescale                         │
│                                                              │
│ 示例:                                                        │
│ DTS = 90000, timescale = 90000 → 1.0秒                      │
│ DTS = 45000, timescale = 90000 → 0.5秒                      │
└─────────────────────────────────────────────────────────────┘
```

### 9.4 索引数据大小估算

```
场景: 2小时30fps视频，关键帧间隔2秒

┌─────────────────────────────────────────────────────────────┐
│ 基本参数：                                                   │
│ • 总帧数 = 30 × 60 × 60 × 2 = 216,000帧                     │
│ • 关键帧数 = 216,000 / 60 = 3,600个                         │
│ • Chunk数（假设每Chunk 10帧）= 21,600个                     │
├─────────────────────────────────────────────────────────────┤
│ 各索引表大小：                                               │
│                                                              │
│ stts (时间映射):                                             │
│ • 条目数: 1（固定帧率）                                      │
│ • 大小: 8 + 4 + 8 = 20字节                                   │
│                                                              │
│ ctts (PTS偏移，有B帧时):                                     │
│ • 条目数: 约216,000（每帧一个条目）                          │
│ • 大小: 8 + 4 + 216,000 × 8 = ~1.7MB                        │
│                                                              │
│ stss (关键帧索引):                                           │
│ • 条目数: 3,600                                              │
│ • 大小: 8 + 4 + 3,600 × 4 = ~14KB                           │
│                                                              │
│ stsc (样本到Chunk映射):                                      │
│ • 条目数: 1（固定每Chunk样本数）                             │
│ • 大小: 8 + 4 + 12 = 24字节                                  │
│                                                              │
│ stsz (样本大小):                                             │
│ • 条目数: 216,000                                            │
│ • 大小: 8 + 4 + 4 + 216,000 × 4 = ~845KB                    │
│                                                              │
│ stco (Chunk偏移):                                            │
│ • 条目数: 21,600                                             │
│ • 大小: 8 + 4 + 21,600 × 4 = ~85KB                          │
├─────────────────────────────────────────────────────────────┤
│ 总计（视频轨道stbl）:                                        │
│ • 无B帧: ~950KB                                             │
│ • 有B帧: ~2.6MB                                             │
│                                                              │
│ 对比文件大小（假设8Mbps码率，2小时）:                        │
│ • 视频数据: ~7.2GB                                           │
│ • 索引开销: ~0.04%（无B帧）或 ~0.1%（有B帧）                │
└─────────────────────────────────────────────────────────────┘
```

│    ✓ 使用faststart标志，确保moov在文件头                    │
│    ✓ 或者使用fMP4格式，提高录制安全性                       │
│    ✓ 关键帧间隔控制在2-4秒，平衡压缩率和Seek性能            │
│                                                             │
│ 2. 流媒体分发时：                                           │
│    ✓ 点播场景使用fMP4 + DASH/HLS                            │
│    ✓ 直播场景使用LL-HLS或LL-DASH                            │
│    ✓ 考虑CMAF统一DASH和HLS格式                              │
│                                                             │
│ 3. 存储归档时：                                             │
│    ✓ 需要多音轨/字幕选择MKV                                 │
│    ✓ 追求最大兼容性选择MP4                                  │
│    ✓ 重要录制使用MKV或TS防止中断损坏                        │
│                                                             │
│ 4. 转封装时注意：                                           │
│    ✓ 使用-c copy避免重新编码                                │
│    ✓ 检查目标格式是否支持源编码                             │
│    ✓ 注意时间戳和同步信息的正确传递                         │
│                                                             │
│ 5. 调试分析时：                                             │
│    ✓ 使用ffprobe查看基本信息                                │
│    ✓ 使用mp4dump查看Box结构                                 │
│    ✓ 使用MediaInfo查看元数据                                │
