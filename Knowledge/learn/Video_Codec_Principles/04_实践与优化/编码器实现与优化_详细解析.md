# 编码器实现与优化

> 从开源编码器选型到参数调优，再到硬件加速的完整实践指南

---

## 核心结论（TL;DR）

**掌握编码器参数调优是提升视频质量和效率的关键，而理解底层优化原理能让你做出更明智的技术决策。**

关键要点：
1. x264 是最成熟的 H.264 编码器，x265 和 SVT-AV1 分别是 H.265/AV1 的生产首选
2. CRF 模式适合大多数场景，直播用 CBR，点播可用两遍 VBR
3. preset 参数是速度-质量的主控开关，tune 针对特定内容优化
4. 硬件编码比软编快 10-50 倍，但质量损失 10-30%（相同码率下）
5. SIMD 优化（SSE/AVX/NEON）是软编码性能的核心，可带来 5-20 倍加速

## 一、开源编码器概览

### 1.1 主流编码器对比

| 编码器 | 标准 | 开发者 | 许可证 | 成熟度 | 速度 | 压缩效率 |
|--------|------|--------|--------|--------|------|---------|
| **x264** | H.264 | VideoLAN | GPL | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **x265** | H.265 | MulticoreWare | GPL | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **libvpx** | VP8/VP9 | Google | BSD | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **libaom** | AV1 | AOM | BSD | ⭐⭐⭐ | ⭐ | ⭐⭐⭐⭐⭐ |
| **SVT-AV1** | AV1 | Intel/Netflix | BSD | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **rav1e** | AV1 | Xiph.org | BSD | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |

### 1.2 各编码器特点

#### x264：H.264 编码的黄金标准

- **优势**：极度成熟、文档完善、社区活跃、性能卓越
- **特点**：唯一能在高速和高质量间自由切换的编码器
- **适用**：几乎所有 H.264 编码场景

```bash
# 查看 x264 版本和编译选项
ffmpeg -hide_banner -h encoder=libx264
```

#### x265：H.265/HEVC 参考实现

- **优势**：功能完整、参数丰富、多线程优化好
- **特点**：preset 体系与 x264 类似但更复杂
- **适用**：4K/HDR 内容、需要更高压缩效率的场景

#### libvpx：VP8/VP9 官方编码器

- **优势**：Google 官方维护、WebRTC 集成良好
- **特点**：两遍编码质量好，单遍编码略逊
- **适用**：WebRTC、YouTube 上传

#### libaom：AV1 参考编码器

- **优势**：压缩效率最高、功能最完整
- **劣势**：速度极慢（生产环境几乎不可用）
- **适用**：质量基准测试、研究用途

#### SVT-AV1：生产级 AV1 编码器

- **优势**：Intel/Netflix 联合优化、速度远超 libaom
- **特点**：针对多核 CPU 深度优化
- **适用**：VOD 编码、流媒体平台

#### rav1e：Rust 编写的 AV1 编码器

- **优势**：内存安全、代码清晰
- **特点**：速度和质量均衡，但不如 SVT-AV1 快
- **适用**：对 Rust 生态友好的项目

### 1.3 编码器选择决策树

```
需要编码 H.264？─→ x264
      │
需要编码 H.265？─→ x265
      │
需要编码 AV1？
      ├─→ 速度优先？─→ SVT-AV1
      ├─→ 质量优先？─→ libaom（离线）或 SVT-AV1 低 preset
      └─→ WebRTC？─→ libvpx
```

## 二、编码器参数调优指南

### 2.1 x264 核心参数

#### preset（速度-质量权衡）

| preset | 编码速度 | 相对质量 | 适用场景 |
|--------|---------|---------|---------|
| ultrafast | ~150 fps | -15% | 实时预览 |
| superfast | ~100 fps | -10% | 低延迟直播 |
| veryfast | ~60 fps | -5% | 普通直播 |
| faster | ~45 fps | -3% | |
| fast | ~35 fps | -2% | |
| **medium** | ~25 fps | 基准 | **默认推荐** |
| slow | ~15 fps | +2% | 高质量点播 |
| slower | ~8 fps | +4% | |
| veryslow | ~3 fps | +6% | 存档、发行 |
| placebo | ~1 fps | +6.5% | 仅测试用 |

> 速度数据基于 1080p 软编码，实际因内容和硬件而异

#### tune（内容类型优化）

| tune | 优化目标 | 适用内容 |
|------|---------|---------|
| film | 高质量电影 | 真实拍摄内容 |
| animation | 动画 | 卡通、动漫 |
| grain | 保留胶片颗粒 | 老电影、艺术效果 |
| stillimage | 静态画面 | 幻灯片、截图 |
| fastdecode | 快速解码 | 低端设备播放 |
| **zerolatency** | 零延迟 | **直播、实时通信** |

#### 核心编码参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| **crf** | 恒定质量因子 (0-51) | 18-28，推荐 23 |
| **keyint** | GOP 长度（最大 I 帧间隔） | 帧率 × 2-10 秒 |
| **bframes** | 连续 B 帧数量 | 3-5（低延迟设 0） |
| **ref** | 参考帧数量 | 3-6 |
| **rc-lookahead** | 码率控制前瞻帧数 | 40-60（低延迟设 0） |

#### x264 完整示例

```bash
# 高质量点播编码
ffmpeg -i input.mp4 -c:v libx264 \
       -preset slow -tune film -crf 18 \
       -x264-params "keyint=250:bframes=5:ref=5" \
       output.mp4

# 低延迟直播编码
ffmpeg -i input.mp4 -c:v libx264 \
       -preset veryfast -tune zerolatency \
       -x264-params "keyint=60:bframes=0:ref=1:rc-lookahead=0" \
       -b:v 4M output.mp4
```

### 2.2 x265 核心参数

#### preset 体系

x265 的 preset 与 x264 类似，但编码时间差异更大：

| preset | 相对 x264 medium | 质量提升 |
|--------|-----------------|---------|
| ultrafast | 0.5x | -20% |
| medium | 3-5x | 基准 |
| slow | 10-15x | +5% |
| veryslow | 30-50x | +8% |

#### 特有参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| **pools** | 线程池配置 | 默认自动 |
| **frame-threads** | 帧级并行线程 | CPU 核心数 |
| **wpp** | 波前并行 | 1（开启） |
| **pmode/pme** | 并行模式决策/运动估计 | 高 preset 开启 |
| **ctu** | CTU 大小 | 64（默认） |
| **max-tu-size** | 最大 TU 大小 | 32 |
| **tu-intra-depth** | 帧内 TU 递归深度 | 1-4 |

#### x265 示例

```bash
# 高质量 4K HDR 编码
ffmpeg -i input.mp4 -c:v libx265 \
       -preset slow -crf 20 \
       -x265-params "pools=+:frame-threads=4:wpp=1" \
       -pix_fmt yuv420p10le \
       output.mp4

# 快速预览编码
ffmpeg -i input.mp4 -c:v libx265 \
       -preset ultrafast -crf 28 \
       output.mp4
```

### 2.3 SVT-AV1 核心参数

#### preset 说明

SVT-AV1 的 preset 范围 0-13，数字越小质量越高、速度越慢：

| preset | 速度描述 | 适用场景 |
|--------|---------|---------|
| 0-2 | 极慢 | 存档、质量优先 |
| 3-5 | 慢 | 高质量 VOD |
| **6-8** | 中等 | **平衡点** |
| 9-11 | 快 | 快速编码 |
| 12-13 | 极快 | 实时预览 |

#### 关键参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| **preset** | 速度-质量权衡 | 6-8 |
| **crf** | 恒定质量 (0-63) | 25-35，推荐 30 |
| **tile-columns/tile-rows** | 分块并行 | 根据分辨率设置 |
| **enable-overlays** | 参考帧叠加 | 1（更好质量） |
| **film-grain** | 胶片颗粒合成 | 内容相关 |

#### SVT-AV1 示例

```bash
# 高质量 VOD
ffmpeg -i input.mp4 -c:v libsvtav1 \
       -preset 4 -crf 28 \
       -svtav1-params "tile-columns=2:tile-rows=1" \
       output.webm

# 快速编码
ffmpeg -i input.mp4 -c:v libsvtav1 \
       -preset 10 -crf 35 \
       output.webm
```

### 2.4 参数影响对照表

| 参数类别 | 调大影响 | 调小影响 |
|---------|---------|---------|
| **CRF/QP** | 文件更小，质量更差 | 文件更大，质量更好 |
| **preset** | 编码更快，质量更差 | 编码更慢，质量更好 |
| **GOP** | 压缩更好，随机访问慢 | 压缩略差，seek 更快 |
| **B 帧数** | 压缩更好，延迟更大 | 延迟更低 |
| **参考帧数** | 压缩更好，内存更多 | 编码更快，质量略降 |

## 三、FFmpeg 实战命令

### 3.1 基础转码

```bash
# 最简单的转码（使用默认参数）
ffmpeg -i input.mp4 -c:v libx264 output.mp4

# 指定码率
ffmpeg -i input.mp4 -c:v libx264 -b:v 5M output.mp4

# 指定质量（CRF 模式）
ffmpeg -i input.mp4 -c:v libx264 -crf 23 output.mp4

# 同时转码音视频
ffmpeg -i input.mp4 -c:v libx264 -crf 23 -c:a aac -b:a 128k output.mp4
```

### 3.2 两遍编码（Two-Pass）

两遍编码能更好地分配码率，适合固定文件大小的场景：

```bash
# 第一遍（分析）
ffmpeg -i input.mp4 -c:v libx264 -preset slow -b:v 5M \
       -pass 1 -an -f null /dev/null

# 第二遍（编码）
ffmpeg -i input.mp4 -c:v libx264 -preset slow -b:v 5M \
       -pass 2 -c:a aac -b:a 128k output.mp4

# x265 两遍编码
ffmpeg -i input.mp4 -c:v libx265 -preset slow -b:v 3M \
       -x265-params "pass=1:stats=x265_2pass.log" \
       -an -f null /dev/null
       
ffmpeg -i input.mp4 -c:v libx265 -preset slow -b:v 3M \
       -x265-params "pass=2:stats=x265_2pass.log" \
       -c:a aac -b:a 128k output.mp4
```

### 3.3 硬件加速编码

#### NVIDIA NVENC

```bash
# H.264 硬编
ffmpeg -i input.mp4 -c:v h264_nvenc -preset p4 -cq 23 output.mp4

# H.265 硬编（更好质量）

# 低延迟硬编
ffmpeg -i input.mp4 -c:v h264_nvenc -preset p1 -tune ll \

```

#### Intel QSV

```bash

ffmpeg -i input.mp4 -c:v h264_qsv -preset medium \
       -global_quality 23 output.mp4

# H.265 QSV
ffmpeg -i input.mp4 -c:v hevc_qsv -preset medium \
       -global_quality 25 output.mp4
```

#### Apple VideoToolbox

```bash

ffmpeg -i input.mp4 -c:v h264_videotoolbox -q:v 65 output.mp4

# H.265 VideoToolbox
ffmpeg -i input.mp4 -c:v hevc_videotoolbox -q:v 60 output.mp4
```


### 3.4 常用滤镜组合


### 3.5 批量处理脚本

```bash
#!/bin/bash
# 批量转码脚本
INPUT_DIR="./input"
OUTPUT_DIR="./output"
CRF=23

mkdir -p "$OUTPUT_DIR"

for file in "$INPUT_DIR"/*.mp4; do
    filename=$(basename "$file" .mp4)
    ffmpeg -i "$file" \
           -c:v libx264 -preset medium -crf $CRF \
           -c:a aac -b:a 128k \
           "$OUTPUT_DIR/${filename}_encoded.mp4"
done
```

## 四、SIMD 优化

### 4.1 SIMD 在编解码中的作用

**SIMD（Single Instruction Multiple Data）** 是编解码性能优化的核心技术，可实现 5-20 倍的加速。

#### 主要 SIMD 指令集

| 指令集 | 平台 | 寄存器宽度 | 并行度 |
|--------|------|-----------|--------|
| SSE2 | x86 | 128 bit | 16×8bit 或 8×16bit |
| AVX2 | x86 | 256 bit | 32×8bit 或 16×16bit |
| AVX-512 | x86 服务器 | 512 bit | 64×8bit |
| NEON | ARM | 128 bit | 16×8bit |

### 4.2 运动估计的 SIMD 加速

运动估计是编码中最耗时的环节，SAD（Sum of Absolute Differences）计算是其核心：

```c
// 标量实现（慢）
int sad_scalar(uint8_t *a, uint8_t *b, int stride) {
    int sum = 0;
    for (int y = 0; y < 8; y++) {
        for (int x = 0; x < 8; x++) {
            sum += abs(a[y*stride + x] - b[y*stride + x]);
        }
    }
    return sum;
}

// SSE2 实现（快约 10 倍）
int sad_sse2(uint8_t *a, uint8_t *b, int stride) {
    __m128i sum = _mm_setzero_si128();
    for (int y = 0; y < 8; y++) {
        __m128i va = _mm_loadl_epi64((__m128i*)(a + y*stride));
        __m128i vb = _mm_loadl_epi64((__m128i*)(b + y*stride));
        sum = _mm_add_epi32(sum, _mm_sad_epu8(va, vb));
    }
    return _mm_cvtsi128_si32(sum) + _mm_extract_epi32(sum, 2);
}
```

## 五、GPU 硬件编码

### 5.1 各平台硬件编码器

| 平台 | 编码器 | API | 支持标准 |
|------|--------|-----|---------|
| **NVIDIA** | NVENC | CUDA/DirectX | H.264, H.265, AV1 |
| **Intel** | QSV/oneVPL | Media SDK | H.264, H.265, VP9, AV1 |
| **AMD** | AMF/VCE | AMF | H.264, H.265, AV1 |
| **Apple** | VideoToolbox | VideoToolbox | H.264, H.265 |

### 5.2 硬编 vs 软编对比

| 维度 | 软编码 (x264) | 硬编码 (NVENC) |
|------|--------------|----------------|
| 编码速度 | 1x | 10-50x |
| 质量（同码率） | 基准 | -10% ~ -30% |
| 质量（同视觉） | 基准 | +20% ~ +50% 码率 |
| CPU 占用 | 高 | 低 |
| 并发能力 | 受限于 CPU | 受限于 GPU 会话数 |
| 成本 | 软件免费 | 需 GPU 硬件 |

### 5.3 NVENC 详解

#### Preset 选择

| Preset | 速度 | 质量 | 用途 |
|--------|------|------|------|
| p1 | 最快 | 最低 | 实时预览 |
| p2-p3 | 快 | 较低 | 低延迟直播 |
| **p4** | 中等 | 中等 | **平衡推荐** |
| p5-p6 | 较慢 | 较高 | 高质量录制 |
| p7 | 最慢 | 最高 | 离线转码 |

#### Tune 参数

| Tune | 优化目标 |
|------|---------|
| hq | 高质量 |
| ll | 低延迟 |
| ull | 超低延迟 |
| lossless | 无损 |

```bash
# NVENC 最佳实践
ffmpeg -i input.mp4 \
       -c:v hevc_nvenc -preset p5 -tune hq \
       -rc vbr -cq 23 -b:v 5M -maxrate 8M \
       -spatial_aq 1 -temporal_aq 1 \
       output.mp4
```

### 5.4 Intel QSV 详解

```bash
# 检查 QSV 可用性
ffmpeg -hide_banner -init_hw_device qsv=qsv:hw \
       -filter_hw_device qsv -i input.mp4 -f null -

# QSV H.265 编码
ffmpeg -hwaccel qsv -i input.mp4 \
       -c:v hevc_qsv -preset medium \
       -global_quality 25 -look_ahead 1 \
       output.mp4
```

### 5.5 Apple VideoToolbox 详解

```bash
# Apple Silicon 上的硬件编码
ffmpeg -i input.mp4 \
       -c:v hevc_videotoolbox \
       -q:v 60 -tag:v hvc1 \
       output.mp4

# 使用硬件解码 + 编码（零拷贝）
ffmpeg -hwaccel videotoolbox -i input.mp4 \
       -c:v hevc_videotoolbox -q:v 60 \
       output.mp4
```

## 六、码率控制实践

### 6.1 码率控制模式对比

| 模式 | 全称 | 特点 | 适用场景 |
|------|------|------|---------|
| **CRF** | Constant Rate Factor | 恒定质量，码率可变 | 点播、存档 |
| **CBR** | Constant Bitrate | 恒定码率 | 直播、流媒体 |
| **VBR** | Variable Bitrate | 可变码率 | 点播 |
| **ABR** | Average Bitrate | 平均码率 | 兼顾两者 |
| **CQP** | Constant QP | 恒定量化参数 | 测试、特殊需求 |

### 6.2 CRF 详解

CRF 是最推荐的点播编码模式，让编码器自动分配码率以达到目标质量。

#### x264 CRF 推荐值

| CRF | 质量描述 | 适用场景 |
|-----|---------|---------|
| 15-17 | 视觉无损 | 存档、母版 |
| **18-20** | 高质量 | **高质量点播** |
| **21-23** | 中高质量 | **普通点播** |
| 24-26 | 中等质量 | 流媒体、节省空间 |
| 27-30 | 较低质量 | 预览、低带宽 |

```bash
# x264 CRF 编码
ffmpeg -i input.mp4 -c:v libx264 -crf 23 output.mp4

# x265 CRF（推荐值比 x264 高 2-4）
ffmpeg -i input.mp4 -c:v libx265 -crf 26 output.mp4

# SVT-AV1 CRF（推荐值更高）
ffmpeg -i input.mp4 -c:v libsvtav1 -crf 30 output.webm
```

### 6.3 CBR 详解

CBR 保持码率稳定，适合网络传输。

```bash
# 严格 CBR
ffmpeg -i input.mp4 -c:v libx264 \
       -b:v 4M -maxrate 4M -bufsize 8M \
       output.mp4

# NVENC CBR
ffmpeg -i input.mp4 -c:v h264_nvenc \
       -rc cbr -b:v 4M output.mp4
```

### 6.4 VBR 与两遍编码

VBR 允许码率波动以优化质量，两遍编码效果更好。

```bash
# VBR 模式
ffmpeg -i input.mp4 -c:v libx264 \
       -b:v 4M -maxrate 6M -bufsize 8M \
       output.mp4

# 两遍 VBR
ffmpeg -i input.mp4 -c:v libx264 -b:v 4M \
       -pass 1 -f null /dev/null
ffmpeg -i input.mp4 -c:v libx264 -b:v 4M \
       -pass 2 output.mp4
```

### 6.5 码率控制选择决策

```
需要固定文件大小？
    ├─→ 是 ─→ 两遍 VBR
    └─→ 否
          │
          ├─→ 直播/实时？─→ CBR
          │
          └─→ 点播/存档？─→ CRF
```

## 七、常见问题与解决

### 7.1 编码质量问题

#### 问题：块效应明显

```bash
# 解决：降低 CRF 或增加码率
ffmpeg -i input.mp4 -c:v libx264 -crf 18 output.mp4

# 或使用去块滤镜（解码后）
ffmpeg -i input.mp4 -vf "deblock=filter=weak" output.mp4
```

#### 问题：运动模糊/拖尾

```bash
# 解决：增加参考帧和 B 帧
ffmpeg -i input.mp4 -c:v libx264 \
       -x264-params "ref=6:bframes=5" \
       -crf 20 output.mp4
```

### 7.2 编码速度问题

#### 问题：编码太慢

```bash
# 使用更快的 preset
ffmpeg -i input.mp4 -c:v libx264 -preset faster output.mp4

# 或使用硬件编码
ffmpeg -i input.mp4 -c:v h264_nvenc output.mp4

# 限制分辨率
ffmpeg -i input.mp4 -vf "scale=1280:720" -c:v libx264 output.mp4
```

#### 问题：多核利用不足

```bash
# x264 多线程
ffmpeg -i input.mp4 -c:v libx264 -threads 8 output.mp4

# x265 多线程
ffmpeg -i input.mp4 -c:v libx265 \
       -x265-params "pools=+:frame-threads=4" output.mp4
```

### 7.3 兼容性问题

#### 问题：视频无法播放

```bash
# 使用更兼容的 profile
ffmpeg -i input.mp4 -c:v libx264 \
       -profile:v high -level 4.1 \
       -pix_fmt yuv420p output.mp4

# H.265 兼容性标签
ffmpeg -i input.mp4 -c:v libx265 -tag:v hvc1 output.mp4
```

## 九、总结

### 9.1 核心要点回顾

1. **编码器选择**：x264（H.264）、x265（H.265）、SVT-AV1（AV1）是生产首选
2. **参数调优**：preset 控制速度-质量权衡，CRF 控制目标质量
3. **码率控制**：点播用 CRF，直播用 CBR，追求精确文件大小用两遍 VBR
4. **硬件加速**：速度提升显著，但质量有损失，适合直播和实时场景
5. **SIMD 优化**：理解原理有助于评估编码器性能差异

### 9.2 实践建议

- **从默认参数开始**：preset medium + CRF 23 是合理起点
- **逐步调优**：先调 preset 和 CRF，再考虑高级参数
- **用数据说话**：用 VMAF 评估实际质量，而非主观判断
- **关注场景需求**：延迟、兼容性、成本比"最高质量"更重要
