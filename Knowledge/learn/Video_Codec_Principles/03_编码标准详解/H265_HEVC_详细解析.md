# H.265/HEVC 详细解析

> H.265/HEVC 是继 H.264 之后的重大技术突破，通过创新的四叉树编码结构和更精细的预测工具，实现了约 50% 的压缩效率提升，成为 4K/8K 超高清视频的核心编码标准。

---

## 核心结论（TL;DR）

**H.265/HEVC 引入 CTU/CU/PU/TU 四叉树编码结构，将块大小从 16×16 扩展到 64×64，配合 35 种帧内预测模式、Merge 模式、SAO 滤波等技术，相比 H.264 节省约 50% 码率，但编码复杂度增加约 2-10 倍。**

关键要点：
1. 64×64 CTU 四叉树递归分割适应不同内容复杂度
2. 35 种帧内预测模式（H.264 仅 9 种）大幅提升预测精度
3. Merge 模式和 AMVP 显著减少运动信息传输开销
4. Tile 和 WPP 提供并行编码能力，缓解复杂度问题

## 1. H.265 的设计目标

### 1.1 发展背景

H.265/HEVC（High Efficiency Video Coding）由 ITU-T VCEG 和 ISO/IEC MPEG 联合开发的 JCT-VC 小组制定，于 **2013年1月** 正式发布第一版。

**核心设计目标**：

| 目标 | 量化指标 |
|------|---------|
| 压缩效率 | 相比 H.264 提升约 50% |
| 分辨率支持 | 支持 4K（3840×2160）和 8K（7680×4320） |
| 并行能力 | 提供 Tile、WPP 等并行处理工具 |
| 位深扩展 | 支持 10bit、12bit 高动态范围 |

### 1.2 应对的技术挑战

```
分辨率增长带来的数据量爆炸：
┌─────────────────────────────────────────────┐
│ SD (720×480)    →  HD (1920×1080) → 4K → 8K │
│ 像素数：34.5万  →  207万          → 830万 → 3320万 │
│                 6×                  4×      4×    │
└─────────────────────────────────────────────┘
```

如果仍使用 H.264：
- 4K@60fps 需要约 **40-60 Mbps**
- 8K@60fps 需要约 **160-240 Mbps**

H.265 目标：将上述码率降低约一半。

### 1.3 技术演进路线

```
H.264/AVC (2003)
    ↓
    + 更大的块结构（64×64）
    + 更多预测模式（35种）
    + Merge/AMVP
    + SAO
    + 并行工具
    ↓
H.265/HEVC (2013)
```

## 2. 四叉树编码结构

### 2.1 核心概念

H.265 引入了全新的分层编码单元体系，这是相对 H.264 最根本的架构变革：

| 单元 | 全称 | 大小范围 | 功能 |
|------|------|---------|------|
| CTU | Coding Tree Unit | 16-64 | 替代宏块的基本处理单元 |
| CU | Coding Unit | 8-64 | 编码决策的基本单元 |
| PU | Prediction Unit | 4-64 | 预测的基本单元 |
| TU | Transform Unit | 4-32 | 变换的基本单元 |

### 2.2 CTU（编码树单元）

CTU 是 H.265 中替代 H.264 宏块的概念：

```
H.264 宏块：固定 16×16
         ↓
H.265 CTU：最大 64×64（可配置 16/32/64）
```

**64×64 CTU 的优势**：
- 对于平坦的大面积区域，一个 CTU 即可覆盖
- 减少边界数量，降低块效应
- 减少模式选择的开销

### 2.3 CU（编码单元）四叉树分割

CU 通过四叉树递归分割，从 CTU 大小分割到最小 8×8：

```
64×64 CTU 四叉树分割示例：

┌───────────────────────────────────────────┐
│                                           │
│                  64×64                    │
│              (不分割,depth=0)              │
│                                           │
└───────────────────────────────────────────┘

                    ↓ 分割

┌───────────────────┬───────────────────────┐
│                   │                       │
│      32×32        │       32×32           │
│    (depth=1)      │     (depth=1)         │
├───────────────────┼───────────────────────┤
│                   │                       │
│      32×32        │       32×32           │
│    (depth=1)      │     (depth=1)         │
└───────────────────┴───────────────────────┘

                    ↓ 继续分割左上角

┌─────────┬─────────┬───────────────────────┐
│  16×16  │  16×16  │                       │
│(depth=2)│(depth=2)│       32×32           │
├─────────┼─────────┤     (depth=1)         │
│  16×16  │  16×16  │                       │
│(depth=2)│(depth=2)│                       │
├─────────┴─────────┼───────────────────────┤
│                   │                       │
│      32×32        │       32×32           │
└───────────────────┴───────────────────────┘
```

**分割深度与 CU 大小对应关系**：

| depth | CU 大小 | 等效 H.264 宏块数 |
|-------|---------|------------------|
| 0 | 64×64 | 16 |
| 1 | 32×32 | 4 |
| 2 | 16×16 | 1 |
| 3 | 8×8 | 1/4 |

### 2.4 PU（预测单元）

PU 定义了预测的分割模式，可以是非对称的：

**帧内 CU 的 PU 分割**：

| 模式 | 图示 | 说明 |
|------|------|------|
| 2N×2N | ▊ | 整个 CU 作为一个 PU |
| N×N | ▐▌ | 仅当 CU 为最小时可用 |

**帧间 CU 的 PU 分割**（8种模式）：

```
2N×2N    2N×N     N×2N     N×N
┌────┐   ┌────┐   ┌──┬──┐  ┌──┬──┐
│    │   ├────┤   │  │  │  ├──┼──┤
│    │   │    │   │  │  │  │  │  │
└────┘   └────┘   └──┴──┘  └──┴──┘

2N×nU    2N×nD    nL×2N    nR×2N
┌────┐   ┌────┐   ┌─┬───┐  ┌───┬─┐
├────┤   │    │   │ │   │  │   │ │
│    │   ├────┤   │ │   │  │   │ │
└────┘   └────┘   └─┴───┘  └───┴─┘
  25%      75%     25% 75%  75% 25%
```

**非对称分割的优势**：更好地匹配物体边界，减少预测残差。

### 2.5 TU（变换单元）

TU 可以独立于 CU 进行四叉树分割，支持残差四叉树（RQT）：

```
CU 内部的 TU 分割：
┌────────────────────┐     ┌────────────────────┐
│                    │     │    TU    │   TU    │
│        CU          │ →   ├─────────┼─────────┤
│                    │     │    TU    │   TU    │
└────────────────────┘     └─────────┴─────────┘
   TU = CU                   TU = CU/2
```

**TU 分割的决策**：
- 高频细节区域：使用小 TU
- 平坦区域：使用大 TU

### 2.6 四叉树 RDO 决策

编码器通过率失真优化（RDO）决定最佳分割：

$$J = D + \lambda \cdot R$$

其中：
- $D$：失真（如 SSE）
- $R$：码率（bits）
- $\lambda$：拉格朗日乘子（与 QP 相关）

**决策流程**：

```
从 CTU 开始，递归尝试：
1. 计算当前深度不分割的 RD Cost
2. 计算分割后四个子 CU 的 RD Cost 之和
3. 选择 Cost 更小的方案
4. 递归直到最小 CU 或 Cost 收敛
```

## 3. 帧内预测增强

### 3.1 预测模式数量扩展

| 特性 | H.264 | H.265 |
|------|-------|-------|
| 角度模式数 | 8 | 33 |
| 非角度模式 | DC | Planar + DC |
| 总模式数 | 9 | 35 |

### 3.2 35 种帧内预测模式

```
模式编号与方向：

                     模式 26 (正上)
                          ↑
         模式 18 ←   ←   ↑   →   → 模式 34
       (正左)    ↖ ↑ ↑ ↑ ↗
                 ← Planar DC →
                 ↙ ↓ ↓ ↓ ↘
         模式 10 ←   ←   ↓   →   → 模式 2
       (左下)        ↓
                  模式 0/1
              (Planar/DC)

角度模式分布：
- 模式 2-17：水平方向附近（9种左下到水平到左上）
- 模式 18-34：垂直方向附近（9种左上到垂直到右上）
```

### 3.3 Planar 模式

Planar 模式（模式 0）用于平滑渐变区域：

```
参考像素：
      T0 T1 T2 T3 ... TN TR
    L0┌─┬─┬─┬─┐
    L1├─┼─┼─┼─┤
    L2├─┼─┼─┼─┤
    L3├─┼─┼─┼─┤
   ...└─┴─┴─┴─┘
    LB

预测值 = 水平方向线性插值 + 垂直方向线性插值
```

**Planar 预测公式**（以 N×N 块为例）：

$$pred[x][y] = \frac{(N-1-x) \cdot L[y] + (x+1) \cdot TR + (N-1-y) \cdot T[x] + (y+1) \cdot LB + N}{2N}$$

### 3.4 DC 模式改进

H.265 的 DC 模式增加了边界平滑处理：

```
对于第一行和第一列，使用相邻参考像素进行额外滤波，
避免 DC 预测值与边界的明显跳变。
```

### 3.5 角度预测的参考像素扩展

为了支持所有角度方向，H.265 扩展了参考像素的获取范围：

```
                    扩展参考像素
                    ↓
        ┌───────────────────────────┐
        │ * * * * * * * * * * * * * │← 上方参考行（含右上扩展）
      * ┌─────────────────────────┐
      * │                         │
      * │     N×N 预测块          │
      * │                         │
      * └─────────────────────────┘
      ↑
   左侧参考列
   （含左下扩展）
```

## 4. 帧间预测增强

### 4.1 Merge 模式

Merge 是 H.265 帧间预测的核心创新，继承邻居块的全部运动信息：

**与 H.264 Skip 的区别**：
- Skip：必须满足残差为零
- Merge：可以有残差，更灵活

**Merge 候选列表构建**：

```
空间候选（最多5个）：
    ┌───┬───┐
    │ B2│ B1│ B0
    ├───┼───┼───┐
    │ A1│Cur│   │
    └───┴───┘   │
    │ A0│       │
    └───┴───────┘

优先级：A1 → B1 → B0 → A0 → B2

时间候选（1个）：
- 共位图像中对应位置或右下位置的块
```

**Merge 索引编码**：
- 仅传输候选索引（0-4）
- 无需传输运动矢量差（MVD）
- 极大节省运动信息码率

### 4.2 AMVP（高级运动矢量预测）

当 Merge 不适用时，使用 AMVP 进行运动矢量预测：

```
AMVP 候选列表构建（2个候选）：

空间候选：
- 左侧组：A0, A1（取第一个具有相同参考帧的）
- 上方组：B0, B1, B2（取第一个具有相同参考帧的）

时间候选：
- 共位块的缩放 MV

如果候选不足，用零矢量填充
```

**AMVP vs Merge**：

| 特性 | Merge | AMVP |
|------|-------|------|
| 候选数量 | 5 | 2 |
| 传输 MVD | 否 | 是 |
| 适用场景 | 运动均匀区域 | 精确运动匹配 |
| 码率开销 | 低 | 中等 |

### 4.3 运动矢量精度

H.265 保持 **1/4 像素精度**，但改进了插值滤波器：

- **亮度半像素**：8 抽头滤波器（H.264 为 6 抽头）
- **亮度 1/4 像素**：7 抽头滤波器
- **色度**：4 抽头滤波器

## 5. 变换编码

### 5.1 变换大小

| 块大小 | 变换类型 | 说明 |
|--------|---------|------|
| 4×4 | DCT/DST | DST 用于帧内 |
| 8×8 | DCT | |
| 16×16 | DCT | |
| 32×32 | DCT | H.264 没有的大变换 |

### 5.2 DST（离散正弦变换）

H.265 为 4×4 帧内预测块引入了 DST：

**为何使用 DST？**

帧内预测的残差在靠近参考边界处通常较小（预测更准确），远离边界处较大。DST 的基函数特性更适合这种统计特性。

```
DCT 基函数：边界处幅值大
    ___/‾‾‾\___

DST 基函数：边界处幅值小
    ___/‾‾\___
```

### 5.3 Transform Skip（变换跳过）

对于某些特殊内容（如屏幕内容、文字），直接传输残差比变换更高效：

```
if (transform_skip_enabled) {
    // 跳过 DCT 变换
    // 直接量化空域残差
    coeff = quantize(residual);
}
```

### 5.4 Sign Data Hiding

一种码率优化技术，利用变换系数的奇偶性隐藏符号信息：

```
如果一个变换块的系数满足条件（首尾系数距离 ≥ 4）：
- 系数绝对值之和的奇偶性编码第一个非零系数的符号
- 避免显式传输该符号位
```

## 6. 并行处理工具

### 6.1 Tile（瓦片）

Tile 将图像分割为独立的矩形区域：

```
帧分割为 3×2 Tiles：
┌─────────┬─────────┬─────────┐
│ Tile 0  │ Tile 1  │ Tile 2  │
├─────────┼─────────┼─────────┤
│ Tile 3  │ Tile 4  │ Tile 5  │
└─────────┴─────────┴─────────┘

特点：
- Tile 之间没有预测依赖
- 可以完全独立编解码
- 熵编码状态在 Tile 边界重置
```

**Tile 的代价**：
- Tile 边界不能跨边界预测，损失压缩效率
- 可能出现可见的 Tile 边界伪影

### 6.2 WPP（波前并行处理）

WPP 提供行级并行，同时保持较好的压缩效率：

```
WPP 并行执行示意（4个线程）：

Time →
CTU Row 0: ████████████████████████
CTU Row 1:   ████████████████████████
CTU Row 2:     ████████████████████████
CTU Row 3:       ████████████████████████
CTU Row 4:         ████████████████████████
...

规则：每行在前一行完成 2 个 CTU 后可以开始
```

**WPP 的优势**：
- 保持行间的预测依赖（压缩效率损失很小）
- 熵编码状态从上一行第二个 CTU 初始化
- 理论最大并行度 = 帧高度 / CTU高度 / 2

### 6.3 Slice（切片）

传统的独立编码单元，H.265 继续支持：

- **Regular Slice**：完全独立，可单独解码
- **Dependent Slice**：依赖前一个 Slice 的熵编码状态

### 6.4 并行度与压缩效率权衡

| 并行工具 | 并行度 | 压缩效率损失 | 典型应用 |
|---------|-------|-------------|---------|
| Slice | 低 | 中等 | 错误恢复 |
| WPP | 中等 | 很小（~1%） | 软件编码 |
| Tile | 高 | 较大（~3-5%） | 硬件编码、4K/8K |

## 7. 样本自适应偏移（SAO）

### 7.1 SAO 概述

SAO（Sample Adaptive Offset）是 H.265 新增的环路滤波技术，在去块滤波之后执行。

**核心思想**：对特定模式的像素添加偏移量，减少振铃效应和块效应。

### 7.2 边界偏移（Edge Offset, EO）

根据像素与相邻像素的关系分类：

```
4 种边界方向：
EO-0：水平    EO-1：垂直    EO-2：135°   EO-3：45°
─ ● ─       │            ╲            ╱
            ●             ╲●         ●╱
            │              ╲         ╱
```

**边界分类（5类）**：

| 类别 | 条件 | 典型偏移 |
|------|------|---------|
| 0 | c == n1 == n2 | 0（不调整） |
| 1 | c < n1 && c < n2 | 正偏移（提亮） |
| 2 | c < n1 && c == n2（或反向） | 小正偏移 |
| 3 | c > n1 && c == n2（或反向） | 小负偏移 |
| 4 | c > n1 && c > n2 | 负偏移（变暗） |

其中 c 是当前像素，n1/n2 是指定方向的相邻像素。

### 7.3 带偏移（Band Offset, BO）

将像素值分为 32 个带（Band），对连续 4 个带应用偏移：

```
像素值范围 [0, 255] 分为 32 个带：
Band 0:   [0,   7]
Band 1:   [8,  15]
...
Band 31: [248, 255]

选择连续 4 个带应用偏移：
如选择 Band 12-15，则像素值在 [96, 127] 范围内的像素被调整
```

### 7.4 SAO 参数传输

每个 CTU 传输：
- SAO 类型（EO-0/1/2/3 或 BO 或 Off）
- 4 个偏移值（EO 类别 1-4 或 BO 的 4 个带）
- BO 的起始带位置

## 8. 去块滤波改进

### 8.1 与 H.264 的区别

| 特性 | H.264 | H.265 |
|------|-------|-------|
| Bs 范围 | 0-4 | 0-2 |
| 滤波粒度 | 4 像素 | 8 像素 |
| 处理顺序 | 宏块 | CTU |

### 8.2 简化的边界强度判断

H.265 简化了 Bs 的判断规则：

```
Bs = 2: CU 边界有 Intra CU
Bs = 1: CU 边界有不同的参考帧，或 MV 差 ≥ 1 整像素，或变换系数非零
Bs = 0: 其他情况
```

### 8.3 滤波强度

根据 Bs、QP 和像素梯度决定是否滤波：

```c
if (Bs > 0) {
    if (d < β) {  // 梯度检查
        if (Bs == 2) {
            strong_filter();  // 修改 3 个像素
        } else {
            normal_filter(); // 修改 1-2 个像素
        }
    }
}
```

## 10. 与 H.264 的改进对比

### 10.1 各工具压缩效率贡献

| 技术改进 | BD-Rate 节省 | 复杂度增加 |
|---------|-------------|-----------|
| 大块结构（64×64） | ~10-15% | 中等 |
| 更多帧内模式（35种） | ~3-5% | 高 |
| Merge 模式 | ~5-8% | 低 |
| AMVP 改进 | ~2-3% | 低 |
| 更大变换（32×32） | ~2-4% | 中等 |
| SAO | ~2-3% | 低 |
| 改进的熵编码 | ~3-5% | 中等 |
| **总计** | **~40-50%** | **2-10×** |

### 10.2 综合对比表

| 特性 | H.264/AVC | H.265/HEVC |
|------|-----------|------------|

| 环路滤波 | Deblock | Deblock + SAO |

## 11. x265 编码参数示例

### 11.1 高质量编码

```bash
# 4K 蓝光级编码
x265 --preset slow --tune grain \
     --crf 18 --profile main10 --level 5.1 \
     --ref 5 --bframes 8 --b-adapt 2 \
     --rd 4 --psy-rd 2.0 --psy-rdoq 1.0 \
     --aq-mode 3 --aq-strength 0.8 \
     --sao --deblock -1:-1 \
     --input input.y4m --output output.265
```

### 11.2 流媒体编码

```bash
# 1080p 流媒体
x265 --preset medium --tune psnr \
     --bitrate 4000 --vbv-maxrate 6000 --vbv-bufsize 8000 \
     --profile main --level 4.1 \
     --keyint 48 --min-keyint 24 \
     --ref 4 --bframes 4 \
     --input input.y4m --output output.265
```

### 11.3 低延迟编码

```bash
# 实时通信
x265 --preset ultrafast --tune zerolatency \
     --bitrate 2000 --vbv-maxrate 2000 --vbv-bufsize 1000 \
     --profile main --level 4.0 \
     --keyint 30 --bframes 0 \
     --no-sao --no-weightp \
     --input input.y4m --output output.265
```

### 11.4 关键参数说明

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| --preset | 速度/质量权衡 | medium |
| --tune | 内容类型优化 | 默认/film/grain |
| --crf | 恒定质量 | 18-28 |
| --ref | 参考帧数 | 3-6 |
| --bframes | B帧数 | 4-8 |
| --rd | RDO 等级 | 3-5 |
| --psy-rd | 心理视觉优化 | 1.0-2.0 |
| --aq-mode | 自适应量化 | 2 或 3 |
| --sao | SAO 开关 | 开启 |
