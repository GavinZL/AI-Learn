# 视频数据的数学表示

> 从像素到张量——理解视频在计算机中的数据结构

---

## 核心结论（TL;DR）

**视频本质上是一个四维张量 $T \times H \times W \times C$，理解其数学结构是掌握编解码的前提。**

关键要点：
1. 单帧图像是 $H \times W \times C$ 的三维张量，视频增加时间维度成为四维张量
2. 色彩子采样（4:2:0）利用人眼特性，可节省 50% 数据量而视觉损失极小
3. 未压缩 1080p@30fps 视频每秒约 187 MB，4K@60fps 超过 1.4 GB——必须压缩
4. I/P/B 帧的概念是视频压缩的基础，将时间冗余转化为可编码的差异

## 1. 像素：视频数据的原子单位

### 1.1 灰度像素：标量表示

对于灰度图像，每个像素是一个标量值：

```python
import numpy as np

# 8-bit 灰度像素
pixel_gray = 128  # 范围 [0, 255]

# 归一化表示
pixel_normalized = 0.5  # 范围 [0.0, 1.0]

# 10-bit 表示
pixel_10bit = 512  # 范围 [0, 1023]
```

### 1.2 彩色像素：向量表示

彩色像素是一个向量，包含多个分量：

**RGB 表示**：
```python
# RGB 像素 (Red, Green, Blue)
pixel_rgb = np.array([255, 128, 64])  # 橙色

# 归一化 RGB
pixel_rgb_norm = np.array([1.0, 0.5, 0.25])
```

**YUV/YCbCr 表示**：
```python
# YUV 像素 (Luma, Chroma-blue, Chroma-red)
pixel_yuv = np.array([188, 98, 176])

# Y: 亮度分量 (0-255)
# U/Cb: 蓝色色度分量
# V/Cr: 红色色度分量
```

### 1.3 RGB 与 YUV 的转换

标准转换公式（BT.601）：

$$\begin{bmatrix} Y \\ U \\ V \end{bmatrix} = \begin{bmatrix} 0.299 & 0.587 & 0.114 \\ -0.169 & -0.331 & 0.500 \\ 0.500 & -0.419 & -0.081 \end{bmatrix} \begin{bmatrix} R \\ G \\ B \end{bmatrix} + \begin{bmatrix} 0 \\ 128 \\ 128 \end{bmatrix}$$

```python
def rgb_to_yuv(r, g, b):
    """RGB 转 YUV (BT.601)"""
    y = 0.299 * r + 0.587 * g + 0.114 * b
    u = -0.169 * r - 0.331 * g + 0.500 * b + 128
    v = 0.500 * r - 0.419 * g - 0.081 * b + 128
    return y, u, v

def yuv_to_rgb(y, u, v):
    """YUV 转 RGB (BT.601)"""
    u, v = u - 128, v - 128
    r = y + 1.402 * v
    g = y - 0.344 * u - 0.714 * v
    b = y + 1.772 * u
    return np.clip([r, g, b], 0, 255).astype(np.uint8)
```

**为什么用 YUV？** 人眼对亮度（Y）远比色度（UV）敏感，分离后可以对 UV 降采样。

## 2. 图像：二维/三维矩阵表示

### 2.1 灰度图像：2D 矩阵

$$I \in \mathbb{R}^{H \times W}$$

```python
# 1080p 灰度图像
height, width = 1080, 1920
gray_image = np.zeros((height, width), dtype=np.uint8)

# 访问像素
pixel_value = gray_image[540, 960]  # 中心点

# 图像就是矩阵，可以进行矩阵运算
brightened = np.clip(gray_image + 50, 0, 255)  # 增加亮度
```

### 2.2 彩色图像：3D 张量

$$I \in \mathbb{R}^{H \times W \times C}$$

其中 $C$ 是通道数（RGB 为 3，RGBA 为 4）。

```python
# 1080p RGB 图像
rgb_image = np.zeros((1080, 1920, 3), dtype=np.uint8)

# 访问像素的各个通道
r = rgb_image[540, 960, 0]
g = rgb_image[540, 960, 1]
b = rgb_image[540, 960, 2]

# 或一次性访问整个像素向量
pixel = rgb_image[540, 960]  # shape: (3,)

# 分离通道
r_channel = rgb_image[:, :, 0]  # shape: (1080, 1920)
g_channel = rgb_image[:, :, 1]
b_channel = rgb_image[:, :, 2]
```

### 2.3 内存布局

图像在内存中的存储方式影响访问效率：

```python
# 交错存储 (Interleaved/Packed)
# RGBRGBRGBRGB... （每个像素的通道连续）
interleaved = np.zeros((H, W, 3))  # numpy 默认

# 平面存储 (Planar)
# RRRR...GGGG...BBBB... （同一通道的像素连续）
planar_r = np.zeros((H, W))
planar_g = np.zeros((H, W))
planar_b = np.zeros((H, W))

# YUV 视频通常使用平面存储
# Y 平面 + U 平面 + V 平面
```

## 3. 视频：四维张量表示

### 3.1 基本结构

$$V \in \mathbb{R}^{T \times H \times W \times C}$$

其中：
- $T$：时间维度（帧数）
- $H$：高度（像素行数）
- $W$：宽度（像素列数）
- $C$：通道数

```python
# 10 秒 30fps 的 1080p RGB 视频
frames = 10 * 30  # 300 帧
video = np.zeros((frames, 1080, 1920, 3), dtype=np.uint8)

# 访问第 100 帧
frame_100 = video[99]  # shape: (1080, 1920, 3)

# 访问第 100 帧的中心像素
pixel = video[99, 540, 960]  # shape: (3,)

# 时间切片：所有帧的同一位置
center_pixels_over_time = video[:, 540, 960, :]  # shape: (300, 3)
```

### 3.2 帧的顺序与索引

```
时间 →
┌─────┬─────┬─────┬─────┬─────┐
│  0  │  1  │  2  │ ... │ T-1 │
└─────┴─────┴─────┴─────┴─────┘
 帧0   帧1   帧2         帧T-1
```

```python
# 帧索引约定
first_frame = video[0]      # 第一帧
last_frame = video[-1]      # 最后一帧

# 帧率决定时间映射
fps = 30
time_seconds = frame_index / fps
frame_index = int(time_seconds * fps)
```

## 4. 色彩子采样：节省数据的关键技术

### 4.1 子采样的生理学基础

人眼视网膜中：
- **视锥细胞（负责色彩）**：约 600 万个，集中在中央凹
- **视杆细胞（负责亮度）**：约 1.2 亿个，分布更广

结论：人眼对亮度变化的分辨率远高于对色彩变化的分辨率。

### 4.2 子采样格式表示法

标准表示：**J:a:b**（基于 4×2 像素块）

- **J**：参考块宽度（通常为 4）
- **a**：第一行色度采样数
- **b**：第二行色度采样数

### 4.3 常见子采样格式

**4:4:4 — 无子采样**
```
亮度 Y：     色度 UV：
○ ○ ○ ○     ○ ○ ○ ○
○ ○ ○ ○     ○ ○ ○ ○
每个像素都有完整的 Y、U、V 值
数据量比例：1 + 1 + 1 = 3
```

**4:2:2 — 水平减半**
```
亮度 Y：     色度 UV：
○ ○ ○ ○     ○   ○  
○ ○ ○ ○     ○   ○  
每行 4 个 Y，2 个 UV
数据量比例：1 + 0.5 + 0.5 = 2（节省 33%）
```

**4:2:0 — 水平垂直各减半**
```
亮度 Y：     色度 UV：
○ ○ ○ ○     ○   ○
○ ○ ○ ○         
每 2×2 的 Y 块共享一个 UV
数据量比例：1 + 0.25 + 0.25 = 1.5（节省 50%）
```

### 4.4 4:2:0 的数学细节

```python
# 4:2:0 格式的数据结构
class YUV420Frame:
    def __init__(self, width, height):
        self.width = width
        self.height = height
        
        # Y 平面：全分辨率
        self.Y = np.zeros((height, width), dtype=np.uint8)
        
        # U/V 平面：1/4 分辨率
        self.U = np.zeros((height // 2, width // 2), dtype=np.uint8)
        self.V = np.zeros((height // 2, width // 2), dtype=np.uint8)
    
    def get_uv_for_pixel(self, y, x):
        """获取像素 (y, x) 对应的 UV 值"""
        return self.U[y // 2, x // 2], self.V[y // 2, x // 2]
    
    def total_bytes(self):
        """总字节数"""
        y_size = self.width * self.height
        uv_size = (self.width // 2) * (self.height // 2) * 2
        return y_size + uv_size  # = 1.5 × width × height
```

### 4.5 子采样的位置问题

色度样本相对于亮度样本的位置有两种主流方案：

**Co-sited（同位）**：
```
Y: ● ● ● ●    UV 采样点与左上角 Y 对齐
   ● ● ● ●    
UV:●   ●      
```

**Interstitial（间位/MPEG-1 风格）**：
```
Y: ● ● ● ●    UV 采样点在四个 Y 的中心
   ● ● ● ●    
UV:  ●   ●    
```

H.264/H.265 使用 co-sited 方案。

### 4.6 为什么 4:2:0 最常用？

| 格式 | 数据量（相对 4:4:4）| 质量损失 | 典型应用 |
|-----|-------------------|---------|---------|
| 4:4:4 | 100% | 无 | 专业后期、图形渲染 |
| 4:2:2 | 67% | 微小 | 广播、专业摄像 |
| 4:2:0 | 50% | 几乎不可察觉 | 流媒体、蓝光、消费级 |

## 5. 存储量计算：理解压缩的必要性

### 5.1 基本公式

未压缩视频数据量：

$$D = W \times H \times fps \times bpp \times duration$$

其中：
- $W$：宽度（像素）
- $H$：高度（像素）
- $fps$：帧率
- $bpp$：每像素比特数
- $duration$：时长（秒）

### 5.2 每像素比特数（bpp）计算

```python
def calculate_bpp(bit_depth, chroma_format):
    """计算每像素比特数"""
    formats = {
        '4:4:4': 3.0,    # 3 个完整通道
        '4:2:2': 2.0,    # Y + 0.5U + 0.5V
        '4:2:0': 1.5,    # Y + 0.25U + 0.25V
    }
    return bit_depth * formats[chroma_format]

# 8-bit 4:2:0
bpp_8bit_420 = calculate_bpp(8, '4:2:0')  # = 12 bits/pixel

# 10-bit 4:2:0
bpp_10bit_420 = calculate_bpp(10, '4:2:0')  # = 15 bits/pixel
```

### 5.3 常见分辨率的数据量

```python
def calculate_bitrate(width, height, fps, bit_depth=8, chroma='4:2:0'):
    """计算未压缩视频比特率"""
    bpp = calculate_bpp(bit_depth, chroma)
    bits_per_second = width * height * fps * bpp
    return bits_per_second

def format_size(bits):
    """格式化显示"""
    mbps = bits / 1e6
    gbph = (bits * 3600) / (8 * 1e9)  # GB per hour
    return f"{mbps:.1f} Mbps ({gbph:.1f} GB/hour)"

# 常见格式的未压缩比特率
resolutions = [
    ("720p 30fps", 1280, 720, 30),
    ("1080p 30fps", 1920, 1080, 30),
    ("1080p 60fps", 1920, 1080, 60),
    ("4K 30fps", 3840, 2160, 30),
    ("4K 60fps", 3840, 2160, 60),
    ("8K 60fps", 7680, 4320, 60),
]

for name, w, h, fps in resolutions:
    bitrate = calculate_bitrate(w, h, fps)
    print(f"{name}: {format_size(bitrate)}")
```

输出：
```
720p 30fps:  332.0 Mbps (149.4 GB/hour)
1080p 30fps: 746.5 Mbps (335.9 GB/hour)
1080p 60fps: 1493.0 Mbps (671.9 GB/hour)
4K 30fps:    2985.9 Mbps (1343.7 GB/hour)
4K 60fps:    5971.9 Mbps (2687.4 GB/hour)
8K 60fps:    23887.9 Mbps (10749.5 GB/hour)
```

### 5.4 未压缩 vs 压缩对比

| 分辨率 | 未压缩码率 | H.264 典型码率 | 压缩比 |
|-------|-----------|---------------|-------|
| 720p@30 | 332 Mbps | 3 Mbps | ~110:1 |
| 1080p@30 | 746 Mbps | 8 Mbps | ~93:1 |
| 4K@30 | 2986 Mbps | 25 Mbps | ~120:1 |
| 4K@60 | 5972 Mbps | 40 Mbps | ~150:1 |

**结论**：没有压缩，视频存储和传输完全不可行。

## 6. 帧类型：理解时间维度的压缩

### 6.1 为什么需要不同帧类型？

视频的时间冗余极高——相邻帧往往 90%+ 的内容相同。

利用这一点：
- **参考帧**：完整编码
- **预测帧**：只编码与参考帧的差异

### 6.2 三种基本帧类型

**I 帧（Intra-coded Frame，关键帧）**
```
特点：
- 完全独立编码，不参考其他帧
- 压缩率最低，体积最大
- 随机访问点（seek 必须到 I 帧）

适用：
- 场景切换
- 定期插入（如每 2 秒一个）
- 直播的起始点
```

**P 帧（Predicted Frame，前向预测帧）**
```
特点：
- 参考前面的 I 帧或 P 帧
- 只编码差异（残差）
- 压缩率高于 I 帧

编码内容：
帧内容 = 参考帧预测 + 残差
P帧存储 = 运动向量 + 残差
```

**B 帧（Bi-directional Frame，双向预测帧）**
```
特点：
- 参考前后两个方向的帧
- 压缩率最高，体积最小
- 增加编码延迟

参考示意：
I ← P ← B → P
    ↑_____|
```

### 6.3 GOP 结构

GOP（Group of Pictures）定义了帧的排列方式：

```
典型 GOP 结构（GOP=12, M=3）:
I B B P B B P B B P B B [I ...]

I: 关键帧
P: 前向预测帧
B: 双向预测帧
```

```python
class GOP:
    """GOP 结构表示"""
    def __init__(self, gop_size=12, b_frames=2):
        self.gop_size = gop_size
        self.b_frames = b_frames
    
    def generate_structure(self):
        """生成 GOP 帧类型序列"""
        structure = ['I']
        p_interval = self.b_frames + 1
        
        for i in range(1, self.gop_size):
            if i % p_interval == 0:
                structure.append('P')
            else:
                structure.append('B')
        
        return structure

# GOP(12, 2) 生成: ['I', 'B', 'B', 'P', 'B', 'B', 'P', 'B', 'B', 'P', 'B', 'B']
```

### 6.4 编码顺序 vs 显示顺序

由于 B 帧需要后向参考，编码顺序与显示顺序不同：

```
显示顺序: I0  B1  B2  P3  B4  B5  P6
编码顺序: I0  P3  B1  B2  P6  B4  B5

原因：必须先编码 P3，B1 和 B2 才能完成编码
```

### 6.5 帧类型与码率分布

典型 I:P:B 体积比约为 **10:3:1**

```python
# 假设场景
i_frame_size = 100  # KB
p_frame_size = 30   # KB  
b_frame_size = 10   # KB

# GOP(12, 2) 的平均帧大小
gop = ['I', 'B', 'B', 'P', 'B', 'B', 'P', 'B', 'B', 'P', 'B', 'B']
total_size = (1 * i_frame_size + 
              3 * p_frame_size + 
              8 * b_frame_size)  # 270 KB

avg_frame_size = total_size / 12  # 22.5 KB
```

## 7. 实际存储格式

### 7.1 YUV 文件格式

原始 YUV 数据常用平面存储：

```python
def read_yuv420_frame(f, width, height):
    """从文件读取一帧 YUV420"""
    y_size = width * height
    uv_size = (width // 2) * (height // 2)
    
    y = np.frombuffer(f.read(y_size), dtype=np.uint8).reshape(height, width)
    u = np.frombuffer(f.read(uv_size), dtype=np.uint8).reshape(height//2, width//2)
    v = np.frombuffer(f.read(uv_size), dtype=np.uint8).reshape(height//2, width//2)
    
    return y, u, v

def write_yuv420_frame(f, y, u, v):
    """写入一帧 YUV420"""
    f.write(y.tobytes())
    f.write(u.tobytes())
    f.write(v.tobytes())

# 使用示例
with open('video.yuv', 'rb') as f:
    y, u, v = read_yuv420_frame(f, 1920, 1080)
```

## 8. 小结：视频数据的层次结构

```
视频文件
├── 容器层（MP4, MKV...）
│   └── 编码流
│       └── GOP 序列
│           ├── I 帧
│           │   └── 宏块/CTU
│           │       └── 块
│           │           └── 像素
│           ├── P 帧
│           │   └── 运动向量 + 残差
│           └── B 帧
│               └── 双向运动向量 + 残差
└── 元数据（时间戳、码率信息等）
```

```
RGB → Y:    Y  = 0.299R + 0.587G + 0.114B
RGB → Cb:   Cb = -0.169R - 0.331G + 0.500B + 128
RGB → Cr:   Cr = 0.500R - 0.419G - 0.081B + 128

未压缩码率 = W × H × fps × bpp

典型压缩比：
  I帧 ≈ 10-20:1
  P帧 ≈ 30-50:1
  B帧 ≈ 50-100:1
```
