# 预测编码（Prediction Coding）

> 通过预测消除冗余，编码"意外"而非"全部"——这是视频压缩效率的核心来源。

---

## 核心结论（TL;DR）

**预测编码的本质是"说差异而非说全部"——与其传输完整的像素值，不如传输"预测值与实际值的差"，这个差值通常更小、更容易压缩。**

关键要点：
1. 帧内预测利用空间相关性，从相邻像素外推当前块
2. 帧间预测利用时间相关性，从参考帧中搜索相似块
3. 残差信号 = 实际值 - 预测值，残差越小压缩越高效
4. 预测模式的选择是率失真优化（RDO）问题

## 1. 预测编码的核心思想

### 1.1 从天气预报说起

假设你要每天告诉朋友今天的气温：

| 方式 | 今天表达 | 明天表达 | 后天表达 | 信息量 |
|------|---------|---------|---------|--------|
| **直接说** | "27度" | "28度" | "26度" | 每天都需要完整数字 |
| **说差异** | "27度"（基准） | "比昨天高1度" | "比昨天低2度" | 差值通常更小 |

视频编码的思路完全一样：**如果能准确预测当前像素/块的值，那么只需要传输"预测错误的部分"（残差），这比传输原始值高效得多。**

### 1.2 预测的数学表达

```
原始值 = 预测值 + 残差
↓
残差 = 原始值 - 预测值
```

编码器传输的是：
- **预测方式**（如何预测）—— 通常只需要几个比特
- **残差值**（预测偏差）—— 经过变换、量化后传输

解码器执行：
```
重建值 = 预测值 + 解码后的残差
```

### 1.3 两种预测维度

| 预测类型 | 利用的相关性 | 参考来源 | 典型场景 |
|---------|-------------|---------|---------|
| **帧内预测** | 空间相关性 | 同一帧内的相邻像素 | I帧、场景切换 |
| **帧间预测** | 时间相关性 | 其他已编码的帧 | P帧、B帧 |

**类比**：帧内预测像是看照片上某个区域，根据周围的颜色猜测中间是什么；帧间预测像是看视频时，根据上一帧猜测这一帧的内容。

## 2. 帧内预测（Intra Prediction）

### 2.1 为什么需要帧内预测

有些情况下，我们无法使用时间维度的信息：

1. **视频的第一帧（I帧）**：没有前序帧可以参考
2. **场景切换**：前一帧和当前帧完全不同，参考没有意义
3. **随机访问点**：需要不依赖其他帧就能解码的关键帧
4. **防止误差传播**：定期插入I帧可以阻断错误累积

此时，只能利用**同一帧内已解码的相邻像素**来预测当前块。

### 2.2 帧内预测的基本流程

```
┌─────────────────────────────────────────────────┐
│                已解码区域                        │
│    ┌───┬───┬───┬───┬───┐                        │
│    │ A │ B │ C │ D │ E │  ← 上方参考像素         │
│    ├───┼───┴───┴───┴───┘                        │
│    │ F │                                        │
│    ├───┤   ┌───────────┐                        │
│    │ G │   │ 当前块    │  ← 待预测               │
│    ├───┤   │ (4x4)     │                        │
│    │ H │   └───────────┘                        │
│    └───┘                                        │
│      ↑                                          │
│   左侧参考像素                                   │
└─────────────────────────────────────────────────┘
```

编码顺序是**从左到右、从上到下**（光栅扫描），因此当前块的**左侧和上方**的像素已经解码完成，可以用作预测参考。

### 2.3 帧内预测模式详解

#### DC 模式（均值预测）

**思想**：用参考像素的平均值填充整个块。

```
预测值 = (A + B + C + D + F + G + H + I) / 8
```

**适用场景**：平坦区域、没有明显方向性的纹理。

**Python 示例**：
```python
def dc_prediction(top_pixels, left_pixels, block_size=4):
    """DC模式：用参考像素均值预测整个块"""
    dc_value = (sum(top_pixels[:block_size]) + 
                sum(left_pixels[:block_size])) // (2 * block_size)
    return [[dc_value] * block_size for _ in range(block_size)]
```

#### Planar 模式（平面拟合）

**思想**：假设块内像素值呈平面分布，同时考虑水平和垂直方向的渐变。

```
pred[y][x] = (H × (x+1) + V × (y+1) + 
              top[x] × (N-1-y) + left[y] × (N-1-x) + N) / (2N)
```

其中 H 和 V 分别是水平和垂直方向的边界值。

**适用场景**：渐变区域、斜向过渡。

**类比**：想象在一张纸上，用左边和上边的颜色"拉"出一个渐变的平面。

#### Angular 模式（方向模式）

**思想**：沿特定角度，将参考像素"投射"到块内。

```
┌──────────────────────────────────────────────────┐
│                 角度方向示意图                    │
│                                                  │
│            2(垂直)    10                        │
│              │       ╱                          │
│       50     │     ╱                            │
│         ╲    │   ╱      18                      │
│           ╲  │ ╱      ╱                         │
│     34 ──── + ──── 26                           │
│           ╱  │ ╲                                │
│         ╱    │   ╲                              │
│              │                                  │
│                                                  │
│    H.265: 从 2° 到 34° 共35种模式               │
│    AV1:   更精细的 56 种模式                     │
└──────────────────────────────────────────────────┘
```

**H.264 的 9 种 4×4 帧内预测模式**：

| 模式号 | 名称 | 方向 |
|-------|------|------|
| 0 | Vertical | 垂直（↓） |
| 1 | Horizontal | 水平（→） |
| 2 | DC | 均值 |
| 3 | Diagonal Down-Left | 对角线左下（↙） |
| 4 | Diagonal Down-Right | 对角线右下（↘） |
| 5 | Vertical-Right | 垂直偏右 |
| 6 | Horizontal-Down | 水平偏下 |
| 7 | Vertical-Left | 垂直偏左 |
| 8 | Horizontal-Up | 水平偏上 |

**标准演进对比**：

| 标准 | 块大小 | 帧内模式数 | 特点 |
|------|--------|-----------|------|
| H.264/AVC | 4×4, 8×8, 16×16 | 9 + 4 | 基础方向模式 |
| H.265/HEVC | 4×4 到 32×32 | 35 | 更精细角度，2°间隔 |
| AV1 | 4×4 到 64×64 | 56 + 递归 | 方向滤波、Paeth |
| H.266/VVC | 4×4 到 64×64 | 67 | 宽角度、MIP |

### 2.4 模式决策：率失真优化（RDO）

编码器如何在 35 种甚至 67 种模式中选择最佳的那个？

**暴力方法**：尝试每种模式，选择效果最好的。但"效果好"怎么定义？

**率失真优化（Rate-Distortion Optimization）**：

$$J = D + \lambda \cdot R$$

- $D$：失真（Distortion），通常用 SSE/SAD 衡量
- $R$：比特率（Rate），编码这个块需要的比特数
- $\lambda$：拉格朗日乘子，平衡质量和码率

**选择使 $J$ 最小的模式。**

**伪代码**：
```python
def select_best_intra_mode(block, reference_pixels, qp):
    lambda_val = get_lambda(qp)
    best_cost = float('inf')
    best_mode = 0
    
    for mode in range(35):  # H.265 的 35 种模式
        # 1. 生成预测块
        pred = generate_intra_prediction(reference_pixels, mode)
        
        # 2. 计算残差
        residual = block - pred
        
        # 3. 变换、量化、熵编码（得到比特数）
        bits = encode_residual(residual, qp)
        
        # 4. 反量化、反变换（得到重建残差）
        recon_residual = decode_residual(bits, qp)
        
        # 5. 重建块
        recon_block = pred + recon_residual
        
        # 6. 计算失真（SSE）
        distortion = sum((block - recon_block) ** 2)
        
        # 7. 率失真代价
        cost = distortion + lambda_val * bits
        
        if cost < best_cost:
            best_cost = cost
            best_mode = mode
    
    return best_mode
```

**快速算法**：实际编码器不会遍历所有模式，而是使用：
- **Rough Mode Decision (RMD)**：先用简单度量（如 SATD）筛选候选模式
- **方向检测**：利用 Sobel 等算子估计梯度方向，只测试相近角度

## 3. 帧间预测（Inter Prediction）

### 3.1 时间冗余的利用

视频的相邻帧通常非常相似——对于 30fps 的视频，每帧之间只有 33ms，画面变化很小。

**类比**：如果让你画一部动画，与其每帧都从头画起，不如在上一帧基础上只修改变化的部分。

帧间预测做的事情：
1. 在参考帧中**找到**和当前块最相似的区域
2. **记录**这个区域相对当前块的位移（运动矢量）
3. **传输**运动矢量 + 残差

### 3.2 运动估计（Motion Estimation）

**目标**：在参考帧中搜索与当前块最匹配的区域。

```
┌──────────────────────────────────────────────────┐
│  当前帧                     参考帧               │
│                                                  │
│  ┌───────┐                 搜索范围              │
│  │当前块 │     运动矢量    ┌─────────────────┐   │
│  │ 8×8   │  ───────────►  │  ···············│   │
│  └───────┘   MV=(12,-3)   │  ·  ┌───────┐  ·│   │
│     ↑                     │  ·  │匹配块 │  ·│   │
│  当前位置                  │  ·  └───────┘  ·│   │
│                           │  ···············│   │
│                           └─────────────────┘   │
└──────────────────────────────────────────────────┘
```

#### 匹配准则

| 准则 | 公式 | 特点 |
|------|------|------|
| SAD | $\sum|a_{ij} - b_{ij}|$ | 最简单，硬件友好 |
| SSE/SSD | $\sum(a_{ij} - b_{ij})^2$ | 更精确，对大误差敏感 |
| SATD | $\sum|H(a-b)|$ | 考虑变换后的能量 |

**SAD（Sum of Absolute Differences）最常用于运动估计**，因为计算简单、硬件实现高效。

#### 搜索算法演进

**1. 全搜索（Full Search / Exhaustive Search）**

```python
def full_search(current_block, reference_frame, search_range=16):
    """暴力搜索：遍历搜索范围内的所有位置"""
    best_mv = (0, 0)
    min_sad = float('inf')
    
    for dy in range(-search_range, search_range + 1):
        for dx in range(-search_range, search_range + 1):
            ref_block = get_block(reference_frame, dx, dy)
            sad = calculate_sad(current_block, ref_block)
            if sad < min_sad:
                min_sad = sad
                best_mv = (dx, dy)
    
    return best_mv
```

- **复杂度**：$O((2S+1)^2)$，搜索范围 $S=16$ 时需要检查 1089 个位置
- **优点**：保证全局最优
- **缺点**：计算量巨大

**2. 三步搜索（Three Step Search, TSS）**

```
Step 1: 搜索范围 ±8，步长 4
        ·   ·   ·
        ·   X   ·     找到最佳点 A
        ·   ·   ·

Step 2: 以 A 为中心，步长 2
        ·   ·   ·
        ·   A   ·     找到最佳点 B
        ·   ·   ·

Step 3: 以 B 为中心，步长 1
        ·   ·   ·
        ·   B   ·     找到最终位置
        ·   ·   ·
```

- **复杂度**：$3 \times 9 = 27$ 次比较
- **加速比**：相比全搜索约 40 倍

**3. 菱形搜索（Diamond Search）**

```
大菱形模式（LDSP）        小菱形模式（SDSP）
      ·                       ·
    · · ·                   · · ·
  · · X · ·     →            · ·
    · · ·
      ·
```

反复用大菱形搜索，直到最佳点位于中心，再用小菱形精细化。

**4. 六边形搜索（Hexagon Search）**

```
    ·   ·
  ·   X   ·
    ·   ·
```

六边形模式在大范围搜索时更高效。

### 3.3 亚像素精度

整像素运动估计的精度有限，实际视频中物体的移动往往不是整像素的。

**解决方案**：插值生成半像素、1/4 像素位置的参考像素。

```
┌──────────────────────────────────────────────────┐
│  整像素位置（原始）                               │
│                                                  │
│     A ─ ─ ─ ─ B        A: 整像素                │
│     │         │        a: 半像素（A和B的平均）    │
│     │    ·    │        b: 1/4像素（A和a的平均）  │
│     │         │                                  │
│     C ─ ─ ─ ─ D                                  │
│                                                  │
│  插值后                                          │
│                                                  │
│     A ─ b ─ a ─ b ─ B                           │
│     │   │   │   │   │                           │
│     c · · · · · · · c                           │
│     │   │   │   │   │                           │
│     a · · · · · · · a                           │
│     │   │   │   │   │                           │
│     c · · · · · · · c                           │
│     │   │   │   │   │                           │
│     C ─ b ─ a ─ b ─ D                           │
└──────────────────────────────────────────────────┘
```

**H.264 的插值滤波器**：
- **半像素**：6-tap 滤波器 `[-1, 5, 20, 20, 5, -1] / 32`
- **1/4像素**：半像素之间线性插值

**H.265 的改进**：
- **亮度**：8-tap 滤波器（更好的频率响应）
- **色度**：4-tap 滤波器

### 3.4 多参考帧预测

传统视频编码只参考前一帧，但有时候参考更早的帧效果更好：

```
帧序列：  [0] [1] [2] [3] [4] [5] [当前帧]
                              ↑
                         可能帧4被遮挡的物体
                         在帧2中完整可见
```

**H.264 引入多参考帧**：
- P 帧可以从最多 16 个参考帧中选择
- 参考帧存储在 DPB（Decoded Picture Buffer）中

### 3.5 双向预测（B 帧）

B 帧可以同时参考**前向**和**后向**的帧，然后对两个预测做加权平均：

```
参考帧 L0        当前帧(B)       参考帧 L1
    │                │               │
    │    MV_L0       │    MV_L1      │
    └───────────────►│◄──────────────┘
                     │
            pred = (pred_L0 + pred_L1) / 2
```

**优势**：
- 更准确的预测（两个方向的信息）
- 适合处理遮挡（被遮挡的区域可能在另一个方向可见）

**B 帧的代价**：
- 编码延迟增加（需要等待后向参考帧）
- 编码复杂度翻倍

## 4. 参考帧管理

### 4.1 DPB（Decoded Picture Buffer）

解码器需要保存已解码的帧，用于后续帧的参考。

```
┌─────────────────────────────────────────────────┐
│                    DPB 示意图                    │
│                                                  │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐       │
│  │帧 0 │ │帧 1 │ │帧 3 │ │帧 4 │ │空槽 │ ...   │
│  │(LTR)│ │(STR)│ │(STR)│ │(STR)│ │     │       │
│  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘       │
│     ↑                                           │
│  长期参考帧                                      │
└─────────────────────────────────────────────────┘
```

### 4.2 长期与短期参考帧

| 类型 | 生命周期 | 典型用途 |
|------|---------|---------|
| **短期参考帧（STR）** | 很快被替换 | 正常预测 |
| **长期参考帧（LTR）** | 长期保留 | 场景重现、防丢包 |

**长期参考帧的应用场景**：
- 会议视频：发言人切换后切回，可以直接参考之前的 LTR
- 直播：网络丢包后，可以基于 LTR 恢复而不需要等待新的 I 帧

### 4.3 参考帧列表

H.264/H.265 使用两个参考帧列表：
- **List0**：主要包含前向参考帧
- **List1**：主要包含后向参考帧

编码时通过 `ref_idx` 指定使用列表中的哪一帧。

## 5. 残差信号与后续流程

### 5.1 残差的特性

```
残差 = 原始块 - 预测块
```

一个好的预测意味着**残差接近零**。

```
原始块              预测块              残差
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│128 129 130 131│ - │128 128 129 130│ = │  0   1   1   1│
│129 130 131 132│   │129 129 130 131│   │  0   1   1   1│
│130 131 132 133│   │130 130 131 132│   │  0   1   1   1│
│131 132 133 134│   │131 131 132 133│   │  0   1   1   1│
└──────────────┘   └──────────────┘   └──────────────┘
                                       大部分是小值！
```

### 5.2 残差为何易压缩

1. **数值范围小**：集中在零附近
2. **能量集中**：大部分残差为零或接近零
3. **低熵**：符号分布不均匀，熵编码效率高

### 5.3 后续处理流程

```
残差信号
    │
    ▼
┌─────────┐
│ 变换编码 │ ← 空间域 → 频域（能量集中）
└────┬────┘
     │
     ▼
┌─────────┐
│  量化   │ ← 去除不重要的高频分量
└────┬────┘
     │
     ▼
┌─────────┐
│ 熵编码  │ ← 无损压缩到接近理论极限
└────┬────┘
     │
     ▼
  码流输出
```
