## 1. 移动端 GPU 架构与优化思维

### 1.1 TBR/TBDR 架构回顾

**结论先行**：移动端 GPU 普遍采用 TBR（Tile-Based Rendering）或 TBDR（Tile-Based Deferred Rendering）架构，其核心思想是将屏幕划分为小块（Tile，通常 16×16 或 32×32 像素），利用片上高速 SRAM（Tile Memory）完成渲染，仅在 Tile 完成后一次性写回外部内存。

```
┌─────────────────────────────────────────────────────────────────┐
│                  TBDR 架构核心优势                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   传统 IMR 架构                    TBDR 架构                      │
│   ┌─────────────┐                 ┌─────────────┐                │
│   │  外部内存    │                 │  Tile Memory│                │
│   │  (高延迟)    │                 │  (片上 SRAM)│                │
│   │             │                 │             │                │
│   │  Color/Depth│  ←频繁读写→     │  临时存储    │                │
│   │  Buffer     │                 │  高速低功耗  │                │
│   └─────────────┘                 └──────┬──────┘                │
│                                          │                       │
│                                          ▼ (Tile 完成后)          │
│                                    ┌─────────────┐               │
│                                    │  外部内存    │               │
│                                    │  (仅1次写入) │               │
│                                    └─────────────┘               │
│                                                                  │
│   带宽节省: 60-80%                                               │
│   功耗降低: 40-60%                                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 移动端 GPU 的核心约束

移动端 GPU 面临三大硬约束，这些约束决定了优化方向：

| 约束维度 | 具体限制 | 对渲染的影响 |
|---------|---------|-------------|
| **带宽** | 10-50 GB/s（vs 桌面 500+ GB/s） | 内存访问成为首要瓶颈 |
| **功耗** | 整机功耗墙 5-10W，GPU 通常 < 3W | 算法需考虑能耗效率 |
| **热量** | 被动散热，热节流（Thermal Throttling） | 持续高负载导致降频 |

## 2. 渲染状态管理优化

### 2.1 状态变更的代价分析

**结论先行**：渲染状态变更是 CPU 端的主要开销来源。每次状态变更都可能导致 GPU 流水线刷新（Pipeline Flush）和驱动层验证，代价远高于单次 Draw Call。

| 状态类型 | 变更代价 | 典型耗时 | 优化策略 |
|---------|---------|---------|---------|
| **Shader Program** | 极高 | 50-200μs | 预编译、排序、PSO 缓存 |
| **Texture Binding** | 高 | 10-50μs | 纹理图集、绑定数组 |
| **Uniform/Constant** | 中 | 5-20μs | UBO/Constant Buffer、批量更新 |
| **Blend/Depth State** | 中 | 5-15μs | 状态排序、减少切换 |
| **Vertex Buffer** | 低 | 2-10μs | VAO/VBO 缓存、交错布局 |

```
┌─────────────────────────────────────────────────────────────────┐
│                  状态变更代价层级                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   代价高 ──────────────────────────────────────────────→ 代价低   │
│                                                                  │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│   │ Shader  │ │ Texture │ │ Uniform │ │  Blend  │ │ Vertex  │   │
│   │Program  │ │  Bind   │ │ Update  │ │  State  │ │ Buffer  │   │
│   │ 切换    │ │ 切换    │ │ 更新    │ │ 切换    │ │ 切换    │   │
│   └─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘   │
│                                                                  │
│   优化优先级: Shader > Texture > Uniform > Blend > Vertex        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 状态排序策略（State Sorting）

**结论先行**：通过将渲染对象按状态分组排序，可以显著减少状态切换次数。推荐排序键（Sort Key）设计：

$$
SortKey = (ShaderID \ll 24) \ | \ (MaterialID \ll 16) \ | \ (TextureID \ll 8) \ | \ Depth
$$

```
┌─────────────────────────────────────────────────────────────────┐
│                  状态排序策略对比                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   未排序 (原始顺序)              排序后 (按 Shader→Material)      │
│   ┌───────────────────┐         ┌───────────────────┐            │
│   │ Obj A: Shader 1   │         │ Obj A: Shader 1   │            │
│   │ Obj B: Shader 2   │         │ Obj C: Shader 1   │  ← 无切换   │
│   │ Obj C: Shader 1   │  ← 切换  │ Obj E: Shader 1   │  ← 无切换   │
│   │ Obj D: Shader 3   │  ← 切换  │ Obj B: Shader 2   │  ← 切换    │
│   │ Obj E: Shader 1   │  ← 切换  │ Obj D: Shader 3   │  ← 切换    │
│   └───────────────────┘         └───────────────────┘            │
│                                                                  │
│   Shader 切换: 4 次              Shader 切换: 2 次 (减少 50%)     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**排序代价 vs 收益分析**：

| 场景 | 对象数量 | 排序耗时 | 节省状态切换 | 净收益 |
|-----|---------|---------|-------------|-------|
| 简单场景 | < 100 | ~0.1ms | 少量 | 中性 |
| 中等场景 | 100-500 | ~0.3ms | 中等 | 正向 |
| 复杂场景 | 500+ | ~0.5ms | 大量 | 显著 |

### 2.3 OpenGL ES 状态管理

#### 减少 glUseProgram / glBindTexture 调用

```glsl
// 优化前: 每个对象单独设置
for (auto& obj : objects) {
    glUseProgram(obj.shaderProgram);        // 频繁切换
    glBindTexture(GL_TEXTURE_2D, obj.texture); // 频繁切换
    glUniformMatrix4fv(obj.mvpLoc, 1, GL_FALSE, obj.mvp);
    glDrawArrays(GL_TRIANGLES, 0, obj.vertexCount);
}

// 优化后: 按 Shader 分组，批量渲染
// 1. 按 Shader ID 排序对象
std::sort(objects.begin(), objects.end(), 
    [](const auto& a, const auto& b) { return a.shaderId < b.shaderId; });

// 2. 批量渲染
GLuint currentShader = 0;
GLuint currentTexture = 0;

for (auto& obj : objects) {
    if (obj.shaderProgram != currentShader) {
        glUseProgram(obj.shaderProgram);
        currentShader = obj.shaderProgram;
    }
    if (obj.texture != currentTexture) {
        glBindTexture(GL_TEXTURE_2D, obj.texture);
        currentTexture = obj.texture;
    }
    glUniformMatrix4fv(obj.mvpLoc, 1, GL_FALSE, obj.mvp);
    glDrawArrays(GL_TRIANGLES, 0, obj.vertexCount);
}
```

#### VAO（Vertex Array Object）缓存

```glsl
// VAO 缓存顶点状态，避免每帧重复配置
```

### 2.4 Metal 状态管理

#### Pipeline State Object 预编译与缓存

```swift
// Metal 的核心理念: 所有状态预编译为 PSO，运行时零开销绑定
class PipelineCache {
    private let device: MTLDevice
    private var cache: [PipelineKey: MTLRenderPipelineState] = [:]
    
    struct PipelineKey: Hashable {
        let vertexFunction: String
        let fragmentFunction: String
        let vertexDescriptor: MTLVertexDescriptor
        let colorPixelFormat: MTLPixelFormat
        let depthPixelFormat: MTLPixelFormat
        let blendEnabled: Bool
    }
    
    func getPipelineState(for key: PipelineKey) -> MTLRenderPipelineState? {
        if let cached = cache[key] {
            return cached
        }
        
        // 创建新的 PSO (应在加载阶段完成，非运行时)
        let descriptor = MTLRenderPipelineDescriptor()
        descriptor.vertexFunction = library.makeFunction(name: key.vertexFunction)
        descriptor.fragmentFunction = library.makeFunction(name: key.fragmentFunction)
        descriptor.vertexDescriptor = key.vertexDescriptor
        descriptor.colorAttachments[0].pixelFormat = key.colorPixelFormat
        descriptor.depthAttachmentPixelFormat = key.depthPixelFormat
        
        if key.blendEnabled {
            descriptor.colorAttachments[0].isBlendingEnabled = true
            descriptor.colorAttachments[0].sourceRGBBlendFactor = .sourceAlpha
            descriptor.colorAttachments[0].destinationRGBBlendFactor = .oneMinusSourceAlpha
        }
        
        guard let pso = try? device.makeRenderPipelineState(descriptor: descriptor) else {
            return nil
        }
        
        cache[key] = pso
        return pso
    }
}
```

#### Descriptor 复用策略

```swift
// 复用 RenderPassDescriptor 避免重复创建
class RenderPassDescriptorPool {
    private var availableDescriptors: [MTLRenderPassDescriptor] = []
    private let maxPoolSize = 10
    
    func acquireDescriptor() -> MTLRenderPassDescriptor {
        if let descriptor = availableDescriptors.popLast() {
            // 重置状态
            return descriptor
        }
        return MTLRenderPassDescriptor()
    }
    
    func releaseDescriptor(_ descriptor: MTLRenderPassDescriptor) {
        guard availableDescriptors.count < maxPoolSize else { return }
        availableDescriptors.append(descriptor)
    }
}

// 使用示例
let descriptor = descriptorPool.acquireDescriptor()
// 配置 descriptor...
let encoder = commandBuffer.makeRenderCommandEncoder(descriptor: descriptor)
// 渲染...
encoder?.endEncoding()
descriptorPool.releaseDescriptor(descriptor)
```

### 3.2 静态批处理（Static Batching）

**适用场景**：静态不移动的几何体（地形、建筑、道具）

**原理**：在离线或加载时将多个静态网格合并为单个顶点缓冲区

```cpp
// 静态批处理实现
class StaticBatch {
private:
    std::vector<Vertex> mergedVertices;
    std::vector<uint16_t> mergedIndices;
    GLuint vao, vbo, ibo;
    
public:
    // 添加静态对象到批次
    void AddStaticObject(const Mesh& mesh, const Transform& transform) {
        uint16_t baseVertex = mergedVertices.size();
        
        // 变换并合并顶点
        for (const auto& vertex : mesh.vertices) {
            Vertex transformed;
            transformed.position = transform * vertex.position;
            transformed.normal = transform.rotation * vertex.normal;
            transformed.uv = vertex.uv;
            mergedVertices.push_back(transformed);
        }
        
        // 合并索引 (偏移)
        for (const auto& index : mesh.indices) {
            mergedIndices.push_back(baseVertex + index);
        }
    }
    
    // 构建 GPU 缓冲区 (一次性)
    void Build() {
        glGenVertexArrays(1, &vao);
        glBindVertexArray(vao);
        
        glGenBuffers(1, &vbo);
        glBindBuffer(GL_ARRAY_BUFFER, vbo);
        glBufferData(GL_ARRAY_BUFFER, 
                     mergedVertices.size() * sizeof(Vertex),
                     mergedVertices.data(), GL_STATIC_DRAW);
        
        glGenBuffers(1, &ibo);
        glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, ibo);
        glBufferData(GL_ELEMENT_ARRAY_BUFFER,
                     mergedIndices.size() * sizeof(uint16_t),
                     mergedIndices.data(), GL_STATIC_DRAW);
        
        // 配置顶点属性...
    }
    
    // 渲染 (单个 Draw Call)
    void Render() {
        glBindVertexArray(vao);
        glDrawElements(GL_TRIANGLES, mergedIndices.size(), GL_UNSIGNED_SHORT, 0);
    }
};

// 使用示例: 将 100 个静态树木合并为 1 个 Draw Call
StaticBatch treeBatch;
for (auto& tree : trees) {
    treeBatch.AddStaticObject(tree.mesh, tree.transform);
}
treeBatch.Build();
// 每帧只需: treeBatch.Render();
```

**静态批处理优缺点**：

| 优点 | 缺点 |
|-----|------|
| 将 N 个 Draw Call 合并为 1 个 | 增加内存占用（顶点数据重复） |
| 运行时零 CPU 开销 | 无法单独剔除（视锥剔除粒度变大） |
| 适合大规模静态场景 | 材质必须相同 |

### 3.3 动态批处理（Dynamic Batching）

**适用场景**：小型动态对象（粒子、UI 元素、小道具）

**原理**：每帧在 CPU 端合并顶点数据，适合顶点数少、变化频繁的对象

```cpp
// 动态批处理 (OpenGL ES)
class DynamicBatch {
private:
    static constexpr size_t MAX_VERTICES = 4096;
    std::vector<Vertex> cpuBuffer;
    GLuint vbo;
    
public:
    void Begin() {
        cpuBuffer.clear();
        cpuBuffer.reserve(MAX_VERTICES);
    }
    
    // 添加动态对象 (每帧调用)
    bool AddObject(const Mesh& mesh, const Transform& transform) {
        if (cpuBuffer.size() + mesh.vertices.size() > MAX_VERTICES) {
            return false; // 批次已满
        }
        
        for (const auto& vertex : mesh.vertices) {
            Vertex transformed;
            transformed.position = transform * vertex.position;
            transformed.uv = vertex.uv;
            transformed.color = mesh.color; // 支持顶点色区分对象
            cpuBuffer.push_back(transformed);
        }
        return true;
    }
    
    void EndAndRender() {
        if (cpuBuffer.empty()) return;
        
        // 更新 GPU 缓冲区 (orphaning 策略)
        glBindBuffer(GL_ARRAY_BUFFER, vbo);
        glBufferData(GL_ARRAY_BUFFER, 
                     cpuBuffer.size() * sizeof(Vertex),
                     nullptr, GL_STREAM_DRAW); // orphan
        glBufferSubData(GL_ARRAY_BUFFER, 0,
                        cpuBuffer.size() * sizeof(Vertex),
                        cpuBuffer.data());
        
        glDrawArrays(GL_TRIANGLES, 0, cpuBuffer.size());
    }
};
```

### 3.4 实例化渲染（Instanced Rendering）

**适用场景**：大量相同几何、不同变换的对象（草地、粒子、士兵群）

**原理**：GPU 端复制几何，CPU 只需提交一次 Draw Call + 实例数据

```glsl
// Vertex Shader (GLSL ES 3.0)

layout(location = 0) in vec3 aPosition;
layout(location = 1) in vec3 aNormal;
layout(location = 2) in vec2 aTexCoord;

// 实例数据 (使用 Instanced Array)
layout(location = 3) in vec4 iModelRow0;  // 模型矩阵第1行
layout(location = 4) in vec4 iModelRow1;  // 模型矩阵第2行
layout(location = 5) in vec4 iModelRow2;  // 模型矩阵第3行
layout(location = 6) in vec4 iModelRow3;  // 模型矩阵第4行
layout(location = 7) in vec4 iColor;      // 实例颜色

uniform mat4 uViewProjection;

out vec3 vWorldPos;
out vec3 vNormal;
out vec2 vTexCoord;
out vec4 vColor;

void main() {

    vWorldPos = worldPos.xyz;
```

```cpp
// C++ 主机端 (OpenGL ES)
struct InstanceData {
    float modelMatrix[16];
    float color[4];
};

class InstancedRenderer {
    GLuint instanceVBO;
    std::vector<InstanceData> instanceData;
    
public:
    void RenderInstanced(const Mesh& mesh, const std::vector<Transform>& transforms) {
        // 准备实例数据
        instanceData.clear();
        for (const auto& t : transforms) {
            InstanceData data;
            t.GetMatrix(data.modelMatrix);
            data.color[0] = t.color.r;
            data.color[1] = t.color.g;
            data.color[2] = t.color.b;
            data.color[3] = t.color.a;
            instanceData.push_back(data);
        }
        
        // 更新实例缓冲区
        glBindBuffer(GL_ARRAY_BUFFER, instanceVBO);
        glBufferData(GL_ARRAY_BUFFER, 
                     instanceData.size() * sizeof(InstanceData),
                     instanceData.data(), GL_DYNAMIC_DRAW);
        
        // 配置实例属性 (每实例更新一次)
        GLsizei stride = sizeof(InstanceData);
        for (int i = 0; i < 4; i++) {
            glEnableVertexAttribArray(3 + i);
            glVertexAttribPointer(3 + i, 4, GL_FLOAT, GL_FALSE, stride,
                                 (void*)(i * sizeof(float) * 4));
            glVertexAttribDivisor(3 + i, 1); // 每 1 个实例更新
        }
        glEnableVertexAttribArray(7);
        glVertexAttribPointer(7, 4, GL_FLOAT, GL_FALSE, stride, (void*)(sizeof(float) * 16));
        glVertexAttribDivisor(7, 1);
        
        // 绑定网格 VAO
        glBindVertexArray(mesh.vao);
        
        // 实例化绘制: 1 个 Draw Call 绘制所有实例
        glDrawArraysInstanced(GL_TRIANGLES, 0, mesh.vertexCount, transforms.size());
    }
};
```

### 5.1 Overdraw 的定义与度量

$$
Overdraw = \frac{\text{实际执行的 Fragment Shader 调用次数}}{\text{屏幕像素数}}

### 5.2 前向渲染中的 Overdraw 控制

#### 前后排序（Front-to-Back Sorting）

对于非 TBDR 架构（或需要 Alpha Test 的情况），按深度从前到后排序可最大化 Early-Z 效率。

```cpp

    [](const RenderObject& a, const RenderObject& b) {
        // 从前到后排序 (距离相机近的优先)
        return a.distanceToCamera < b.distanceToCamera;

for (auto& obj : opaqueObjects) {
    obj.Render();

// 后渲染透明物体 (从后到前)
std::sort(transparentObjects.begin(), transparentObjects.end(),
    [](const RenderObject& a, const RenderObject& b) {
        // 从后到前排序 (距离相机远的优先)
        return a.distanceToCamera > b.distanceToCamera;
    });

}
```

### 5.5 视锥体裁剪（Frustum Culling）

```cpp
// 视锥体裁剪实现
struct Frustum {
    Plane planes[6]; // 左、右、上、下、近、远
};

bool Frustum::Intersects(const BoundingSphere& sphere) const {
    for (int i = 0; i < 6; i++) {
        float distance = planes[i].Distance(sphere.center);
        if (distance < -sphere.radius) {
            return false; // 完全在平面外侧
        }
    }
    return true;
}

// 场景遍历
void CullScene(const SceneNode& node, const Frustum& frustum) {
    if (!frustum.Intersects(node.bounds)) {
        return; // 完全在视锥外，跳过
    }
    
    if (node.IsLeaf()) {
        visibleObjects.push_back(node.object);
    } else {
        for (auto& child : node.children) {
            CullScene(child, frustum);
        }
    }
}
```

### 6.2 Load/Store Action 最佳实践

#### 具体场景配置表

| 附件类型 | Load Action | Store Action | 说明 |
|---------|-------------|--------------|------|
| **颜色附件 (最终显示)** | `.clear` | `.store` | 每帧清除，保存结果 |
| **颜色附件 (中间纹理)** | `.dontCare` | `.store` | 将被完全覆盖 |
| **Depth Buffer (临时)** | `.clear` | `.dontCare` | 不需要保留深度 |
| **Depth Buffer (阴影)** | `.clear` | `.store` | 需要用于后续阴影计算 |
| **Stencil Buffer** | `.clear` | `.dontCare` | 通常不需要保留 |
| **MSAA 颜色** | `.clear` | `.multisampleResolve` | 解析到非 MSAA 纹理 |
| **Memoryless 深度** | `.clear` | `.dontCare` | 仅 Tile Memory 存在 |

```swift
// Metal Render Pass 优化配置
func createOptimizedRenderPassDescriptor(
    colorTexture: MTLTexture,
    depthTexture: MTLTexture?
) -> MTLRenderPassDescriptor {
    let descriptor = MTLRenderPassDescriptor()
    
    // 颜色附件: 清除并存储
    descriptor.colorAttachments[0].texture = colorTexture
    descriptor.colorAttachments[0].loadAction = .clear
    descriptor.colorAttachments[0].clearColor = MTLClearColor(
        red: 0.1, green: 0.1, blue: 0.1, alpha: 1.0
    )
    descriptor.colorAttachments[0].storeAction = .store
    
    // 深度附件: 清除但不存储 (关键优化!)
    if let depthTexture = depthTexture {
        descriptor.depthAttachment.texture = depthTexture
        descriptor.depthAttachment.loadAction = .clear
        descriptor.depthAttachment.clearDepth = 1.0
        descriptor.depthAttachment.storeAction = .dontCare  // ⭐ 节省带宽
    }
    
    return descriptor
}
```

### 6.3 Memoryless 纹理

**原理**：Memoryless 纹理仅存在于 Tile Memory 中，不占用系统内存。

```swift
// Memoryless 深度纹理创建
func createMemorylessDepthTexture(width: Int, height: Int) -> MTLTexture? {
    let descriptor = MTLTextureDescriptor.texture2DDescriptor(
        pixelFormat: .depth32Float,
        width: width,
        height: height,
        mipmapped: false
    )
    
    // ⭐ 关键: 设置 storageMode 为 .memoryless
    descriptor.storageMode = .memoryless
    descriptor.usage = .renderTarget
    
    return device.makeTexture(descriptor: descriptor)
}

// 使用 Memoryless 纹理的 Render Pass
func renderWithMemorylessDepth(
    commandBuffer: MTLCommandBuffer,
    colorTexture: MTLTexture
) {
    let memorylessDepth = createMemorylessDepthTexture(
        width: colorTexture.width,
        height: colorTexture.height
    )!
    
    let renderPassDescriptor = MTLRenderPassDescriptor()
    renderPassDescriptor.colorAttachments[0].texture = colorTexture
    renderPassDescriptor.colorAttachments[0].loadAction = .clear
    renderPassDescriptor.colorAttachments[0].storeAction = .store
    
    renderPassDescriptor.depthAttachment.texture = memorylessDepth
    renderPassDescriptor.depthAttachment.loadAction = .clear
    renderPassDescriptor.depthAttachment.storeAction = .dontCare  // Memoryless 必须 .dontCare
    
    let encoder = commandBuffer.makeRenderCommandEncoder(
        descriptor: renderPassDescriptor
    )!
    
    // 渲染...
    
    encoder.endEncoding()
    // Memoryless 纹理在此自动丢弃，零内存开销!
}
```

```swift
    let descriptor = MTLRenderPassDescriptor()
    
    // 创建 MSAA 纹理 (Memoryless)
    let msaaTextureDescriptor = MTLTextureDescriptor.texture2DDescriptor(
        pixelFormat: resolveTexture.pixelFormat,
        width: resolveTexture.width,
        height: resolveTexture.height,
        mipmapped: false
    )
    msaaTextureDescriptor.textureType = .type2DMultisample
    msaaTextureDescriptor.sampleCount = sampleCount
    msaaTextureDescriptor.storageMode = .memoryless  // ⭐ On-Chip
    msaaTextureDescriptor.usage = .renderTarget
    
    let msaaTexture = device.makeTexture(descriptor: msaaTextureDescriptor)!
    
    // 配置 MSAA 渲染
    descriptor.colorAttachments[0].texture = msaaTexture
    descriptor.colorAttachments[0].resolveTexture = resolveTexture
    descriptor.colorAttachments[0].loadAction = .clear
    descriptor.colorAttachments[0].storeAction = .multisampleResolve  // ⭐ 片上解析
    descriptor.colorAttachments[0].clearColor = MTLClearColor(
        red: 0, green: 0, blue: 0, alpha: 1
    )
    
    // 深度附件同样 Memoryless
    let depthDescriptor = MTLTextureDescriptor.texture2DDescriptor(
        pixelFormat: .depth32Float,
        width: resolveTexture.width,
        height: resolveTexture.height,
        mipmapped: false
    )
    depthDescriptor.textureType = .type2DMultisample
    depthDescriptor.sampleCount = sampleCount
    depthDescriptor.storageMode = .memoryless
    depthDescriptor.usage = .renderTarget
    
    let msaaDepth = device.makeTexture(descriptor: depthDescriptor)!
    descriptor.depthAttachment.texture = msaaDepth
    descriptor.depthAttachment.loadAction = .clear
    descriptor.depthAttachment.storeAction = .dontCare
    
    return descriptor
}
```

**MSAA 性能对比**：

| 传统 MSAA (Resolve 到内存) | 4x | 高 | 桌面 GPU |
| On-Chip MSAA (Tile 内 Resolve) | 1x (几乎免费) | 高 | 移动端 TBDR |

```metal
// Programmable Blending: 在 Fragment Shader 中读取当前颜色
fragment float4 programmableBlend(
    VertexOut in [[stage_in]],
    texture2d<float> sourceTexture [[texture(0)]],
    sampler sourceSampler [[sampler(0)]],
    
    // ⭐ 读取当前 Framebuffer 颜色 (Tile Memory 内)
    texture2d<float, access::read_write> colorAttachment [[texture(1)]]
) {
    float4 sourceColor = sourceTexture.sample(sourceSampler, in.texCoord);
    
    // 读取当前颜色 (仅在 TBDR Tile Memory 内有效)
    float4 destColor = colorAttachment.read(uint2(in.position.xy));
    
    // 自定义混合 (例如 Soft Light)
    float4 result = softLightBlend(sourceColor, destColor);
    
    return result;
}
```

## 7. Shader 优化

### 7.1 精度选择

**结论先行**：移动端 GPU 的半精度（half/fp16）计算能力通常是全精度（float/fp32）的 2 倍，且功耗更低。合理选择精度是 Shader 优化的首要任务。

| 精度 | GLSL | MSL | 计算速度 | 功耗 | 适用场景 |
|-----|------|-----|---------|------|---------|
| **highp/float** | `highp` | `float` | 1x | 高 | 位置、深度、大数计算 |
| **mediump/half** | `mediump` | `half` | 2x | 低 | 颜色、UV、法线、大部分计算 |
| **lowp/fixed** | `lowp` | - | 4x | 最低 | 简单颜色插值 |

```glsl
// GLSL ES 精度声明
precision mediump float;  // 默认 mediump

// 需要高精度的地方显式声明
uniform highp mat4 uViewProjection;
attribute highp vec3 aPosition;

varying mediump vec2 vTexCoord;
varying mediump vec3 vNormal;
varying lowp vec4 vColor;
```

```metal
// Metal 精度选择
#include <metal_stdlib>
using namespace metal;

// 默认使用 half (fp16)
struct VertexOut {
    float4 position [[position]];  // 位置需要 float
    half3 worldNormal;              // 法线可用 half
    half2 texCoord;                 // UV 可用 half
    half4 color;                    // 颜色可用 half
};

fragment half4 fragmentShader(VertexOut in [[stage_in]]) {
    // 纹理采样返回 half4
    half4 texColor = baseColorMap.sample(sampler, in.texCoord);
    
    // 光照计算使用 half
    half3 lightDir = normalize(half3(1.0h, 1.0h, 1.0h));
    half NdotL = max(dot(in.worldNormal, lightDir), 0.0h);
    
    return half4(texColor.rgb * NdotL, texColor.a);
}
```

### 7.2 分支与条件语句的代价

**结论先行**：GPU 使用 SIMD 执行，分支会导致线程发散（Divergence），降低执行效率。

```glsl
// 反模式: 动态分支 (导致 SIMD 发散)
void main() {
    vec4 color;
    if (materialType == 0) {
        color = CalculatePBR();      // 部分线程执行
    } else if (materialType == 1) {
        color = CalculateBlinnPhong(); // 部分线程执行
    } else {
        color = CalculateLambert();    // 部分线程执行
    }
    fragColor = color;
}

// 优化方案 1: 静态分支 (基于 uniform，所有线程一致)
uniform int uMaterialType;

void main() {
    vec4 color;
    if (uMaterialType == 0) {        // 所有线程走同一路径
        color = CalculatePBR();
    } else {
        color = CalculateLambert();
    }
    fragColor = color;
}

// 优化方案 2: 避免分支，使用 mix/step
void main() {
    vec4 pbrColor = CalculatePBR();
    vec4 lambertColor = CalculateLambert();
    
    // 使用 mix 替代分支 (如果两个计算都不太重)
    vec4 color = mix(lambertColor, pbrColor, step(0.5, materialBlend));
    fragColor = color;
}

// 优化方案 3: Shader 变体 (Uber Shader)
#ifdef USE_PBR
    color = CalculatePBR();
#else
    color = CalculateLambert();
#endif
```

### 7.4 数学函数近似

| 函数 | 精确版本 | 近似版本 | 精度损失 | 速度提升 |
|-----|---------|---------|---------|---------|
| `pow(x, y)` | 内置 | `exp(y * log(x))` | 低 | 1.5x |
| `normalize(v)` | `v / length(v)` | `v * rsqrt(dot(v,v))` | 低 | 2x |
| `length(v)` | `sqrt(dot(v,v))` | `dot(v,v)` (比较用) | 高 | 3x |
| `smoothstep` | 内置 | `x*x*(3-2*x)` | 无 | 1.2x |
| `sin/cos` | 内置 | 泰勒级数近似 | 中 | 2x |

```glsl
// 快速归一化 (使用倒数平方根)
vec3 fastNormalize(vec3 v) {
    return v * inversesqrt(dot(v, v));
}

// 快速平滑步
float fastSmoothstep(float edge0, float edge1, float x) {
    float t = clamp((x - edge0) / (edge1 - edge0), 0.0, 1.0);
    return t * t * (3.0 - 2.0 * t);
}
```

### 9.3 常见反模式与修正方案

| 反模式 | 问题 | 修正方案 |
|-------|------|---------|
| **每帧创建 PSO** | 严重卡顿 | 加载时预编译并缓存 |
| **深度 Buffer storeAction = .store** | 浪费带宽 | 改为 .dontCare |
| **未压缩纹理** | 带宽爆炸 | 使用 ASTC/ETC2 |
| **无 Mipmap** | 缓存未命中 | 启用 Mipmap |
| **过度 Draw Call** | CPU 瓶颈 | 批处理、实例化 |
| **Alpha Test 滥用** | HSR 失效 | 改用 Alpha Blend 或 Alpha-to-Coverage |
| **动态分支** | SIMD 发散 | 静态分支或 Shader 变体 |
| **高精度滥用** | 计算浪费 | 使用 mediump/half |
| **多 Render Pass** | 带宽浪费 | 合并 Pass，使用 MRT |

## 总结

移动端图形渲染优化的核心在于**理解 TBDR 架构特性**，围绕**减少带宽**、**消除 Overdraw**、**充分利用 Tile Memory** 三大原则展开。

关键要点：

1. **架构理解是前提**：TBDR 的 Tile Memory 和 HSR 是移动端优化的核心机制
2. **带宽是首要瓶颈**：优化纹理压缩、Mipmap、Load/Store Action 的收益最高
3. **状态管理决定 CPU 开销**：排序、批处理、PSO 缓存是降低 CPU 负载的关键
4. **精度选择影响计算效率**：合理使用 half/mediump 可提升 2x 性能
5. **工具驱动优化**：使用 GPU Capture 定位瓶颈，数据驱动决策
