## 1. GPU 渲染管线概述

### 1.1 图形渲染管线的基本流程

**结论先行**：现代 GPU 渲染管线是一个高度并行化的数据流处理系统，将三维场景描述转换为二维像素输出。理解管线的每个阶段及其数据流动，是掌握渲染架构的基础。

图形渲染管线通常分为四个主要阶段：

| 阶段 | 处理内容 | 坐标空间 | 关键操作 |
|-----|---------|---------|---------|
| **应用阶段 (Application)** | 场景管理、视锥剔除、Draw Call 提交 | 世界空间 | CPU 端逻辑 |
| **几何阶段 (Geometry)** | 顶点变换、投影、裁剪、屏幕映射 | 模型 → 世界 → 观察 → 裁剪 → NDC → 屏幕 | Vertex Shader、Tessellation、Geometry Shader |
| **光栅化阶段 (Rasterization)** | 图元分解为片元、属性插值 | 屏幕空间 | 三角形遍历、透视校正插值 |
| **像素/片元阶段 (Fragment)** | 着色计算、深度测试、颜色混合 | 屏幕空间 | Fragment Shader、ROP (Render Output Unit) |

```

├─────────────────────────────────────────────────────────────────────────┤
```

**直观类比**：渲染管线就像一条现代化的汽车装配线——每个工作站（管线阶段）专注于特定任务，产品在传送带（数据流）上依次经过各站，最终成为成品（像素输出）。与装配线不同的是，GPU 管线可以同时处理成千上万个"产品"（顶点/片元）的并行处理。

### 1.2 为什么渲染架构如此重要

**带宽是瓶颈**。在现代 GPU 中，计算能力的增长速度远超内存带宽的增长速度。这意味着：**数据搬运比数据计算更昂贵**。

```
│         │                        ╭────╯                         │
│     100 ┤                  ╭────╯                               │

│      10 ┤      ╭────╯        ╭────╯                             │
│         │╭────╯        ╭────╯                                   │
│       1 ┼┴──────┬─────┴─────┬─────┬─────┬─────┬─────┬────→      │
│         2000   2005       2010  2015  2020  2025                │

│                                                                 │
```

渲染架构设计的核心目标就是：**最小化内存带宽消耗**。

### 1.3 主流 GPU 架构分类总览

现代 GPU 架构主要分为三大类：

| 架构 | 全称 | 代表 GPU | 核心特点 | 适用场景 |
|-----|------|---------|---------|---------|
| **IMR** | Immediate Mode Rendering | NVIDIA RTX、AMD Radeon | 逐三角形立即渲染，高并行度 | 桌面/工作站 |
| **TBR** | Tile-Based Rendering | ARM Mali (部分)、Qualcomm Adreno (部分) | 分块渲染，减少带宽 | 移动端 |
| **TBDR** | Tile-Based Deferred Rendering | Apple GPU、Imagination PowerVR | 分块+延迟渲染，消除 Overdraw | 移动端/嵌入式 |

## 2. IMR（Immediate Mode Rendering）

### 2.1 IMR 的工作原理

**结论先行**：IMR（即时渲染模式）是最传统的 GPU 架构，每个三角形一旦进入管线就立即走完整个渲染流程，包括光栅化和像素写入。这种"来一个处理一个"的方式简单直接，但会带来显著的内存带宽压力。

**直观类比**：IMR 就像快餐店的流水线——每个订单（三角形）到达后立即开始制作，完成后立即送出。优点是响应快、延迟低；缺点是高峰期（复杂场景）会造成拥堵（带宽瓶颈）。

### 2.2 IMR 的内存访问模式

IMR 架构中，Framebuffer（包括 Color Buffer 和 Depth Buffer）存储在**外部显存/内存**中。每个片元处理都需要读写这些缓冲区，导致大量内存访问。

```
┌─────────────────────────────────────────────────────────────────┐
│                  IMR 内存访问模式分析                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   场景: 1920×1080 分辨率，渲染一个复杂场景                        │
│                                                                 │
│   带宽消耗计算:                                                  │
│   ┌──────────────────────────────────────────────────────┐     │
│   │  Color Buffer: 1920 × 1080 × 4 bytes = 8.3 MB        │     │
│   │  Depth Buffer: 1920 × 1080 × 4 bytes = 8.3 MB        │     │
│   │  每帧基础占用: ~16.6 MB                               │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                 │
│   假设场景平均 Overdraw = 3x (每个像素被3个三角形覆盖)            │
│                                                                 │
│   每帧内存访问量:                                                │
│   ┌──────────────────────────────────────────────────────┐     │
│   │  Depth 读取: 1920 × 1080 × 3 × 4 bytes = 24.9 MB     │     │
│   │  Depth 写入: 1920 × 1080 × 3 × 4 bytes = 24.9 MB     │     │
│   │  Color 读取: 1920 × 1080 × 3 × 4 bytes = 24.9 MB     │     │
│   │  Color 写入: 1920 × 1080 × 3 × 4 bytes = 24.9 MB     │     │
│   │  纹理采样: 假设平均 100 MB                            │     │

│   │  总计: ~200 MB/帧                                    │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                 │
│   @ 60 FPS: 200 MB × 60 = 12 GB/s 的内存带宽需求                │
│   @ 144 FPS: 200 MB × 144 = 28.8 GB/s                          │
│                                                                 │
│   这只是基础渲染，实际游戏/应用可能更高!                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 IMR 的优缺点

| 优点 | 缺点 |
|-----|------|
| **延迟低** - 三角形立即处理，无需等待 | **带宽消耗大** - 频繁访问外部内存 |
| **实现简单** - 管线直接，易于理解 | **功耗高** - 内存访问是功耗大户 |
| **几何吞吐高** - 适合大规模几何场景 | **Overdraw 浪费** - 被遮挡像素仍被着色 |
| **灵活性强** - 支持复杂的渲染状态切换 | **不适合移动端** - 受限于移动设备的功耗和带宽 |

### 2.4 代表性 GPU

IMR 架构主要应用于桌面和高性能计算 GPU：

- **NVIDIA GeForce/RTX 系列**：从最早的 GeForce 256 到最新的 RTX 4090
- **AMD Radeon 系列**：从 Radeon HD 到 RX 7000 系列
- **Intel Arc 系列**：Intel 独立显卡
- **数据中心 GPU**：NVIDIA A100/H100、AMD MI 系列

## 3. TBR（Tile-Based Rendering）

### 3.1 Tiling 的基本思想

**结论先行**：TBR（分块渲染）的核心思想是将屏幕划分为小块（Tile，通常 16×16 或 32×32 像素），先将几何数据分块存储，再逐 Tile 渲染。这样可以将 Framebuffer 操作限制在**片上 SRAM**（Tile Memory）中，大幅减少外部内存访问。

**直观类比**：TBR 就像画家作画——传统方式（IMR）是随机在画布各处画几笔，每次都要走到画布前；TBR 则是将画布分成小块，一次专注完成一小块，减少走动（内存访问）次数。

```
┌─────────────────────────────────────────────────────────────────┐
│                    屏幕 Tile 划分示意图                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
```

### 3.2 TBR 的工作流程

TBR 分为两个主要阶段：**Binning Pass**（分块阶段）和 **Rendering Pass**（渲染阶段）。

### 3.3 On-Chip Tile Memory 的作用

Tile Memory 是 TBR 架构的核心优化点。它是 GPU 芯片上的高速 SRAM，访问速度比外部内存快 10-100 倍。

```
┌─────────────────────────────────────────────────────────────────┐
│                  Tile Memory 架构对比                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   IMR 架构 (无 Tile Memory)                                      │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   片元着色器 ──→ 深度测试 ──→ 颜色混合 ──→ 显存         │   │
│   │        │                                     ↑          │   │
│   │        └─────────────────────────────────────┘          │   │
│   │                    频繁读写外部内存                      │   │
│   │                    (高延迟/高功耗)                       │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│   TBR 架构 (有 Tile Memory)                                      │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   片元着色器 ──→ 深度测试 ──→ 颜色混合 ──→ Tile Memory  │   │
│   │        │                                     ↑          │   │
│   │        └─────────────────────────────────────┘          │   │
│   │                    片上 SRAM 访问                        │   │
│   │                    (低延迟/低功耗)                       │   │
│   │                                                         │   │
│   │   Tile 完成后:                                          │   │
│   │   Tile Memory ────────────→ 显存 (仅一次写入)           │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│   带宽节省分析:                                                  │
│   ┌──────────────────────────────────────────────────────┐     │
│   │                                                      │     │
│   │  IMR: 每个片元都要读写外部内存                        │     │
│   │       Overdraw 3x → 3次深度读取 + 3次深度写入        │     │
│   │                     3次颜色读取 + 3次颜色写入        │     │
│   │                                                      │     │
│   │  TBR: 所有操作在 Tile Memory 完成                     │     │
│   │       Overdraw 3x → 0次外部深度访问 (片上完成)       │     │
│   │                     0次外部颜色访问 (片上完成)       │     │
│   │       最终: 1次颜色写入外部内存                      │     │
│   │                                                      │     │
│   │  带宽节省: 约 60-80%                                 │     │
│   │                                                      │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 TBR 的优缺点

| 优点 | 缺点 |
|-----|------|
| **带宽节省** - 减少 60-80% 内存带宽 | **延迟增加** - 需要 Binning Pass |
| **功耗降低** - 片上 SRAM 访问更节能 | **内存占用** - 需要存储 Tile List |
| **适合移动端** - 低功耗场景优势明显 | **几何遍历两次** - Vertex Shader 执行两次 |
| **MSAA 几乎免费** - 在 Tile Memory 中解析 | **不适合大规模几何** - Binning 开销随几何增长 |

### 3.5 代表性 GPU

TBR 架构主要应用于移动端 GPU：

- **ARM Mali** (部分型号)：Mali-G 系列采用 TBR 或混合架构
- **Qualcomm Adreno** (部分)：Adreno 早期型号

## 4. TBDR（Tile-Based Deferred Rendering）

### 4.1 TBDR = TBR + HSR

**结论先行**：TBDR（分块延迟渲染）在 TBR 的基础上增加了 **HSR（Hidden Surface Removal，隐藏面消除）** 阶段。HSR 在光栅化后、Fragment Shader 执行前进行，**只对可见片元执行 Fragment Shader**，彻底消除 Overdraw 的着色浪费。

**直观类比**：TBDR 就像装修房子时先规划好哪些墙面会被家具挡住——只给可见的墙面刷漆，被挡住的墙面完全不处理，而不是刷完漆再被遮住。

```
┌─────────────────────────────────────────────────────────────────┐
│              TBR vs TBDR 关键区别                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   TBR (分块渲染)                                                 │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐     │
│   │ Vertex  │───→│ Tiling  │───→│Rasterize│───→│Fragment │     │
│   │ Process │    │         │    │         │    │ Shader  │     │
│   └─────────┘    └─────────┘    └─────────┘    └────┬────┘     │
│                                                      │          │
│                                                      ▼          │
│                                                 ┌─────────┐     │
│                                                 │  Tile   │     │
│                                                 │ Resolve │     │
│                                                 └─────────┘     │
│                                                                 │
│   问题: 所有片元都执行 Fragment Shader，包括被遮挡的            │
│                                                                 │
│   ─────────────────────────────────────────────────────────     │
│                                                                 │
│   TBDR (分块延迟渲染)                                            │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐     │
│   │ Vertex  │───→│ Tiling  │───→│Rasterize│───→│   HSR   │     │
│   │ Process │    │         │    │         │    │ (隐面消除)│     │
│   └─────────┘    └─────────┘    └─────────┘    └────┬────┘     │
│                                                      │          │
│                                                      ▼          │
│                                                 ┌─────────┐     │
│                                                 │Fragment │     │
│                                                 │ Shader  │     │
│                                                 │(仅可见)  │     │
│                                                 └────┬────┘     │
│                                                      │          │
│                                                      ▼          │
│                                                 ┌─────────┐     │
│                                                 │  Tile   │     │
│                                                 │ Resolve │     │
│                                                 └─────────┘     │
│                                                                 │
│   优势: 只有可见片元才执行 Fragment Shader，彻底消除 Overdraw   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 HSR（隐藏面消除）的原理

HSR 的核心思想是：**在 Fragment Shader 之前确定每个像素位置的最靠前片元**。

```
┌─────────────────────────────────────────────────────────────────┐
│                    HSR 工作原理详解                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   场景: 三个重叠的三角形投射到同一个 Tile                         │
│                                                                 │
│   深度值 (越小越靠前):                                           │
│   - 三角形 A (红色):   Z = 0.8 (最远)                           │
│   - 三角形 B (绿色):   Z = 0.5 (中间)                           │
│   - 三角形 C (蓝色):   Z = 0.3 (最近)                           │
│                                                                 │
│   传统 Early-Z 方案:                                             │
│   ┌──────────────────────────────────────────────────────┐     │
│   │                                                      │     │
│   │  1. 光栅化三角形 A → 生成片元 (Z=0.8)                │     │
│   │     → 深度测试通过 → 执行 Fragment Shader            │     │
│   │     → 写入 Color/Depth                             │     │
│   │                                                      │     │
│   │  2. 光栅化三角形 B → 生成片元 (Z=0.5)                │     │
│   │     → 深度测试通过 (0.5 < 0.8) → 执行 FS             │     │
│   │     → 写入 Color/Depth                             │     │
│   │                                                      │     │
│   │  3. 光栅化三角形 C → 生成片元 (Z=0.3)                │     │
│   │     → 深度测试通过 (0.3 < 0.5) → 执行 FS             │     │
│   │     → 写入 Color/Depth                             │     │
│   │                                                      │     │
│   │  结果: 3个片元都执行了 FS，2个被覆盖 (浪费!)          │     │
│   │                                                      │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                 │
│   TBDR HSR 方案:                                                 │
│   ┌──────────────────────────────────────────────────────┐     │
│   │                                                      │     │
│   │  1. 光栅化所有三角形 → 生成所有片元                   │     │
│   │     但不执行 Fragment Shader!                        │     │
│   │     只记录: 位置(X,Y) + 深度(Z) + 三角形ID            │     │
│   │                                                      │     │
│   │  2. HSR 阶段 (硬件实现)                               │     │
│   │     对每个像素位置，找出最小深度 (Z=0.3)              │     │
│   │     标记: 只有三角形 C 的片元可见                     │     │
│   │                                                      │     │
│   │  3. 执行 Fragment Shader                              │     │
│   │     只有三角形 C 的片元执行 FS                        │     │
│   │                                                      │     │
│   │  结果: 1个片元执行 FS，0 浪费                         │     │
│   │                                                      │     │
│   └──────────────────────────────────────────────────────┘     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 HSR 与 Early-Z 的区别

| 特性 | Early-Z | HSR (TBDR) |
|-----|---------|-----------|
| **执行时机** | 光栅化后，FS 前 | 批量处理，延迟到 FS 前 |
| **消除范围** | 部分 Overdraw | 完全消除 Overdraw |
| **依赖条件** | 需要按深度排序 | 无需排序，硬件自动处理 |
| **Shader 修改深度** | 失效 | 失效 |
| **实现复杂度** | 较简单 | 较复杂，需要 Tile 存储 |

### 4.4 TBDR 的完整工作流程

```
┌─────────────────────────────────────────────────────────────────┐
│                  TBDR 完整工作流程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   阶段 1: Vertex Processing (顶点处理)                           │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  输入: 顶点缓冲区                                          │   │
│   │  处理: Vertex Shader (变换、动画、光照)                    │   │
│   │  输出: 裁剪空间顶点位置 + 顶点属性                          │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│   阶段 2: Tiling / Binning (分块)                                │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  输入: 变换后的三角形                                      │   │
│   │  处理: 确定每个三角形覆盖哪些 Tile                         │   │
│   │  输出: Tile List (每个 Tile 的三角形列表)                  │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│   阶段 3: Rasterization (光栅化)                                 │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  输入: Tile 内的三角形                                     │   │
│   │  处理: 生成片元 (位置、深度、属性插值)                     │   │
│   │  注意: 此时不执行 Fragment Shader!                        │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│   阶段 4: HSR (隐藏面消除) ⭐ 关键步骤                            │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  输入: Tile 内所有片元                                     │   │
│   │  处理: 对每个像素位置，找出最靠前片元                      │   │
│   │  输出: 可见片元列表                                        │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│   阶段 5: Fragment Processing (片元处理)                         │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  输入: 可见片元 (仅!)                                      │   │
│   │  处理: Fragment Shader (纹理、光照、着色)                  │   │
│   │  输出: 片元颜色 + 深度                                     │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│   阶段 6: Tile Resolve (Tile 回写)                               │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  输入: Tile Memory 中的颜色/深度数据                       │   │
│   │  处理: 写入外部 Framebuffer                                │   │
│   │  注意: 每个 Tile 只写一次!                                 │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.4 Load/Store Action 最佳实践

| 附件类型 | Load Action | Store Action | 说明 |
|---------|-------------|--------------|------|
| **颜色附件 (最终显示)** | `.clear` | `.store` | 每帧清除，保存结果 |
| **颜色附件 (中间纹理)** | `.dontCare` | `.store` | 将被完全覆盖 |
| **Depth Buffer (临时)** | `.clear` | `.dontCare` | 不需要保留深度 |
| **Depth Buffer (阴影)** | `.clear` | `.store` | 需要用于后续阴影计算 |
| **MSAA 颜色** | `.clear` | `.multisampleResolve` | 解析到非 MSAA 纹理 |
| **Memoryless 深度** | `.clear` | `.dontCare` | 仅 Tile Memory 存在 |

```swift
// 最佳实践示例：优化的 Render Pass 配置
func createOptimizedRenderPassDescriptor(
    colorTexture: MTLTexture,
    depthTexture: MTLTexture?
) -> MTLRenderPassDescriptor {
    let descriptor = MTLRenderPassDescriptor()
    
    // 颜色附件：每帧清除，需要显示
    descriptor.colorAttachments[0].texture = colorTexture
    descriptor.colorAttachments[0].loadAction = .clear
    descriptor.colorAttachments[0].clearColor = MTLClearColor(
        red: 0.1, green: 0.1, blue: 0.1, alpha: 1.0
    )
    descriptor.colorAttachments[0].storeAction = .store
    
    // 深度附件：每帧清除，不需要保留
    if let depthTexture = depthTexture {
        descriptor.depthAttachment.texture = depthTexture
        descriptor.depthAttachment.loadAction = .clear
        descriptor.depthAttachment.clearDepth = 1.0
        descriptor.depthAttachment.storeAction = .dontCare  // ⭐ 节省带宽
    }
    
    return descriptor
}
```

### 5.7 Programmable Blending

Apple GPU 支持 Programmable Blending，允许 Fragment Shader 读取当前像素的颜色值，实现复杂混合效果。

```metal
// Programmable Blending 示例
#include <metal_stdlib>
using namespace metal;

fragment float4 programmableBlendShader(
    VertexOut in [[stage_in]],
    texture2d<float> sourceTexture [[texture(0)]],
    sampler sourceSampler [[sampler(0)]],
    
    // ⭐ 颜色附件作为纹理输入 (Programmable Blending)
    texture2d<float, access::read_write> colorAttachment [[texture(1)]]
)
{
    float4 sourceColor = sourceTexture.sample(sourceSampler, in.texCoord);
    
    // 读取当前 Framebuffer 中的颜色
    float4 destColor = colorAttachment.read(uint2(in.position.xy));
    
    // 自定义混合 (例如：Photoshop 混合模式)
    // 这里实现 "Overlay" 混合模式
    float4 result;
    for (int i = 0; i < 3; i++) {
        if (destColor[i] < 0.5) {
            result[i] = 2.0 * sourceColor[i] * destColor[i];
        } else {
            result[i] = 1.0 - 2.0 * (1.0 - sourceColor[i]) * (1.0 - destColor[i]);
        }
    }
    result.a = sourceColor.a;
    
    // 写回 (在 TBDR 中，这仍然在 Tile Memory 内完成)
    return result;
}
```

```
│                            │ × Projection Matrix (P)            │
│                            ▼                                    │
│   裁剪空间 (Clip Space)                                          │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  经过投影后的齐次坐标                                      │   │
│   │  v_clip = P × v_view = P × V × M × v_model               │   │
│   │         = MVP × v_model                                  │   │

│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 各变换矩阵的数学定义

**Model Matrix（模型矩阵）**：将顶点从模型空间变换到世界空间

$$
M = T \cdot R \cdot S
$$

其中：
- $S$：缩放矩阵
- $R$：旋转矩阵（可以是 $R_x \cdot R_y \cdot R_z$）
- $T$：平移矩阵

**View Matrix（视图矩阵）**：将顶点从世界空间变换到观察空间

使用 LookAt 方法构造：

$$
V = \begin{bmatrix} 
s_x & s_y & s_z & -\mathbf{s} \cdot \mathbf{eye} \\
u_x & u_y & u_z & -\mathbf{u} \cdot \mathbf{eye} \\
-f_x & -f_y & -f_z & \mathbf{f} \cdot \mathbf{eye} \\
0 & 0 & 0 & 1 
\end{bmatrix}
$$

其中：
- $\mathbf{f} = normalize(target - eye)$（前方向）
- $\mathbf{s} = normalize(cross(\mathbf{f}, up))$（右方向）
- $\mathbf{u} = cross(\mathbf{s}, \mathbf{f})$（上方向）

**Projection Matrix（投影矩阵）**：将顶点从观察空间变换到裁剪空间

透视投影：

$$
P = \begin{bmatrix} 
\frac{1}{aspect \cdot tan(fov/2)} & 0 & 0 & 0 \\
0 & \frac{1}{tan(fov/2)} & 0 & 0 \\
0 & 0 & -\frac{f+n}{f-n} & -\frac{2fn}{f-n} \\
0 & 0 & -1 & 0 
\end{bmatrix}
$$

### 6.3 透视除法与视口变换

**透视除法**：将齐次坐标转换为 NDC

$$
\mathbf{v}_{NDC} = \begin{bmatrix} x_{clip}/w_{clip} \\ y_{clip}/w_{clip} \\ z_{clip}/w_{clip} \end{bmatrix}
$$

**视口变换**：将 NDC 映射到屏幕像素坐标

$$
x_{screen} = \frac{x_{NDC} + 1}{2} \cdot width + x_{viewport}
$$

$$
y_{screen} = \frac{y_{NDC} + 1}{2} \cdot height + y_{viewport}
$$

```metal
    float logDepth = log(w / near) / log(far / near);
    return logDepth;
}

// 反向深度示例
float calculateReverseZ(float linearDepth, float near, float far) {
    // 将 [near, far] 映射到 [1, 0] 而非 [0, 1]
    return (far - linearDepth) / (far - near);
}
```

### 7.5 纹理与 Buffer 的最佳实践

```swift
// 纹理存储模式选择指南
func createTextureWithOptimalStorageMode(
    width: Int,
    height: Int,
    pixelFormat: MTLPixelFormat,
    usage: MTLTextureUsage
) -> MTLTexture? {
    let descriptor = MTLTextureDescriptor.texture2DDescriptor(
        pixelFormat: pixelFormat,
        width: width,
        height: height,
        mipmapped: false
    )
    descriptor.usage = usage
    
    // 根据用途选择存储模式
    if usage.contains(.renderTarget) {
        // 渲染目标：如果是临时附件，使用 memoryless
        // 否则使用 private
        descriptor.storageMode = .private
    } else if usage.contains(.shaderRead) && !usage.contains(.shaderWrite) {
        // 只读纹理：使用 private，GPU 可以优化布局
        descriptor.storageMode = .private
    } else {
        // 需要 CPU 访问：使用 shared
        descriptor.storageMode = .shared
    }
    
    return device.makeTexture(descriptor: descriptor)
}
```

### 7.6 MSAA 优化（On-Chip MSAA Resolve）

TBDR 架构使得 MSAA 解析可以在 Tile Memory 中完成，几乎免费。

```swift
// MSAA 优化配置
func createMSAARenderPassDescriptor(
    resolveTexture: MTLTexture,
    sampleCount: Int = 4
) -> MTLRenderPassDescriptor {
    let descriptor = MTLRenderPassDescriptor()
    
    // 创建 MSAA 纹理 (memoryless!)
    let msaaTextureDescriptor = MTLTextureDescriptor.texture2DDescriptor(
        pixelFormat: resolveTexture.pixelFormat,
        width: resolveTexture.width,
        height: resolveTexture.height,
        mipmapped: false
    )
    msaaTextureDescriptor.textureType = .type2DMultisample
    msaaTextureDescriptor.sampleCount = sampleCount
    msaaTextureDescriptor.storageMode = .memoryless  // ⭐ 关键!
    msaaTextureDescriptor.usage = .renderTarget
    
    let msaaTexture = device.makeTexture(descriptor: msaaTextureDescriptor)!
    
    // 配置颜色附件
    descriptor.colorAttachments[0].texture = msaaTexture
    descriptor.colorAttachments[0].resolveTexture = resolveTexture
    descriptor.colorAttachments[0].loadAction = .clear
    descriptor.colorAttachments[0].storeAction = .multisampleResolve  // ⭐ 片上解析
    
    // 深度附件 (memoryless)
    let depthDescriptor = MTLTextureDescriptor.texture2DDescriptor(
        pixelFormat: .depth32Float,
        width: resolveTexture.width,
        height: resolveTexture.height,
        mipmapped: false
    )
    depthDescriptor.textureType = .type2DMultisample
    depthDescriptor.sampleCount = sampleCount
    depthDescriptor.storageMode = .memoryless
    depthDescriptor.usage = .renderTarget
    
    let msaaDepth = device.makeTexture(descriptor: depthDescriptor)!
    descriptor.depthAttachment.texture = msaaDepth
    descriptor.depthAttachment.loadAction = .clear
    descriptor.depthAttachment.storeAction = .dontCare
    
    return descriptor
}
```

### 7.8 常见反模式与修正方案

| 反模式 | 问题 | 修正方案 |
|-------|------|---------|
| **Depth Buffer storeAction = .store** | 不必要的内存写入 | 改为 `.dontCare` |
| **每物体一个 Render Pass** | 频繁 Tile Flush | 合并到单个 Render Pass |
| **在 FS 中采样 colorAttachment** | 强制 Flush | 使用 Programmable Blending 或 MRT |
| **LoadAction = .load 颜色附件** | 不必要的内存读取 | 改为 `.clear` 或 `.dontCare` |
| **未使用 Memoryless 的临时附件** | 浪费内存和带宽 | 设置 `storageMode = .memoryless` |
| **MSAA 后手动解析** | 额外的内存操作 | 使用 `storeAction = .multisampleResolve` |
| **频繁切换 Pipeline State** | 增加开销 | 按 Pipeline 排序渲染对象 |
| **Fragment Shader 中分支发散** | 降低 SIMD 效率 | 避免在 uniform 控制的分支中做大量工作 |
