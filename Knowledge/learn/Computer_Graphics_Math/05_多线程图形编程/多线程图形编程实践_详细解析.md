## 1. 图形 API 多线程模型概述

### 1.1 为什么图形编程需要多线程

**结论先行**：现代图形渲染涉及 CPU 端繁重的场景准备工作和 GPU 端的并行执行，单线程架构已成为性能瓶颈。多线程图形编程的核心目标是**最大化 CPU 与 GPU 的并行度**，将 CPU 从"等待 GPU"的状态中解放出来。

#### 渲染帧的工作拆分

一个典型的渲染帧包含以下阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│                    渲染帧流水线时序图                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  CPU: [场景更新]→[视锥剔除]→[排序]→[命令编码]→[提交]→[等待]    │
│       │         │       │      │       │      │                │
│       ▼         ▼       ▼      ▼       ▼      ▼                │
│  GPU:          [顶点处理]→[光栅化]→[片段处理]→[帧缓冲]         │
│                                                                 │
│  单线程问题: CPU 编码命令时 GPU 空闲，GPU 执行时 CPU 等待        │
│                                                                 │
│  多线程优化: CPU 编码第N+1帧时，GPU 执行第N帧                   │
│            [CPU: N+1]───────────────→                           │
│            [GPU:     N]───────────────→                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**CPU 端任务**（可多线程并行）：

| 任务类型 | 计算复杂度 | 并行潜力 | 典型耗时 |
|---------|-----------|---------|---------|
| 场景图遍历与更新 | O(n) | 高 | 1-3ms |
| 视锥剔除（Frustum Culling） | O(n) | 极高 | 0.5-2ms |
| 遮挡剔除（Occlusion Culling） | O(n log n) | 中 | 1-4ms |
| 渲染状态排序 | O(n log n) | 中 | 0.5-1ms |
| 命令编码（Command Encoding） | O(n) | 极高 | 2-8ms |
| Uniform Buffer 更新 | O(n) | 高 | 0.5-2ms |

**GPU 端任务**（硬件并行）：
- 顶点处理、光栅化、片段着色、帧缓冲写入

#### 单线程瓶颈分析

```
┌─────────────────────────────────────────────────────────────────┐
│                    单线程 vs 多线程 CPU 利用率                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  单线程模式（CPU 利用率 ~30%）:                                  │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                     │

│  │ (3ms)   │    │ (8ms)   │    │ (3ms)   │                     │
│  └─────────┘    └─────────┘    └─────────┘                     │
│       ▲                              ▲                         │
│       └──────── 11ms/帧 ────────────┘                          │
│                                                                 │
│  多线程模式（CPU 利用率 ~80%）:                                  │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                     │
│  │ 线程1   │───→│ 线程1   │───→│ 线程1   │  (主渲染线程)       │
│  │ 编码命令 │    │ 编码命令 │    │ 编码命令 │                     │
│  └─────────┘    └─────────┘    └─────────┘                     │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                     │
│  │ 线程2   │───→│ 线程2   │───→│ 线程2   │  (剔除线程)         │
│  │ 视锥剔除 │    │ 视锥剔除 │    │ 视锥剔除 │                     │
│  └─────────┘    └─────────┘    └─────────┘                     │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                     │
│  │ 线程3   │───→│ 线程3   │───→│ 线程3   │  (资源加载线程)     │
│  │ 加载纹理 │    │ 加载纹理 │    │ 加载纹理 │                     │
│  └─────────┘    └─────────┘    └─────────┘                     │
│                                                                 │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**直观类比**：单线程渲染就像一个人既要炒菜（CPU 编码）又要等烤箱（GPU 执行），期间只能干等。多线程渲染则像餐厅厨房——有人专门备菜（剔除）、有人专门炒菜（编码）、有人专门看管烤箱（同步），各工序流水线作业。

### 1.2 主流图形 API 的线程模型对比

**结论先行**：OpenGL ES 诞生于单核时代，线程支持有限；Metal 和 Vulkan 从设计之初就考虑多核 CPU，提供原生的多线程命令编码能力。

| 特性 | OpenGL ES | Metal | Vulkan |
|-----|-----------|-------|--------|
| **设计年代** | 1992 (OpenGL 1.0) | 2014 | 2016 |
| **线程模型** | 全局状态机 | 无状态命令编码 | 无状态命令编码 |
| **上下文绑定** | 单线程绑定 | 无绑定要求 | 无绑定要求 |
| **命令编码** | 单线程（隐式） | 多线程并行 | 多线程并行 |
| **资源线程安全** | 共享上下文有限支持 | 显式同步 | 显式同步 |
| **API 复杂度** | 低 | 中 | 高 |
| **CPU 开销** | 高（驱动验证） | 低 | 低 |

#### Vulkan：完全多线程

Vulkan 提供比 Metal 更细粒度的多线程控制：
- **Command Pool 每线程独立**：避免分配时的线程竞争
- **Secondary Command Buffer**：可并行录制，主 Buffer 统一提交
- **显式同步原语**：Semaphore、Fence、Event 精细控制

## 2. Metal 多线程命令编码

### 2.1 Metal 命令提交架构

**结论先行**：Metal 的命令提交遵循 `MTLCommandQueue → MTLCommandBuffer → MTLCommandEncoder` 三层架构。理解哪些对象线程安全、哪些不安全，是避免崩溃和竞态条件的关键。

#### 命令提交层次结构

```

│              Metal 命令提交层次结构                              │

│  ┌─────────────────────────────────────────────────────────┐   │

│  │  - 创建 Buffer/Texture                                   │   │

│       │                                                         │
│       ▼                                                         │
│  Layer 2: CommandQueue (线程安全)                               │
```

#### 线程安全性详细表格

| 对象/操作 | 线程安全 | 说明 |
|---------|---------|------|
| **MTLDevice** | ✅ 安全 | 可跨线程创建资源 |
| **MTLCommandQueue** | ✅ 安全 | 可多线程创建 CommandBuffer |
| **MTLCommandBuffer** | ⚠️ 部分 | 创建安全，但每个 Buffer 只能单线程编码 |
| **MTLRenderCommandEncoder** | ❌ 不安全 | 必须单线程创建和使用 |
| **MTLComputeCommandEncoder** | ❌ 不安全 | 必须单线程创建和使用 |
| **MTLBuffer** | ⚠️ 需同步 | GPU 读取时 CPU 不应写入 |
| **MTLTexture** | ⚠️ 需同步 | 同上 |
| **MTLRenderPipelineState** | ✅ 安全 | 不可变对象，创建后只读 |
| **MTLComputePipelineState** | ✅ 安全 | 不可变对象，创建后只读 |
| **MTLDepthStencilState** | ✅ 安全 | 不可变对象 |
| **MTLSamplerState** | ✅ 安全 | 不可变对象 |
| **commit()** | ✅ 安全 | 可从任意线程提交 |
| **waitUntilCompleted()** | ⚠️ 阻塞 | 会阻塞调用线程 |

### 2.2 并行命令编码（Parallel Render Command Encoder）

**结论先行**：`MTLParallelRenderCommandEncoder` 允许在单个 Render Pass 内将渲染命令拆分到多个线程并行编码，适用于大规模几何体或分块渲染场景。

#### MTLParallelRenderCommandEncoder 使用方式

```swift
// Swift 示例：并行渲染命令编码
class ParallelRenderer {
    private let device: MTLDevice
    private let commandQueue: MTLCommandQueue
    private let pipelineState: MTLRenderPipelineState
    
    func renderParallel(view: MTKView, scene: Scene) {
        guard let commandBuffer = commandQueue.makeCommandBuffer(),
              let renderPassDescriptor = view.currentRenderPassDescriptor else {
            return
        }
        
        // 1. 创建并行渲染编码器
        guard let parallelEncoder = commandBuffer.makeParallelRenderCommandEncoder(
                descriptor: renderPassDescriptor) else {
            return
        }
        
        // 2. 定义渲染区域划分（例如：屏幕分为4个tile）
        let tileCount = 4
        let dispatchGroup = DispatchGroup()
        
        for tileIndex in 0..<tileCount {
            dispatchGroup.enter()
            
            // 每个 tile 在一个独立线程编码
            DispatchQueue.global().async { [weak self] in
                guard let self = self else {
                    dispatchGroup.leave()
                    return
                }
                
                // 3. 获取子编码器（每个线程一个）
                let subEncoder = parallelEncoder.makeRenderCommandEncoder()
                
                // 4. 设置共享的渲染状态
                subEncoder.setRenderPipelineState(self.pipelineState)
                
                // 5. 设置 viewport（限制到当前 tile 区域）
                let viewport = self.calculateViewportForTile(tileIndex, tileCount: tileCount)
                subEncoder.setViewport(viewport)
                
                // 6. 编码该 tile 的绘制命令
                self.encodeDrawCallsForTile(subEncoder, tileIndex: tileIndex, scene: scene)
                
                // 7. 结束子编码器
                subEncoder.endEncoding()
                
                dispatchGroup.leave()
            }
        }
        
        // 8. 等待所有 tile 编码完成
        dispatchGroup.wait()
        
        // 9. 结束并行编码器
        parallelEncoder.endEncoding()
        
        // 10. 提交
        commandBuffer.present(view.currentDrawable!)
        commandBuffer.commit()
    }
    
    private func calculateViewportForTile(_ tileIndex: Int, tileCount: Int) -> MTLViewport {
        // 简化示例：水平分割屏幕
        let width = Double(viewportWidth) / Double(tileCount)

                          width: width, height: Double(viewportHeight),
                          znear: 0.0, zfar: 1.0)
    }
```

### 2.3 多 CommandBuffer 并行提交

**结论先行**：除了 Parallel Render Encoder，Metal 还支持完全独立的 CommandBuffer 在多线程并行编码，通过 `enqueue` 控制提交顺序，通过 `commit` 触发执行。

#### 多线程创建 CommandBuffer 的模式

```swift
// Swift 示例：多线程并行编码多个 CommandBuffer
class MultiBufferRenderer {
    private let device: MTLDevice
    private let commandQueue: MTLCommandQueue
    
    // 使用信号量控制并发数，避免内存爆炸
    private let encodingSemaphore = DispatchSemaphore(value: 3)
    
    func renderWithMultipleBuffers(scene: Scene, view: MTKView) {
        guard let drawable = view.currentDrawable else { return }
        
        // 将场景拆分为多个渲染任务
        let renderTasks = splitSceneIntoTasks(scene)
        var commandBuffers: [MTLCommandBuffer] = []
        let lock = NSLock()
        
        let dispatchGroup = DispatchGroup()
        
        for task in renderTasks {
            encodingSemaphore.wait()  // 控制并发数
            dispatchGroup.enter()
            
            DispatchQueue.global().async { [weak self] in
                defer {
                    self?.encodingSemaphore.signal()
                    dispatchGroup.leave()
                }
                
                guard let self = self else { return }
                
                // 每个线程创建自己的 CommandBuffer
                guard let commandBuffer = self.commandQueue.makeCommandBuffer() else {
                    return
                }
                
                // 设置调试标签
                commandBuffer.label = "RenderTask-\(task.id)"
                
                // 编码该任务的命令
                self.encodeTask(task, into: commandBuffer, view: view)
                
                // 收集 CommandBuffer
                lock.lock()
                commandBuffers.append(commandBuffer)
                lock.unlock()
            }
        }
        
        // 等待所有编码完成
        dispatchGroup.wait()
        
        // 按依赖顺序提交
        submitInOrder(commandBuffers, finalDrawable: drawable)
    }
    
    private func encodeTask(_ task: RenderTask, into buffer: MTLCommandBuffer, view: MTKView) {
        // 创建渲染编码器
        guard let encoder = buffer.makeRenderCommandEncoder(
            descriptor: view.currentRenderPassDescriptor!) else {
            return
        }
        
        // 编码任务特定的绘制命令
        encoder.setRenderPipelineState(task.pipelineState)
        encoder.setVertexBuffer(task.vertexBuffer, offset: 0, index: 0)
        encoder.drawPrimitives(type: .triangle, vertexStart: 0, vertexCount: task.vertexCount)
        
        encoder.endEncoding()
    }
    
    private func submitInOrder(_ buffers: [MTLCommandBuffer], finalDrawable: CAMetalDrawable) {
        // 按 ID 排序确保正确顺序
        let sortedBuffers = buffers.sorted { 
            ($0.label ?? "") < ($1.label ?? "") 
        }
        
        // 使用 enqueue 确保 GPU 执行顺序
        for (index, buffer) in sortedBuffers.enumerated() {
            if index == sortedBuffers.count - 1 {
                // 最后一个 buffer 负责 present
                buffer.present(finalDrawable)
            }
            buffer.commit()
        }
    }
}
```

```cpp
class GLMultiThreadContext {
private:
    EGLDisplay display;
    EGLConfig config;
    EGLContext mainContext;
    EGLContext loaderContext;
    EGLSurface surface;
    
public:
    bool initialize() {
        // 1. 获取默认显示
        display = eglGetDisplay(EGL_DEFAULT_DISPLAY);
        eglInitialize(display, nullptr, nullptr);
        
        // 2. 选择配置
        const EGLint attribs[] = {
            EGL_RENDERABLE_TYPE, EGL_OPENGL_ES3_BIT,
            EGL_SURFACE_TYPE, EGL_WINDOW_BIT,
            EGL_BLUE_SIZE, 8,
            EGL_GREEN_SIZE, 8,
            EGL_RED_SIZE, 8,
            EGL_DEPTH_SIZE, 24,
            EGL_NONE
        };
        EGLint numConfigs;
        eglChooseConfig(display, attribs, &config, 1, &numConfigs);
        
        // 3. 创建主渲染上下文
        const EGLint contextAttribs[] = {
            EGL_CONTEXT_CLIENT_VERSION, 3,
            EGL_NONE
        };
        mainContext = eglCreateContext(display, config, EGL_NO_CONTEXT, contextAttribs);
        
        // 4. 创建共享的加载上下文（共享主上下文的所有数据）
        loaderContext = eglCreateContext(display, config, mainContext, contextAttribs);
        
        return mainContext != EGL_NO_CONTEXT && loaderContext != EGL_NO_CONTEXT;
    }
    
    // 在主渲染线程调用
    bool makeMainContextCurrent(EGLSurface draw, EGLSurface read) {
        return eglMakeCurrent(display, draw, read, mainContext);
    }
    
    // 在资源加载线程调用
    bool makeLoaderContextCurrent(EGLSurface draw, EGLSurface read) {
        // 加载线程可以使用 pbuffer surface（离屏）
        return eglMakeCurrent(display, draw, read, loaderContext);
    }
    
    // 解绑上下文（线程结束前必须调用）
    bool clearCurrentContext() {
        return eglMakeCurrent(display, EGL_NO_SURFACE, EGL_NO_SURFACE, EGL_NO_CONTEXT);
    }
};
```

```cpp
class GLResourceLoader {
private:
    EGLDisplay display;
    EGLContext loaderContext;
    EGLSurface pbufferSurface;
    std::thread loaderThread;
    std::queue<std::function<void()>> taskQueue;
    std::mutex queueMutex;
    std::condition_variable condition;
    bool running = true;
    
public:
    // 在主线程初始化
    void initialize(EGLDisplay disp, EGLContext sharedContext, EGLConfig config) {
        display = disp;
        
        // 创建离屏 surface（PBuffer）
        const EGLint pbufferAttribs[] = {
            EGL_WIDTH, 1,
            EGL_HEIGHT, 1,
            EGL_NONE
        };
        pbufferSurface = eglCreatePbufferSurface(display, config, pbufferAttribs);
        
        // 创建共享上下文
        const EGLint contextAttribs[] = {
            EGL_CONTEXT_CLIENT_VERSION, 3,
            EGL_NONE
        };
        loaderContext = eglCreateContext(display, config, sharedContext, contextAttribs);
        
        // 启动加载线程
        loaderThread = std::thread(&GLResourceLoader::loaderLoop, this);
    }
    
    // 提交异步加载任务
    void loadTextureAsync(const std::string& path, GLuint& outTexture) {
        std::lock_guard<std::mutex> lock(queueMutex);
        taskQueue.push([this, path, &outTexture]() {
            outTexture = loadTextureInternal(path);
        });
        condition.notify_one();
    }
    
private:
    void loaderLoop() {
        // 绑定加载上下文到当前线程
        eglMakeCurrent(display, pbufferSurface, pbufferSurface, loaderContext);
        
        while (running) {
            std::function<void()> task;
            {
                std::unique_lock<std::mutex> lock(queueMutex);
                condition.wait(lock, [this] { return !taskQueue.empty() || !running; });
                if (!running) break;
                task = std::move(taskQueue.front());
                taskQueue.pop();
            }
            task();  // 执行加载任务
        }
        
        // 清理
        eglMakeCurrent(display, EGL_NO_SURFACE, EGL_NO_SURFACE, EGL_NO_CONTEXT);
    }
    
    GLuint loadTextureInternal(const std::string& path) {
        // 在加载线程中：加载图片、创建纹理、上传数据
        // 创建的纹理自动与主上下文共享
        GLuint texture;
        glGenTextures(1, &texture);
        glBindTexture(GL_TEXTURE_2D, texture);
        // ... 上传纹理数据
        return texture;
    }
};
```

```cpp
// C++: 使用 PBO 异步纹理上传
class AsyncTextureUploader {
private:
    GLuint pboIds[2];  // 双缓冲 PBO
    int currentPBO = 0;
    
public:
    void initialize() {
        // 创建两个 PBO 用于双缓冲
        glGenBuffers(2, pboIds);
        
        // 预分配 PBO 内存（例如 4K 纹理）
        for (int i = 0; i < 2; i++) {
            glBindBuffer(GL_PIXEL_UNPACK_BUFFER, pboIds[i]);
            glBufferData(GL_PIXEL_UNPACK_BUFFER, 4096 * 4096 * 4, 
                        nullptr, GL_STREAM_DRAW);
        }
        glBindBuffer(GL_PIXEL_UNPACK_BUFFER, 0);
    }
    
    // 异步上传纹理
    void uploadTextureAsync(const void* imageData, int width, int height, 
                           GLuint textureId) {
        int pboIndex = currentPBO;
        currentPBO = (currentPBO + 1) % 2;
        int nextPBO = currentPBO;
        
        // 1. 绑定当前 PBO 并映射内存
        glBindBuffer(GL_PIXEL_UNPACK_BUFFER, pboIds[pboIndex]);
        
        // 2. 将图像数据复制到 PBO（DMA 传输，非阻塞）
        void* mappedBuffer = glMapBufferRange(GL_PIXEL_UNPACK_BUFFER, 0,
                                              width * height * 4,
                                              GL_MAP_WRITE_BIT | GL_MAP_UNSYNCHRONIZED_BIT);
        if (mappedBuffer) {
            memcpy(mappedBuffer, imageData, width * height * 4);
            glUnmapBuffer(GL_PIXEL_UNPACK_BUFFER);
        }
        
        // 3. 绑定纹理并设置像素源为 PBO（0 表示从 PBO 偏移 0 读取）
        glBindTexture(GL_TEXTURE_2D, textureId);
        glTexSubImage2D(GL_TEXTURE_2D, 0, 0, 0, width, height,
                       GL_RGBA, GL_UNSIGNED_BYTE, nullptr);  // nullptr = 从 PBO 读取
        
        // 4. 异步处理下一个 PBO（如果上一帧的数据已准备好）
        glBindBuffer(GL_PIXEL_UNPACK_BUFFER, pboIds[nextPBO]);
        // 这里可以开始准备下一帧数据...
        
        // 5. 解绑 PBO
        glBindBuffer(GL_PIXEL_UNPACK_BUFFER, 0);
    }
};
```

```cpp
// C++: 后台 Shader 编译
class AsyncShaderCompiler {
private:
    std::thread compileThread;
    std::queue<CompileTask> taskQueue;
    std::mutex queueMutex;
    std::condition_variable condition;
    std::atomic<bool> running{true};
    
    struct CompileTask {
        std::string vertexSource;
        std::string fragmentSource;
        std::function<void(GLuint)> onComplete;  // 回调主线程
    };
    
public:
    void start(EGLDisplay display, EGLContext sharedContext, EGLConfig config) {
        compileThread = std::thread([this, display, sharedContext, config]() {
            // 创建离屏上下文
            EGLint pbufferAttribs[] = { EGL_WIDTH, 1, EGL_HEIGHT, 1, EGL_NONE };
            EGLSurface pbuffer = eglCreatePbufferSurface(display, config, pbufferAttribs);
            EGLint contextAttribs[] = { EGL_CONTEXT_CLIENT_VERSION, 3, EGL_NONE };
            EGLContext ctx = eglCreateContext(display, config, sharedContext, contextAttribs);
            eglMakeCurrent(display, pbuffer, pbuffer, ctx);
            
            while (running) {
                CompileTask task;
                {
                    std::unique_lock<std::mutex> lock(queueMutex);
                    condition.wait(lock, [this] { return !taskQueue.empty() || !running; });
                    if (!running) break;
                    task = std::move(taskQueue.front());
                    taskQueue.pop();
                }
                
                // 编译 Shader
                GLuint program = compileShader(task.vertexSource, task.fragmentSource);
                
                // 回调主线程（注意：这里只是传递 program ID，实际使用需要在主线程）
                if (task.onComplete) {
                    task.onComplete(program);
                }
            }
            
            eglMakeCurrent(display, EGL_NO_SURFACE, EGL_NO_SURFACE, EGL_NO_CONTEXT);
            eglDestroyContext(display, ctx);
            eglDestroySurface(display, pbuffer);
        });
    }
    
    void compileAsync(const std::string& vs, const std::string& fs,
                     std::function<void(GLuint)> callback) {
        std::lock_guard<std::mutex> lock(queueMutex);
        taskQueue.push({vs, fs, callback});
        condition.notify_one();
    }
    
private:
    GLuint compileShader(const std::string& vs, const std::string& fs) {
        GLuint vertexShader = glCreateShader(GL_VERTEX_SHADER);
        const char* vsSource = vs.c_str();
        glShaderSource(vertexShader, 1, &vsSource, nullptr);
        glCompileShader(vertexShader);
        
        GLuint fragmentShader = glCreateShader(GL_FRAGMENT_SHADER);
        const char* fsSource = fs.c_str();
        glShaderSource(fragmentShader, 1, &fsSource, nullptr);
        glCompileShader(fragmentShader);
        
        GLuint program = glCreateProgram();
        glAttachShader(program, vertexShader);
        glAttachShader(program, fragmentShader);
        glLinkProgram(program);
        
        glDeleteShader(vertexShader);
        glDeleteShader(fragmentShader);
        
        return program;
    }
};
```

```cpp
// C++: 使用 Fence 同步 GPU 完成
class GLSyncManager {
private:
    GLsync fence = nullptr;
    
public:
    // 在加载线程上传纹理后插入 Fence
    void insertFenceAfterUpload() {
        if (fence) {
            glDeleteSync(fence);
        }
        fence = glFenceSync(GL_SYNC_GPU_COMMANDS_COMPLETE, 0);
        glFlush();  // 确保命令提交到 GPU
    }
    
    // 在主线程等待 GPU 完成
    bool waitForGPUCompletion(uint64_t timeoutNanos = 1000000000) {  // 1秒
        if (!fence) return true;
        
        GLenum result = glClientWaitSync(fence, GL_SYNC_FLUSH_COMMANDS_BIT, timeoutNanos);
        
        switch (result) {
            case GL_ALREADY_SIGNALED:
                // GPU 已完成
                return true;
            case GL_CONDITION_SATISFIED:
                // 等待后 GPU 完成
                return true;
            case GL_TIMEOUT_EXPIRED:
                // 超时
                return false;
            case GL_WAIT_FAILED:
                // 错误
                return false;
            default:
                return false;
        }
    }
    
    // 非阻塞检查
    bool isGPUComplete() {
        if (!fence) return true;
        
        GLint status;
        GLsizei length;
        glGetSynciv(fence, GL_SYNC_STATUS, 1, &length, &status);
        return status == GL_SIGNALED;
    }
    
    ~GLSyncManager() {
        if (fence) {
            glDeleteSync(fence);
        }
    }
};
```

## 4. GPU-CPU 同步机制

### 4.1 同步的必要性与代价

**结论先行**：GPU-CPU 同步是性能杀手，每次同步都意味着 CPU 等待 GPU 完成工作，破坏了 CPU-GPU 并行性。理想的渲染管线应该实现「CPU 永远领先 GPU 1-2 帧」。

```
┌─────────────────────────────────────────────────────────────────┐
│              GPU-CPU 同步的代价                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  无同步（理想情况）:                                            │
│  CPU: [帧1编码]─→[帧2编码]─→[帧3编码]─→[帧4编码]─→...         │
│  GPU:          [帧1执行]─→[帧2执行]─→[帧3执行]─→...           │
│       ───────────────────────────────────────────────→         │
│       完全并行，CPU 利用率 100%                                 │
│                                                                 │
│  每帧同步（glFinish/waitUntilCompleted）:                       │
│  CPU: [帧1编码]─→[等待GPU]─→[帧2编码]─→[等待GPU]─→...         │
│  GPU:          [帧1执行]          [帧2执行]                    │
│       ───────────────────────────────────────────────→         │
│       CPU 等待 50% 时间，利用率 50%                             │
│                                                                 │
│  数据回读同步（read pixels）:                                   │
│  CPU: [帧1编码]─→[回读等待]─→[处理数据]─→[帧2编码]            │
│  GPU:          [帧1执行]─→[回读阻塞]                          │
│       ───────────────────────────────────────────────→         │
│       回读可能阻塞 GPU 数毫秒，严重影响性能                     │
│                                                                 │
│  同步类型代价对比:                                              │
│  ┌─────────────────────┬─────────────────┬─────────────────────┐│
│  │ 同步操作            │ 典型耗时        │ 使用场景            ││
│  ├─────────────────────┼─────────────────┼─────────────────────┤│
│  │ glFinish            │ 5-16ms          │ 仅在调试时使用      ││
│  │ waitUntilCompleted  │ 5-16ms          │ 截图、资源释放      ││
│  │ glReadPixels        │ 2-10ms          │ 避免，使用 PBO      ││
│  │ Fence Sync          │ 0.01-0.1ms      │ 推荐的轻量同步      ││
│  │ Triple Buffering    │ 0ms (无阻塞)    │ 默认推荐方案        ││

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 Triple Buffering（三重缓冲）

**结论先行**：三重缓冲通过增加一帧的延迟，换取 CPU 和 GPU 永不互相等待。这是移动端渲染的默认推荐配置。

#### 原理与实现

```
┌─────────────────────────────────────────────────────────────────┐
│              双缓冲 vs 三缓冲时序对比                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  双缓冲（CPU 可能等待 GPU）:                                    │
│                                                                 │
│  Buffer:   [A]      [B]      [A]      [B]      [A]             │
│            │        │        │        │        │               │
│  CPU:      [编码A]─→[编码B]─→[等待]─→[编码A]─→[编码B]          │
│  GPU:             [执行A]─→[执行B]─→[执行A]                    │
│                     ↑                 ↑                        │
│                   GPU忙             CPU等待                    │
│                                                                 │
│  帧延迟: 1 帧                                                   │
│  风险: 如果 GPU 执行慢，CPU 会阻塞等待                          │
│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  三缓冲（CPU 永不等待 GPU）:                                    │
│                                                                 │
│  Buffer:   [A]      [B]      [C]      [A]      [B]      [C]    │
│            │        │        │        │        │        │      │
│  CPU:      [编码A]─→[编码B]─→[编码C]─→[编码A]─→[编码B]─→[编码C]│
│  GPU:             [执行A]─→[执行B]─→[执行C]─→[执行A]           │
│                     ↑        ↑        ↑                        │
│                   三个缓冲区轮转，CPU 始终有可用的 Buffer       │
│                                                                 │
│  帧延迟: 2 帧（延迟增加一帧）                                   │
│  优势: CPU 和 GPU 完全并行，无阻塞                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 信号量控制（dispatch_semaphore / MTLEvent）

```swift
// Swift: 使用信号量实现 Triple Buffering
class TripleBufferRenderer {
    private let device: MTLDevice
    private let commandQueue: MTLCommandQueue
    private let maxFramesInFlight = 3
    private let frameSemaphore: DispatchSemaphore
    private var currentFrameIndex = 0
    
    // 三缓冲 Uniform Buffer
    private var uniformBuffers: [MTLBuffer] = []
    
    init() {
        device = MTLCreateSystemDefaultDevice()!
        commandQueue = device.makeCommandQueue()!
        frameSemaphore = DispatchSemaphore(value: maxFramesInFlight)
        
        // 创建三个 Uniform Buffer
        for _ in 0..<maxFramesInFlight {
            let buffer = device.makeBuffer(length: 1024, options: .storageModeShared)!
            uniformBuffers.append(buffer)
        }
    }
    
    func draw(view: MTKView) {
        // 1. 等待信号量（确保不超过 3 帧在飞行中）
        frameSemaphore.wait()
        
        // 2. 获取当前帧的 buffer
        let uniformBuffer = uniformBuffers[currentFrameIndex]
        
        // 3. 更新 Uniform 数据（CPU 写入）
        updateUniforms(buffer: uniformBuffer)
        
        // 4. 创建 CommandBuffer
        guard let commandBuffer = commandQueue.makeCommandBuffer() else { return }
        
        // 5. 编码渲染命令
        guard let encoder = commandBuffer.makeRenderCommandEncoder(
            descriptor: view.currentRenderPassDescriptor!) else { return }
        
        encoder.setVertexBuffer(uniformBuffer, offset: 0, index: 0)
        // ... 更多编码
        encoder.endEncoding()
        
        // 6. 设置完成回调（释放信号量）
        commandBuffer.addCompletedHandler { [weak self] _ in
            self?.frameSemaphore.signal()
        }
        
        // 7. 提交
        commandBuffer.present(view.currentDrawable!)
        commandBuffer.commit()
        
        // 8. 轮转到下一帧
        currentFrameIndex = (currentFrameIndex + 1) % maxFramesInFlight
    }
}
```

```cpp
// C++: OpenGL ES 使用 Fence 实现 Triple Buffering
class GLTripleBuffer {
private:
    static constexpr int MAX_FRAMES_IN_FLIGHT = 3;
    GLsync fences[MAX_FRAMES_IN_FLIGHT] = {};
    int currentFrame = 0;
    
public:
    void renderFrame() {
        // 1. 等待当前帧的 fence（确保 GPU 已完成使用该帧资源）
        if (fences[currentFrame]) {
            GLenum result = glClientWaitSync(fences[currentFrame], 
                                            GL_SYNC_FLUSH_COMMANDS_BIT,
                                            1000000000);  // 1秒超时
            if (result == GL_TIMEOUT_EXPIRED) {
                // 处理超时
            }
            glDeleteSync(fences[currentFrame]);
            fences[currentFrame] = nullptr;
        }
        
        // 2. 使用当前帧的 buffer 更新 uniform 数据
        updateUniforms(currentFrame);
        
        // 3. 渲染
        glBindBuffer(GL_UNIFORM_BUFFER, uniformBuffers[currentFrame]);
        glDrawArrays(...);
        
        // 4. 插入 fence（标记 GPU 完成时间）
        fences[currentFrame] = glFenceSync(GL_SYNC_GPU_COMMANDS_COMPLETE, 0);
        
        // 5. 轮转到下一帧
        currentFrame = (currentFrame + 1) % MAX_FRAMES_IN_FLIGHT;
    }
```

```swift
        // GBuffer 渲染（写入位置、法线、材质信息）
        encoder.setRenderPipelineState(gBufferPipeline)
        encoder.drawPrimitives(type: .triangle, vertexStart: 0, vertexCount: vertexCount)
        
        // 更新 fence，标记 GBuffer 写入完成
        encoder.updateFence(gBufferFence!)
    }
    
    func renderLightingPass(encoder: MTLRenderCommandEncoder) {
        // 等待 GBuffer 写入完成
        encoder.waitForFence(gBufferFence!)
        
        // 光照计算（读取 GBuffer）
        encoder.setRenderPipelineState(lightingPipeline)
        encoder.setFragmentTexture(gBufferPosition, index: 0)
        encoder.setFragmentTexture(gBufferNormal, index: 1)
        encoder.drawPrimitives(type: .triangle, vertexStart: 0, vertexCount: 3)
    }
```

```cpp
// C++: 使用帧标记追踪资源使用
class FrameIndexTracker {
private:
    static constexpr int MAX_FRAMES_IN_FLIGHT = 3;
    uint64_t currentFrameIndex = 0;
    std::array<uint64_t, MAX_FRAMES_IN_FLIGHT> frameCompletionMarkers{};
    
public:
    uint64_t getCurrentFrameIndex() const {
        return currentFrameIndex;
    }
    
    void advanceFrame() {
        currentFrameIndex++;
    }
    
    // 标记某帧已完成
    void markFrameCompleted(uint64_t frameIndex) {
        frameCompletionMarkers[frameIndex % MAX_FRAMES_IN_FLIGHT] = frameIndex;
    }
    
    // 检查某资源是否还在被 GPU 使用
    bool isResourceInUse(uint64_t lastUsedFrame) const {
        // 如果资源最近使用的帧与当前帧差距小于 MAX_FRAMES_IN_FLIGHT
        // 说明 GPU 可能还在使用
        return (currentFrameIndex - lastUsedFrame) < MAX_FRAMES_IN_FLIGHT;
    }
    
    // 安全释放资源
    void safeReleaseResource(uint64_t lastUsedFrame, std::function<void()> releaseAction) {
        if (!isResourceInUse(lastUsedFrame)) {
            releaseAction();
        } else {
            // 延迟到帧完成后再释放
            queueDeferredRelease(lastUsedFrame + MAX_FRAMES_IN_FLIGHT, releaseAction);
        }
    }
};
```

```swift
// Swift: 异步资源加载管线
class AsyncResourceLoader {
    private let device: MTLDevice
    private let loadingQueue = DispatchQueue(label: "resource.loading", qos: .utility)
    private var pendingResources: [String: ResourceLoadTask] = [:]
    private let lock = NSLock()
    
    struct ResourceLoadTask {
        let url: URL
        let completion: (MTLTexture?) -> Void
        var isCancelled = false
    }
    
    // 异步加载纹理
    func loadTextureAsync(from url: URL, completion: @escaping (MTLTexture?) -> Void) {
        let key = url.absoluteString
        
        lock.lock()
        // 检查是否已在加载中
        if pendingResources[key] != nil {
            lock.unlock()
            return
        }
        
        let task = ResourceLoadTask(url: url, completion: completion)
        pendingResources[key] = task
        lock.unlock()
        
        loadingQueue.async { [weak self] in
            guard let self = self else { return }
            
            // 1. 从磁盘加载数据
            guard let imageData = try? Data(contentsOf: url),
                  let image = UIImage(data: imageData),
                  let cgImage = image.cgImage else {
                self.completeTask(key: key, texture: nil)
                return
            }
            
            // 2. 创建纹理描述符
            let textureDescriptor = MTLTextureDescriptor.texture2DDescriptor(
                pixelFormat: .rgba8Unorm,
                width: cgImage.width,
                height: cgImage.height,
                mipmapped: true)
            
            // 3. 在主线程创建 Metal 资源（MTLDevice 线程安全）
            DispatchQueue.main.async {
                guard let texture = self.device.makeTexture(descriptor: textureDescriptor) else {
                    self.completeTask(key: key, texture: nil)
                    return
                }
                
                // 4. 上传纹理数据
                self.uploadTextureData(cgImage, to: texture)
                
                // 5. 完成任务
                self.completeTask(key: key, texture: texture)
            }
        }
    }
    
    private func completeTask(key: String, texture: MTLTexture?) {
        lock.lock()
        let task = pendingResources.removeValue(forKey: key)
        lock.unlock()
        
        if let task = task, !task.isCancelled {
            DispatchQueue.main.async {
                task.completion(texture)
            }
        }
    }
}
```

#### Buffer Pool：避免频繁分配

```swift
// Swift: Buffer Pool 实现
class BufferPool {
    private let device: MTLDevice
    private let bufferSize: Int
    private var availableBuffers: [MTLBuffer] = []
    private var inUseBuffers: Set<MTLBuffer> = []
    private let lock = NSLock()
    
    init(device: MTLDevice, bufferSize: Int, initialCapacity: Int = 10) {
        self.device = device
        self.bufferSize = bufferSize
        
        // 预创建 buffer
        for _ in 0..<initialCapacity {
            if let buffer = device.makeBuffer(length: bufferSize, options: .storageModeShared) {
                availableBuffers.append(buffer)
            }
        }
    }
    
    func acquire() -> MTLBuffer? {
        lock.lock()
        defer { lock.unlock() }
        
        if let buffer = availableBuffers.popLast() {
            inUseBuffers.insert(buffer)
            return buffer
        }
        
        // 池耗尽，创建新 buffer
        if let buffer = device.makeBuffer(length: bufferSize, options: .storageModeShared) {
            inUseBuffers.insert(buffer)
            return buffer
        }
        
        return nil
    }
    
    func release(_ buffer: MTLBuffer) {
        lock.lock()
        defer { lock.unlock() }
        
        if inUseBuffers.remove(buffer) != nil {
            availableBuffers.append(buffer)
        }
    }
}

// 使用示例
class PooledRenderer {
    private let uniformBufferPool: BufferPool
    
    func render() {
        // 获取 buffer
        guard let uniformBuffer = uniformBufferPool.acquire() else { return }
        
        // 使用 buffer 渲染
        updateUniforms(in: uniformBuffer)
        encodeDrawCalls(using: uniformBuffer)
        
        // 在 GPU 完成后归还 buffer
        commandBuffer.addCompletedHandler { [weak self] _ in
            self?.uniformBufferPool.release(uniformBuffer)
        }
    }
}
```

#### Texture Cache：纹理复用

```swift
// Swift: 纹理缓存实现 LRU 策略
class TextureCache {
    private let device: MTLDevice
    private var cache: [String: CachedTexture] = [:]
    private let lock = NSLock()
    private let maxCacheSize: Int
    
    struct CachedTexture {
        let texture: MTLTexture
        var lastUsedFrame: UInt64
        var referenceCount: Int
    }
    
    func getTexture(key: String, descriptor: MTLTextureDescriptor) -> MTLTexture? {
        lock.lock()
        defer { lock.unlock() }
        
        if let cached = cache[key] {
            // 更新使用信息
            cache[key]?.lastUsedFrame = currentFrame
            cache[key]?.referenceCount += 1
            return cached.texture
        }
        
        // 创建新纹理
        guard let texture = device.makeTexture(descriptor: descriptor) else {
            return nil
        }
        
        // 如果缓存已满，清理最久未使用的
        if cache.count >= maxCacheSize {
            evictLRU()
        }
        
        cache[key] = CachedTexture(
            texture: texture,
            lastUsedFrame: currentFrame,
            referenceCount: 1)
        
        return texture
    }
    
    private func evictLRU() {
        guard let oldestKey = cache.min(by: { $0.value.lastUsedFrame < $1.value.lastUsedFrame })?.key else {
            return
        }
        cache.removeValue(forKey: oldestKey)
    }
    
    func releaseTexture(key: String) {
        lock.lock()
        defer { lock.unlock() }
        
        if var cached = cache[key] {
            cached.referenceCount -= 1
            if cached.referenceCount <= 0 {
                // 可选：立即释放或延迟释放
                // cache.removeValue(forKey: key)
            }
            cache[key] = cached
        }
    }
}
```

```swift
// Swift: 使用 Aliasing 复用内存
class AliasingRenderer {
    private let device: MTLDevice
    private var heap: MTLHeap?
    private var gBufferPosition: MTLTexture?
    private var gBufferNormal: MTLTexture?
    private var lightingResult: MTLTexture?
    private var gBufferFence: MTLFence?
    
    func setupAliasedHeap() {
        // 计算需要的总大小（最大同时使用的资源）
        let gBufferSize = 1920 * 1080 * 4  // Position (RGBA32F)
        let normalSize = 1920 * 1080 * 4    // Normal (RGBA8)
        let lightingSize = 1920 * 1080 * 4  // Lighting (RGBA16F)
        
        // 创建 heap，启用 aliasing
        let descriptor = MTLHeapDescriptor()
        descriptor.size = gBufferSize + normalSize + max(lightingSize, 0)
        descriptor.storageMode = .private
        
        heap = device.makeHeap(descriptor: descriptor)
    }
    
    func renderWithAliasing(view: MTKView) {
        guard let commandBuffer = commandQueue.makeCommandBuffer() else { return }
        
        // 1. GBuffer Pass
        let gBufferEncoder = commandBuffer.makeRenderCommandEncoder(descriptor: gBufferDescriptor)!
        renderGBuffer(gBufferEncoder)
        
        // 标记 GBuffer 完成
        gBufferEncoder.updateFence(gBufferFence!)
        gBufferEncoder.endEncoding()
        
        // 2. Lighting Pass（复用 GBuffer 的内存）
        let lightingEncoder = commandBuffer.makeRenderCommandEncoder(descriptor: lightingDescriptor)!
        
        // 等待 GBuffer 完成后再复用内存
        lightingEncoder.waitForFence(gBufferFence!)
        
        // 现在可以安全地复用 GBuffer 的内存
        renderLighting(lightingEncoder)
        lightingEncoder.endEncoding()
    }
}
```

### 6.2 竞态条件

#### 多线程同时修改共享 Buffer

```swift
// ❌ 错误：多线程同时修改共享 Buffer
class UnsafeBufferAccess {
    private var sharedBuffer: MTLBuffer?
    
    func updateFromThreadA() {
        DispatchQueue.global().async {
            let ptr = self.sharedBuffer?.contents()
            // ❌ 线程A写入
            memcpy(ptr, dataA, size)
        }
    }
    
    func updateFromThreadB() {
        DispatchQueue.global().async {
            let ptr = self.sharedBuffer?.contents()
            // ❌ 线程B同时写入！
            memcpy(ptr, dataB, size)
        }
    }
}

// ✅ 正确：每个线程使用独立的 Buffer
class SafeBufferAccess {
    private var bufferPool: [MTLBuffer] = []
    private let lock = NSLock()
    
    func updateFromThreadA() {
        let buffer = acquireBuffer()
        let ptr = buffer.contents()
        memcpy(ptr, dataA, size)
        // 使用完成后释放回池
    }
    
    func updateFromThreadB() {
        let buffer = acquireBuffer()  // 获取不同的 buffer
        let ptr = buffer.contents()
        memcpy(ptr, dataB, size)
    }
    
    private func acquireBuffer() -> MTLBuffer {
        lock.lock()
        defer { lock.unlock() }
        // 从池中获取可用 buffer
        return bufferPool.popLast() ?? createNewBuffer()
    }
}
```

### 7.1 渲染线程 + 逻辑线程分离

```cpp
// C++: 双缓冲场景数据实现
template<typename T>
class DoubleBufferedData {
private:
    std::array<T, 2> buffers;
    std::atomic<int> writeIndex{0};
    std::atomic<int> readIndex{1};
    std::mutex swapMutex;
    
public:
    // 逻辑线程调用：获取写入缓冲
    T& getWriteBuffer() {
        return buffers[writeIndex.load()];
    }
    
    // 渲染线程调用：获取读取缓冲
    const T& getReadBuffer() const {
        return buffers[readIndex.load()];
    }
    
    // 帧边界交换缓冲
    void swap() {
        std::lock_guard<std::mutex> lock(swapMutex);
        int currentWrite = writeIndex.load();
        writeIndex.store(readIndex.load());
        readIndex.store(currentWrite);
    }
};

// 使用示例
struct SceneData {
    std::vector<Transform> objectTransforms;
    CameraData camera;
    LightData lights[MAX_LIGHTS];
};

class Engine {
    DoubleBufferedData<SceneData> sceneData;
    
    void logicThread() {
        while (running) {
            auto& writeBuffer = sceneData.getWriteBuffer();
            
            // 更新场景数据
            updateGameLogic(writeBuffer);
            updatePhysics(writeBuffer);
            updateAnimations(writeBuffer);
            
            // 等待渲染线程完成上一帧
            syncWithRenderThread();
            
            // 交换缓冲
            sceneData.swap();
        }
    }
    
    void renderThread() {
        while (running) {
            const auto& readBuffer = sceneData.getReadBuffer();
            
            // 从场景数据生成渲染命令
            encodeRenderCommands(readBuffer);
            
            // 提交 GPU
            submitToGPU();
        }
    }
};
```

```cpp
enum class RenderCommandType {
    SetViewport,
    SetScissor,
    SetRenderPipeline,
    SetVertexBuffer,
    SetIndexBuffer,
    SetUniformBuffer,
    SetTexture,
    Draw,
    DrawIndexed

struct RenderCommand {
    RenderCommandType type;
    union {
        struct { float x, y, width, height; } viewport;
        struct { uint32_t pipelineId; } setPipeline;
        struct { uint32_t bufferId; uint32_t slot; } setVertexBuffer;
        struct { uint32_t vertexCount; uint32_t instanceCount; } draw;
        // ... 其他命令参数
    } data;

// 命令录制器（逻辑线程使用）
class CommandRecorder {
private:
    std::vector<RenderCommand> commands;
    
public:
    void setViewport(float x, float y, float w, float h) {
        commands.push_back({
            RenderCommandType::SetViewport,
            .viewport = {x, y, w, h}
        });
    }
    
    void setRenderPipeline(uint32_t pipelineId) {
        commands.push_back({
            RenderCommandType::SetRenderPipeline,
            .setPipeline = {pipelineId}
        });
    }
    
    void draw(uint32_t vertexCount, uint32_t instanceCount = 1) {
        commands.push_back({
            RenderCommandType::Draw,
            .draw = {vertexCount, instanceCount}
        });
    }
    
    std::vector<RenderCommand> finishRecording() {
        return std::move(commands);
    }
};

class MetalCommandPlayback {
private:
    MTLDevice device;
    MTLRenderCommandEncoder encoder;
    ResourceManager resources;  // 管理 buffer/texture/pipeline
    
public:
    void playback(const std::vector<RenderCommand>& commands) {
        for (const auto& cmd : commands) {
            switch (cmd.type) {

                        cmd.data.viewport.x, cmd.data.viewport.y,

                }
                case RenderCommandType::SetRenderPipeline: {

                }
                case RenderCommandType::Draw: {
                    encoder.drawPrimitives(
                        .triangle,
                        vertexStart: 0,
                        vertexCount: cmd.data.draw.vertexCount,
                        instanceCount: cmd.data.draw.instanceCount
                    );
                    break;
                }
                // ... 其他命令
            }
        }
    }
};
```

```cpp
// C++: 简化版 Job System 与渲染集成

class JobSystem {
public:
    using JobFunction = std::function<void()>;
    
    struct Job {
        JobFunction func;
        std::atomic<int> unfinishedJobs{0};
        Job* parent = nullptr;
    };
    
    void initialize(uint32_t workerCount) {
        for (uint32_t i = 0; i < workerCount; i++) {
            workers.emplace_back([this] { workerLoop(); });
        }
    }
    
    void submit(Job* job) {
        {
            std::lock_guard<std::mutex> lock(queueMutex);
            jobQueue.push(job);
        }
        condition.notify_one();
    }
    
    void wait(Job* job) {
        while (job->unfinishedJobs.load() > 0) {
            // 帮助执行其他任务
            if (Job* nextJob = getNextJob()) {
                execute(nextJob);
            }
        }
    }
    
private:
    std::vector<std::thread> workers;
    std::queue<Job*> jobQueue;
    std::mutex queueMutex;
    std::condition_variable condition;
    
    void workerLoop() {
        while (true) {
            Job* job = getNextJob();
            if (job) {
                execute(job);
            }
        }
    }
    
    Job* getNextJob() {
        std::lock_guard<std::mutex> lock(queueMutex);
        if (jobQueue.empty()) return nullptr;
        Job* job = jobQueue.front();
        jobQueue.pop();
        return job;
    }
    
    void execute(Job* job) {
        job->func();
        
        if (job->parent) {
            job->parent->unfinishedJobs.fetch_sub(1);
        }
        job->unfinishedJobs.fetch_sub(1);
    }
};

// 渲染任务图示例
class RenderTaskGraph {
public:
    struct Task {
        std::string name;
        std::function<void()> execute;
        std::vector<Task*> dependencies;
        std::atomic<int> unfinishedDeps{0};
    };
    
    void addTask(const std::string& name, std::function<void()> exec) {
        Task* task = new Task{name, exec, {}, {}};
        tasks.push_back(task);
    }
    
    void addDependency(Task* before, Task* after) {
        after->dependencies.push_back(before);
        after->unfinishedDeps.fetch_add(1);
    }
    
    void submitToJobSystem(JobSystem& js) {
        for (auto* task : tasks) {
            JobSystem::Job* job = new JobSystem::Job{
                [task] {
                    // 等待依赖完成
                    for (auto* dep : task->dependencies) {
                        while (dep->unfinishedDeps.load() > 0) {
                            std::this_thread::yield();
                        }
                    }
                    
                    // 执行任务
                    task->execute();
                    task->unfinishedDeps.fetch_sub(1);
                },
                {0},
                nullptr
            };
            js.submit(job);
        }
    }
    
private:
    std::vector<Task*> tasks;
};
```

### Metal 多线程检查清单

```
□ CommandEncoder 是否只在单线程使用？
□ 是否使用 Triple Buffering 避免 CPU 等待？
□ Buffer/Texture 更新是否使用正确的同步机制？
□ PipelineState 是否在加载时预创建？
□ 是否使用信号量控制在飞帧数？
□ 跨线程资源访问是否加锁或使用原子操作？
□ 是否避免在渲染循环中动态分配资源？
□ 是否使用 Validation Layer 检查潜在问题？
```

### OpenGL ES 多线程检查清单

```
□ 每个线程是否有独立的 Context？
□ Context 是否正确绑定和解绑？
□ 共享资源是否使用 Fence Sync 同步？
□ 后台加载线程是否使用 PBuffer Surface？
□ 纹理上传是否使用 PBO 异步传输？
□ Shader 编译是否在后台线程完成？
□ 是否避免在主线程调用 glFinish？
```

### 性能优化检查清单

```
□ CPU 是否领先 GPU 1-2 帧？
□ 是否避免每帧 GPU 数据回读？
□ 是否使用 Buffer Pool 减少分配？
□ 是否启用 Metal Heap 和 Aliasing？
□ 是否使用 Parallel Render Encoder？
□ 是否对大量实例化渲染使用 ICB？
□ 是否使用 Instruments 分析 GPU 时间线？
```
