## 1. 案例一：大规模直播平台

### 1.1 场景描述

- **需求**：支持百万级并发的游戏/电商直播
- **延迟要求**：普通观众 3-10s，互动场景 1-3s
- **质量要求**：1080p60fps，自适应码率
- **DRM 需求**：付费内容需要加密

### 1.2 架构方案

```
┌──────────┐    RTMP     ┌──────────┐    内部协议    ┌──────────┐
│ OBS/编码器 │──────────→│ 源站服务器 │──────────────→│ 转码集群  │
└──────────┘   推流      │  (SRS)   │              │(FFmpeg)  │
                         └──────────┘              └──────────┘
                                                        │
                                            多码率 HLS/DASH 分片
                                                        │
                                                        ▼
                                                  ┌──────────┐
                                                  │CDN 边缘节点│ → 百万观众
                                                  │(HLS/DASH) │
                                                  └──────────┘
```

### 1.3 关键配置

**SRS 源站配置**：
```conf
listen 1935;
max_connections 1000;

vhost __defaultVhost__ {
    hls {
        enabled on;
        hls_fragment 3;        # 3秒分片（降低延迟）
        hls_window 30;         # 保留30秒的分片
        hls_path /tmp/hls;
    }
    
    http_remux {
        enabled on;
        mount [vhost]/[app]/[stream].flv;  # HTTP-FLV 低延迟拉流
    }
    
    transcode {
        enabled on;
        ffmpeg /usr/bin/ffmpeg;
        engine 720p {
            enabled on;
            vcodec libx264;
            vbitrate 2500;
            vfps 30;
            vwidth 1280;
            vheight 720;
            vpreset veryfast;
            acodec aac;
            abitrate 128;
            output rtmp://127.0.0.1/live/[stream]_720p;
        }
    }
}
```

**FFmpeg 多码率转码命令**：
```bash
ffmpeg -i rtmp://source/live/stream \
  -filter_complex "[0:v]split=3[v1][v2][v3]; \
    [v1]scale=1920:1080[v1out]; \
    [v2]scale=1280:720[v2out]; \
    [v3]scale=854:480[v3out]" \
  -map "[v1out]" -c:v:0 libx264 -b:v:0 5000k -preset veryfast \
  -map "[v2out]" -c:v:1 libx264 -b:v:1 2500k -preset veryfast \
  -map "[v3out]" -c:v:2 libx264 -b:v:2 1000k -preset veryfast \
  -map 0:a -c:a aac -b:a 128k \
  -f hls -hls_time 3 -master_pl_name master.m3u8 \
  -var_stream_map "v:0,a:0 v:1,a:0 v:2,a:0" \
  /output/stream_%v.m3u8
```

### 1.4 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 推流协议 | RTMP | 生态最成熟，OBS 原生支持 |
| 分发协议 | HLS + HTTP-FLV | HLS 兼容性最好，HTTP-FLV 低延迟 |
| 分片时长 | 3s | 平衡延迟和 CDN 缓存效率 |
| CDN | 多 CDN 切换 | 容灾和成本优化 |

## 2. 案例二：视频会议系统

### 2.1 场景描述

- **需求**：支持 2-100 人视频会议
- **延迟要求**：< 300ms
- **特性**：屏幕共享、文字聊天、录制
- **网络**：企业内网 + 远程公网混合

### 2.2 架构方案

```
┌──────────┐                    ┌──────────┐                    ┌──────────┐
│ 浏览器A   │ ←── WebRTC ───→  │   SFU    │ ←── WebRTC ───→  │ 浏览器B   │
│(Chrome)  │                   │(LiveKit/ │                    │(Safari)  │
└──────────┘                   │mediasoup)│                    └──────────┘
                                │          │
┌──────────┐ ←── WebRTC ───→  │   信令   │ ←── WebRTC ───→  ┌──────────┐
│ 移动端C   │                   │ WebSocket│                    │ 桌面端D   │
└──────────┘                   └──────────┘                    └──────────┘
                                     │
                              ┌──────┴──────┐
                              │   录制/直播   │
                              │  FFmpeg 转码  │
                              │   → HLS/S3   │
                              └─────────────┘
```

### 2.3 LiveKit SFU 配置示例

```yaml
# livekit.yaml
port: 7880
rtc:
  port_range_start: 50000
  port_range_end: 60000
  use_external_ip: true
  tcp_port: 7881         # TCP 降级端口
  
turn:
  enabled: true
  domain: turn.example.com
  tls_port: 5349
  udp_port: 3478

room:
  max_participants: 100
  empty_timeout: 300     # 空房间 5 分钟后关闭

media:
  # Simulcast 配置
  simulcast:
    enabled: true
    layers:
      - quality: HIGH
        width: 1920
        height: 1080
        bitrate: 3000000
      - quality: MEDIUM
        width: 1280
        height: 720
        bitrate: 1500000
      - quality: LOW
        width: 640
        height: 360
        bitrate: 500000
```

### 2.4 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 通信协议 | WebRTC | 唯一支持浏览器原生双向通信 |
| 架构 | SFU | 平衡质量和服务器成本 |
| Simulcast | 3 层 | 适配不同网络的参与者 |
| TURN | 开启 | 企业网络可能封 UDP |
| 录制 | 服务端录制 | 不占用参与者带宽 |

## 3. 案例三：跨区域远程制作

### 3.1 场景描述

- **需求**：将现场 4K 信号通过公网传回制作中心
- **延迟要求**：< 1s
- **质量**：广播级（4K@60fps, 30-50Mbps）
- **网络**：跨城市公网，RTT 30-80ms

### 3.2 架构方案

```
现场A ──SRT──→ 公网 ──→ 制作中心（SRT Listener）
(摄像机+编码器)                    │
                            ┌──────┘
                            ▼
现场B ──SRT──→ 公网 ──→ SRT 接收 → 切换台/导播
                            │
                            └──→ 录制 + 转码 → CDN → 观众
```

### 3.3 关键配置

**FFmpeg SRT 推流（现场端）**：
```bash
# 4K 60fps SRT 推流
ffmpeg -i /dev/video0 \
  -c:v libx264 -preset ultrafast -tune zerolatency \
  -b:v 35000k -maxrate 40000k -bufsize 70000k \
  -g 60 -keyint_min 60 \
  -c:a aac -b:a 256k \
  -f mpegts \
  "srt://production-center.example.com:9000?\
mode=caller&\
latency=500000&\
pbkeylen=32&\
passphrase=BroadcastSecret2026&\
maxbw=50000000&\
oheadbw=25"
```

**SRT 接收（制作中心）**：
```bash
# SRT Listener 接收并解码
ffplay "srt://:9000?mode=listener&latency=500000&passphrase=BroadcastSecret2026"

# 或者转为 SDI 输出到切换台
ffmpeg -i "srt://:9000?mode=listener&latency=500000" \
  -c:v rawvideo -pix_fmt uyvy422 \
  -f decklink "DeckLink 4K Extreme"
```

### 3.4 SRT 参数调优

| 参数 | 值 | 理由 |
|------|---|------|
| `latency` | 500ms | ≥ 4×RTT(80ms)，留足重传空间 |
| `passphrase` | 强密码 | AES-256 加密，防公网窃听 |
| `maxbw` | 50Mbps | 限制发送带宽 |
| `oheadbw` | 25% | 预留重传带宽 |
| `payloadsize` | 1316 | 适配 7×188(TS) = 1316 |

## 4. 案例四：超低延迟互动直播

### 4.1 场景描述

- **需求**：电商直播 + 实时互动（连麦、抢购倒计时同步）
- **延迟要求**：主播间连麦 < 500ms，观众 < 2s
- **规模**：10 万级观众

### 4.2 架构方案

```
主播A ←──WebRTC──→ ┌───────────────┐ ←──WebRTC──→ 连麦主播B
                    │  WebRTC SFU   │
                    │  (信令+路由)   │
                    └───────┬───────┘
                            │
                     WebRTC → RTMP 转换
                            │
                            ▼
                    ┌───────────────┐
                    │   媒体服务器    │
                    │   (SRS/NME)   │
                    └───────┬───────┘
                            │
                    LL-HLS 生成（2s Part）
                            │
                            ▼
                    ┌───────────────┐
                    │   CDN 分发     │ ──→ 10 万观众（LL-HLS 播放器）
                    └───────────────┘
```

### 4.3 关键配置

**SRS WebRTC → RTMP 转换**：
```conf
vhost __defaultVhost__ {
    rtc {
        enabled on;
        # WebRTC 推流
        rtmp_to_rtc on;
        rtc_to_rtmp on;
    }
    
    hls {
        enabled on;
        hls_fragment 2;
        hls_window 10;
    }
}
```

**观众端 hls.js 低延迟配置**：
```javascript
const hls = new Hls({
  lowLatencyMode: true,
  liveSyncDuration: 3,
  liveMaxLatencyDuration: 5,
  liveDurationInfinity: true,
  highBufferWatchdogPeriod: 1,
});
```

## 5. 案例五：大规模视频监控

### 5.1 场景描述

- **需求**：1000+ 路摄像头实时监控
- **延迟要求**：< 2s
- **存储**：30 天循环录制
- **回放**：任意时间段回放

### 5.2 架构方案

```
摄像头群组                      媒体服务器集群              客户端
┌─────────┐  RTSP/RTP     ┌──────────────┐           ┌──────────┐
│ IP Camera │──────────────→│   流媒体网关   │──HLS──→  │ Web 监控台│
│ (H.264)  │               │  (接入转换)   │           └──────────┘
└─────────┘               │              │──RTSP──→ ┌──────────┐
                           │   FFmpeg     │           │ VLC/专业  │
┌─────────┐  RTSP/RTP     │   转码/录制    │           │  客户端   │
│ NVR 群组  │──────────────→│              │           └──────────┘
└─────────┘               └──────┬───────┘
                                  │
                           录制存储（MP4/HLS 分片）
                                  │
                                  ▼
                           ┌──────────────┐
                           │  对象存储/NAS  │ ← 30天循环
                           └──────────────┘
```

### 5.3 FFmpeg 监控流转换

```bash
# RTSP 摄像头 → HLS 分片 + 录制
ffmpeg -rtsp_transport tcp -i rtsp://camera:554/stream \
  -c:v copy -c:a copy \
  -f hls -hls_time 4 -hls_list_size 15 \
  -hls_flags delete_segments+append_list \
  /output/live/cam01.m3u8 \
  -c:v copy -c:a copy \
  -f segment -segment_time 3600 -segment_format mp4 \
  -strftime 1 /archive/cam01_%Y%m%d_%H%M%S.mp4
```

## 6. 通用工程实践

### 6.1 FFmpeg 常用传输命令速查

```bash
# === 推流 ===
# RTMP 推流
ffmpeg -re -i input.mp4 -c copy -f flv rtmp://server/live/key

# SRT 推流
ffmpeg -re -i input.mp4 -c copy -f mpegts "srt://server:9000?mode=caller"

# RTP 推流
ffmpeg -re -i input.mp4 -c:v libx264 -f rtp rtp://server:5004

# === 拉流转推 ===
# RTMP → HLS
ffmpeg -i rtmp://source/live/key -c copy -f hls -hls_time 4 out.m3u8

# RTMP → SRT
ffmpeg -i rtmp://source/live/key -c copy -f mpegts "srt://:9000?mode=listener"

# SRT → RTMP
ffmpeg -i "srt://source:9000?mode=caller" -c copy -f flv rtmp://dest/live/key

# === 录制 ===
# RTMP 流录制为 MP4
ffmpeg -i rtmp://source/live/key -c copy -movflags +faststart output.mp4
```

### 6.2 关键编码参数

| 场景 | 预设 | 关键参数 |
|------|------|---------|
| **超低延迟** | ultrafast + zerolatency | `-preset ultrafast -tune zerolatency -g 30` |
| **低延迟直播** | veryfast + zerolatency | `-preset veryfast -tune zerolatency -g 60` |
| **标准直播** | veryfast | `-preset veryfast -g 120 -bf 3` |
| **高质量录制** | medium | `-preset medium -crf 18 -bf 3` |

### 6.3 容错与重连

```bash
# RTMP 推流自动重连
ffmpeg -re -i input.mp4 \
  -c copy -f flv \
  -reconnect 1 -reconnect_at_eof 1 \
  -reconnect_streamed 1 -reconnect_delay_max 5 \
  rtmp://server/live/key

# SRT 推流超时设置
ffmpeg -re -i input.mp4 \
  -c copy -f mpegts \
  "srt://server:9000?mode=caller&connect_timeout=5000&rcvlatency=200000"
```

## 7. 故障排查手册

### 7.1 常见问题与解决方案

| 问题 | 可能原因 | 排查步骤 | 解决方案 |
|------|---------|---------|---------|
| 推流失败 | 服务器未开放端口 | `telnet server 1935` | 检查防火墙规则 |
| 画面花屏 | 丢包/关键帧丢失 | Wireshark 抓包分析 | 降低码率/增加 FEC |
| 延迟持续增大 | 编码速度跟不上 | 检查 CPU 使用率 | 降低编码预设 |
| 音视频不同步 | 时间戳不对齐 | 检查 PTS/DTS | `-async 1` 或重新封装 |
| CDN 播放卡顿 | 源站带宽不足 | 检查回源带宽 | 增加 CDN 回源带宽 |
| SRT 大量 Drop | Latency 设置过低 | 检查 `srt_bstats` | 增大 Latency |

### 7.2 网络诊断命令

```bash
# 测试到服务器的 RTT
ping -c 100 server.example.com

# 测试 UDP 连通性
nc -u -z server.example.com 9000

# 测试带宽
iperf3 -c server.example.com -u -b 10M

# 测试丢包率
mtr --report server.example.com
```

## 8. 监控指标体系

### 8.1 关键指标

| 层级 | 指标 | 告警阈值 | 说明 |
|------|------|---------|------|
| **推流端** | 推流成功率 | < 99% | 推流连接的成功比例 |
| **推流端** | 推流码率 | 偏离目标 >20% | 编码器输出码率 |
| **服务端** | 转码延迟 | > 200ms | 转码引入的额外延迟 |
| **服务端** | 分片生成延迟 | > 500ms | HLS 分片的生成时间 |
| **CDN** | 首帧时间 | > 3s | 从请求到首帧显示 |
| **CDN** | 缓冲率 | > 1% | 播放过程中的缓冲事件 |
| **播放端** | 端到端延迟 | 超过目标 | 采集到播放的总延迟 |
| **播放端** | 卡顿率 | > 0.5% | 卡顿时间占总播放时间 |
| **SRT** | Too-Late Drop | > 0.1% | 超时丢弃的包比例 |
| **WebRTC** | 丢包率 | > 2% | RTP 包丢失比例 |
