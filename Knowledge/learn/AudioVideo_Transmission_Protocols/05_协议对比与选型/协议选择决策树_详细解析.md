# 协议选择决策树 详细解析

> 根据业务场景需求，系统化地选择最适合的音视频传输协议

---

## 核心结论（TL;DR）

**协议选择的本质是场景驱动的工程决策。** 不需要从零开始分析每个协议的优劣——只需要回答几个关键问题：你的延迟容忍度是多少？需要支持多少并发用户？是单向还是双向通信？网络环境是内网还是公网？沿着决策树走下来，答案自然浮现。

## 1. 决策树总览

```
开始
  │
  ├─ 延迟要求 < 500ms？
  │   ├─ YES → 需要双向通信？
  │   │         ├─ YES → WebRTC（P2P 或 SFU）
  │   │         └─ NO  → 专业广播级传输？
  │   │                   ├─ YES → SRT
  │   │                   └─ NO  → WebRTC（WHEP 拉流）
  │   │
  │   └─ NO → 延迟要求 < 3s？
  │           ├─ YES → 推流还是拉流？
  │           │         ├─ 推流 → RTMP 或 SRT
  │           │         └─ 拉流 → LL-HLS / LL-DASH / WebRTC(WHEP)
  │           │
  │           └─ NO → 大规模分发（1000+并发）？
  │                   ├─ YES → HLS / DASH
  │                   └─ NO  → RTMP / HTTP-FLV
```

## 2. 第一层：延迟需求

**这是最关键的决策维度。** 不同的延迟范围对应完全不同的技术方案：

| 延迟范围 | 场景 | 可选协议 |
|---------|------|---------|
| **< 200ms** | 实时对话、远程操控 | WebRTC |
| **200ms - 1s** | 互动直播、连麦、远程制作 | WebRTC, SRT |
| **1s - 3s** | 低延迟直播、电商直播 | SRT, RTMP, LL-HLS/LL-DASH |
| **3s - 10s** | 普通直播 | LL-HLS, LL-DASH, RTMP |
| **10s - 30s** | 大规模直播、点播 | HLS, DASH |
| **无要求** | 纯点播 | HLS, DASH |

### 为什么延迟是第一决策维度？

因为延迟需求直接决定了底层传输协议的选择：
- **< 1s** → 必须用 UDP 系（TCP 的队头阻塞无法满足）
- **1-3s** → UDP 或 TCP 都可以
- **> 3s** → HTTP/TCP 是最经济的选择（CDN 完美适配）

## 3. 第二层：通信模式

| 模式 | 特征 | 推荐协议 |
|------|------|---------|
| **双向实时** | 两端都需要发送和接收 | WebRTC |
| **单向推流** | 一端推流到服务器 | RTMP, SRT, WHIP |
| **单向拉流** | 多端从服务器拉流 | HLS, DASH, WHEP |
| **推流 + 拉流** | 推流到服务器，再分发 | RTMP/SRT 推 + HLS/DASH 拉 |

### 双向 vs 单向的技术影响

**双向通信**需要：
- NAT 穿透（ICE/STUN/TURN）
- 双向媒体协商（SDP Offer/Answer）
- 同时编码和解码
→ 只有 WebRTC 原生支持

**单向传输**简单得多：
- 明确的发送方和接收方
- 不需要 NAT 穿透（发送方连接服务器即可）
- 只需单向编解码
→ RTMP/SRT/HLS/DASH 都适用

## 4. 第三层：规模需求

| 并发规模 | 架构 | 推荐方案 |
|---------|------|---------|
| **1v1** | P2P | WebRTC |
| **2-20人** | SFU | WebRTC + SFU |
| **20-100人** | SFU + Simulcast | WebRTC + SFU（如 LiveKit） |
| **100-1万** | 混合架构 | WebRTC SFU + HLS 兜底 |
| **1万-100万** | CDN 分发 | HLS / DASH |
| **100万+** | 多 CDN 分发 | HLS / DASH + 多 CDN |

### 规模与成本的关系

```
WebRTC SFU 成本 ≈ O(N × 带宽) + O(CPU)
  100 人会议：约 500 Mbps 出带宽 + 8核服务器

HLS CDN 成本 ≈ O(N × CDN 单价)
  10 万观众：约 400 Gbps CDN 带宽（4Mbps × 100K）
  但 CDN 有缓存，实际回源带宽远小于此
```

## 5. 第四层：网络环境

| 网络环境 | 特征 | 推荐协议 |
|---------|------|---------|
| **内网/专线** | 低延迟、低丢包 | 任意（RTP 直推效率最高） |
| **同城公网** | 中等延迟、低丢包 | WebRTC, SRT, RTMP |
| **跨区域公网** | 较高延迟、中等丢包 | SRT（可配延迟）, HLS/DASH |
| **跨洲际** | 高延迟、有丢包 | SRT + 高 Latency 或 CDN(HLS/DASH) |
| **移动弱网** | 高抖动、高丢包 | WebRTC（自适应）, HLS（ABR） |
| **UDP 被封** | 只有 TCP/HTTP 可用 | RTMP(S), HLS, DASH |

### 网络约束下的特殊考虑

**UDP 被封锁的场景**（部分企业网络、某些国家）：
- WebRTC → 降级到 TURN over TCP（延迟增加）
- SRT → 不可用（纯 UDP）
- 只能选择 RTMP 或 HLS/DASH

**高丢包网络**（> 5%）：
- TCP 系协议（RTMP）→ 严重卡顿
- SRT → 需要增大 Latency 或启用 FEC
- WebRTC → 自动降码率 + FEC
- HLS/DASH → ABR 降码率

## 6. 第五层：安全需求

| 安全级别 | 需求 | 推荐方案 |
|---------|------|---------|
| **无加密** | 内网传输 | 任意协议原始版本 |
| **传输加密** | 防窃听 | RTMPS, SRT(AES), WebRTC(DTLS) |
| **内容保护** | 防盗版 | HLS(FairPlay), DASH(Widevine/PlayReady) |
| **端到端加密** | 即使服务器也无法解密 | WebRTC + E2EE（Insertable Streams） |

## 7. 典型场景速查表

### 7.1 直播场景

| 场景 | 推流协议 | 分发协议 | 延迟 | 说明 |
|------|---------|---------|------|------|
| **游戏直播** | RTMP | HLS | 3-10s | 稳定优先 |
| **电商直播** | RTMP | LL-HLS/HTTP-FLV | 1-3s | 低延迟+互动 |
| **赛事直播** | SRT | HLS/DASH | 3-15s | 高质量+DRM |
| **连麦直播** | WebRTC(主播间) + RTMP(推CDN) | HLS(观众) | 混合 | 混合架构 |
| **超低延迟直播** | WebRTC(WHIP) | WebRTC(WHEP) | <1s | 小规模 |

### 7.2 通信场景

| 场景 | 协议 | 架构 | 延迟 |
|------|------|------|------|
| **1v1 语音/视频通话** | WebRTC | P2P | <300ms |
| **小型会议（3-10人）** | WebRTC | SFU | <500ms |
| **大型会议（10-100人）** | WebRTC | SFU+Simulcast | <500ms |
| **网络研讨会（100-1000人）** | WebRTC(演讲者)+HLS(观众) | 混合 | 混合 |

### 7.3 专业传输场景

| 场景 | 协议 | 配置 | 说明 |
|------|------|------|------|
| **远程制作** | SRT | Latency=200-500ms | 替代卫星 |
| **贡献传输** | SRT | Latency=500-2000ms | 跨区域 |
| **监控** | RTSP/RTP | 持续流 | 低成本设备 |
| **无人机图传** | 自定义RTP 或 SRT | 最低延迟模式 | 极端条件 |

## 8. 混合协议架构

### 8.1 最常见的混合架构

**大规模低延迟直播**：

```
主播端 ──RTMP──→ 源站服务器 ──转码──→ CDN ──HLS/DASH──→ 99% 观众
                      │                        │
                      └──WebRTC──→ 连麦用户      └──LL-HLS──→ 低延迟观众
```

**跨区域专业传输 + 大规模分发**：

```
现场 ──SRT──→ 制作中心 ──编辑──→ 编码器 ──RTMP──→ CDN ──HLS──→ 观众
```

**视频会议 + 直播**：

```
参与者 ←─WebRTC─→ SFU ──RTMP──→ CDN ──HLS──→ 旁听观众
```

### 8.2 协议转换节点

| 转换 | 工具 | 说明 |
|------|------|------|
| RTMP → HLS | nginx-rtmp, SRS | 最常见转换 |
| SRT → RTMP | FFmpeg, SRS | SRT 推流到传统 CDN |
| WebRTC → HLS | Janus/MediaSoup + FFmpeg | 会议录制/直播 |
| RTMP → WebRTC | SRS, LiveKit | 低延迟拉流 |
| HLS → WebRTC | ❌ 不常见 | 方向错误 |

## 9. 决策检查清单

在最终确定协议之前，逐项检查：

### 技术维度

- [ ] 延迟需求是否明确？（量化到具体范围）
- [ ] 是否需要双向通信？
- [ ] 最大并发用户数是多少？
- [ ] 目标网络环境是什么？（内网/公网/移动/跨境）
- [ ] 是否需要穿透防火墙/NAT？
- [ ] 需要支持哪些编解码器？
- [ ] 是否需要加密/DRM？

### 生态维度

- [ ] 团队是否有该协议的开发经验？
- [ ] 是否有成熟的开源实现可用？
- [ ] CDN 供应商是否支持？
- [ ] 客户端设备是否原生支持？（特别是 iOS/Safari）

### 成本维度

- [ ] 服务器/CDN 成本预算是否足够？
- [ ] 开发和维护成本是否可接受？
- [ ] 是否需要付费许可（某些 DRM/编解码器）？

### 未来兼容维度

- [ ] 协议是否仍在积极发展？
- [ ] 是否需要考虑向新协议迁移？（如 RTMP → SRT/WHIP）
- [ ] 是否需要支持 CMAF 统一格式？
