# 协议性能对比 详细解析

> 六大音视频传输协议的多维横向对比——延迟、带宽、可靠性、扩展性、安全性全面量化分析

---

## 核心结论（TL;DR）

**没有"最好"的音视频传输协议，只有"最适合"的。** 每个协议都是特定约束条件下的最优解：WebRTC 极致低延迟但难以扩展，HLS/DASH 极致扩展但延迟较高，RTMP 生态成熟但受限于 TCP，SRT 在专业传输中达到最佳的延迟-可靠性平衡。选择协议的本质是在**延迟、可靠性、扩展性、成本**四个维度上做取舍。

## 1. 对比维度定义

为了公平对比，我们定义以下量化维度：

| 维度 | 定义 | 评分标准（1-5） |
|------|------|----------------|
| **端到端延迟** | 从采集到播放的时间 | 1=最高延迟, 5=最低延迟 |
| **传输可靠性** | 在丢包网络中的表现 | 1=完全不容忍丢包, 5=完美抗丢包 |
| **扩展性** | 支持的并发观众规模 | 1=点对点, 5=百万级 |
| **安全性** | 加密和内容保护能力 | 1=无加密, 5=强制加密+DRM |
| **生态成熟度** | 工具、文档、社区支持 | 1=新兴, 5=极其成熟 |
| **实现复杂度** | 开发集成的难度 | 1=极复杂, 5=极简单 |
| **成本效益** | 部署和运维成本 | 1=极高, 5=极低 |

## 2. 延迟对比

### 2.1 端到端延迟量化

| 协议 | 最低延迟 | 典型延迟 | 最高延迟 | 延迟可控性 |
|------|---------|---------|---------|-----------|
| **WebRTC** | 50ms | 100-300ms | 500ms | ★★★★★ |
| **SRT** | 100ms | 200ms-1s | 2s | ★★★★★ |
| **RTMP** | 500ms | 1-3s | 5s+ | ★★☆☆☆ |
| **LL-HLS** | 1s | 2-4s | 6s | ★★★☆☆ |
| **LL-DASH** | 1s | 2-4s | 6s | ★★★☆☆ |
| **HLS** | 6s | 10-20s | 30s+ | ★★☆☆☆ |
| **DASH** | 4s | 8-15s | 30s+ | ★★☆☆☆ |

### 2.2 延迟构成分解

```
延迟 = 采集 + 编码 + 协议开销 + 网络传输 + 协议缓冲 + 解码 + 渲染

各协议的"协议开销 + 协议缓冲"对比：
  WebRTC:  ~20ms（Jitter Buffer）
  SRT:     ~200ms（可配置 Latency Buffer）
  RTMP:    ~100ms（TCP 缓冲） + 网络波动时数秒
  LL-HLS:  ~1-2s（Part 下载 + 缓冲）
  HLS:     ~6-18s（分片时长 × 缓冲分片数）
```

### 2.3 延迟稳定性

延迟的**稳定性**比绝对值更重要（人对延迟变化比延迟本身更敏感）：

| 协议 | 延迟方差 | 尾部延迟（P99） | 说明 |
|------|---------|----------------|------|
| **WebRTC** | 低 | 适中（NACK 重传） | Jitter Buffer 自适应 |
| **SRT** | 极低 | 低（ARQ 窗口限制） | Latency 参数保证上限 |
| **RTMP** | 高 | 很高（TCP 重传累积） | 队头阻塞导致延迟尖峰 |
| **HLS/DASH** | 低 | 适中 | HTTP 稳定但基础延迟高 |

## 3. 可靠性与抗弱网对比

### 3.1 不同丢包率下的表现

**模拟测试条件**：1080p@4Mbps 视频流，RTT=50ms

| 丢包率 | WebRTC | SRT | RTMP | HLS |
|--------|--------|-----|------|-----|
| **0%** | 完美 | 完美 | 完美 | 完美 |
| **1%** | 偶尔 NACK 恢复 | ARQ 恢复 | TCP 重传，略卡 | 无影响 |
| **3%** | NACK+FEC 恢复 | ARQ 恢复 | 明显卡顿 | 偶尔缓冲 |
| **5%** | 降码率+FEC | ARQ+FEC 恢复 | 严重卡顿 | 频繁缓冲 |
| **10%** | 明显降质 | 部分 Too-Late 丢弃 | 几乎不可用 | 长时间缓冲 |
| **20%** | 基本不可用 | 大量丢弃 | 不可用 | 严重断流 |

### 3.2 抗丢包机制对比

| 协议 | ARQ | FEC | 容忍丢失 | 自适应 |
|------|-----|-----|---------|--------|
| **WebRTC** | NACK | ULPFEC/FlexFEC | PLI/FIR | GCC/TCC |
| **SRT** | 周期 NACK | 矩阵 FEC | Too-Late Drop | 应用层回调 |
| **RTMP** | TCP 内置 | ❌ | ❌ | ❌ |
| **HLS/DASH** | TCP 内置 | ❌ | ❌ | ABR 切码率 |

### 3.3 弱网恢复速度

| 协议 | 中断后恢复时间 | 恢复机制 |
|------|-------------|---------|
| **WebRTC** | 100ms-1s | NACK + PLI 请求关键帧 |
| **SRT** | 50ms-500ms | ARQ 窗口内自动恢复 |
| **RTMP** | 1-10s | TCP 拥塞窗口重建 |
| **HLS** | 2-10s | 重新请求分片 + ABR 降级 |

## 4. 扩展性对比

### 4.1 并发规模

| 协议 | 单源支持并发 | 架构扩展后 | 典型部署 |
|------|------------|-----------|---------|
| **WebRTC P2P** | 1-5 | - | 1v1 通话 |
| **WebRTC SFU** | 50-500 | 数千（级联 SFU） | 视频会议 |
| **RTMP** | 100-1000（单服务器） | 数万（CDN） | 推流+分发 |
| **SRT** | 1-10 | 数百（中继链） | 专业传输 |
| **HLS** | ∞（CDN 缓存） | 百万级 | 大规模直播 |
| **DASH** | ∞（CDN 缓存） | 百万级 | 大规模直播 |

### 4.2 CDN 兼容性

| 协议 | CDN 支持 | 缓存友好 | 边缘计算需求 |
|------|---------|---------|-------------|
| **WebRTC** | ⚠️ 需专用 CDN | ❌ | 高（SFU） |
| **SRT** | ⚠️ 有限支持 | ❌ | 中（中继） |
| **RTMP** | ✅ 推流支持好 | ❌（长连接） | 低 |
| **HLS** | ✅ 完美 | ✅（静态文件） | 极低 |
| **DASH** | ✅ 完美 | ✅（静态文件） | 极低 |

## 5. 安全性对比

| 协议 | 传输加密 | 是否强制 | DRM 支持 | 认证机制 |
|------|---------|---------|---------|---------|
| **WebRTC** | DTLS-SRTP | ✅ 强制 | ❌ | 证书指纹验证 |
| **SRT** | AES-128/256 | ⚠️ 可选 | ❌ | Passphrase |
| **RTMP** | TLS (RTMPS) | ⚠️ 可选 | ❌ | Stream Key |
| **HLS** | TLS + AES/DRM | ⚠️ 可选 | ✅ FairPlay | Token/DRM License |
| **DASH** | TLS + CENC/DRM | ⚠️ 可选 | ✅ Widevine/PlayReady | Token/DRM License |

## 9. 成本对比

### 9.1 基础设施成本

| 协议 | 服务器成本 | CDN 成本 | 开发成本 |
|------|-----------|---------|---------|
| **WebRTC P2P** | 极低（仅 STUN/TURN） | 无 | 高 |
| **WebRTC SFU** | 高（CPU+带宽） | 专用CDN贵 | 很高 |
| **SRT** | 低（轻量中继） | 有限支持 | 中 |
| **RTMP** | 中 | 推流标准价 | 低 |
| **HLS/DASH** | 低（HTTP 服务器） | 标准 CDN 价格 | 中 |

### 9.2 带宽成本

| 协议 | 额外带宽开销 | 原因 |
|------|------------|------|
| **WebRTC** | +10-30%（FEC+NACK） | 冗余和重传 |
| **SRT** | +5-20%（ARQ+FEC） | 可配置冗余度 |
| **RTMP** | +3-5%（TCP 开销） | TCP 头部和重传 |
| **HLS/DASH** | +5-10%（多码率存储） | 需要编码多个码率 |

## 10. 未来趋势

### 10.1 协议融合趋势

```
当前状态：
  推流 → RTMP → 服务器 → 转码 → CDN → HLS/DASH → 观众

未来趋势：
  1. WHIP/WHEP 替代 RTMP 推流（WebRTC 标准化推拉流）
  2. CMAF 统一 HLS 和 DASH 分片格式
  3. WebTransport（基于 QUIC）替代 WebSocket 信令
  4. MoQ（Media over QUIC）统一实时和分发
```

### 10.2 新兴协议

| 协议 | 定位 | 状态 |
|------|------|------|
| **WHIP** | WebRTC 标准化推流 | IETF Draft |
| **WHEP** | WebRTC 标准化拉流 | IETF Draft |
| **MoQ** | 基于 QUIC 的统一协议 | 早期开发 |
| **WebTransport** | 基于 QUIC 的 Web 传输 | Chrome 已支持 |
| **RIST** | 专业广播传输（SRT 竞争者） | VSF TR-06 |
