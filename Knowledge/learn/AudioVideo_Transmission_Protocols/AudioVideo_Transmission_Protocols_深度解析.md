# 音视频传输协议深度解析

> 系统性梳理音视频传输协议的核心原理、协议架构、应用场景与工程实践，构建从传输基础到协议选型的完整知识图谱

---

## 核心结论（TL;DR）

**音视频传输协议的核心目标是：在不可靠的网络环境中，将编码后的音视频数据以尽可能低的延迟、尽可能高的质量传递到接收端。**

这看似简单的目标，背后蕴含着网络工程中最深刻的权衡——**延迟 vs 可靠性 vs 扩展性**的"不可能三角"：

1. **低延迟**意味着不能等待重传，必须容忍一定丢包（UDP 路线）
2. **高可靠**意味着需要重传或冗余，必然增加延迟（TCP 路线或 FEC）
3. **大规模**意味着需要 CDN 分发，必然引入额外的转码和分段延迟（HTTP 路线）

**一句话理解音视频传输**：如果说视频编解码是"如何把一幅画压缩成最小的信封"，那么传输协议就是"选择用什么交通工具、走哪条路线把信封送到对方手中"——不同的场景需要不同的"物流方案"。

## 第一部分：为什么需要专用的音视频传输协议

### 1.1 音视频数据的特殊性

音视频数据与普通文件传输有着根本性的差异，理解这些差异是理解所有传输协议设计决策的基础：

| 特性 | 文件传输 | 音视频传输 |
|------|---------|-----------|
| **时效性** | 无实时性要求 | 每帧有严格的播放截止时间 |
| **容错性** | 不允许任何丢失 | 可容忍少量丢失（人耳/眼有掩蔽效应） |
| **顺序性** | 可乱序接收后重组 | 必须按时序播放，迟到等于丢失 |
| **码率特征** | 固定或突发 | 持续且波动（I帧远大于P/B帧） |
| **交互性** | 无交互需求 | 可能需要双向实时交互 |

**核心洞察**：TCP 的"保证交付"特性对音视频反而是一把双刃剑——当一个包丢失时，TCP 会停下来等待重传（队头阻塞），而此时后续的音视频帧已经错过了播放时间窗口，导致整体体验的卡顿远比丢一帧更严重。

### 1.2 网络物理约束：第一性原理

所有传输协议的设计决策，都可以追溯到三个基本的物理约束：

**约束一：光速限制了传播延迟**

```
北京 → 纽约的光纤距离 ≈ 17,000 km
光在光纤中的传播速度 ≈ 200,000 km/s（约2/3光速）
单程传播延迟 ≈ 85ms
```

这意味着，即使是完美的网络，北京和纽约之间的音视频通话延迟也不可能低于 170ms（往返），而人类对话的舒适延迟阈值是 150ms。这就是为什么全球化 RTC 必须依赖边缘节点中继。

**约束二：带宽是共享的有限资源**

家庭网络带宽通常在 50-500 Mbps 之间共享，而一路 1080p 高质量视频需要 4-8 Mbps。当多个设备同时使用网络时，可用带宽会剧烈波动，协议必须具备自适应能力。

**约束三：丢包是不可避免的**

互联网的路由器在拥塞时会主动丢包（AQM），WiFi 信道存在干扰和碰撞，移动网络在切换基站时会产生瞬时断连。典型的丢包率：

| 网络类型 | 典型丢包率 | 抖动范围 |
|---------|-----------|---------|
| 有线局域网 | < 0.01% | < 1ms |
| 家庭 WiFi | 0.1% - 2% | 5-50ms |
| 4G 移动网络 | 1% - 5% | 20-100ms |
| 跨洲际公网 | 0.5% - 3% | 10-80ms |

### 1.3 协议设计的核心权衡

基于上述约束，所有音视频传输协议本质上都在做同一个权衡：

```
                    低延迟
                   /      \
                  /   选择   \
                 /    区域    \
                /____________\
          高可靠             大规模
```

- **WebRTC**：极致低延迟（< 500ms），适合小规模 P2P
- **RTMP**：较低延迟（1-3s），适合推流到服务器
- **SRT**：低延迟 + 高可靠（可配置），适合专业传输
- **HLS/DASH**：高扩展性（CDN 分发），但延迟较高（3-30s）

## 第二部分：协议发展时间线

音视频传输协议的演进反映了互联网基础设施和应用场景的变迁：

```
1996 ─── RTP/RTCP (RFC 1889)
          │  实时传输的基石，定义了时间戳和序列号机制
          │
2002 ─── RTSP (RFC 2326, 更新于2016 RFC 7826)
          │  流媒体会话控制，VCR 式操作
          │
2003 ─── RTMP (Adobe/Macromedia)
          │  Flash 时代的直播标准，基于 TCP
          │
2009 ─── HLS (Apple)
          │  HTTP 分片流，开启了 ABR 时代
          │
2011 ─── WebRTC (Google 开源)
          │  浏览器原生 P2P 实时通信
          │
2012 ─── MPEG-DASH (ISO/IEC 23009-1)
          │  国际标准的自适应流，供应商中立
          │
2017 ─── SRT (Haivision 开源)
          │  安全可靠传输，面向专业广播
          │
2020 ─── LL-HLS (Apple)
          │  低延迟 HLS，目标 2s 以内
          │
2022 ─── WebTransport (W3C/IETF)
          │  基于 QUIC 的下一代 Web 传输
          │
2024 ─── WHIP/WHEP (IETF Draft)
              WebRTC 的标准化推流/拉流协议
```

## 第三部分：全局协议分类图谱

### 3.1 按传输模型分类

音视频传输协议可以从传输模型的维度进行 MECE 分类：

**分类一：基于传输层协议**

| 底层协议 | 代表协议 | 核心特征 |
|---------|---------|---------|
| **UDP 系** | RTP/RTCP, WebRTC, SRT | 低延迟、需自行处理可靠性 |
| **TCP 系** | RTMP, RTSP/RTP-over-TCP | 可靠传输、有队头阻塞问题 |
| **HTTP 系** | HLS, DASH | CDN 友好、延迟较高 |
| **QUIC 系** | WebTransport, MoQ | 融合 UDP 低延迟 + TLS 安全性 |

**分类二：按应用场景**

| 场景 | 协议选择 | 延迟要求 | 规模 |
|------|---------|---------|------|
| **实时通信（RTC）** | WebRTC | < 500ms | 小规模 P2P/SFU |
| **实时推流** | RTMP, SRT | 1-3s | 单路推流 |
| **大规模直播** | HLS, DASH | 3-30s | 百万级观众 |
| **专业广播传输** | SRT, RIST | 可配置 | 点对点 |
| **点播回放** | HLS, DASH | 无要求 | CDN 分发 |

## 第四部分：各协议核心特征速查表

| 协议 | 底层传输 | 端到端延迟 | 扩展性 | 加密 | 抗丢包 | 适用场景 |
|------|---------|-----------|--------|------|--------|---------|
| **RTP/RTCP** | UDP | 极低（< 100ms） | 灵活 | 可选(SRTP) | 无（需上层处理） | 基础协议层 |
| **RTMP** | TCP | 低（1-3s） | 中 | RTMPS(TLS) | TCP 保证 | 推流到服务器 |
| **SRT** | UDP | 可配（200ms-2s） | 中 | AES-128/256 | ARQ + FEC | 跨公网专业传输 |
| **WebRTC** | UDP(DTLS) | 极低（< 500ms） | 低-中 | 强制(DTLS-SRTP) | NACK + FEC | P2P 通信/会议 |
| **HLS** | HTTP/TCP | 高（6-30s） | 极高 | AES/DRM | TCP 保证 | 大规模直播/点播 |
| **DASH** | HTTP/TCP | 高（3-30s） | 极高 | DRM(CENC) | TCP 保证 | 大规模直播/点播 |
| **LL-HLS** | HTTP/TCP | 中（2-4s） | 极高 | AES/DRM | TCP 保证 | 低延迟大规模直播 |

## 第五部分：协议间的关系网络

音视频传输协议并非孤立存在，它们之间存在着丰富的层次关系和组合使用模式：

### 5.1 层次关系

```
WebRTC ──使用──→ RTP/RTCP ──运行于──→ UDP
  │                  │
  └──信令层──→ SDP + ICE/STUN/TURN
  
RTMP ──运行于──→ TCP

SRT ──运行于──→ UDP ──自实现──→ ARQ + FEC

HLS ──运行于──→ HTTP ──运行于──→ TCP
  │
  └──分片格式──→ TS 或 fMP4

DASH ──运行于──→ HTTP ──运行于──→ TCP
  │
  └──分片格式──→ fMP4 (CMAF)
```

### 5.2 典型组合场景

**场景：大规模低延迟直播**

```
主播端 ──RTMP/SRT──→ 源站服务器 ──转码──→ CDN ──HLS/DASH──→ 百万观众
                                    │
                                    └──WebRTC──→ 连麦用户
```

**场景：视频会议**

```
用户A ←──WebRTC──→ SFU服务器 ←──WebRTC──→ 用户B
                      │
                      ←──WebRTC──→ 用户C
```

## 第六部分：关键技术横切面

无论使用哪种协议，以下技术是共同的基础设施：

### 6.1 拥塞控制

| 算法 | 使用协议 | 核心思想 |
|------|---------|---------|
| **GCC** | WebRTC | 基于延迟梯度的带宽估计 |
| **BBR** | QUIC/TCP | 基于带宽和 RTT 的模型驱动 |
| **NADA** | WebRTC（备选） | 基于显式拥塞通知 |
| **SRT CC** | SRT | 基于 ARQ 的实时拥塞控制 |

### 6.2 前向纠错（FEC）

FEC 通过发送冗余数据包，使接收端无需重传即可恢复丢失的包：

```
发送端：原始包 [1] [2] [3] + FEC包 [1⊕2⊕3]
接收端：丢失 [2]，用 [1] ⊕ [3] ⊕ [FEC] 恢复 [2]
```

### 6.3 自适应码率（ABR）

ABR 算法根据网络状况动态切换视频质量：

| 策略 | 代表 | 优点 | 缺点 |
|------|------|------|------|
| **基于带宽** | 吞吐量估计 | 简单直接 | 反应滞后 |
| **基于缓冲** | BBA/BOLA | 稳定性好 | 启动慢 |
| **混合策略** | MPC/Pensieve | 综合最优 | 复杂度高 |
