# HDR色彩管理深度解析

> 系统性梳理颜色空间、色彩管理和HDR的核心概念、技术原理与实践应用

---

## 核心结论（TL;DR）

**HDR色彩管理的核心目标是：在从内容创作到最终显示的完整链路中，确保色彩信息的一致性和准确性。**

现代HDR工作流基于以下关键支柱：

1. **广色域标准（Rec. 2020）**：覆盖CIE 1931色域的75.8%，远超SDR的35.9%
2. **感知量化（PQ/ST 2084）**：基于人眼视觉特性的非线性编码，支持最高10000尼特
3. **ACES色彩管理系统**：提供从输入到输出的标准化转换流程
4. **色调映射（Tonemapping）**：将高动态范围场景映射到显示设备可呈现的范围

## 第一部分：基础概念

### 1.1 光与颜色的本质

**光是物理概念，颜色是感知概念。**

光是一种电磁波，可见光的波长范围约为400-700nm。我们使用**光谱能量分布（SPD, Spectral Power Distribution）**来描述光的物理特性——以波长为横坐标、辐射度为纵坐标作图。

颜色则是人脑基于可见光谱对视锥细胞刺激所产生的一种视觉感知。人眼有三种视锥细胞，分别对短波（S，约450nm）、中波（M，约550nm）、长波（L，约600nm）的光最敏感。

> 📌 **关键洞察**：由于条件等色（Metamerism）现象，不同光谱分布的光可能产生相同的颜色感知。这意味着颜色与光的关系是一对多的。

### 1.2 CIE 1931 色彩空间体系

CIE（国际照明委员会）于1931年建立了现代色彩科学的基础框架：

#### 颜色匹配函数（Color Matching Functions）

通过等色匹配实验，CIE定义了三个颜色匹配函数 $r(\lambda)$、$g(\lambda)$、$b(\lambda)$。对于任意可见光，可以计算其三刺激值（Tristimulus Values）：

$$
R = \int_{380}^{780} I(\lambda) \cdot r(\lambda) \, d\lambda \\
G = \int_{380}^{780} I(\lambda) \cdot g(\lambda) \, d\lambda \\
B = \int_{380}^{780} I(\lambda) \cdot b(\lambda) \, d\lambda
$$

#### CIE 1931 XYZ 色彩空间

为解决RGB空间存在负值的问题，CIE定义了XYZ色彩空间：

- **X、Z**：仅包含色度信息
- **Y**：同时包含亮度和色度信息（与亮度函数匹配）

XYZ空间的三个基向量是"想象色"（Imaginary Color），在物理世界中无法真实构造，但数学上全为正值。

#### CIE 1931 xy 色度图

将XYZ空间投影到 $X + Y + Z = 1$ 平面，再投影到XY平面，得到二维色度图：

$$
x = \frac{X}{X + Y + Z}, \quad y = \frac{Y}{X + Y + Z}
$$

色度图的马蹄形边界称为**光谱轨迹（Spectral Locus）**，表示各波长单色光的位置。轨迹内部的所有颜色都是人眼可感知的。

> 🔗 详细内容：[色度图与颜色匹配函数详解](./色度图_详细解析.md)

### 1.3 定义RGB颜色空间

一个完整的RGB颜色空间需要三个组成部分：

| 组成部分 | 定义 | 作用 |
|---------|------|------|
| **三原色（Primaries）** | 在xy色度图中指定R、G、B的坐标 | 定义色域范围 |
| **白点（White Point）** | 指定纯白色(1,1,1)在色度图中的位置 | 确定颜色空间的方向和缩放 |
| **传递函数（Transfer Functions）** | OETF和EOTF | 线性与非线性信号之间的转换 |

#### 常见颜色空间对比

| 颜色空间 | 三原色 | 白点 | 传递函数 | 色域覆盖 |
|---------|--------|------|---------|---------|
| **sRGB** | R(0.64,0.33), G(0.30,0.60), B(0.15,0.06) | D65 | sRGB OETF/EOTF | ~35.9% CIE 1931 |
| **Rec. 709** | 同sRGB | D65 | Rec. 709 OETF / BT.1886 EOTF | ~35.9% CIE 1931 |
| **DCI-P3** | R(0.680,0.320), G(0.265,0.690), B(0.150,0.060) | D63(偏绿) | Gamma 2.6 | ~53.6% CIE 1931 |
| **Rec. 2020** | R(0.708,0.292), G(0.170,0.797), B(0.131,0.046) | D65 | PQ或HLG | ~75.8% CIE 1931 |

> 🔗 详细内容：[颜色空间的数学定义](./颜色空间_数学定义.md)

## 第二部分：传递函数与图像状态

### 2.1 传递函数的分类

传递函数定义了线性与非线性信号之间的转换关系：

```
场景线性光 → [OETF] → 非线性视频信号 → [EOTF] → 显示光亮度
```

| 类型 | 全称 | 方向 | 应用场景 |
|------|------|------|---------|
| **OETF** | Opto-Electronic Transfer Function | 光→电 | 摄像机拍摄、图像编码存储 |
| **EOTF** | Electro-Optical Transfer Function | 电→光 | 显示器解码、图像显示 |
| **OOTF** | Opto-Optical Transfer Function | 光→光 | 整个系统的端到端转换 |

### 2.2 Gamma校正的历史与原理

Gamma校正最初是为了补偿CRT显示器的非线性响应特性。现代LCD虽然可以线性输出，但为了兼容历史内容，仍沿用Gamma编码。

**Gamma编码的核心优势**：

人眼视觉系统（HVS）对暗部细节更敏感。使用Gamma编码可以在有限的位深下，将更多的编码空间分配给暗部区域。

sRGB的传递函数公式：

$$
V = \begin{cases}
12.92 \cdot L & L \leq 0.0031308 \\
1.055 \cdot L^{1/2.4} - 0.055 & L > 0.0031308
\end{cases}
$$

其中 $L$ 是线性场景亮度（0-1），$V$ 是编码后的视频信号值。

### 2.3 图像状态（Image State）

| 图像状态 | 定义 | 特点 |
|---------|------|------|
| **Scene-Referred** | 记录场景实际光照值 | 线性、高动态范围、与显示设备无关 |
| **Display-Referred** | 针对特定显示设备定义 | 非线性、有限动态范围、与显示设备相关 |

渲染流程通常遵循：

```
Scene-Referred (线性HDR) → Tonemapping → Display-Referred (非线性SDR/HDR)
```

## 第三部分：SDR渲染管线

### 3.1 SDR Color Pipeline 概览

```
┌─────────────────────────────────────────────────────────────────────┐
│                        SDR Color Pipeline                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  [sRGB纹理] ──sRGB EOTF──▶ [线性工作空间] ──渲染计算──▶ [HDR场景值]   │
│       ↑                        (sRGB Linear)                          │
│       │                                                              │
│       │                    ┌──────────────────┐                      │
│       │                    │  后处理效果       │                      │
│       │                    │  - Bloom         │                      │
│       │                    │  - Color Grading │                      │
│       │                    └──────────────────┘                      │
│       │                                                              │
│       │                              ↓                               │
│       │                    [曝光 + Tonemapping]                      │
│       │                    (HDR → SDR范围映射)                        │
│       │                              ↓                               │
│       └──────────────────── [sRGB OETF] ─────────────────▶ [显示器]  │
│                                                              ↓       │
│                                                        [sRGB EOTF]   │
│                                                              ↓       │
│                                                        [显示亮度]    │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 为什么需要Tonemapping

SDR显示器的亮度范围有限（通常100-300尼特），而HDR渲染的场景亮度可能高达数千尼特。直接截断（clip）会导致：

1. 高亮区域细节完全丢失
2. 整体画面对比度降低
3. 中灰值过亮

Tonemapping使用S型曲线将高动态范围映射到有限范围，同时保持视觉上的对比度：

$$
V_{out} = \frac{V_{in}(1 + \frac{V_{in}}{C^2})}{1 + V_{in}}
$$

其中 $C$ 是控制曲线形状的参数。

### 3.3 常见Tonemapping算法

| 算法 | 特点 | 应用 |
|------|------|------|
| **Reinhard** | 简单、全局操作 | 早期渲染 |
| **Hable Filmic** | 拟合胶片响应曲线 | 神秘海域2 |
| **ACES Filmic** | 影视行业标准 | 现代3A游戏 |
| **Uncharted 2** | 可参数化调整 | 广泛使用的改进版 |

## 第四部分：HDR显示技术

### 4.1 HDR相比SDR的优势

| 特性 | SDR | HDR |
|------|-----|-----|
| 色域覆盖 | ~35.9% CIE 1931 | ~75.8% CIE 1931 |
| 峰值亮度 | 100-300 尼特 | 1000-10000 尼特 |
| 位深 | 8-bit | 10-bit/12-bit |
| 传递函数 | Gamma (~2.2) | PQ / HLG |
| 亮度确定性 | 相对值(0-1) | 绝对值(尼特) |

### 4.2 PQ（Perceptual Quantizer）传递函数

PQ（ST 2084）由杜比开发，基于Barten Ramp模型——描述人眼在不同亮度下可察觉的最小亮度差。

**PQ的核心特性**：

1. **感知均匀性**：在任意亮度级别，量化步长都接近人眼的感知阈值
2. **绝对亮度编码**：输出值直接对应0-10000尼特的绝对亮度
3. **高效编码**：12-bit PQ可在全亮度范围内避免可见色阶（banding）

PQ EOTF公式：

$$
L = \left(\frac{\max(N^{1/m_2} - c_1, 0)}{c_2 - c_3 \cdot N^{1/m_2}}\right)^{1/m_1}
$$

其中 $N$ 是归一化信号值(0-1)，$L$ 是输出亮度（尼特），常数 $m_1=0.1593$, $m_2=78.84375$, $c_1=0.8359$, $c_2=18.8516$, $c_3=18.6875$。

### 4.3 HLG（Hybrid Log-Gamma）传递函数

HLG由BBC和NHK开发，设计目标是兼容SDR显示：

- **HDR显示**：使用完整的对数曲线
- **SDR显示**：使用近似Gamma 2.4的曲线，可直接显示合理效果

HLG更适合广播电视场景，PQ更适合影视制作和游戏。

> 🔗 详细内容：[HDR显示标准与技术](./HDR显示标准_详细解析.md)

## 第五部分：ACES色彩管理系统

### 5.1 ACES的核心价值

ACES（Academy Color Encoding System）是奥斯卡科学技术委员会推出的开放式色彩管理标准，旨在解决影视制作中设备繁多、标准混乱的问题。

**ACES的三大核心优势**：

1. **统一工作流**：从拍摄到放映的端到端标准化
2. **广色域工作空间**：AP0包含所有人眼可见颜色
3. **设备无关性**：中间文件可在不同设备间无损交换

### 5.2 ACES颜色空间

| 颜色空间 | 三原色 | 用途 |
|---------|--------|------|
| **ACES2065-1 (AP0)** | R(0.7347,0.2653), G(0.0,1.0), B(0.0001,-0.077) | 图像交换、存档 |
| **ACEScg (AP1)** | R(0.713,0.293), G(0.165,0.830), B(0.128,0.044) | CG/VFX渲染工作空间 |
| **ACEScc/cct/proxy** | 同AP1 | 不同位深/编码的变体 |

**为什么选择ACEScg作为渲染工作空间？**

研究表明，三原色越接近光谱轨迹的颜色空间，渲染结果越接近光谱渲染的ground truth。ACEScg的AP1三原色经过优化，比sRGB更适合光照计算。

### 5.3 ACES转换流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Input     │───▶│   Look      │───▶│ Reference   │───▶│   Output    │
│ Transform   │    │  Transform  │    │  Rendering  │    │   Device    │
│   (IDT)     │    │   (LMT)     │    │  Transform  │    │  Transform  │
└─────────────┘    └─────────────┘    │   (RRT)     │    │   (ODT)     │
                                      └──────┬──────┘    └─────────────┘
                                             │
                                      Output Transform
```

| 转换阶段 | 功能 | 在游戏中的应用 |
|---------|------|--------------|
| **IDT** | 输入设备转换（如sRGB→ACES） | sRGB纹理解码到线性 |
| **LMT** | 外观调整（创意LUT） | 可选的风格化调整 |
| **RRT** | 参考渲染转换（Scene→Output Referred） | Tonemapping核心 |
| **ODT** | 输出设备转换 | 针对SDR/HDR显示器的最终映射 |

### 5.4 ACES Output Transform详解

ACES 1.0使用Two-Stage Tone Scale（RRT + ODT各有一次Tone Scale），ACES 2.0改进为Single Stage Tone Scale（SSTS）。

**HDR ODT支持**：
- 1000 nit PQ
- 2000 nit PQ  
- 4000 nit PQ

**SDR ODT支持**：
- sRGB (100 nit)
- Rec. 709 (100 nit)
- DCI-P3 (48 nit影院/100 nit显示)

## 第六部分：游戏中的HDR实现

### 6.1 现代HDR管线架构

以使命召唤：二战和UE4为例，现代HDR管线通常包含以下阶段：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        HDR Color Pipeline                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  [场景渲染] ──▶ [后处理] ──▶ [曝光] ──▶ [色域扩展] ──▶ [Color Grading]  │
│  sRGB Linear    HDR Linear   Rec.2020      AP1空间                        │
│                                                                          │
│                              ↓                                           │
│                    ┌─────────────────┐                                   │
│                    │   Tone Curve    │  ← RRT等效                        │
│                    │  (S曲线映射)    │                                   │
│                    │  0-10000 nit    │                                   │
│                    └────────┬────────┘                                   │
│                             ↓                                            │
│                    ┌─────────────────┐                                   │
│                    │ Display Mapping │  ← ODT等效                        │
│                    │  (Range Reduce) │                                   │
│                    └────────┬────────┘                                   │
│                             ↓                                            │
│              ┌──────────────┴──────────────┐                            │
│              ↓                              ↓                            │
│       [SDR路径]                       [HDR路径]                          │
│    BT.1886 OETF                       PQ OETF                            │
│    Rec.709/sRGB                       Rec.2020                           │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.2 关键实现细节

#### Universal CLUT（Color Lookup Table）

现代游戏引擎通常使用3D LUT来加速颜色转换：

- **输入**：曝光后的线性场景颜色
- **包含操作**：Color Grading + Tone Curve + Display Mapping
- **输出**：Output-referred颜色值

#### HDR Range Reduction

由于HDR显示器实际亮度峰值通常只有1000-2000尼特，需要将0-10000尼特映射到设备实际范围：

**BT.2390 EETF**：使用Hermite曲线在ICtCp空间进行亮度范围调整，可根据用户校准参数动态调整。

#### UI渲染

UI通常需要特殊处理：

- **SDR路径**：直接混合到sRGB空间
- **HDR路径**：UI亮度限制在300尼特以内，转换到BT.2020后应用PQ编码

### 6.3 UE4与COD实现对比

| 特性 | UE4原生实现 | COD: WWII |
|------|------------|-----------|
| Tonemapping | 简化版ACES | 自定义Tone Curve |
| HDR Range Reduction | 固定1000/2000 nit | 用户可校准BT.2390 |
| CLUT生成 | 每帧Async Compute | 类似 |
| UI处理 | Offscreen Buffer混合 | Backbuffer直接混合 |
| 色域扩展 | Expand Gamut参数 | 固定Wide Gamut |

## 第七部分：色彩管理管线设计

### 7.1 设计原则

1. **线性工作流**：所有光照计算必须在线性空间进行
2. **广色域中间空间**：使用ACEScg或Rec.2020作为工作空间
3. **设备无关的Scene-Referred数据**：资产和中间文件保存为Scene-Referred
4. **一致的View Transform**：使用标准化的Output Transform

### 7.2 推荐管线配置

#### 资产制作阶段

| 资产类型 | 推荐格式 | 颜色空间 |
|---------|---------|---------|
| 颜色纹理 | PNG/TGA | sRGB (带sRGB OETF) |
| 法线/粗糙度 | PNG/TGA | Linear |
| HDR环境贴图 | EXR/Radiance | Linear, Scene-Referred |
| LUT | CUBE/3DL | 根据用途指定 |

#### 渲染阶段

```
输入纹理 ──▶ sRGB EOTF解码 ──▶ Linear sRGB ──▶ 转换到AP1 ──▶ 渲染计算
                                                              ↓
                                                        [ACEScg工作空间]
                                                              ↓
                                    后处理效果（Bloom/SSAO等）─▶ Color Grading
                                                              ↓
                                                    Output Transform (RRT+ODT)
                                                              ↓
                                    ┌─────────────────────────┴─────────────────────────┐
                                    ↓                                                   ↓
                              SDR显示器                                           HDR显示器
                              sRGB OETF                                           PQ OETF
```

### 7.3 调试与验证

- **使用校准图像**：ACES提供参考图像用于验证实现正确性
- **灰阶测试**：确保中性色在各阶段保持中性
- **饱和度测试**：验证广色域渲染不会过度饱和
- **亮度测量**：使用色度计验证输出亮度符合预期

## 第八部分：总结与展望

### 8.1 关键要点回顾

1. **颜色空间是三维概念**：由三原色、白点、传递函数共同定义
2. **传递函数服务于感知**：OETF/EOTF优化编码效率，匹配HVS特性
3. **HDR需要新的编码方式**：PQ基于感知量化，HLG追求兼容性
4. **ACES提供标准化框架**：从输入到输出的完整色彩管理方案
5. **Tonemapping是艺术+科学**：在动态范围压缩中保持视觉美感

### 8.2 未来趋势

- **更高亮度**：4000+ nit显示器逐渐普及
- **更完善的标准**：ACES 2.0、ICtCp等新技术
